## Section 08. Slack Integration and Client Boundary

Section 01 fixed the system boundary and made Skywalker a retrieval backend behind MCP rather than a conversational agent. Section 02 fixed the entry contract and the identity-scoping path. Sections 04 through 06 then fixed the runtime behavior of the backend itself: route selection, UKB integration, common reranking, and abstain behavior. This section now moves upward one layer and defines the first client surface that Skywalker itself must actually live inside: a Slack bot that employees invoke through slash commands, direct messages, and channel mentions.

That matters because Slack is not just another transport. It is the user-facing surface where loosely phrased questions, conversational follow-ups, partial context, and ordinary workplace interaction all arrive in one stream. If this section were written as though Slack merely forwarded text into Skywalker and echoed text back out, it would erase the part of the system that actually has to behave intelligently on the client side. That would be a mistake. Slack is where the architecture stops looking like a retrieval program on paper and starts looking like a tool a real employee would use.

This section therefore does two things at once. It defines the Slack bot as a concrete subsystem with its own boundaries, contracts, and failure semantics. And it defines the line that must not blur: Skywalker owns scoped retrieval and evidence selection, while the Slack application owns conversational handling, turn interpretation, and message construction. That division of labor is not aesthetic. It is what keeps the retrieval backend deterministic and what keeps the Slack surface useful.

The system is also intentionally asymmetric across client surfaces. QuickSuite already brings its own chat-agent runtime and consumes Skywalker through Amazon MCP Gateway on its Federate-inbound route from that side. Slack does not. In Slack, the team is explicitly building the bot, the model-layer prompt, and the interaction surface. That means the Slack section has more surface area than QuickSuite, not because Slack is architecturally more important than retrieval, but because it owns more of the user-facing work. Both client paths share the same transport — Amazon MCP Gateway — and the same Skywalker MCP server; what differs is the inbound auth shape (CloudAuth from the Slack inline-agent orchestrator's AAA application on `api.mcp.asbx.aws.dev/ca/mcp/...`, with CloudAuth OBO carrying the orchestrator's identity and TransitiveAuth carrying the human end-user's identity through to Skywalker; Federate OAuth from QuickSuite on `api.mcp.asbx.aws.dev/federate/mcp/...`, with identity in MCP tool arguments).

### 1. What this section owns

This section owns the Slack-facing subsystem that begins when Slack delivers a user event to the Slack application boundary and ends when the user receives a Slack reply shaped from Skywalker output. In practical terms, that means it owns Slack event intake, surface normalization across slash commands, direct messages, and channel mentions, alias-aware request construction for Skywalker, the Slack-side model layer that calls Skywalker MCP tools, the prompt and response behavior of that model layer, and the translation of Skywalker evidence or abstain packages into Slack-visible replies.

It also owns the conversational boundary. Skywalker is not being extended into a multi-turn retrieval brain that stores and reasons over conversational history itself. The Slack application is the layer that carries conversational continuity, interprets follow-up turns, and decides how to turn one Slack utterance into one or more Skywalker tool calls. That is a direct consequence of the larger design philosophy already fixed in the series. The retrieval system is deterministic, relatively narrow, and identity-aware. The conversational layer is where the flexibility lives.

This section owns the Slack-side identity handoff as well. For Slack specifically, the stable identifier the integration has available is the **Amazon alias**, resolved by the Slack application from the Slack user identity through an internal resolver (the specific resolver implementation is captured in API_02 outstanding items; the architecturally important point is that the alias is what crosses the MCP boundary, not raw Slack user IDs). That fact is operationally important because it fixes which Skywalker MCP entry path the Slack integration uses: the alias-based path. The Slack surface is therefore not responsible for recreating PAPI behavior or for synthesizing scope attributes itself. Its job is to carry the alias truthfully into the alias-based MCP tool path.

This section also owns the architectural rule that the Slack integration is responsible for getting the alias material it needs through its **own integration-specific channel**, not through any MCP-side propagation mechanism. Amazon MCP Gateway does not propagate the originating end-user identity to the Skywalker MCP server on this path (the gateway's published delegated-identity patterns are all flagged "in progress" on the BuilderHub MCP Gateway concepts page as of this section's writing). Slack carries the alias as part of the MCP tool arguments it sends, supplied from its own Slack-user-to-alias resolution. This is symmetric with how QuickSuite (Section 09) and the UAT inline-agent orchestrator (Section 05) carry identity material in their own ways.

This section owns the fact that Slack is a distinct frontend, not a disguised internal test client. That means it owns the difference between a backend abstaining and a Slack conversation failing. It owns the difference between a routed retrieval failure and a user-visible reply. It owns how sources, policy links, and uncertainty are presented. It owns the decision to let the Slack application write the user-facing answer rather than asking Skywalker to produce Slack-ready prose.

This section does not own retrieval correctness. It does not own the PAPI integration, the controlled FAQ corpus, the UKB arm, reranking, threshold derivation, or the backend abstain rules themselves. Those were already fixed earlier. It also does not own the internal shape of the MCP server contract beyond how the Slack application consumes it. The Slack subsystem is a client of Skywalker, not an alternate implementation of it.

It also does not own runtime human review as part of the request path. Human-in-the-loop review is in scope once the system is live in production because that is currently the only practical way to evaluate behavior against subject-matter expectations, but that review loop is not part of how one Slack message is handled online. It is a post-response evaluation discipline, not a request-time branch.

### 2. Inputs, outputs, and contracts

The primary input to this section is a Slack event from one of three supported surfaces: a slash command, a direct message, or a channel mention. Those surfaces are different in UX but the section should normalize them into one internal turn contract quickly. The Slack subsystem should not carry three separate orchestration paths any longer than necessary. By the time the application is deciding what to do, it should be reasoning over one normalized representation of a user turn rather than over Slack's raw event diversity.

That normalized turn contract needs, at minimum, the raw user message text, the Slack surface type, the Slack user identity, the Amazon alias available for that user in the Slack integration, the channel or thread context needed to route the eventual reply, and the immediately relevant conversation context visible on the Slack side. The key point is that the Slack application does not need every raw Slack field. It needs the subset that determines who is speaking, what they asked, where the answer has to go, and what prior conversational context should be taken into account.

The second input is the Skywalker MCP contract established earlier in the series. Slack is not allowed to bypass MCP and call backend subsystems directly. It should consume Skywalker through the MCP surface that already exposes the supported search tools. That matters for discipline as much as for code reuse. Once the Slack integration starts talking around the MCP contract, the architecture stops having a real backend boundary.

The default Slack-to-MCP contract should therefore be alias-first. Slack normally has the Amazon alias, so the normal tool call is the search path that accepts alias and lets Skywalker resolve PAPI internally. The employee-ID path exists for other clients and remains part of the MCP contract, but it is not the primary Slack route. The explicit scoped-metadata short-circuit path is different. That path should remain available to the Slack application when, and only when, it already has authoritative scoped values for the current turn and using them is better than forcing another alias-to-PAPI round-trip. The important point is that the shortcut is optional and deliberate, not the default path.

That means the Slack-side contract has two internal layers rather than one. The first is the normalized turn object the Slack event handler gives to the model layer. The second is the narrower MCP tool call the model layer gives to Skywalker. The model layer is the only thing in this section allowed to bridge those layers. The Slack event handler itself should not be constructing retrieval arguments from raw events, and Skywalker should not be handed raw Slack structure as though it were a search request. The bridge belongs in the model layer because that is where conversational interpretation already lives.

The normalized turn object also needs a prompt-facing contract, because prompting is not an implementation footnote on this path. The Slack-side model layer should receive, per turn, the current user utterance, the normalized surface metadata, the alias context, the conversation excerpt it is allowed to use, and the tool descriptions that define how Skywalker may be called. This section is not trying to freeze the exact prompt text, but it is freezing the architecture that the prompt is part of the subsystem contract. A Slack application that does not know the supported tool shapes, the importance of scoped correctness, and the fact that backend abstention is valid is not operating inside the architecture even if the code still runs.

The MCP response contract back into the Slack application should be treated the same way. The application should not be receiving one raw block of prose and deciding for itself whether that prose was confident, weak, or abstaining. It should receive a structured Skywalker result that tells it whether the backend judged the request answerable, what evidence and source links survived, what policy links are attached, and whether the backend explicitly abstained. That is what lets the Slack layer stay conversational without becoming epistemically loose. The model is then free to shape wording, but not free to invent what kind of backend result it received.

The Slack-side output contract to Skywalker is therefore not "here is a Slack conversation, do something with it." It is much narrower and cleaner. The Slack application sends one current user question plus identity material sufficient for one of the supported Skywalker search tools. Skywalker returns either a structured answerable package or a structured abstain package. The Slack application then turns that backend result into Slack-visible behavior.

The output of this section, from an architectural perspective, is a Slack reply event rather than a retrieval result. That reply can take several forms while still preserving one contract shape. It can be an answer built from a Skywalker evidence package. It can be an explicit abstain message built from a Skywalker abstain package. It can be a system failure message when the Slack application could not obtain a trustworthy backend result at all. But in every case, the Slack output is owned by the Slack application, not by Skywalker.

That contract distinction is crucial. Skywalker does not need to know whether the final message is a direct message with rich formatting, a terse slash-command response, or a threaded reply to a mention in a public channel. The Slack integration does. The Slack subsystem is where the backend package becomes a user-facing message.

A second contract distinction matters just as much. The Slack application should treat Skywalker output as evidence-shaped input, not as a final statement to be echoed. That was already implicit in the broader architecture and should be explicit here. The output of MCP feeds into clients. It is not written for direct end-user display. The Slack layer is therefore not a renderer over backend prose. It is an application that receives structured backend output and composes the final Slack answer from it.

### 3. Fixed decisions

The first fixed decision is that Slack is a real client surface, not a thin transport. The team is explicitly building the prompting, the interaction logic, and the Slack-specific response behavior on this path. That is the correct shape because Slack is the place where conversational behavior actually has to exist.

The second fixed decision is that the supported Slack surfaces are slash commands, direct messages, and channel mentions. Those are all in scope from the start. The architecture should not be written around only one of them and then hope the others can be added without changing the shape of the subsystem.

The third fixed decision is that the Slack integration consumes Skywalker exclusively through MCP, **routed through Amazon MCP Gateway** at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry-id}/{skywalker-server-id}` with **CloudAuth** inbound auth using credentials minted by the Slack inline-agent orchestrator's AAA application (suggested name `SkywalkerSlackOrchestrator`, finalized at onboarding). The orchestrator is registered as a CloudAuth-modeled AAA application in ServiceLens with AAA relationships to MCP Gateway's `InvokeMcp` endpoint and to Skywalker's MCP server resource per the OBO Decision Guide's "OBO Required (Shared / Cross-Boundary)" path. **CloudAuth OBO is on by default on this combination** — MCP Gateway invokes Skywalker on behalf of the orchestrator's AAA application identity, and Skywalker's AAA permission evaluation sees the orchestrator's principal rather than the gateway's. **TransitiveAuth additionally carries the human end-user's identity** as a separate TA token; the orchestrator initiates the TA token from the Slack-resolved alias plus an `AgenticContext`, and Skywalker validates the TA token server-side and reads the human alias from the TA claims rather than from MCP tool arguments. The Slack integration does not call the controlled FAQ corpus directly. It does not query UKB directly. It does not reimplement reranking, thresholding, or scope resolution in the client layer. Those behaviors remain inside Skywalker. This is the same gateway product Section 09 uses for QuickSuite (on its Federate-inbound route) and Section 05 uses for UAT (also on the CloudAuth-inbound route, with the orchestrator's own AAA application). Source: BuilderHub [MCP Gateway User Guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/), [UnifiedAuth MCP Server Builder Guidance](https://w.amazon.com/bin/view/Dev.CDO/UnifiedAuth/Agentic/MCP/Overview/), and grounding contract [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md).

Earlier drafts of this section described the Slack integration as using SigV4 inbound at `/mcp/{registry-id}/{skywalker-server-id}`. That path is no longer available for new MCP servers per the [MCP server vendor guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/howto-vendor/) — AWSAuth-inbound for CloudAuth-protected services was withdrawn for new servers after April 24, 2026, and the grace window for existing servers closed May 15, 2026. CloudAuth-inbound is the supported successor and is also the path with native OBO + TA support. The migration is captured in API_14.

The fourth fixed decision is that the Slack identity signal is the **Amazon alias only**. Slack's Slack-user-to-alias resolver produces the alias before the MCP call leaves the Slack inline-agent orchestrator; the orchestrator then **initiates a TransitiveAuth token carrying that alias** as the canonical channel for the human identity into Skywalker. The Slack action group declares one function (`skywalker_search_by_alias`) and not the explicit-scope path because Slack does not hold authoritative `country`/`level`/`role` for the user — Skywalker resolves those on its own side through PAPI, with the alias arriving via TA. The MCP tool's `arguments.alias` shape is preserved in the contract for clients that do not propagate TA (notably QuickSuite on the Federate-inbound path, see Section 09 / API_08); on the Slack and UAT paths, Skywalker treats the TA-supplied alias as the canonical identity and falls back to `arguments.alias` only if TA propagation fails. The launch posture is fail-closed when TA is missing: the orchestrator is responsible for initiator setup before going live, and a TA-missing failure surfaces as a system-failure error rather than a silent degradation to argument-supplied identity. The employee-ID and explicit-scope paths remain part of the core MCP contract for other clients (UAT uses `skywalker.search.by_explicit_scope`, future callers may use `skywalker.search.by_employee_id`), but Slack does not.

The fifth fixed decision is that multi-turn handling belongs to the Slack application rather than to Skywalker. This is both a technological and an architectural decision. The retrieval backend is deterministic, structured, and relatively narrow. It is not the right place to do meaningful multi-turn interpretation. Large models already do that work well enough that the system should let them do it, rather than dragging the retrieval layer into conversational state management it is not well suited to own.

The sixth fixed decision is that the Slack-side conversational model is Claude Sonnet 4.6. That model is the system layer responsible for interpreting the turn, deciding how to use Skywalker, and composing the final Slack-facing response from grounded backend output. This fixes the current model surface without changing the backend boundary.

The model is invoked through Amazon Bedrock Inline Agents rather than through the Converse API. The Slack application calls `InvokeInlineAgent` with one action group that declares the Skywalker retrieval tools with `RETURN_CONTROL` as its executor. When the agent decides to call retrieval, Bedrock returns control to the Slack application; the application then performs the real MCP `tools/call` against Skywalker and feeds the structured backend result back into the next inline-agent turn. This keeps Skywalker consumption on the MCP contract (decision three) while avoiding a Lambda-backed action group for a tool that already has a well-defined MCP surface. The Converse API is not used on this path.

Every `InvokeInlineAgent` call on the Slack path carries `streamingConfigurations: { streamFinalResponse: true }`. The agent's final response text streams back as multiple `chunk` events on the completion stream rather than as one consolidated chunk after generation completes. This applies to both the first turn (where the agent may produce a clarification reply directly) and the second turn (where the agent composes the grounded final answer from the Skywalker evidence package). The `returnControl` event itself is sync and discrete — only generated narrative text streams. `applyGuardrailInterval` is left at the documented default of 50 characters and is effectively dormant at launch because no Bedrock Guardrail is configured. Streaming requires `bedrock:InvokeModelWithResponseStream` on the Slack application's IAM execution role in addition to the inline-agent invoke permission. The MCP boundary itself stays sync — Skywalker's evidence envelope is structured rather than narrative, the gateway's HTTP transport on our auth combination does not stream, and there is no time-to-first-token to optimize on a fixed-shape JSON response. Streaming is a rendering-layer property that lives between the inline agent and the Slack surface; it does not change anything below the MCP boundary. See [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for the full streaming contract.

The action group is declared in the `InvokeInlineAgent` call with `actionGroupExecutor: { customControl: "RETURN_CONTROL" }` and a `functionSchema` that exposes **one** function corresponding to the one MCP tool the Slack application uses. `skywalker_search_by_alias` takes `query_text` and `alias` and is the Slack identity path per decision four; the explicit-scope path is not on the Slack action group because Slack does not hold authoritative scope and PAPI resolves it on Skywalker's side. The function name mirrors the MCP tool name with dots replaced by underscores so the inline-agent schema and the MCP tool registry stay legible against each other. On each turn the application calls `InvokeInlineAgent` with `foundationModel` set to the Claude Sonnet 4.6 model ID, `instruction` set to the stable Slack behavioral prompt described below, `sessionId` derived from the Slack thread context for channel mentions, the Slack IM channel for DMs, or the correlation between a slash command and its follow-up thread, and `inputText` carrying the user's current turn. `idleSessionTTLInSeconds` starts at 900 seconds (15 minutes), revisited against real usage.

The return-control loop has four explicit steps. The first call emits either a direct reply (clarification turn — text streams as `chunk` events) or a `returnControl` event naming the tool and arguments the agent chose. On return-control, the Java code maps the function name to the corresponding MCP tool name (`skywalker_search_by_alias` → `skywalker.search.by_alias`), builds the MCP `tools/call` request with arguments matching the MCP input schema, **initiates a TransitiveAuth token carrying the human alias resolved earlier in the request lifecycle** plus an `AgenticContext` describing the agent involvement, and **issues the request through Amazon MCP Gateway** at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry-id}/{skywalker-server-id}` with **CloudAuth** credentials minted by the orchestrator's AAA application. The TA token rides alongside CloudAuth via the CloudAuth MCP SDK. MCP Gateway validates the inbound CloudAuth credentials, performs Bindle authorization (`canInvoke` on `MCPGateway::{skywalker-server-id}` for the orchestrator's AAA application principal), and forwards the call to Skywalker on the gateway's CloudAuth-outbound leg with **CloudAuth OBO** (so Skywalker sees the orchestrator's AAA application identity rather than the gateway's) and the TA token preserved for downstream validation. Skywalker validates the TA token server-side and reads the human alias from the TA claims; PAPI then resolves the scope triple from that alias as on every other request. Skywalker's response — the structured backend result envelope from the MCP contract — is returned through MCP Gateway as a single sync JSON-RPC result and serialized to match the action group's function return shape, then fed back via a second `InvokeInlineAgent` call using the same `sessionId` with `inlineSessionState.returnControlInvocationResults` populated and `streamFinalResponse: true` set again. The agent then produces the final grounded assistant message with citations as a token stream. Bolt consumes the streamed `chunk` events, buffers them, and posts/edits the Slack message at a 2.5-second cadence using `chat.update` (decision ten and §6). When the MCP call itself fails at the transport level (CloudAuth authorization failure at the gateway, AAA-permission failure, TA token validation failure on Skywalker, gateway-side 5xx, timeout), the return-control result passed back to the agent explicitly indicates the failure — the behavioral prompt instructs the model that this is a system-failure state rather than permission to answer from memory. **The MCP Gateway response is sync; the inline-agent response streams; the Slack reply is rate-limited progressive `chat.update` calls converging on a final complete Block Kit message.** Starting inference parameters are `temperature: 0` and `maxTokens: 1024`, adjusted against real evaluation.

That decision also fixes a prompt-ownership boundary. The stable behavioral prompt for the Slack application belongs to this subsystem, not to Skywalker and not to Slack event plumbing. The prompt needs to teach the model at least four things that are architectural rather than stylistic: that Skywalker is the source of scoped retrieval truth, that backend abstention must be preserved rather than overwritten, that sources and policy links should remain attached to the answer when present, and that the model is not allowed to replace missing grounding with general background fluency. Prompt text may evolve. Prompt ownership should not.

The seventh fixed decision is that Skywalker abstention is a valid backend outcome on the Slack path. The Slack application does not treat an abstain package as a backend failure. It treats it as a grounded statement that the retrieval system did not assemble enough evidence to justify an answer-shaped handoff. The abstain presentation must also be structurally distinct in the Slack surface from cases where the application itself declines to engage. Users need to be able to tell the difference between "Skywalker could not find a grounded answer for your specific situation" and "the bot is being cagey." The precise wording and visual treatment are rendering concerns, but the distinction they convey is an architectural one, and Slack abstain messages that blur into generic non-response behavior violate this contract.

The eighth fixed decision is that Slack does not ask Skywalker to maintain conversation state. Any conversation history the Slack application uses remains on the Slack side of the boundary. The retrieval backend still receives one request at a time and remains stateless from turn to turn.

The ninth fixed decision is the citation requirement, and it is the strongest architectural rule in this section. On any answerable backend result, the final user-facing Slack message must be constructible from the evidence package returned by Skywalker, and the message must surface citations to the specific evidence candidates or source or policy links that support its claims. This is not a stylistic preference. It is a grounding discipline enforced at the architecture layer because the alternative — a strong general model producing confident, plausible, unsupported answers on travel, events, and expense policy — is the single highest-risk failure mode for the system's accuracy target. The precise citation format is a surface-specific rendering concern and may vary across slash commands, direct messages, and channel mentions. The requirement that citations exist, trace back to the backend evidence package, and appear in the user-facing message is not negotiable. A Slack implementation that produces an answer without traceable citations to the backend evidence package has violated this section's contract regardless of whether the final wording happens to be correct. The model is allowed to phrase the answer; it is not allowed to produce an ungrounded answer that looks grounded.

The tenth fixed decision is the Slack-surface latency target. The end-to-end experience from user message receipt to final Slack reply delivery targets under 4 seconds p95. Because the Skywalker retrieval pipeline itself is budgeted at 250 to 450 milliseconds, the client-side interpretation, tool call, generation, and Slack transport together must fit in roughly 3.5 seconds. This is a real constraint on prompt length, tool-call chaining, model response token budget, and Slack message construction, and it shapes several downstream implementation choices even though those choices are not this section's direct owner. A sustained breach of the 4-second budget is an architecture-class event for the Slack path, not a tuning inconvenience.

### 4. Alternatives considered or still live

One alternative would have made Slack a thin wrapper that simply forwarded the user's message into Skywalker and displayed whatever came back. That approach is rejected. It would force Skywalker output to become user-facing prose, erase the distinction between evidence packages and final answers, and leave no proper home for the multi-turn interpretation work that Slack actually requires.

A second alternative would have pushed conversation-state ownership into Skywalker itself. That approach is also rejected for the current design. It would blur the system boundary that earlier sections deliberately established and would force a deterministic retrieval backend to become responsible for behaviors that a conversational model already handles better.

A third alternative would have left both the alias-based path and the explicit scoped-metadata short-circuit path on the Slack action group, letting the model layer choose. That alternative is rejected for Slack at launch. The Slack application does not hold authoritative `country`/`level`/`role` for the user — its identity surface is the Slack-user-to-alias resolver — so adding the explicit-scope path to the Slack action group would expose the model to a tool it cannot honestly populate. The explicit-scope path remains in the core MCP contract for clients that do hold authoritative scope (UAT in Section 05), and remains available to Slack if the integration ever grows that capability, but is not on the launch action group.

A fourth alternative would be to build the Slack bot as a lightweight custom webhook service rather than using an established Slack framework. That alternative remains live only in the abstract. The more realistic live decision is between Bolt for Java and any Amazon-internal Slack framework that may be required or strongly preferred. The architecture should preserve that choice as open until the framework requirement is confirmed. The important architectural point is not the library name. It is that the Slack subsystem should live comfortably in the Java-heavy environment the rest of the program already assumes.

A fifth alternative would be to let the Slack application answer directly from its own model reasoning when Skywalker abstains or when the MCP call fails. That alternative is rejected. It would defeat the entire point of building a scoped retrieval backend with explicit abstention. The Slack application is allowed to phrase the answer. It is not allowed to replace grounded retrieval with freeform confidence theater.

A sixth alternative would be to treat all three Slack surfaces as separate product behaviors with distinct internal orchestration. That remains an implementation detail rather than a preferred architecture. The cleaner posture is to normalize them early into one turn model and let the output formatting vary by surface only where Slack itself forces that distinction.

A seventh alternative — keeping the Slack reply sync, sent as a single complete `chat.postMessage` after the inline agent's full final response has arrived — was the original launch posture and was reversed. The reasons for the reversal are pinned in [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md). The summary: total latency does not change either way, but time-to-first-token does, and at the 4-second p95 end-to-end target (decision ten) the dominant phase is model generation, where streaming buys real perceived-latency improvement on a chat surface. Skywalker's MCP boundary stays sync regardless because the evidence envelope is structured rather than narrative; streaming lives one layer above the MCP boundary, between the inline agent and Slack, and uses the same rate-limited `chat.update` pattern other internal Bedrock-agent Slack bots have already validated in production. The cost is a more elaborate failure-handling surface (mid-stream errors require a final-state write that overwrites partial text) and a coarser cadence than the per-token feel an SSE surface would give, because Slack's `chat.update` rate limit caps update frequency at roughly one per second per channel; both are acceptable trade-offs for the time-to-first-token win.

### 5. Assumptions inherited from upstream sections

This section inherits the system boundary from Section 01 (Skywalker is the retrieval backend behind MCP, not the client-side brain — the Slack subsystem sits above the MCP boundary and consumes Skywalker rather than redefining it) and the MCP contract and identity-scoping paths from Section 02 (alias-based, employee-ID, and explicit scoped-metadata search all exist, and the backend, not the Slack application, is where PAPI is called and where location, level, and manager-versus-IC scope become retrieval truth).

It also inherits the controlled FAQ arm from Section 03, the runtime routing behavior from Section 04, the UKB black-box general arm from Section 06, and the common reranking and abstain layer from Section 07. The Slack subsystem therefore never decides which retrieval arm should run, never compares evidence scores, and never invents its own abstention rule — those responsibilities live below it. Two principles from the earlier sections matter here specifically. Context-free answers are wrong-shaped rather than neutral, which is especially important on Slack because it is the easiest surface on which to drift into casualness; tone may be conversational, but identity-specific correctness may not. And structured backend abstention is part of normal behavior that the Slack integration must handle as healthy rather than as a rare embarrassment.

### 6. End-to-end data flow for this section

The first step is Slack event intake. A user issues a slash command, sends a direct message, or mentions the Slack bot in a channel. The Slack application receives the event and immediately normalizes the surface-specific payload into one internal turn object. That object records the message text, the Slack user, the Amazon alias available through the Slack integration, and the return target for the eventual reply.

The three Slack surfaces map to three distinct platform subscriptions. Slash commands are configured per command at the Slack app level; Slack POSTs the command payload to the command's Request URL when a user invokes `/skywalker`. Channel mentions arrive through the Events API as `app_mention` events. Direct messages arrive through the Events API as `message.im` events filtered to IM channels and non-bot users. Slack also sends an initial `url_verification` event when a Request URL is configured; the application must echo the `challenge` field to confirm ownership of the endpoint. Every inbound HTTP request from Slack is verified against the app's signing secret — Slack sends an `X-Slack-Signature` header computed by HMAC-SHA256 over the request body combined with a timestamp header, and the application recomputes the signature using its signing secret and rejects non-matching or stale requests. The raw request body must be read before any JSON deserialization because parsing first invalidates the signature check. Outbound messages are sent via the `chat.postMessage` Web API call, or, for slash commands that need quick turnaround, via the `response_url` included in the command payload; threaded replies set `thread_ts` so replies thread cleanly under the triggering message.

The Slack OAuth scopes required on the bot are `commands` (to receive slash command invocations), `app_mentions:read` (to receive `app_mention` events), `im:history`, `im:read`, and `im:write` (to receive and reply in DMs), `chat:write` (to post messages), and `users:read` plus `users:read.email` (to resolve Slack user identity to the Amazon alias through profile email). Three secrets must be provisioned: `SLACK_BOT_TOKEN` (starts with `xoxb-`, used by the Web API client), `SLACK_SIGNING_SECRET` (used to verify inbound HTTP requests in HTTP mode), and, if the application runs in Socket Mode instead, `SLACK_APP_TOKEN` (starts with `xapp-` and carries the `connections:write` scope). The choice between HTTP mode (Slack POSTs events to a public HTTPS endpoint we host, with signing-secret verification on every request) and Socket Mode (the app opens an outbound WebSocket to Slack and receives events over that socket, with no public inbound endpoint) is a deployment-environment decision that remains open; Socket Mode removes the public-endpoint requirement which matters for an internal-only Amazon service, and is therefore the likely choice once the deployment constraints are confirmed.

The framework is Bolt for Java. The core dependency is `com.slack.api:bolt`, with `com.slack.api:bolt-servlet` added for HTTP-mode deployments on servlet containers or `com.slack.api:bolt-socket-mode` plus a WebSocket client implementation added for Socket Mode. Minimum JVM is Java 8. The framework is organized around an `App` instance to which handlers are attached via builder patterns: `app.command("/skywalker", (req, ctx) -> {...})` for the slash command, `app.event(AppMentionEvent.class, ...)` for channel mentions, and `app.message(...)` filtered to IM channels for direct messages. Request signing verification is handled automatically by Bolt when the signing secret is provided to the `AppConfig`; Socket Mode uses the app token instead. Slack requires a 3-second acknowledgement on commands and interactive payloads — Bolt handles this automatically on `ack()` calls, but the retrieval work must happen after ack either on a background executor or via `response_url` edits, which is load-bearing for the 4-second end-to-end budget because it means the user sees an immediate "working on it" response and the final grounded answer is posted or edited in when the Skywalker call returns.

The second step is turn interpretation on the Slack side. The model layer reads the current turn together with the immediately relevant conversation context held on the Slack side of the boundary. This is where multi-turn understanding lives. If the user has asked a follow-up question, referred to something mentioned one turn earlier, or framed the request conversationally rather than as a clean search query, the Slack application is responsible for turning that into a coherent current question for Skywalker. The backend is not asked to reconstruct conversation state from fragments.

That interpretation step also includes prompt assembly. The Slack application should combine a stable behavioral instruction layer with per-turn context rather than relying on a monolithic freeform prompt. The stable layer should teach the model the boundary it must preserve: Skywalker is the source of scoped retrieval truth, abstention is valid, sources and policy links are not optional decoration, and the model should not answer beyond grounded backend support. The per-turn layer should carry only what changed on this turn: the user's current message, the immediately relevant conversation slice, the alias or scoped identity material, and the result of any Skywalker tool call. This keeps the Slack integration legible when prompt behavior needs to be revisited later.

The third step is MCP tool selection. The Slack application chooses the alias-based Skywalker search path. This is a fixed choice at launch per §3 decision four — the action group declares only `skywalker_search_by_alias`, so the inline agent has no other tool to choose from. Skywalker resolves `(country, level, role)` from the alias on its own side through PAPI.

The fourth step is the MCP `tools/call` against Skywalker's MCP server, **routed through Amazon MCP Gateway**. The Slack inline-agent orchestrator initiates a TransitiveAuth token from the Slack-resolved human alias plus an `AgenticContext`, then POSTs to `https://api.mcp.asbx.aws.dev/ca/mcp/{registry-id}/{skywalker-server-id}` with **CloudAuth** credentials minted by the orchestrator's AAA application; the TA token rides alongside CloudAuth via the CloudAuth MCP SDK. The request body is the standard JSON-RPC envelope from API_01 with `name: "skywalker.search.by_alias"` and `arguments: { query_text, alias }`. The `arguments.alias` field is preserved for contract compatibility with non-TA clients but is not the canonical identity channel for this path — Skywalker reads the alias from the TA token. MCP Gateway validates the inbound CloudAuth credentials, applies Bindle-based authorization (`canInvoke` on `MCPGateway::{skywalker-server-id}` for the orchestrator's AAA application principal), and forwards the request to Skywalker on the gateway's CloudAuth-outbound leg with **CloudAuth OBO** and the TA token preserved. Skywalker validates the request against the Section 02 entry contract, validates the TA token and reads the human alias from the TA claims, calls PAPI to resolve scope, and proceeds to retrieval.

The fifth step is Skywalker execution. Skywalker resolves scope through PAPI, decides whether the request is FAQ-only or dual-arm, retrieves evidence, normalizes candidates, reranks them, and returns either an answerable evidence package or a structured abstain package. **The MCP response itself is a single sync JSON-RPC result** — Skywalker's evidence envelope is structured rather than narrative, so there is no time-to-first-token to optimize at this layer, and the gateway's HTTP transport on our CloudAuth-inbound combination does not expose SSE framing. MCP Gateway returns the response unchanged through the same connection. Streaming on the Slack surface happens one layer above the MCP boundary, between the inline agent and Slack itself (see step seven and [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md)).

The sixth step is Slack-side answer construction. The Slack application consumes the Skywalker result and turns it into a Slack-visible reply. If the backend returned an answerable package, the Slack application writes the user-facing answer from that grounded package and must include traceable citations to the supporting evidence candidates or source and policy links in the response form appropriate to the Slack surface. Any claim in the user-facing message must be attributable to the returned evidence package; the model is not permitted to supplement grounded evidence with unsupported model-generated policy content. If the backend returned abstain, the Slack application writes an abstain-shaped reply that is structurally distinct from an application-level non-answer and that preserves the backend's judgment rather than fabricating coverage. The Slack application is responsible for phrasing and for citation placement. The backend remains responsible for the evidence judgment.

That construction step should preserve a strict internal order. First the application reads the backend result type. Then it reads the surviving evidence and link metadata. Then it composes the Slack-visible answer. The order matters because it prevents the model from beginning with a conversational answer impulse and only afterward checking whether the backend package supported that impulse. The architecture should force the opposite posture: backend result first, user-facing wording second. That is the only reliable way to keep a strong general model from sounding more certain than the evidence package actually warrants.

The seventh step is response delivery. The Slack subsystem posts the reply back to the originating surface, preserving the correct return path for slash commands, DMs, or channel mentions. The same backend package may therefore be rendered somewhat differently depending on the Slack surface, but that variation happens after grounding, not before. **Replies are progressively streamed to the user.** On user message receipt, the bot posts an initial placeholder via `chat.postMessage` and captures its `ts`. As the inline-agent's final-response chunks arrive on the Bedrock event stream, Bolt buffers them and calls `chat.update` against that `ts` at a rate-limited cadence — 2.5 seconds at launch, SSM-tunable — with a partial Block Kit rendering. When the inline-agent stream ends, a final `chat.update` writes the complete Block Kit response (section block carrying the answer body, context block listing source titles and URLs from the evidence package's `citations[]`). The 2.5-second cadence sits comfortably under Slack's documented `chat.update` rate limit of approximately 1 call per second per channel and avoids visual flicker that a tighter cadence would produce. See [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for the streaming contract; the rate-limited `chat.update` pattern is the same one used by other internal Bedrock-agent Slack bots (notably the SAIL Slack Bot's progressive-streaming phase).

The concrete reply transports differ by surface. Slash commands POST to the `response_url` included in the command payload within 3 seconds for the first acknowledgement, using `response_type: "ephemeral"` by default and `"in_channel"` when the user explicitly asks for a public answer; `chat.postMessage` handles any follow-up beyond the 30-minute `response_url` TTL. Direct messages go out as `chat.postMessage` calls with `channel` set to the IM channel ID. Channel mentions go out as `chat.postMessage` calls with `thread_ts` set to the triggering message's `ts` so replies thread cleanly under the user's mention rather than flattening into the parent channel. The default message format on every surface is Block Kit: a `section` block carrying the answer body followed by a `context` block listing source titles and source URLs drawn from the evidence package's `title` and `source_url` fields. That default satisfies the citation requirement from decision nine above without requiring surface-specific rendering logic to hand-roll its own citation placement. During streaming the message alternates between two states — an in-progress state showing accumulated text plus a "still typing" indicator (a trailing ellipsis or a `context` block with a "generating" hint), and the final state with the complete Block Kit response. The transition from in-progress to final happens on the terminating event of the inline-agent completion stream. On stream error mid-generation, a final `chat.update` writes a structurally-distinct error message that overwrites whatever partial text was visible; this is the same final-state-write primitive the success path uses, just terminating in a different shape (decision seven and §7).

That distinction matters operationally. A DM can tolerate a fuller answer body because the entire surface is already private and conversational. A channel mention may need a tighter reply because the surface is shared and the user did not open a dedicated workspace for a long answer. A slash command may eventually want the cleanest structured presentation of all because the user intentionally invoked a tool-like surface. Those are rendering decisions, not retrieval decisions, and it is healthy for them to live here rather than in Skywalker.

The eighth step is post-response capture for later review. Because human-in-the-loop subject-matter review is in scope once the system is in production, the Slack surface should preserve enough traceability that a reviewer can understand what question was asked, what backend route was taken, and what answer or abstain message the user received. That is not part of the online answer path, but it is part of why the Slack integration must preserve backend traceability rather than flattening everything into one ephemeral chat message.

The trace bundle should remain close to the actual runtime truth of the turn. The reviewer should be able to see the normalized user request, the Skywalker route outcome, whether the backend answered or abstained, and the Slack message the user finally saw. That is enough to let the team reason about whether a problem came from retrieval, from client-side interpretation, or from Slack-specific phrasing. Without that separation, later review would misdiagnose many issues because the only visible artifact would be the final chat message.

The important feature of this flow is that it gives each layer one honest job. Slack handles interaction. Skywalker handles retrieval. The Slack application translates between conversational turns and backend tool usage. The backend translates between scoped requests and evidence judgments. Neither layer is forced to impersonate the other.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

The first failure rule is that Slack event-handling failure is not the same as backend retrieval failure. If the Slack application cannot verify, normalize, or route the incoming event correctly, the system should fail at the Slack boundary rather than pretending that Skywalker returned nothing useful. The user-facing message in that case should reflect an integration problem, not an evidence problem.

The second failure rule is that missing alias prevents normal Slack-to-Skywalker invocation. Since alias is the default identity signal on the Slack path, the Slack subsystem should not quietly improvise when it lacks that field. It should fail explicitly rather than sending an under-scoped request into Skywalker and hoping the backend can infer what was missing.

The third failure rule is that MCP or backend-call failure is not permission for the Slack application to answer from model intuition. If the Slack application cannot obtain a grounded backend result, it should return a system-failure reply rather than crossing the boundary and pretending the model itself is now the source of policy truth.

The fourth failure rule is that backend abstention is not a Slack failure. If Skywalker returns an abstain package, the Slack application should convert that into a user-facing non-answer that preserves the backend's judgment rather than treating it as an outage. A healthy retrieval backend is allowed to say that the current evidence is not strong enough.

The fifth failure rule is that multi-turn ambiguity remains owned by the Slack application. If the user's message is conversationally incomplete in a way the Slack application cannot responsibly resolve into a backend call, then the application should handle that at the Slack layer instead of pushing ambiguity downward and forcing Skywalker to operate on an ill-formed question. The retrieval backend is not the place to fix conversational incompleteness.

A sixth failure rule follows from the same boundary. If the Slack application has enough conversational context to know that the current turn is underspecified for a safe backend call, then the correct failure shape is not silent tool use with bad assumptions. The correct shape is still a Slack-side interaction decision. The architecture should prefer one more client-side clarification turn over a backend call built on guessed intent or guessed scope. That is not because the backend is fragile. It is because the backend is doing exactly what it should do: insist on questions that are specific enough to retrieve against responsibly.

The non-goals in this section matter because Slack is where architectural drift is most tempting. This section does not turn Slack into a second retrieval system. It does not let the client bypass Skywalker. It does not put PAPI logic into the Slack layer. It does not make Slack own threshold tuning, policy interpretation, or confidence scoring. It also does not make the Slack application a persistent backend session manager. Conversation continuity remains a client concern, but that does not imply building a new retrieval-state substrate inside Skywalker.

It also does not make the Slack layer the place where source truth is repaired. If the backend returned weak sources, missing policy links, or a structured abstain, the Slack application is allowed to present that well. It is not allowed to patch over it by widening beyond the returned evidence or by constructing a cleaner answer than the evidence package supports. Good Slack behavior here means honest translation, not cosmetic rescue.

A second non-goal is evaluation harness design. That work is out of scope for this architecture series even though production review by subject-matter experts is in scope operationally. The Slack subsystem should preserve enough traceability that evaluation can happen later, but this section is not the place to define an evaluation platform.

A third non-goal is Slack-specific authentication design. The current program posture is that if a user has reached the tool surfaces that can call the service, they are already highly authenticated, and the deeper MCP-side auth specifics are not a major focus at this stage. The Slack integration should therefore record the identity material it receives and use the supported MCP contract rather than trying to become an auth architecture chapter.

### 8. Calibration surfaces and what would cause re-litigation

The first calibration surface is the Slack-side prompt itself. The architecture is fixed that the Slack application uses a model layer and that the model must treat Skywalker output as grounded input rather than as optional guidance. What is not fixed forever is the exact wording that best teaches the model to do that reliably. Re-litigation would be justified if the model repeatedly overstates weak backend output, suppresses abstention too aggressively, or fails to use sources and policy links coherently.

The second calibration surface is how much prior conversation context the Slack application should feed into its turn interpretation. The architecture is fixed that multi-turn handling belongs on the Slack side rather than in Skywalker. What remains open is how much history is enough before the model starts drifting or overfitting to stale earlier turns. Re-litigation would be justified if follow-up questions are frequently misread or if too much conversation history is clearly degrading current-turn interpretation.

The third calibration surface is the Slack-user-to-alias resolver itself. The architecture is fixed that the alias is what crosses the MCP boundary; the resolver implementation (a small internal service keyed on Slack user ID, Slack email, or another correlation) is captured as an open item in API_02. Re-litigation is justified if production review shows alias resolution is unreliable or if the resolver becomes a meaningful operational dependency on its own.

The fourth calibration surface is response shaping by Slack surface. The architecture does not require three separate internal orchestration paths, but the user experience may still benefit from different output treatments across slash commands, DMs, and channel mentions. Re-litigation would be justified if one surface clearly needs different brevity, formatting, or source presentation to remain usable.

The fifth calibration surface is the abstain presentation strategy. The backend already decides whether to abstain. The Slack layer still needs to decide how to present that abstention. Re-litigation would be justified if users systematically misread abstention as a crash, a refusal, or an empty answer rather than as an evidence-based limit.

The sixth calibration surface is human review feedback once the system is live. Because subject-matter expert review is the current realistic way to assess production quality, repeated review findings should be treated as legitimate triggers for reopening Slack-side prompt behavior, tool-use guidance, or surface-specific response shaping.

A seventh calibration surface is the balance between grounded completeness and conversational brevity. Slack is a workplace surface, not a formal report surface, so the Slack application has to answer in a form people will actually read. At the same time, this system is not allowed to trade away grounding in exchange for polish. Re-litigation would be justified if reviewers consistently find that answers are technically grounded but too compressed to be useful, or pleasantly phrased but too loose about what the backend actually supported.

### 9. Open questions, if any

The Slack framework is pinned as Bolt for Java at launch, with deployment-topology and hosting details captured in [API_02](done/API_02_Slack_Platform_Surface.md) and [API_03](done/API_03_Bolt_for_Java_Framework.md). What remains open is HTTP mode vs. Socket Mode for the deployment topology, and whether an Amazon-internal Slack framework exists that the team must use instead of Bolt. Until that is confirmed, Bolt for Java is the working assumption.

The model-layer invocation shape is pinned in [API_05](done/API_05_Bedrock_Inline_Agents.md) (Bedrock Inline Agents with a single `RETURN_CONTROL` action group; the return-control handler performs a real MCP `tools/call` against Skywalker's core surface per [API_01](done/API_01_MCP_Server_Protocol.md)). What remains open is the exact Claude Sonnet 4.6 model ID and regional availability.

Whether the Slack application should ever grow capability to hold authoritative scope and use the explicit scoped-metadata search path is not a launch question and is not pinned. The launch action group declares only `skywalker_search_by_alias`. If the Slack integration ever gains a reliable source of `country`/`level`/`role` for the user beyond what PAPI already returns, that becomes an architecture-level question for §3 decision four rather than a calibration adjustment.

Whether the three Slack surfaces should eventually diverge more in response formatting than the current architecture assumes remains a UX open question.

The exact reviewer-facing trace bundle the Slack subsystem should preserve once human-in-the-loop review begins in production remains open.

Whether the Slack surface should eventually expose any explicit user affordance for seeing the supporting sources and policy links outside the main answer body remains open.

The Slack-user-to-Amazon-alias resolution mechanism is a real dependency that is not yet pinned. It is most likely an internal Amazon directory service keyed by Slack user ID or email; see API_02's open items. This is the one Slack-side dependency that may require its own internal-only documentation before coding can start.

### Closing position

The Slack subsystem is the first place where Skywalker has to feel like a product instead of a backend. That does not mean the system boundary should blur. It means the line between retrieval and conversation needs to be sharper here than anywhere else.

Skywalker remains the scoped retrieval backend behind MCP. The Slack application remains the conversational layer that interprets the turn, calls the MCP tools, and turns grounded backend output into Slack-visible replies. Multi-turn handling lives with the Slack application. Scoped retrieval and abstain judgment live with Skywalker. That is the correct split. It makes the backend deterministic, keeps the Slack surface useful, and gives the architecture one honest place for each kind of work.
