## Section 07. Reranking, Candidate Unification, and Abstain Behavior

Section 04 fixed the online runtime path that decides whether a request stays inside the controlled FAQ arm or is widened to both retrieval arms. Section 06 fixed the integration boundary around UKB as the black-box general arm. This section now takes the output of those earlier decisions and defines the next load-bearing layer: the common scoring surface.

That surface matters because the two-arm design is only coherent if both arms eventually re-enter one disciplined judgment layer. The FAQ arm returns controlled evidence from a corpus Skywalker owns. The UKB arm returns general-domain evidence from a surface Skywalker does not control internally. If those two streams simply arrive side by side and the system pretends their raw rankings or local confidence signals are comparable, the architecture collapses into folklore. This section exists to prevent that collapse.

The reranking layer is where heterogeneous retrieval becomes one runtime decision rather than two parallel opinions. It decides what evidence should survive into the final evidence package, what evidence should be treated as too weak or too unstable to support an answer, and when the right behavior is not to push harder but to abstain. Because Skywalker sits behind MCP and feeds agents rather than rendering the final conversational answer itself, the abstain decision here is not a refusal to help. It is a structured statement that the retrieval backend does not currently have enough grounded evidence to justify an answer-shaped package.

This section therefore fixes four things that the rest of the system needs to be able to trust. It fixes what common candidate unification actually means once both arms have returned. It fixes the role and limits of the common reranker. It fixes the shape of the ranked evidence package that leaves the retrieval backend. And it fixes abstention as a first-class, valid system outcome rather than an embarrassing edge case.

### 1. What this section owns

This section owns the common scoring and evidence-selection layer that begins after the online runtime has already produced one normalized candidate pool and ends when Skywalker has either produced a reranked evidence package or declared that it should abstain. In practical terms, that means it owns the reranker-facing candidate contract, the reranker invocation itself, the post-rerank shortlist construction, the answerability judgment over that shortlist, the explicit abstain path, the post-rerank linked-item expansion that surfaces author-curated FAQ chains alongside winning anchors, and the final retrieval result package returned upward through Skywalker.

It also owns the real meaning of candidate unification. Section 04 already fixed the fact that both arms must converge into one common candidate pool before downstream scoring. This section fixes what that convergence is for. Candidate unification here does not mean pretending the FAQ arm and the UKB arm are identical systems. It means requiring that both arms present evidence in a common enough form that one scoring surface can judge them honestly without inventing false certainty from arm-local ranking metadata.

This section owns the separation between route choice and answerability. The route policy asks which arms should run. The reranking and abstain layer asks whether the returned evidence, once those arms have run, is actually strong enough to support an answer-shaped package. That distinction is load-bearing. A request can be routed correctly and still deserve abstention. A request can also be routed conservatively, widen into both arms, and then still end with one clear evidence winner. The system needs both decisions, and they are not the same decision.

This section also owns the backend-facing definition of confidence, but only in the narrow architectural sense that Skywalker must decide whether the evidence package it has assembled is usable. It does not own the agent's wording, tone, or escalation phrasing. It does not own Slack-side or QuickSuite-side conversation handling. It does not own the user's final rendered response. It owns only the retrieval backend's judgment about whether the current evidence set is strong enough to pass upward as answerable.

This section does not own retrieval routing thresholds, variant matching, or arm invocation concurrency. Section 04 already fixed those concerns. It does not own the internal behavior of UKB. Section 06 already fixed that boundary. It does not own PAPI resolution or any other scope-establishment logic. Section 02 already fixed that. It also does not own the final conversational answer generation, because Skywalker is not the agent. If a later agent chooses to answer from a weak evidence package anyway, that would be an error above this layer, not a silent permission granted by it.

The practical effect of this ownership boundary is simple. By the time a request leaves this section, the rest of the system should no longer be asking which arm produced what and which arm should be trusted by default. It should be holding one of two outcomes: a ranked evidence package that is explicit enough to support downstream answering, or a structured abstain package that is explicit enough to support downstream non-answer behavior.

### 2. Inputs, outputs, and contracts

The primary input to this section is the normalized candidate pool emitted by the Section 04 runtime orchestration layer. That pool already represents the output of the chosen route. On FAQ-only requests, it contains candidates from the controlled FAQ arm only — produced by a hybrid (BM25 + FAISS cosine) query against AOSS, with the BM25 leg and the vector leg fused by the `skywalker-faq-hybrid` search pipeline before the candidate hits the runtime. On dual-arm requests, it contains candidates from both the controlled FAQ arm (still hybrid-fused) and the UKB general arm (whose internal retrieval is opaque to Skywalker). In both cases, the pool has already been normalized into one runtime schema rather than two arm-native payloads.

The second input is the route record. This matters because reranking is not being done in a vacuum. The reranking layer needs to know whether the request reached it through an FAQ-only route, a normal dual-arm route, or a single-arm fallback path where one arm failed or timed out. Those facts do not automatically determine the outcome, but they are part of the runtime truth that later diagnosis and later abstain analysis need to preserve.

The third input is the scoped request snapshot. The reranking layer is not allowed to revisit the user's identity resolution or metadata scope, but it still needs the active scope values in its envelope because those values make the evidence package intelligible later. A reranked result should remain auditable against the exact country, level, and manager-versus-IC context that governed the search.

The reranker-facing candidate contract has to be stricter than "a list of things that looked relevant upstream." Every candidate passed into the common reranker needs a stable runtime candidate identifier, a source-arm marker retained in metadata, a stable source identifier where one exists, a rerankable evidence text payload, source title where available, source URL where available, attached policy-link metadata when available, the arm-local rank or ordering metadata that explains how the candidate reached this point, and the scope snapshot that applied to the request. These fields do not all become part of the reranker input text, but they do belong to the reranking envelope.

A second contract point is equally important. The reranker must score evidence surfaces, not answer surfaces. That means a candidate from the FAQ arm is still passed as evidence text from the controlled retrieval corpus, not as a pre-written final answer. It also means a candidate from UKB is passed as the retrieved evidence UKB returned, not as an arm-native conclusion Skywalker is supposed to forward unexamined. The reranker is comparing evidence packages, not arm-local assertions of authority.

The output of this section is one ranked evidence decision package. When the request is answerable, that package contains the final ordered shortlist, the route metadata that explains how the shortlist was reached, the evidence-strength judgment that says the package is usable, and the source and policy metadata that downstream agents need to answer from it. When the request is not answerable, the output is instead an abstain package containing the same route context, the evidence that was examined, and a structured reason why the backend judged the result too weak to pass upward as answerable.

The key contract distinction is that abstention is not represented here as an empty list. Empty evidence is one path to abstention, but not the only one. The section needs a contract rich enough to distinguish no evidence, thin evidence, single-arm-fallback evidence, and evidence that remained too ambiguous to justify answerability. Without that distinction, the agent above Skywalker would receive only "nothing came back," which is both operationally unhelpful and architecturally false.

### 3. Fixed decisions

The first fixed decision is that there is one common reranking surface for both retrieval arms. Skywalker does not compare FAQ-arm scores to UKB-arm scores directly and does not ask the later agent to resolve that problem for it. Both arms produce evidence. That evidence converges. One common reranker scores the converged set. This is the layer that makes the two-arm design mathematically honest rather than rhetorically convenient.

The second fixed decision is that raw arm-local ranking metadata is not treated as a common confidence language. The FAQ arm returns a hybrid-fused score combining BM25 and FAISS cosine contributions; that fused score is meaningful within the FAQ arm but is not commensurate with UKB's positional ordering, which itself reflects an opaque internal ranking from a black-box service. Skywalker is allowed to preserve those values for provenance. It is not allowed to treat them as directly comparable evidence of truth.

The third fixed decision is that a strong FAQ route does not bypass reranking. FAQ-only means the request does not need the UKB arm on that path. It does not mean that the FAQ arm's retrieval output should be trusted without the same common scoring and answerability discipline that the rest of the system uses. This is important because the controlled FAQ arm is high-value and high-control, not magically self-certifying.

The fourth fixed decision is that source-arm identity is preserved in metadata and excluded from the reranker text surface. The reranker should judge evidence content, not prestige labels. If Skywalker prepends "FAQ arm" or "UKB arm" into the reranker text itself, it risks teaching the scoring surface to reward source identity rather than actual relevance. Source-arm identity still matters for telemetry, audit, and later explanation, but it belongs in the surrounding envelope rather than the scored text.

The fifth fixed decision is that reranker input should include the candidate's meaningful evidence payload and may include source title when that title materially helps interpretation. The text surface should not be starved to the point of ambiguity, and it should not be bloated with metadata that was never meant to be scored as evidence. The common scoring surface should see the evidence as a strong reader would see it: enough context to know what the passage is, not so much scaffolding that the scoring surface is overwhelmed by administrative noise.

The sixth fixed decision is that the reranker's output is not a calibrated probability of correctness. It is a relative relevance judgment over the candidate set it was shown. That makes it valuable and necessary, but it also means the system cannot simply rename the top rerank score to "confidence" and pretend the problem is solved. Answerability judgment has to remain a separate backend decision built from reranked evidence quality rather than from one raw number treated as if it were a probability.

The seventh fixed decision is that routing threshold and abstain threshold are separate decisions. The routing-gate thresholds ask whether the request is confidently inside the Top 50 question set strongly enough to stay FAQ-only. The abstain threshold asks whether the evidence package, after retrieval and reranking, is strong enough to pass upward as answerable. These are different jobs and they should stay different. A request can be strongly FAQ-like and still not produce answerable evidence. A request can also be weakly FAQ-like, go dual-arm, and still produce an excellent evidence package.

The eighth fixed decision is that Skywalker is allowed to abstain. This sounds trivial, but it should be recorded plainly because many retrieval systems quietly behave as if every request must terminate in a positive package. In this system, abstention is not a defect in the control flow. It is one of the legitimate outputs of the retrieval backend. The agents above Skywalker can only behave responsibly if the backend is allowed to say that the current evidence does not justify an answer-shaped package.

The ninth fixed decision is that the output of this section is still evidence, not prose. Even when the request is answerable, Skywalker returns a ranked evidence package plus an answerability signal. It does not take over the agent's job of conversational response construction. That boundary preserves the architecture already fixed in Section 01 and prevents the reranking layer from quietly turning into a hidden answer generator.

The tenth fixed decision is the reranker model and hosting. The common scoring surface is **Cohere Rerank 4 Pro, self-hosted on SageMaker using `ml.p5.4xlarge` (H100) instances in a 2-instance always-on fleet for production**. Naming both the model and the hosting substrate is an architectural commitment: the system depends on one common rerank surface serving both arms, the 32K-token context window of Rerank 4 removes the chunk-size pressure that Rerank 3.5's 4096-token window imposed, and the H100 hosting is what makes sub-300 ms p50 latency feasible on Skywalker's 20K-token-scale payloads. A reranker migration is an architecture-class event because it would affect the scoring contract both arms converge on.

The concrete invocation surface is a SageMaker real-time endpoint serving Cohere Rerank v4.0 Pro (`rerank-v4.0-pro`), subscribed through AWS Marketplace listing `prodview-du2svpomxs5vw` (product ID `prod-b3hko54dqpujq`) with acceptance of the EULA and the $3.50/host-hour software fee. Deployment runs as the standard three-call SageMaker control-plane sequence — `CreateModel` bound to the Marketplace Model Package ARN with `enableNetworkIsolation: true` and a `VpcConfig` placing ENIs in the query service's VPC, `CreateEndpointConfig` with a single `ProductionVariant` declaring `ml.p5.4xlarge`, `initialInstanceCount: 2` (production) or `1` (beta), `inferenceAmiVersion: al2-ami-sagemaker-inference-gpu-2`, `modelDataDownloadTimeoutInSeconds: 1200`, and `containerStartupHealthCheckTimeoutInSeconds: 600` to accommodate large-weight download and GPU health checks, and `CreateEndpoint` binding named endpoints (`skywalker-rerank-prod-a`, `skywalker-rerank-prod-b`, `skywalker-rerank-beta`) to the config. Production uses two distinct endpoints fronted by client-side round-robin rather than a single multi-instance endpoint because two endpoints give independent failure domains and independent rolling updates. Auto-scaling is not used — SageMaker cold start on a fresh `ml.p5.4xlarge` is ~3–5 minutes, which is longer than any burst window we expect; instead we run always-on capacity. Sustained throughput per instance is ~12–20 QPS; cost is ≈ $9,486/month/instance including the flat Cohere Marketplace fee, producing a ≈ $18,972/month production HA fleet on-demand.

Runtime invocation goes through the separate SageMaker Runtime service (`runtime.sagemaker.us-east-1.amazonaws.com`), not the control-plane SageMaker service — they are distinct AWS services with distinct SDK clients, distinct IAM actions, and distinct endpoints. The query service's IAM execution role carries `sagemaker:InvokeEndpoint` scoped to the three endpoint ARNs and nothing broader; authentication is SigV4 against the role's temporary credentials. Request traffic reaches the endpoint through a SageMaker Runtime VPC interface endpoint (`com.amazonaws.us-east-1.sagemaker.runtime`) so it never traverses the public internet; the endpoint policy restricts invokable endpoint ARNs as a defense-in-depth layer on top of the IAM policy.

The on-the-wire request body is a standard Cohere Rerank JSON payload: `model: "rerank-v4.0-pro"`, `query` set to the user's query text, `documents` set to an array of candidate text strings in the order of the common candidate pool, `top_n: 5` (matching `/skywalker/runtime/retrieval/shortlist_size`), `max_tokens_per_doc: 4096` (the Cohere default, unchanged because Skywalker's 1000-token child ceiling means 20 candidates sit well inside the 32K-token context window), and `api_version: 2`. Per-document rendering is deliberately minimal: each document is the candidate's `title` if a usable title exists, followed by a blank line, followed by the candidate's `text`. **No arm prefix, no rank, no source URL appears in the scored text**, consistent with decisions four and five above — source-arm identity stays in the surrounding envelope, not in the content the reranker judges. UKB candidates without a usable title omit the title line and render the text alone. The response returns `results[]` containing `index` (the position in the input `documents` array, which Skywalker uses to recover the original child chunk) and `relevance_score` in `[0, 1]`.

Invocation from Java uses `software.amazon.awssdk:sagemakerruntime` holding one long-lived `SageMakerRuntimeClient` per JVM with a connection pool sized to the sustained QPS target, and `ClientOverrideConfiguration` setting both `apiCallTimeout` and `apiCallAttemptTimeout` to the 350 ms budget read from `/skywalker/runtime/rerank/evidence_timeout_ms` in SSM, plus a retry policy of `numRetries(1)` with `FixedDelayBackoffStrategy` of 50 ms — one retry absorbs a single transient TCP reset without breaching the budget, and any failure beyond that falls through to the reranker-failure fallback path rather than attempting further retries. Endpoint updates (new instance type, new model version, new config) use `UpdateEndpoint` against a new named `EndpointConfig` rather than delete-and-recreate; SageMaker performs a blue-green rollout with automatic rollback on health-check failure, and we keep an immutable config history (`skywalker-rerank-v4-pro-config-v1`, `-v2`, `…`) so manual rollback is always a one-call `UpdateEndpoint` pointing back at the previous config.

Monitoring emits the standard SageMaker CloudWatch metrics under `AWS/SageMaker` dimensioned by `EndpointName` and `VariantName`: `Invocations`, `Invocation4XXErrors`, `Invocation5XXErrors`, `InvocationsPerInstance`, `ModelLatency` (time inside the container on the prediction itself, excluding network), `OverheadLatency` (SageMaker-side authentication, routing, response framing), and the per-instance resource metrics `CPUUtilization`, `GPUUtilization`, `GPUMemoryUtilization`, `MemoryUtilization`, `DiskUtilization`. Three alarms sit on top of those metrics: `Invocation5XXErrors > 0` for three consecutive minutes (endpoint-side failure), `ModelLatency` p95 above 300 ms for five consecutive minutes (latency drift against the 350 ms budget), and `GPUUtilization` sustained above 85% (capacity pressure ahead of saturation).

The eleventh fixed decision is the abstain rule's launch posture. The composite is two branches, not more:

- **`NO_USABLE_EVIDENCE`**: the reranker returns an empty shortlist, or the orchestration layer never produced enough candidates to hand to the reranker in the first place.
- **`EVIDENCE_TOO_WEAK_AFTER_RERANK`**: the top reranked candidate's `relevance_score` falls below a configured absolute floor. Launch default `FLOOR = 0.30`, calibration-active in Section 10 C-02 and C-12.

That is the entire abstain composite. Earlier drafts of this section included a "top-two margin with substantive disagreement" branch and a "single-arm fallback with thin surviving pool" branch; both have been removed. The disagreement branch required a semantic judgment about whether two close-scoring candidates support the same answer or conflicting answers, which Skywalker cannot make without an LLM-side verifier it does not run. The fallback-thinness branch conflated arm identity with answerability, which contradicts the decision that the reranker scores candidates on a common surface without regard to arm origin. Both branches are dropped. The cross-encoder relevance score on a 20-candidate shortlist is the backbone signal; the floor value is where calibration lives.

The target operational abstain rate at launch sits in the 5 to 15 percent band for Top 50 traffic. An abstain rate materially outside that band is a tripwire for recalibration rather than a signal to let the system drift.

The twelfth fixed decision is post-rerank linked-item expansion. After the abstain decision settles in the answerable direction, each anchor in the reranked shortlist may carry a precomputed depth-2 chain of `linked_parent_ids` (Section 03, API_06). The runtime issues one AOSS `terms` query on `parent_id` against the union of the winners' own parent_ids and the parent_ids in their chains, with **no scope filter** on this expansion query — sibling expansion within an anchor's own parent inherits scope from the original retrieval, and linked-parent expansion is curatorial: an author asserting "B is contextually relevant alongside A" is the link, and that assertion stands regardless of B's own `country`/`level`/`role` tags. Linked items never enter the rerank pool; they are author-curated rideralongs that surface only when their anchor wins reranking. Each segment in the rendered evidence text carries a Unicode superscript citation marker (`¹`, `²`, `³`) and the MCP envelope record gains a parallel `citations[]` field that resolves each marker to its source. The carve-out from D-02's spirit applies only to the post-rerank expansion query; the original retrieval query continues to apply the full scope filter unchanged.

### 4. Alternatives considered or still live

One alternative would have been to skip the common reranker entirely and rely on each arm's own local ranking. On paper that looks simpler. In practice it simply relocates the difficult problem. The system would still need a way to compare the top FAQ candidate against the top UKB candidate on dual-arm requests, and it would still have no honest common scale for doing so. That alternative is rejected because it saves one explicit scoring layer only by forcing the rest of the runtime to compare incommensurable outputs.

A second alternative would have been to use separate rerankers for separate arms and compare only their top outputs afterward. That alternative is more disciplined than raw arm-local comparison, but it still fails at the same point. Two separate rerankers operating on different candidate sets still do not produce one obviously comparable decision surface unless Skywalker adds yet another meta-judgment layer above them. The current architecture is cleaner: one common reranker, one candidate universe, one evidence comparison stage.

A third alternative would have been to encode a hard source prior and say that FAQ-arm evidence should always beat UKB-arm evidence when the two are close. The appeal of that idea is obvious. Skywalker controls the FAQ arm and does not control UKB, so a hard trust prior sounds comforting. The cost is that it would distort the common scoring surface just when dual-arm competition is supposed to be most useful. The current design rejects a hard source prior as a fixed rule. A source-aware prior remains a live alternative only if later evidence shows systematic near-tie behavior that the common reranker cannot resolve satisfactorily.

A fourth alternative would have been to treat the reranker's top score as the abstain decision directly. That alternative is attractive because it produces one simple threshold. It is also too coarse. A top score means different things depending on the spread of the shortlist, the route by which the shortlist was formed, the number of surviving candidates, and the amount of degradation the request experienced before reranking. The architecture therefore rejects one-score answerability as the fixed baseline.

A fifth alternative would have been to ban abstention and always return the best evidence set available, leaving all caution to the agent above Skywalker. That would simplify the retrieval backend at the cost of making it less honest. The current architecture rejects that. The backend is the one place that actually sees the route, the candidate pool, the reranked order, and the fallback states together. It is therefore the right place to emit a first-class abstain signal.

A sixth alternative remains live around shortlist diversity. The current design does not fix a hard per-source or per-arm cap inside the reranked shortlist. That is deliberate because overly eager diversity shaping can suppress genuinely strong evidence clusters. But it remains possible that later traffic will show one source document or one arm dominating the shortlist in a way that reduces the value of the returned evidence package. If that happens, diversity controls become a serious live alternative rather than a theoretical one.

A seventh alternative remains live around reranker-failure fallback behavior. On reranker failure, the current architecture leans toward returning a clearly marked retrieval-order package rather than pretending no evidence exists. That is a design direction, not yet an irrevocable doctrine. If later evidence shows that those retrieval-order packages are too noisy or too unstable for agent use, the system may need to convert reranker failure into hard abstain instead.

### 5. Assumptions inherited from upstream sections

This section inherits the system boundary from Section 01 (Skywalker feeds agents rather than replacing them, so this section's output is a backend evidence package with an answerability judgment, not a final user-facing response) and the canonical scoped request from Section 02 (location, level, and manager-versus-IC role are part of correctness — this layer is not allowed to treat a context-free hit as a substitute for scoped correctness).

It also inherits the controlled FAQ arm from Section 03 (FAQ-arm candidates are retrieval evidence, not free-floating authored answers, and are scored as such), the route policy and candidate convergence from Section 04 (this section is not re-deciding whether both arms should have run; it judges the resulting evidence package), and the UKB normalization boundary from Section 06 (UKB candidates arrive as honest general-arm evidence packages with preserved provenance). A deeper premise from the earlier sections applies here as well: the system does not hide uncertainty in prose — provenance, route state, and arm identity are preserved so later human review can tell the difference between a weak FAQ retrieval, a weak UKB retrieval, a bad convergence, and a genuinely hard question.

### 6. End-to-end data flow for this section

The data flow begins when the normalized candidate pool arrives from the online orchestration layer. At that moment the route decision has already happened, the participating arms have already returned or failed, and the runtime holds one candidate universe plus one route record that explains how that universe came to exist.

The first step is reranker document assembly. For each candidate, the system constructs the reranker-facing evidence surface from the candidate title where useful and the candidate evidence text payload. Source-arm markers, arm-local scores, route notes, and other administrative fields remain outside the scored text. They stay attached to the candidate envelope for telemetry and audit, but they do not become part of the content the reranker is asked to judge.

The second step is common reranking. The reranker receives the live query and the assembled evidence surfaces and produces one ordered result over the candidate universe it was shown. This is the first point in the request where FAQ-arm and UKB-arm evidence are being judged on a truly common surface rather than on two independent upstream retrieval systems.

The third step is shortlist formation. The system takes the reranked output and constructs a bounded ranked evidence shortlist. This shortlist is not simply "everything that survived reranking." It is the smaller package the backend is willing to pass upward as the evidence basis for answering. The shortlist must remain wide enough to preserve supporting context and narrow enough that the agent above Skywalker is not handed a shapeless dump.

The fourth step is answerability judgment. The system examines the reranked shortlist in light of the route record and the evidence shape that reached it. This is where abstention becomes possible. A request with no surviving candidates after convergence clearly abstains. A request with one weak candidate and no meaningful support may also abstain. A request that reached reranking through a single-arm fallback route and still emerged with one clearly dominant, well-grounded candidate package may be answerable. The architecture does not flatten those cases into one score threshold because they are not the same situation.

The fifth step is abstain classification when the answerability test fails. The section classifies the abstain reason using the two-branch vocabulary fixed in §3: `NO_USABLE_EVIDENCE` when the shortlist is empty after reranking (or was never populated), and `EVIDENCE_TOO_WEAK_AFTER_RERANK` when the top candidate's `relevance_score` falls below the absolute floor. No other abstain categories exist at launch.

The sixth step is post-rerank sibling and linked-parent expansion. The answerable path does not return child-chunk text directly. For each candidate in the final shortlist, the system reads two things from the child: the child's own `parent_id`, and the child's `linked_parent_ids` array (the precomputed depth-2 chain materialized at ingest from the COREx custom "next" field per Section 03). The runtime computes the union of all these parent_ids — the anchors' own parents plus the parents of every Q&A in every anchor's chain — deduplicated, and issues one AOSS `terms` filter query against that union (API_06). The query carries no scope filter; sibling expansion within an anchor's own parent inherits scope from the original retrieval, and linked-parent expansion is curatorial. Client-side, it groups returned children by `parent_id`, sorts each group by `child_order`, and concatenates the child `text` fields using a separator derived from `split_type` (`"\n\n"` on `paragraph_boundary`, empty string on `size_forced`). The reconstructed parent text replaces the child's own text for each parent_id; the anchor's own metadata (source URL, policy links, title) remains attached to the anchor's evidence record. This is the "rerank on small chunks, answer from big chunks" pattern that keeps reranker signal-to-noise high while delivering full contextual evidence to the downstream agent — extended to the chain of author-curated linked Q&As when the anchor has one.

The seventh step is evidence-text assembly with citations. For each anchor in the shortlist, the system builds the rendered `text` field by concatenating, in order: the anchor's own reconstructed parent text, then the reconstructed parent text of each linked Q&A in `linked_parent_ids` order, separated by `"\n\n"`. Each segment is suffixed with a Unicode superscript citation marker — `¹` for the anchor, `²` for the first linked Q&A, `³` for the second. In parallel, the runtime builds the anchor's `citations[]` envelope field: one entry per rendered segment in the same order, each carrying `marker`, `source_id`, `title`, `source_url`, `policy_links` for that segment. The first entry is always the anchor itself; subsequent entries are the linked Q&As in chain order. Anchor envelope fields outside of `text` and `citations[]` continue to refer to the anchor's own Q&A. The `citations[]` field is absent on UKB candidates and on FAQ candidates whose anchor has no chain (purely additive to the existing envelope).

If a parent's returned sibling count is less than its recorded `child_count`, the reconstruction utility returns what is present in `child_order` order, logs a structured warning, and emits a CloudWatch metric. If a linked Q&A returns no chunks at all (genuine storage-level integrity issue — depublish race, missing children), the linked segment is dropped from concatenation and `LinkedItemSuppressed` fires; the answer still goes out with the surviving segments. `LinkedItemSuppressed` does **not** fire for scope mismatch, because the expansion query is unfiltered by scope — author-asserted contextual relationships override the scope filter on the expansion query specifically. Integrity-induced abstention is intentionally not a branch of the composite rule; a storage-level inconsistency is a system-maintenance concern, not a user-facing one. If a linked-segment text exceeds `linked_text_token_cap` (default 8000, calibration-active under Section 10 C-23), it is truncated at the nearest paragraph boundary, an inline `[truncated]` marker is appended, and `LinkedSegmentTruncated` fires; the segment's citation marker remains attached.

The eighth step is output package assembly. If the request is answerable, the system returns the evidence package — ranked anchors with their concatenated reconstructed-parent text (with inline citation markers), the parallel `citations[]` resolution table per anchor, the route record, and the answerability signal marked positive. If the request is not answerable, the system returns the abstain package with the route and provenance context plus the abstain classification (one of the two values). In both cases the output remains a retrieval-backend artifact, not a conversational answer.

One property of this flow is especially important. It means that answerability is decided after the two-arm design has already converged, not inside each arm independently. That preserves the architecture's main advantage. The system does not ask each arm whether it thinks it found something good enough. It asks whether the unified evidence package, once judged on one common surface, is good enough.

Another property matters just as much. The flow preserves route-truth. If UKB failed and the FAQ arm carried the request anyway, that should remain visible in the output. If the request abstained because the only surviving evidence came from a single-arm fallback route, that should also remain visible. The backend is not only selecting evidence. It is preserving how that evidence came to be trusted or not trusted.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

The first failure rule is that reranker failure is not the same thing as retrieval failure. By the time the request reaches this section, the system may still have a meaningful candidate set even if the common reranker is unavailable. The architecture should therefore treat reranker failure as its own reranker-failure fallback rather than automatically collapsing the request into the fiction that no evidence was ever found.

The second failure rule is that a reranker-failure fallback path does not inherit normal answerability assumptions. If the system chooses to return a retrieval-order evidence package on reranker failure, it should do so with a reranker-failure fallback marker and with a more conservative abstain posture than it would use under normal reranked operation. The backend should not silently pretend that retrieval order and reranked order are equivalent.

The third failure rule is that an empty candidate pool after convergence is an immediate abstain, not a later agent problem. The backend already knows that neither arm produced enough usable evidence to populate a common scoring surface. Passing that emptiness upward as if it were merely an unfortunate but still answerable state would be dishonest.

The fourth failure rule is that a non-empty shortlist is not automatically answerable. This is where many systems go wrong. They treat "something survived" as sufficient proof that an answer should be attempted. Skywalker should not do that. Weak evidence packages and brittle single-arm-fallback packages are exactly the cases where abstention should remain available.

The fifth failure rule is that disagreement between arms is not, by itself, a failure. Dual-arm requests are allowed to produce candidates from different retrieval surfaces that emphasize different parts of the answer space. The reranking layer exists partly to resolve that situation. Only when the final shortlist remains too unstable or too thin to justify answerability should the disagreement contribute to abstention.

The sixth failure rule is that missing provenance on a candidate is a structural defect, not a cosmetic omission. A candidate whose text cannot be tied back to an arm, source identifier, or request context is dangerous in a system where later trust depends on knowing where evidence came from. The section should therefore reject such candidates from the answerable shortlist even if their text looked superficially relevant.

Abstain behavior itself should remain first-class and legible. The backend should be able to say, in effect, that the request was properly scoped, that the designed retrieval route was followed or fell back to a known partial route, that the evidence was examined on the common scoring surface, and that the resulting package was still not strong enough to support an answer-shaped handoff. That is a useful outcome, not an embarrassing one.

The non-goals help keep the section from becoming a hidden policy engine. This section does not decide the final wording the user sees. It does not perform multi-turn clarification. It does not rewrite the query in hopes of saving a weak request. It does not try to resolve policy contradictions by itself. It does not convert abstention into escalation prose. Those responsibilities belong to the layers above Skywalker.

A second non-goal is equally important. This section does not retrofit a source-of-truth hierarchy that says FAQ is always right or UKB is always suspect. The architecture already encoded where control lives and where black-box leverage lives. This section's job is not to re-litigate that distinction by smuggling a hard source preference into the scoring stage.

### 8. Calibration surfaces and what would cause re-litigation

The first calibration surface is the size of the reranker candidate set. Too small and the common scoring surface may never see enough diversity to rescue weak retrieval ordering. Too large and the reranker is asked to score a noisy pool whose tail contributes cost without improving evidence quality. Re-litigation would be justified if the common reranking surface is consistently starved of useful alternatives or consistently flooded with candidates that never survive into the shortlist.

The second calibration surface is the size of the final evidence shortlist. The backend needs to pass enough evidence upward that the agent can answer responsibly, but not so much that the output package becomes diffuse. Re-litigation would be justified if the current shortlist size routinely leaves the agent with either too little grounding or too much loosely ranked material.

The third calibration surface is the abstain rule itself. The architecture is fixed that abstention is valid and that it is not the same decision as routing. What is not fixed yet is the exact composite rule that should map reranked evidence quality, route degradation, and shortlist shape into answerable versus abstain. Re-litigation would be justified when real judged traffic reveals that the system is either abstaining too often on clearly answerable requests or answering too often on clearly weak ones.

The fourth calibration surface is reranker-failure fallback behavior. Returning a clearly marked retrieval-order package may be the right launch posture because it preserves evidence when the common reranker is temporarily unavailable. Re-litigation would be justified if those packages prove too unstable for safe agent use or, conversely, if they routinely prove good enough that the system should formalize them as a supported fallback rather than as an emergency path.

The fifth calibration surface is source diversity inside the final shortlist. The current architecture resists imposing hard caps too early. Re-litigation would be justified if later observation shows that the shortlist is often dominated by one source document or one arm in ways that reduce the overall usefulness of the evidence package.

The sixth calibration surface is whether source-arm information should ever influence scoring indirectly through a source prior or tie-breaker. The current design excludes source-arm labels from reranker text and does not apply a hard source preference. Re-litigation would be justified only if later evidence shows that near-tie situations systematically produce the wrong winner and that a carefully constrained source-aware tie-break rule would improve behavior without distorting the common scoring surface.

The seventh calibration surface is the exact candidate text presented to the reranker. Title plus body is the current architectural direction because it gives the scorer enough semantic footing without drowning it in metadata. Re-litigation would be justified if later evaluation shows that titles are consistently unhelpful noise or, on the other side, that title context is so important that body-only scoring is systematically weaker.

The eighth calibration surface is the per-linked-segment text-token cap (`linked_text_token_cap`, default 8000, held in SSM at `/skywalker/runtime/retrieval/linked_text_token_cap` per Section 10 C-23). The cap protects downstream agent context windows from pathological link patterns where one linked Q&A's reconstructed parent text is very long. Re-litigation is justified if the default systematically truncates linked segments that turn out to be load-bearing to the answer, or in the opposite direction if it never truncates and is therefore not earning its existence as a knob.

### 9. Open questions, if any

The reranker service and invocation contract are pinned in [API_10](done/API_10_SageMaker_Rerank_4_Pro.md): Cohere Rerank 4 Pro (`rerank-v4.0-pro`) self-hosted on SageMaker `ml.p5.4xlarge` endpoints, invoked via AWS SDK for Java v2 `sagemakerruntime` `InvokeEndpointRequest`. Two always-on production endpoints for HA, one beta endpoint. What remains open is the Marketplace Model Package ARN (surfaces after subscription acceptance) and whether real measured p95 latency at production volume stays within the 350 ms reranker timeout.

The exact size of the reranker input pool and the final ranked shortlist are calibration-active. Launch defaults: 20 candidates into rerank (10 per arm on dual-arm; 20 from the survivor on single-arm-fallback; 20 FAQ on FAQ-only), shortlist size 5. Both move against judged-traffic evidence per Section 10 C-03 and C-04.

The reranker-failure fallback posture is settled architecturally (a clearly marked retrieval-order evidence package with `reranker_state: RERANKER_FAILURE_FALLBACK`). What remains open is whether real operational traffic justifies converting reranker failure to a hard abstain instead; that is a Section 10 re-litigation trigger, not a launch decision.

The abstain reason taxonomy is pinned at two values in API_01 (`NO_USABLE_EVIDENCE`, `EVIDENCE_TOO_WEAK_AFTER_RERANK`). What remains open is whether reviewer feedback shows these two classes produce distinguishable user-facing abstain messages from QuickSuite's chat-agent runtime and the Slack application; refinement of the vocabulary is a Section 10 C-09 trigger.

Whether the system eventually needs a source-diversity rule or a source-aware tie-break rule remains intentionally open. The current design delays both until the common reranker and plain shortlist formation have been observed under real traffic.

Whether later judged traffic will show a meaningful class of unresolved cross-arm disagreement that deserves its own explicit abstain category remains an open evaluation question, but any future abstain category would need a measurable signal Skywalker can compute without an LLM-side verifier.

The specific numeric threshold (absolute floor) is calibration-active with a launch default of 0.30 (API_10). This number exists so the system can run at launch and is explicitly expected to move against subject-matter review and judged-traffic analysis per Section 10 C-02 and C-12.

### Closing position

The two-arm architecture only works if it converges into one honest scoring and answerability layer. This section is that layer. It takes the normalized candidate universe created by the runtime path, scores it on one common reranking surface, constructs one bounded evidence shortlist, and decides whether that shortlist is strong enough to pass upward as answerable or should instead produce a structured abstain.

That is the right place for these decisions. Retrieval arms can disagree. Black-box and owned evidence can coexist. Route policy can be conservative or broad. None of that becomes a trustworthy backend until the system has one place where heterogeneous evidence is judged together and one place where the backend is allowed to say that the current package is not good enough. This section makes that judgment layer explicit.
