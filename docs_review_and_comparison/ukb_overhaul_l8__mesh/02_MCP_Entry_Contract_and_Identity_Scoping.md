## Section 02. MCP Entry Contract and Identity Scoping

Section 01 fixed the runtime boundary. Skywalker is the identity-aware retrieval backend behind an MCP surface, not the conversational layer, and later sections are not allowed to drift across that line. This section takes the next load-bearing step. It defines how requests are allowed to enter the system, what identity and scope information the system must have before retrieval begins, what the MCP surface is actually promising to callers, and how the entry path produces one canonical internal request shape regardless of whether the caller starts with an alias, an employee identifier, or already-resolved scope.

That is not a packaging concern. It is one of the places where correctness either survives or quietly fails. Travel, events, and expense answers are not generic knowledge responses. They are scoped responses. The same question can produce different correct answers depending on geography, approximate level, and whether the employee is asking as a manager or as an individual contributor. In this system, context-free is not neutral. It is wrong-shaped. The entry contract therefore has to be explicit about what information is required, where it comes from, and what the system refuses to do when that information is absent.

This section also fixes another discipline that later sections depend on. Retrieval is not allowed to improvise identity semantics after the fact. PAPI resolution, when it is needed, happens at the front of the request path. Scope is established before similarity scoring, before retrieval, before reranking, and before any answerability judgment. The routing logic and the two-arm retrieval design only make sense if the request entering those stages already carries the scope that makes the search space correct for the employee asking.

### 1. What this section owns

This section owns the contract between external callers and the internal Skywalker query path. Concretely, it owns the MCP search surface, the supported request modes, request validation at the contract boundary, identity-to-scope resolution through PAPI when the caller does not supply scope directly, normalization of those inputs into one canonical internal request form, and the conditions under which a request is permitted to proceed into routing and retrieval.

It also owns the vocabulary and responsibility split for the current scoping dimensions. The project description established three cornerstone metadata pieces for retrieval: location, level, and type, where type means manager versus individual contributor. This section makes that usable at the contract level. Location is represented here as the country or equivalent geography field that retrieval can actually filter on. Type is represented here as the role field whose current active values distinguish manager from individual contributor. If the upstream identity source later becomes richer, that can broaden the shape of the contract, but it does not change the current fact that the request pipeline needs a country-like geography value, a level value, and a manager-versus-IC role value before retrieval starts.

This section does not own the routing threshold that decides whether a query is Top 50 FAQ enough to take the FAQ-only path. That belongs downstream in the query pipeline. It does not own candidate retrieval, reranking, or retrieval-layer abstain logic. It does not own the Slack-side or QuickSuite-side conversational behavior, prompt design, or multi-turn memory. It does not own PAPI itself. It does not own caller authentication in the deeper platform sense beyond the assumption that callers reaching this MCP surface are already strongly authenticated by the surrounding internal environment. And it does not own the exact wording of the final user-facing response.

The practical output of this section is therefore narrow and decisive. A request either enters Skywalker in a valid supported mode and is transformed into one canonical scoped internal request, or it fails at the boundary and does not proceed. Later sections can build richer logic on top of that request object. They are not permitted to re-invent it.

### 2. Inputs, outputs, and contracts

The MCP surface exists to let more than one caller use one retrieval backend without each caller rebuilding policy-aware retrieval for itself. The contract therefore has to reflect the real entry conditions of the integration surfaces rather than an abstract ideal. In the current design there are three supported request modes, and each one corresponds to an actual integration need already described in the project.

The first mode accepts a user question plus an Amazon alias. This is the expected Slack-side entry mode. Slack gives us the alias, not the full scoping tuple, so Skywalker uses that alias to resolve the employee's scope through PAPI before retrieval begins.

The second mode accepts a user question plus an employee identifier. This path is available for any caller that already holds the stable employee identifier rather than the alias. Skywalker uses that identifier to resolve the same scoping tuple through PAPI as the alias path does.

The third mode accepts a user question plus the already-resolved scope fields and bypasses PAPI entirely. In the current design that means a request containing the query, the employee identifier, the employee's country, the employee's level, and the employee's role or type. This path exists because some callers will already know the scope they want Skywalker to use, or will want to exercise the system with explicit scope for testing and controlled integration purposes. When the decisive scope is already present, the architecture should not force a needless PAPI round-trip.

The surface therefore supports three search contracts rather than one vaguely overloaded payload. The alias path is for integrations that naturally hold an alias and need Skywalker to resolve scope. The employee-ID path is for integrations that naturally hold the employee identifier and need the same resolution. The explicit-scope path is for integrations that already possess the decisive scoping fields and want Skywalker to use them directly without a PAPI round-trip. All three paths produce the same downstream result from this section: one canonical scoped request ready for routing and retrieval.

At the MCP surface itself, these three entry modes map to three distinct tools: `skywalker.search.by_alias`, `skywalker.search.by_employee_id`, and `skywalker.search.by_explicit_scope`. All three are discoverable via `tools/list` and invokable via `tools/call` under the JSON-RPC 2.0 transport that MCP defines. Skywalker targets MCP protocol revision `2024-11-05` at launch because that is the revision that the QuickSuite MCP-connector implementation supports today and that Amazon MCP Gateway carries through transparently; the revision moves only as a coordinated update across the gateway, the Skywalker MCP server, and any client expectations.

All three production client paths consume those three tools through **Amazon MCP Gateway** as the transport (see Sections 05, 08, and 09). The gateway's endpoint at `api.mcp.asbx.aws.dev` is a public DNS name with auth enforced at the gateway; the architecture inherits a public-with-auth posture from the gateway, which is the BuilderHub Golden Path's recommended shape for cross-fabric Amazon-internal connectivity (see [API_13](done/API_13_Network_Fabric_and_Internet_Exposure_Posture.md)). The Coral service backing the gateway is not directly addressable on its own public DNS; the gateway is the authenticated termination, and the closed set of supported auth shapes (SigV4, CloudAuth, Federate OAuth) keeps the authorization boundary against any caller external to Amazon's auth platforms.

Slack's IAM-fleet-hosted inline-agent orchestrator and the UAT inline-agent orchestrator reach MCP Gateway on its **CloudAuth-inbound route** at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry-id}/{skywalker-server-id}`, with each orchestrator registered as a CloudAuth-modeled AAA application in ServiceLens and granted `canInvoke` on `MCPGateway::{skywalker-server-id}` plus an AAA relationship to MCP Gateway's `InvokeMcp` and to Skywalker's MCP server resource. CloudAuth OBO is on by default on this combination — MCP Gateway invokes Skywalker on behalf of the calling orchestrator's AAA application identity — and **TransitiveAuth carries the human end-user's identity through the call chain** as a separate TA token alongside CloudAuth, so Skywalker reads the human alias server-side from the TA claims rather than from MCP tool arguments. QuickSuite reaches MCP Gateway on its **Federate-OAuth-inbound route** at `https://api.mcp.asbx.aws.dev/federate/mcp/{registry-id}/{skywalker-server-id}` with a Federate Prod Service Profile; MCP Gateway does not currently propagate originating end-user identity downstream on the Federate-inbound + CloudAuth-outbound combination, so QuickSuite continues to carry identity as MCP tool arguments through its own integration-specific channel. No client has a wrapper Lambda in front of the MCP server; no client sees a task-oriented translation tool different from the three internal tools; every client reads the same `tools/list` and chooses among the three tools directly. The three-tool surface plus the per-route identity-channel discipline is what keeps the MCP boundary clean across all three production paths. Grounding contract: [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md).

Earlier drafts of this section described Slack and UAT as using SigV4 inbound at `/mcp/{registry-id}/{skywalker-server-id}` with identity carried as MCP tool arguments. That path is no longer available for new MCP servers. Per the [MCP server vendor guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/howto-vendor/): "AWSAuth inbound for CloudAuth-protected services is no longer available for new MCP servers created after April 24, 2026. Existing servers that already use this flow will continue to work until May 15, 2026. New servers must use CloudAuth inbound directly." Both deadlines have passed. The CloudAuth-inbound path is the BuilderHub-supported successor, and the OBO + TransitiveAuth pair is the documented mechanism for surfacing the calling AAA application's identity and the originating human's identity to the downstream service. The migration is a structural improvement over the SigV4 shape: identity is carried through verified auth claims rather than through MCP tool arguments the orchestrator chose to inject.

The concrete tool shapes matter because the MCP contract is the load-bearing external surface of Skywalker, and this section is the right place for that detail to live rather than hiding behind a pointer.

Each tool is declared with an `inputSchema` that MCP clients consume via `tools/list`. `skywalker.search.by_alias` requires `query_text` (string, `minLength: 1`) and `alias` (string), with an optional `correlation_id` (UUID-format string) that the caller may supply for cross-system tracing. `skywalker.search.by_employee_id` requires the same `query_text` plus an `employee_id` string and carries the same optional `correlation_id`. `skywalker.search.by_explicit_scope` requires `query_text`, `employee_id`, `country`, `level`, and `role`; `role` is an enum with values `MANAGER` and `INDIVIDUAL_CONTRIBUTOR`. The final wire encodings for `country` and `level` — whether country is ISO 3166-1 alpha-2, alpha-3, or an internal code, and which canonical format the level string takes — are finalized as the implementation settles; the architectural commitment is to the three decisive dimensions, not to a specific string format.

All three tools return the same output envelope through `structuredContent`, and a consistent `content[]` text rendition for MCP clients that prefer the unstructured block. The envelope has six top-level fields.

`result_kind` is an enum with two values: `ANSWERABLE` when evidence was found and shortlisted, or `ABSTAIN` when the retrieval path ran safely but could not produce an answer Skywalker can defend. This field is the single distinction agents consume to decide between grounded answer rendering and abstain rendering; they are not permitted to infer the distinction from `evidence` length alone.

`route` describes what retrieval path actually ran. It contains `path` (enum: `FAQ_ONLY`, `DUAL_ARM`, or `SINGLE_ARM_FALLBACK`), `surviving_arms` (array of `FAQ` and/or `UKB`), `reranker_state` (enum: `NORMAL` or `RERANKER_FAILURE_FALLBACK`), and per-arm candidate counts `faq_counts` and `ukb_counts`. Route metadata is not operational trivia — it is part of the answer's provenance and is inspected during post-hoc review of decisions.

`scope_snapshot` echoes the `country`, `level`, and `role` that actually shaped retrieval. This exists because the three entry modes take different paths to the scope triple, and auditability requires that the final values used for filtering are always visible in the response regardless of how they were obtained.

`evidence` is an array of candidate records, populated when `result_kind` is `ANSWERABLE`. Each candidate carries `candidate_id`, `source_arm` (enum: `FAQ` or `UKB`), `source_id` (the arm-native identifier for the underlying source document or passage), `title`, `text` (for FAQ candidates: the anchor's reconstructed parent text concatenated with each linked Q&A's reconstructed parent text in chain order, separated by `"\n\n"`, with each segment suffixed by a Unicode superscript citation marker — `¹` for the anchor, `²`, `³` for linked Q&As in chain order; for UKB candidates: the native UKB passage with no markers), `source_url`, `policy_links` (array of URIs), `arm_local_rank` (the arm's own pre-rerank position for provenance), `rerank_score` (the Rerank 4 Pro relevance score that won this candidate its place on the shortlist), and `citations` (an ordered array, present only on FAQ candidates whose anchor has linked items, where each entry resolves one inline marker in `text` to its source — fields per entry: `marker` (the rendered Unicode superscript), `source_id`, `title`, `source_url`, `policy_links`; the first entry is always the anchor itself). The candidate's top-level `source_id`, `title`, `source_url`, and `policy_links` continue to refer to the anchor's own Q&A; per-segment provenance for linked items lives only in `citations`. The `evidence` array may still be present on abstain results for auditability, but agents are not allowed to render abstain responses from it.

`abstain_reason` is populated when `result_kind` is `ABSTAIN`. At launch it takes one of two values: `NO_USABLE_EVIDENCE` (reranker returned an empty shortlist, or not enough candidates ever reached the reranker) or `EVIDENCE_TOO_WEAK_AFTER_RERANK` (the top candidate's rerank score fell below the absolute floor held in SSM). The enum is deliberately narrow; earlier drafts had four reasons and we dropped the two that either required semantic judgment Skywalker does not have or conflated arm identity with answerability.

`correlation_id` propagates the caller-supplied (or boundary-assigned) trace identifier back into the response. Every downstream log line, metric, and event carries this identifier so a request can be reconstructed end-to-end from its arrival at the MCP boundary through PAPI resolution, routing, retrieval, reranking, and envelope emission.

The error model splits two ways and this distinction is load-bearing rather than cosmetic. **JSON-RPC protocol errors** surface as the standard `{code, message}` error object on the JSON-RPC response and indicate that the request never entered Skywalker's retrieval path: unknown tool name, malformed arguments that fail schema validation, authentication failure at the transport layer. **Tool-execution errors** surface as a normal JSON-RPC result whose body carries `isError: true` and indicate that the request reached Skywalker but could not be safely fulfilled: PAPI unable to resolve identity into a usable scope triple, caller-supplied scope that fails validation, total retrieval failure where both arms died and no fallback could produce a candidate pool. **A backend abstain is not an error** — it is a successful result with `result_kind: ABSTAIN`. Agents that conflate abstain with error will render the wrong user experience on a substantial fraction of live traffic, which is why the contract makes the difference structural rather than something the agent has to infer.

The transport choice at launch is **Amazon MCP Gateway over HTTP** for all three production callers (Slack inline-agent orchestrator, UAT inline-agent orchestrator, QuickSuite via its own MCP connector registration), with the gateway terminating MCP's HTTP transport on its inbound side. MCP Gateway validates inbound auth (CloudAuth from Slack and UAT, Federate OAuth from QuickSuite), performs Bindle-based authorization on `MCPGateway::{skywalker-server-id}`, propagates inbound auth as CloudAuth on the outbound leg with **CloudAuth OBO carrying the calling AAA application's identity** on the Slack and UAT paths, and forwards the request to Skywalker's CloudAuth-protected MCP server. On the Slack and UAT paths, **TransitiveAuth additionally carries the originating human end-user's identity** as a separate TA token; Skywalker validates the TA token server-side and reads the human alias from the TA claims. On the QuickSuite path, MCP Gateway does not currently propagate originating end-user identity (TA delegation on the Federate-inbound + CloudAuth-outbound combination is not yet published per the [MCP Gateway concepts](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/concepts/) page), so identity rides as MCP tool arguments — `arguments.alias` or `arguments.country`/`level`/`role` — supplied by the QuickSuite chat-agent integration. STDIO transport is not used because all three callers are remote consumers, and direct HTTP-to-MCP-server connections are not used because MCP Gateway is the documented internal paved path for hosting and consuming MCP servers across Amazon. See [API_13](done/API_13_Network_Fabric_and_Internet_Exposure_Posture.md) for the gateway's authorization-boundary posture and [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md) for the CloudAuth OBO + TransitiveAuth integration.

**Skywalker's MCP responses are sync at the MCP boundary itself.** Skywalker's MCP server returns a single JSON-RPC result; MCP Gateway returns a single sync HTTP response; clients consume the structured envelope as one complete object. This is true symmetrically across Slack, UAT, and QuickSuite, and is a property of the evidence envelope being structured rather than narrative — there is no time-to-first-token to optimize on a fixed-shape JSON object, and the gateway's HTTP transport on our auth combinations does not expose SSE framing. **Streaming on the user-facing surface is a separate, higher-layer concern.** On the Slack and UAT paths, where Skywalker itself runs the inline agent that composes the final user-facing text, the inline agent's final response streams back from Bedrock as `chunk` events and the surface-specific renderer (rate-limited `chat.update` for Slack, SSE for UAT) writes those chunks progressively to the user. On the QuickSuite path, Skywalker does not run the agent at all — QuickSuite's chat-agent runtime does — and any streaming behavior on that surface is QuickSuite's own concern, not Skywalker's. See [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for the streaming contract that sits one layer above the MCP boundary.

The immediate output of this section is not the final MCP answer envelope returned to an agent. The immediate output is the internal request object that later runtime stages consume. That internal request object has one purpose: remove variation at the edge so the routing and retrieval layers do not care how the request originally entered the system. By the time this section finishes, the downstream pipeline should be holding one canonical representation of the request.

That canonical request needs to contain, at minimum, the following fields.

The first field is the original `query_text`. Skywalker is not permitted to silently replace it with some rewritten paraphrase at the MCP boundary. Query rewriting, if it ever exists later, is a downstream decision and not part of this section's contract.

The second field is the stable caller identity as known at this boundary. In the alias path, the alias is the incoming identifier and the employee identifier may be discovered as part of PAPI resolution or internal normalization. In the employee-ID path and the explicit-scope path, the employee identifier is already present. The canonical request should preserve enough identity to support downstream diagnostics and traceability.

The third field is the resolved country. The project description originally called this dimension location, which is the right architectural word, but retrieval needs a value it can actually filter on. At the current contract boundary, that concrete value is country or the equivalent geography code used by the downstream indexes.

The fourth field is the resolved level. The retrieval path needs this because entitlement and policy applicability can vary by level, and the system has already taken the position that those differences are part of correctness.

The fifth field is the resolved role or type. The current active meaning of this field is manager versus individual contributor. The contract should be written in a way that preserves this semantic truth rather than flattening it into a generic string with no documented meaning.

The sixth field is request metadata that the rest of the pipeline needs for control and traceability rather than retrieval semantics. Correlation identifiers belong here. Route metadata about how the request entered the system belongs here. Caller-supplied environment or integration markers can belong here if the implementation needs them. These fields help the system explain itself under failure and under human review later, but they should not be allowed to mutate the retrieval semantics established by the scoping fields.

One more contract decision deserves to be made explicit. This section is the place where the system decides whether a request is safe to send into retrieval at all. That means malformed requests are rejected here, missing scope is rejected here, and unresolved identity is rejected here. The later retrieval stages should receive a valid canonical request or nothing.

The contract can be summarized more precisely in internal terms. Every canonical internal request contains the original query text exactly as it arrived at the boundary. It contains the stable employee identity used for traceability and downstream control, whether that identity was supplied directly or recovered during lookup. It preserves the alias when alias mode was used because that is still useful diagnostic context even after normalization. It contains the resolved country field that makes the location dimension usable for retrieval, the resolved level field, and the resolved role field whose current semantics distinguish manager from individual contributor. It also carries the request mode and the correlation identifier so the rest of the system can explain how the request entered Skywalker and follow it across later runtime steps.

The MCP response contract itself remains intentionally narrow in this section. Section 01 already fixed that Skywalker returns structured retrieval output for agents to consume rather than polished final chat prose. This section adds only the boundary consequence of that decision: entry-mode variability is erased before retrieval begins, so later stages and later callers receive consistent downstream behavior regardless of which supported search mode was used.

### 3. Fixed decisions

Several decisions in this area are no longer exploratory. Later sections should treat them as settled unless the architecture is explicitly re-litigated.

The first fixed decision is that the MCP surface supports three explicit request modes. The system is not pretending that alias-based entry, employee-ID-based entry, and direct-scope entry are the same thing just because they all eventually become one internal request object. They are distinct contract surfaces because the information the caller possesses is meaningfully different in each case.

The second fixed decision is that PAPI resolution happens immediately after the MCP boundary when the caller did not supply direct scope. There is no intermediate stage where the request is allowed to proceed into routing or retrieval with unresolved identity scope. PAPI is first in the execution order after request acceptance because the retrieval system cannot safely operate without the scoping tuple it provides.

The third fixed decision is that the current scoping tuple is enough for the system being built now. Country represents the location dimension at the retrieval boundary. Level is an active correctness dimension. Role represents the manager-versus-individual-contributor distinction. Later versions of the project may discover that other metadata dimensions matter, but this section is not allowed to undercut the current position by keeping the active contract vague.

The fourth fixed decision is that direct caller-supplied scope is a real first-class path, not a debugging loophole. If a caller submits a request with the explicit scoping fields that the contract requires, Skywalker bypasses PAPI and uses those values to construct the canonical internal request. The whole point of that path is to avoid forcing a dependency call when the authoritative fields are already present.

The fifth fixed decision is that the direct-scope path includes the employee identifier, not just the scope fields. This keeps the path aligned with the other entry modes in terms of traceability and caller identity, even though the retrieval semantics come from the supplied scope.

The sixth fixed decision is that this section does not introduce a PAPI cache. The request path performs a live PAPI lookup when scope is absent and proceeds from there. The architecture does not add a cache layer in front of PAPI for this first implementation.

The seventh fixed decision is that this section fails closed on missing or unusable scope. It is not allowed to degrade into unscoped retrieval just because an alias lookup failed or a PAPI response came back incomplete. If the scoping tuple cannot be produced, the request stops here.

The eighth fixed decision is that authentication is not a major architectural focus in this section beyond the practical assumption that callers reaching this MCP surface are already strongly authenticated by the surrounding Amazon internal environment. That is not the same thing as saying security does not matter. It means this document series is not spending its budget inventing a new identity platform inside a system that is already operating behind authenticated internal surfaces.

The ninth fixed decision is that multi-turn conversation state does not live here. The request boundary accepts one request, produces one canonical internal request object, and passes it downstream. The conversation-aware behavior belongs to the agent layer above Skywalker, not to the MCP contract itself.

### 4. Alternatives considered or still live

This section is not all settled doctrine. Some alternatives have been considered and rejected. Others remain legitimately open and should stay visible rather than being buried in implementation defaults.

The first contract alternative is one generic search method with a tagged identity object instead of three explicit search modes. That design is attractive in one narrow sense. It reduces the number of public entry points. One method could theoretically accept a payload that says, in effect, "here is a query plus either an alias, an employee ID, or a fully scoped identity bundle." The reason that design is not the current baseline is clarity. MCP is the public contract of the service. Explicit search modes make it obvious to callers which inputs are valid for which path. That is especially useful when different integration surfaces naturally possess different identity information.

That alternative is not dead forever. If tool proliferation later becomes more painful than contract explicitness, the surface could collapse into one tagged union. But the burden of proof should be on the more abstract interface, not on the explicit one. Right now the explicit contract is cleaner.

The second alternative is mandatory PAPI resolution on every request, including requests that already provide the decisive scoping fields. That design would produce one kind of runtime path, which is superficially tidy, but it would defeat the purpose of the direct-scope contract. The whole reason to expose a direct-scope entry path is that some callers will already have the fields Skywalker needs. Forcing those callers through PAPI anyway would turn a deliberate contract decision into wasted work.

The third alternative is an alias-only world in which every caller is normalized to alias before Skywalker sees the request. That alternative is closer to the launch reality than it appears — both Slack and QuickSuite arrive with the alias as the natural identity, and the employee-ID path stays in the contract for future callers that hold the stable employee identifier rather than the alias. The MCP surface preserves the three modes because doing so keeps the contract honest about which fields are actually present at each entry point. Collapsing to alias-only would foreclose the explicit-scope short-circuit and make the boundary worse, not better.

The fourth alternative is a direct-scope contract that accepts only the scoping tuple and no stable caller identity. That is weaker than the current design because it loses traceability. Even when PAPI is bypassed, the system still benefits from knowing who the request is about. The current direct-scope path therefore keeps the employee identifier as part of the contract rather than collapsing the request down to query plus scope alone.

The fifth alternative is introducing a PAPI cache at the boundary. That idea surfaced naturally because PAPI-derived scope is the first step in the live path and the data does not change frequently. It is not the current design. The architecture deliberately keeps the first implementation simpler here and resolves identity live when scope is absent. If that later becomes an operational pain point, the system can revisit it with evidence rather than speculation.

The sixth alternative is treating scope as a soft signal that later retrieval or reranking can use opportunistically rather than as a hard precondition to search. This alternative is already rejected by the project's own correctness model. Scope is not decorative personalization. It is part of whether the answer belongs to the employee asking. That means the boundary has to produce scope up front rather than leaving the rest of the system to guess at its importance.

The seventh and still-live alternative concerns trust. Once a caller provides explicit country, level, and role directly, should Skywalker trust that data absolutely, or should it validate or audit those fields against some authoritative source later? The architecture has not frozen that answer yet. The direct-scope path exists. The remaining question is whether it is purely authoritative on its own terms or authoritative only for execution while still subject to some later trust policy.

### 5. Assumptions inherited from upstream sections

This section inherits the system boundary from Section 01: Skywalker is the retrieval backend behind MCP, PAPI resolution belongs before retrieval, and identity-aware correctness is part of the answer rather than optional personalization. The MCP entry contract therefore does not carry conversational memory, prompt instructions, or multi-turn state, and the canonical internal request produced here must be good enough for both retrieval arms downstream. The three current scope dimensions — country, level, and manager-versus-IC role — are the decisive fields this contract is built around, and no later section is allowed to treat them as optional enrichment.

### 6. End-to-end data flow for this section

The data flow through this section is short, but it is one of the most important control loops in the system because every later stage relies on its output being both valid and uniform.

The flow begins when an internal caller invokes one of the supported MCP search modes through Amazon MCP Gateway. The caller is the Slack inline-agent orchestrator (alias-based entry on the CloudAuth route, with the human alias arriving server-side via TransitiveAuth), the UAT inline-agent orchestrator (explicit-scope entry on the CloudAuth route, with the human alias additionally arriving via TransitiveAuth for audit and future policy use), QuickSuite's chat-agent runtime via its MCP connector (alias or explicit-scope entry on the Federate route, with identity in MCP tool arguments), or another approved future caller.

The MCP boundary first performs contract validation on the incoming request shape. That means it checks that the request contains a non-empty query and the fields required by the chosen entry mode. Alias mode must contain an alias. Employee-ID mode must contain an employee identifier. Explicit-scope mode must contain the employee identifier plus the direct scoping tuple. Requests that do not satisfy the entry contract do not proceed.

Once the request is syntactically valid, the boundary assigns a correlation identifier and records the entry mode. Those values do not change retrieval semantics, but they are part of the runtime truth of the request and should be present in the canonical request object that leaves this section.

The next step depends on the entry mode.

In the alias path, Skywalker sends the alias to PAPI and asks for the identity-derived fields the retrieval system needs. PAPI responds with the employee information from which country, level, and role are taken. Skywalker normalizes those fields into the contract vocabulary used by the downstream query path. It also preserves the alias as an incoming identity marker and preserves or resolves the stable employee identifier if that becomes available through the lookup.

Concretely, the PAPI call is `peopleSearchV3` with a term expression on `login` keyed to the alias (or on `personId` for the employee-ID path), returning a single `EmployeeV3` record. Skywalker reads `basicInfo.login`, `basicInfo.personId`, `workLocation.country`, `job.jobLevel`, and whichever PAPI field carries the manager-versus-IC signal; it normalizes those into the canonical `country`, `level`, `role` triple used downstream. CloudAuth with TransitiveAuth is the authentication path; the endpoint comes from SuperStar service discovery (`papi-beta.amazon.com`, `papi-gamma.amazon.com`, `papi.amazon.com`). Client-level retries (3 attempts), 30-second timeout, and fail-closed behavior on persistent failure are all properties of the PAPI client library rather than of this section's contract — but the architectural commitment here is that this layer fails closed and does not fabricate scope when PAPI cannot resolve identity.

In the employee-ID path, Skywalker sends the employee identifier to PAPI and performs the same resolution and normalization process. The difference is only the lookup key, not the shape of the resulting canonical internal request.

In the explicit-scope path, Skywalker does not call PAPI. It validates the supplied scope fields, normalizes them into the same canonical vocabulary used by the other two paths, preserves the supplied employee identifier, and constructs the same internal request object directly.

That normalization step matters. Later stages should not have to interpret a mixture of raw PAPI values, caller-specific field spellings, and integration-specific conventions. By the time this section finishes, the downstream pipeline should receive one canonical object whose country, level, and role fields are already in the vocabulary the retrieval system expects.

The complete section-level flow can be stated directly. First, the caller submits one of the supported request shapes: query plus alias, query plus employee identifier, or query plus employee identifier and explicit scope. Second, the boundary validates that the chosen request mode is complete. Third, the boundary assigns the correlation identifier and records the request mode. Fourth, the system resolves scope either by calling PAPI with the supplied identity key or by accepting the directly supplied scope in the explicit-scope path. Fifth, the system normalizes the identity and scope fields into the shared runtime vocabulary. Sixth, it constructs one canonical request object containing the query, identity, scope, and boundary metadata. Seventh, that canonical request is handed to the downstream routing and retrieval logic.

From the perspective of the rest of Skywalker, the important thing is that steps one through six disappear behind this section. The routing layer should not need conditional logic that says, in effect, "if the request came from Slack, look in one place for scope, but if it came from QuickSuite, look somewhere else." That is exactly the kind of drift this section is meant to prevent.

One more boundary flow matters here even though it does not change the internal request object. The final retrieval result leaving Skywalker later will still carry route metadata that reflects which MCP entry mode was used. That is not because the retrieval logic should branch differently later. It is because auditability matters. When a human reviewer or engineer needs to understand why a request behaved the way it did, knowing whether the request came in through alias lookup, employee-ID lookup, or direct scope is operationally useful.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

The failure behavior of this section is governed by one principle: do not let the rest of the system pretend a request is safely scoped when it is not.

The first failure class is malformed contract input. If a caller selects alias mode and omits the alias, that request fails at the MCP boundary. If a caller selects employee-ID mode and omits the employee identifier, that request fails at the boundary. If a caller selects the explicit-scope path and omits any required scope field, that request fails at the boundary. These are not retriable retrieval events. They are invalid contract uses.

The second failure class is identity lookup failure. If the request depends on PAPI and PAPI cannot resolve the caller identity into the scoping tuple the system requires, the request does not proceed into routing or retrieval. It fails closed. The system is not allowed to quietly continue with partially known identity or with no scope at all.

The third failure class is incomplete or unusable scoping data. This can happen in two ways. PAPI can return data that does not map cleanly into the current contract vocabulary. Or the explicit-scope path can receive values that are missing, malformed, or outside the allowed vocabulary. In either case the request stops here. The right corrective action is to repair the input or the upstream identity data, not to send the query into retrieval and hope reranking compensates for broken scope.

The fourth failure class is internal normalization failure. If this section cannot normalize a request into the canonical internal form that later stages require, the request fails here rather than passing an ambiguous hybrid object downstream.

Abstain behavior in this section is different from abstain behavior later in the retrieval pipeline. This section does not abstain because the evidence was weak. It can only refuse to construct a safe retrieval request. That distinction matters. A request that fails here did not reach the stage where answerability was judged. It was stopped because the system could not establish the scope that makes retrieval valid.

That means the user-visible or caller-visible surface should distinguish between boundary failure and retrieval abstain even if both eventually produce a no-answer experience. One means the request could not be safely scoped. The other means it was safely scoped and still did not produce adequate evidence.

The non-goals in this area are important because boundary sections tend to attract responsibilities they should not own.

This section does not own deeper authentication design. It assumes the surrounding internal platforms already know who is calling and that the MCP surface is not public in the general sense.

It does not own conversational continuity, session memory, or multi-turn context. Those behaviors belong to the agents above Skywalker, and later sections are not allowed to sneak them back into the MCP contract.

It does not own retrieval routing or the Top 50 similarity decision. It only produces the valid scoped request that makes that downstream decision meaningful.

It does not own reranking, answerability scoring, or the retrieval-layer abstain threshold.

It also does not own a fallback strategy that deliberately performs generic retrieval when scope is unavailable. That is not an omitted feature. It is a rejected posture.

### 8. Calibration surfaces and what would cause re-litigation

This section has fewer numeric knobs than the retrieval sections will have, but it still has calibration surfaces in the broader architectural sense. These are the places where the current design is intentionally chosen but may later need to be revisited with evidence.

The first calibration surface is the public MCP shape itself. Three explicit search modes are the current baseline because they make the contract legible and keep caller responsibilities obvious. Re-litigation would be justified if the explicit tool surface proves materially harder for callers to integrate than a single tagged contract would be, or if the number of supported request modes grows enough that the surface becomes fragmented.

The second calibration surface is the direct-scope trust model. Today the design permits callers to bypass PAPI when they supply the decisive fields directly. What remains open is how absolute that trust should be. Re-litigation would be justified if real integration behavior shows that direct caller-supplied scope is frequently wrong, inconsistent, or operationally risky.

The third calibration surface is the normalization vocabulary for country, level, and role. This section assumes those values can be normalized into one shared runtime contract. Re-litigation would be justified if the upstream identity data or the integration surfaces expose meaningful variation that the current contract cannot represent cleanly.

The fourth calibration surface is the identity key policy. The current design deliberately accepts both alias and employee identifier because that matches the actual integration surfaces. Re-litigation would be justified only if the environment standardizes more strongly on one stable key and the dual-key surface stops earning its keep.

The fifth calibration surface is whether the explicit-scope path should remain employee-ID-based only or should later permit alias plus direct scope as well. The current contract is written around `employee_id + scope` because that is the direct-shape path that has actually been put on the table. Re-litigation would be justified if a real caller has direct scope but only alias-level identity and the current contract forces unnecessary translation.

One thing is not a calibration surface here: adding a PAPI cache by default. That is not being treated as a tuning knob for this section. It would need to be reintroduced as an explicit architectural decision if it ever comes back into scope.

### 9. Open questions, if any

The MCP tool names and the shared output schema are now pinned in API_01 (tool names `skywalker.search.by_alias`, `skywalker.search.by_employee_id`, `skywalker.search.by_explicit_scope`; shared output envelope with `result_kind`, `route`, `scope_snapshot`, `evidence`, `abstain_reason`, `correlation_id`). What remains genuinely open is the final value encoding for the scope fields — specifically country (ISO alpha-2 vs. alpha-3 vs. an internal code), level (the canonical Amazon level string format), and role (`MANAGER` / `INDIVIDUAL_CONTRIBUTOR` enum values). The architectural commitment is the three decisive dimensions. The wire encoding of each dimension is finalized as the implementation settles.

The final trust posture for direct caller-supplied scope remains open. The direct-scope path is real and fixed. What remains undecided is whether the system treats those values as fully authoritative by contract or as execution-authoritative while still subject to later validation or audit.

Whether the direct-scope path should eventually permit alias plus scope as an additional first-class entry mode is open. The current architecture does not need that expansion to proceed, but the question is real enough to keep visible.

The error-code taxonomy exposed through the MCP boundary is partially pinned in API_01 (JSON-RPC protocol errors for malformed input, unknown tool, and auth failure; tool-execution errors with `isError: true` for PAPI unresolved identity, post-resolution scope failure, and total retrieval failure). What remains open is the specific error code numbers chosen for the JSON-RPC protocol-error branch.

The concrete PAPI invocation shape — which `peopleSearchV3` expression template the alias and employee-ID paths use, the fields Skywalker reads from the response, and the CloudAuth / TransitiveAuth plumbing — is captured at the reference level against the PAPI PDF already in this directory. Final wiring depends on whether Skywalker re-uses the existing `PapiClient` singleton pattern documented there or adopts a JVM-native equivalent. Either way, the architectural commitment in this section does not depend on which SDK we wrap.

### Closing position

The MCP entry contract and the identity-scoping path are where Skywalker decides whether a request is valid for retrieval at all. The system accepts three real request modes because the integration surfaces actually possess different kinds of identity information. It resolves scope through PAPI immediately when scope was not supplied directly. It bypasses PAPI when the caller already supplied the decisive scope. It normalizes every accepted request into one canonical internal form carrying query text, identity, country, level, role, and boundary metadata. And it fails closed when that canonical scoped request cannot be built safely.

That is the point of this section. Later runtime logic should be allowed to assume that a request entering routing and retrieval is already valid, already scoped, and already normalized. If later sections find themselves compensating for fuzzy identity entry or half-resolved scope, then this section has failed its job.
