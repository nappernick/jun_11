## Section 04. Query Pipeline and Online Decision Flow

Section 01 fixed the system boundary. Section 02 fixed the entry contract and scoping path. Section 03 fixed the controlled FAQ corpus as a real, implemented subsystem. This section defines the online runtime path that sits on top of those decisions — the place where the architecture stops being descriptive and starts being operational.

One scoped request arrives. The system decides whether it is close enough to the Top 50 question set to deserve an FAQ-first route, whether one arm or both should be invoked, how two retrieval arms that were not built together converge into one candidate pool, and how the path fails without quietly inventing confidence it did not earn. A controlled FAQ arm and a black-box UKB arm are only a coherent architecture if the orchestration layer above them is disciplined about exactly those questions — otherwise every later success or failure gets blamed on the wrong layer.

### 1. The orchestration boundary

This section owns the request-time orchestration that begins after the canonical scoped request exists (Section 02's output) and ends when one normalized candidate pool is ready for the common reranking surface (Section 07's input). Concretely: the routing gate and its branch policy, the concurrency model for arm invocation, the FAQ-arm query construction against the evidence surface Section 03 publishes, the UKB-arm invocation contract's runtime use, normalization of arm-native outputs into the common candidate envelope, route metadata as a first-class artifact, and the reranker handoff.

It also owns runtime statelessness. The online path knows the current request, the current scoped identity, and the currently published retrieval surfaces. It does not accumulate conversational memory, invent per-session retrieval context, or mutate behavior based on prior turns (Section 01 Decision 7).

It does not own the MCP entry contract (Section 02), the evidence corpus or its publication (Section 03), the static variant set ([API_11] — a manually maintained S3 artifact this layer only reads), UKB's internals ([API_09] — a black box behind its own MCP surface), the reranker's scoring semantics or the abstain floor (Section 07), or response rendering (Sections 05, 08, 09). A bug about a query routed to the wrong arm files here; a bug about a correctly routed pool scored wrongly files against Section 07; a bug about missing FAQ evidence on a correctly constructed query files against Section 03.

### 2. Inputs, outputs, and contracts

**Input one — the canonical scoped request.** By the time a request reaches this layer it carries the original query text, the stable caller identity, the resolved country, level, and employee class, and the boundary metadata explaining how that scope was obtained. This layer is not allowed to revisit whether scoping was necessary; it inherits that decision and uses it.

**Input two — the variant set in memory.** Loaded at service boot from `s3://skywalker-config-{stage}/variants/top_50.json` ([API_11]): a JSON object whose header carries `schema_version` (1 — unknown versions are refused), `embedding_model` (`cohere.embed-v4:0` — must match the query-time embedding or cosine similarities are meaningless), `embedding_dimension` (1024), `embedding_input_type` (`search_document`), and `generated_at`; and whose `variants[]` array carries 50 entries of `id`, `text`, and a pre-computed 1024-float `embedding`. The service validates the header, holds the variants in memory, and exposes `skywalker.variants.count` and `skywalker.variants.schema_version` as boot metrics. **A missing, malformed, or invalid variant file fails service startup.** Header validation alone has a hole worth naming: `embedding_model` is a string self-report, and Cohere ships point updates to Embed v4 — regenerate the variants under a slightly different model version and cosine quality degrades silently, with no detector except "routing feels off." Boot therefore also runs an **embedding-drift canary**: embed one known variant text live through the runtime's own embedding path, compare cosine against that variant's stored vector, and fail startup (or alarm, per stage policy) when drift exceeds tolerance (SSM `/skywalker/runtime/gate/canary_drift_tolerance`, launch 0.01). The canary is the real check; the header string is documentation. One validator nit closed with it: the header records the foundation model (`cohere.embed-v4:0`) while the runtime invokes through the `us.cohere.embed-v4:0` inference profile — a literal string comparison would false-mismatch, so validation compares model lineage, and the canary compares behavior. There is no fallback, because silently starting with an empty variant set would route every request dual-arm — a hidden quality regression rather than a visible outage. Variant matching is similarity-only: a strong match is evidence the query is *shaped like* a Top 50 question, not a claim about which one. Updates are manual (re-embed, upload; S3 versioning preserves history) and propagate by rolling restart — no hot reload, a deliberate simplicity choice for an artifact that changes on a human cadence.

**Input three — the published FAQ evidence surface.** The runtime resolves the live physical index by reading the SSM pointer `/skywalker/ingestion/faq_evidence/live_index` (Section 03 Decision 7 — there is no alias on AOSS; the pointer is the contract) and queries that index. Candidates come back as whole-node fragments in the schema Section 03 fixed.

**Input four — the UKB arm.** An external MCP surface returning passages in its own shape, with its own score semantics and its own personalization. The orchestration layer assumes none of that is commensurate with the FAQ arm's output — which is why the common candidate envelope exists.

**Output — the routed retrieval package.** Four parts. The **route record**: which path ran (`FAQ_ONLY`, `DUAL_ARM`, `SINGLE_ARM_FALLBACK`), which arms survived, whether the gate exited at cosine or invoked stage 2, and what degradations applied. The **normalized candidate pool**: every candidate in the common envelope regardless of source arm. The **per-candidate provenance**: source arm, arm-local identifier, arm-local rank, source metadata — preserved not so later stages can compare raw arm-local scores (they must not) but so the system can explain what it did under human review. The **reranker handoff**: one ordered package of normalized candidates plus route metadata, never two unrelated stacks of arm-specific output.

The envelope contract is frozen field-level in [API_01]: `candidate_id`, `source_arm`, `source_id`, `title`, `text`, `source_url`, `policy_links`, `arm_local_rank`, `rerank_score` (populated later by Section 07). Two normalization disciplines bind this layer. First, normalize what is real: a field absent for one arm stays absent or explicitly null rather than being fabricated for symmetry — an envelope that hides which fields were truly present makes later diagnosis materially harder. Second, preserve thinness: a dual-arm request returning eight FAQ candidates and one UKB candidate is a different runtime condition from four strong candidates per arm, even though both "succeeded," so arm-level counts (`faq_counts`, `ukb_counts`) ride in the route record rather than being flattened away.

### 3. Fixed decisions

**Decision 1 — The gate runs before arm selection.** The online path does not call both arms and retroactively decide whether the query "really was" FAQ-shaped. The gate is a routing mechanism, not a reporting mechanism. Binds the request lifecycle ordering. Reopens only with the always-query-both alternative (§4).

**Decision 2 — The hybrid gate architecture.** Stage 1 is cosine similarity between the query embedding and each in-memory variant embedding — pure in-process vector math, effectively free. Stage 2 is a **SageMaker-hosted Cohere Rerank v4 cross-encoder** check against the 50 variant texts, invoked only when the top cosine lands in the ambiguity band — roughly 20% of expected traffic. Routing rules, driven by SSM control-plane values ([API_07]): `top_cosine > cosine_high_threshold` (launch 0.80) → FAQ-only, stage 2 skipped; `top_cosine < cosine_low_threshold` (launch 0.30) → dual-arm, stage 2 skipped; otherwise stage 2 runs and its top `relevance_score` against `rerank_floor` (launch 0.50) decides. This buys cross-encoder accuracy exactly where bi-encoder cosine is least trustworthy — paraphrase, negation, entity substitution, compound queries — while paying its latency only on the ambiguous slice. All reranking in Skywalker, this gate stage included, runs Cohere Rerank v4 on SageMaker (Section 01 Decision 3; [API_10] Part B's Bedrock Rerank 3.5 gate is dead and flagged stale). These three thresholds are the most calibration-active values in the system; treating them as hard-coded constants is a bug. Binds the route distribution every downstream quality measurement sits on. Reopens per §8 threshold evidence; the two-stage *shape* reopens only via §4's alternatives.

**Decision 3 — Concurrency on the dual-arm path.** FAQ and UKB invocations run in parallel; dual-arm is part of the intended design, not a slow fallback. Binds Section 01 Decision 9's latency budget. Reopens with it.

**Decision 4 — Reranking is the common scoring surface.** Provenance and arm-local rank are preserved for traceability, but native scores are never compared across arms. Binds the envelope contract and Section 07. Reopens never; the alternative was rejected (§4).

**Decision 5 — PAPI does not reappear in this layer, and scope never loosens.** Scope arrived resolved; retrieval uses it on every path; no arm gets a looser request because it would prefer one. Binds the FAQ query's filter clause and the UKB `targetUser` discipline. Reopens never.

**Decision 6 — One query embedding per request.** The runtime embeds the query once through the `us.cohere.embed-v4:0` inference profile with `input_type: search_query` (the same profile discipline ingestion uses — the bare model ID is not invokable, Section 03 §2) and reuses that vector for both the gate's cosine stage and the FAQ hybrid query's k-NN leg. UKB does not share it; UKB embeds internally. Binds the embedding-model coupling in Section 03 Decision 8. Reopens with it.

**Decision 7 — No query rewriting.** The query text handed to retrieval is the text the user produced. The gate decides where to send it, not what it says. If rewriting ever enters the system it does so as an explicit architectural decision, not a quiet embellishment. Binds the evaluability of routing (judged traffic compares like with like). Reopens only by that explicit decision.

**Decision 8 — The FAQ arm runs hybrid retrieval from launch.** Every FAQ-evidence call scores both a BM25 lexical leg and a FAISS k-NN vector leg, fused through the `skywalker-faq-hybrid` search pipeline (`min_max` normalization, `arithmetic_mean` combination — created and maintained by ingestion, Section 03 §2). Hybrid is architecture-class; the per-leg weights are the calibration surface. The reasoning: vector-only retrieval is fragile on identifier-shaped tokens — policy codes, vendor names, currency abbreviations — that BM25 catches reliably, while BM25-only retrieval misses the paraphrase and conceptual matches vector similarity catches reliably, and Top 50 traffic contains both shapes. The BM25 leg scores the `text` field only — the implemented mapping carries no analyzed `title` field (an earlier draft scored `title` with a boost; there is nothing to score, `OpenSearchIndexManager.java:177-197`). One asymmetry is named rather than left for a reviewer to find: tenet 4 was wielded hard to kill chunking, and at ~56 documents with `k: 40` over `size: 20` the vector leg retrieves a large fraction of the corpus anyway — an exhaustive filter-and-score-everything design would suffice today and would carry BM25's identifier matching for free, making HNSW machinery over this corpus over-engineering by the same standard that killed the chunker. The machinery is kept anyway, for two stated reasons: the FAISS index comes with the AOSS vector-collection substrate rather than being built (the platform choice carries it, tenet 5), and the query shape is sized for corpus growth that the rebuild-from-scratch publication model makes cheap to re-index into. That is a deliberate, acknowledged exception to tenet 4's strict reading — accepted, not unnoticed. Binds the query shape, the pipeline dependency, and Section 03's analyzer calibration surface. Reopens only if judged traffic shows one leg contributing nothing — and the cheap direction (reweight via SSM) is exhausted before the expensive one (drop a leg).

**Decision 9 — Route metadata is a first-class artifact.** The runtime records the gate outcome (cosine-confident / stage-2 decided / stage-2 widened on failure), the path, arm survival, per-arm counts, and degradations — because a two-arm runtime that cannot explain its routing cannot be calibrated or reviewed. Binds the response envelope's `route` object and Section 10's calibration loop, which consumes exactly this metadata. Reopens never.

**Decision 10 — Shadow-routing on FAQ-only traffic.** *(Committed June 10, 2026 — promoted from an open question; Section 10 D-45.)* On an SSM-tunable sample of FAQ-only requests (`/skywalker/runtime/gate/shadow_sample_pct`, launch 0.10), the runtime fires the UKB arm and a shadow rerank of the combined pool **asynchronously, after the user's response has already been returned** — off the latency path entirely — and logs the comparative outcome: would a UKB candidate have outscored the served FAQ winner on the common surface? The commitment exists because of a measurement hole the architecture otherwise cannot see. The FAQ-rescue path fires only when FAQ retrieval is empty or unusable (§7), not when it is mediocre-but-passing: a query that is variant-similar but corpus-poor routes FAQ-only, retrieves a weak document, clears the 0.30 abstain floor, and answers from inferior evidence while UKB's better answer is never consulted — and nothing in the route record, the abstain telemetry, or the review traces can reveal it. The system can see when FAQ-only abstains; without shadow data it cannot see when FAQ-only *answered worse than dual-arm would have*. That blindness is not incidental: the gate threshold calibration (Section 10 C-01/C-15/C-16) is partially blind without exactly this evidence, because thresholds tuned only on abstain signals optimize against half the failure surface. The short-circuit's latency and control benefits are real; its quality cost was previously invisible, and a cost you cannot measure is not a cost you have accepted. Sampled shadow traffic costs extra UKB and rerank calls on a tenth of FAQ-only volume — accepted under the cost premise (Section 01 §5). Binds the gate calibration's evidence supply and §8 surface nine. Reopens on shadow data itself: consistently safe short-circuits sample down; consistently worse ones move the thresholds or re-litigate toward always-query-both (§4).

### 4. Alternatives considered

**Cosine-only gate.** Simpler, and meaningfully less accurate on paraphrase and negation — exactly where Top 50 questions arrive disguised. The hybrid gate pays a contingent latency cost on ~20% of traffic to recover that accuracy. Rejected.

**Always running the cross-encoder.** Adds the stage-2 cost to every query, including the ~80% cosine already classifies confidently. Rejected: the return on latency spend is smallest at the confident ends of the cosine distribution.

**Strict single-arm gate** (FAQ-like → FAQ-only; otherwise → UKB-only). Rejected because it discards the controlled corpus precisely on the near-miss cases where having it compete in the reranker helps most.

**Always-query-both with the gate as diagnostics.** Still live, not adopted. It would remove a branch from the online path at the cost of UKB latency and black-box candidates on every request. Re-litigate if runtime evidence shows the FAQ-only short-circuit buys little beyond implementation complexity.

**Arm selection by native retrieval score.** Rejected: it assigns meaning to scores from fundamentally different systems. The common reranking surface exists to avoid exactly this.

**Session-aware retrieval inside Skywalker.** Rejected; collapses the Section 01 boundary.

### 5. Assumptions inherited from upstream

From Section 01: the two-arm asymmetry (tenet 1), the latency budget (Decision 9), statelessness (Decision 7), and hybrid-at-launch (§6 there, Decision 8 here). From Section 02: requests arrive valid, scoped in the corpus vocabulary, and normalized — this layer never repairs identity. From Section 03: the evidence surface is whole-node fragments behind the SSM live pointer, complete and scope-filterable by construction; the corpus carries everybody-values as real data, which makes this section's filter semantics (below) load-bearing rather than decorative. The scoping triple — country, level, employee class — is sufficient for routing and retrieval today; this section invents no additional dimensions.

### 6. End-to-end data flow

The runtime receives the canonical scoped request and packages query text, scoped identity, and boundary metadata into one request context every downstream step references.

**Query embedding.** One call through `us.cohere.embed-v4:0` with `input_type: search_query`; the 1024-float vector is held in the request context.

**Gate stage 1 — cosine.** The runtime scores the query vector against all 50 in-memory variant embeddings, takes the top score, and records the matched variant ID for logging.

**Gate decision.** Above the high threshold: FAQ-only, stage 2 skipped. Below the low threshold: dual-arm, stage 2 skipped. In the band: **stage 2** — a Cohere Rerank v4 call on SageMaker with the standard Cohere Rerank payload: `query` set to the user text, `documents` set to the 50 variant texts in stable order, `top_n: 1` (only the top score matters), `max_tokens_per_doc: 4096`, `api_version: 2`; the response's top `relevance_score` against the rerank floor decides the route. The call is bounded by `/skywalker/runtime/gate/rerank_timeout_ms` (launch 300 ms); timeout or transport failure falls through to dual-arm and is recorded as a route-widening event, not an error — a gate failure never stops a request. Whether stage 2 runs against a dedicated lighter endpoint or shares the evidence reranker's endpoint is folded into the pending instance bake-off (Section 01 §9): the gate's payload (one query against 50 short texts) is far lighter than the evidence stage's, which argues for a dedicated small endpoint with an independent failure domain, but the call lands with the bake-off because fleet shape and instance economics are one decision. Every stage-2 outcome logs score, top variant, and resulting route.

**Arm invocation.** FAQ-only invokes the evidence surface alone; dual-arm invokes both arms concurrently. The route changes which arms run, never how scope works.

The **FAQ-evidence query** resolves the live index through the SSM pointer and posts a hybrid query with `search_pipeline=skywalker-faq-hybrid`: a `match` clause on `text` (the BM25 leg) and a `knn` clause on `embedding` carrying the query vector with over-retrieval `k` (SSM `/skywalker/runtime/retrieval/knn_overretrieve_k`, launch 40) and a FAISS `efficient_filter` that pre-filters scope inside the vector search. The scope filter implements Section 02's contract per dimension: `country` must contain the employee's geography label **or** `"Global"`; `level` must contain the employee's `L<n>` **or** `"All Job Levels"`; `role` must contain the employee's employee-class label **or** `"All Employee Classes"` — three `terms` clauses, each carrying the employee value and the everybody-value. A filter demanding exact matches alone would exclude every globally scoped document, inverting the corpus's coverage model. A `content_type: "Skywalker FAQ"` term rides as defense in depth against the enumeration-narrowing question (Section 03 §9). The pipeline fuses legs with weights set on the request (`[bm25_weight, vector_weight]`, launch `[0.3, 0.7]`, SSM `/skywalker/runtime/retrieval/hybrid_bm25_weight`) — 0.3/0.7 reflects that semantic recall does most of the work on a small, conceptually scoped corpus while BM25 catches what embedding blurs. If the search pipeline is unavailable, the runtime degrades to a plain `knn`-only query against the same index, recorded as `RETRIEVAL_VECTOR_ONLY_FALLBACK` in route metadata — degraded, visible, and never the silent default.

The **UKB invocation** is an MCP `tools/call` to `retrieve` with `maxResults` set from the per-arm budget and `targetUser` populated from the resolved identity so UKB applies its native personalization ([API_09], Section 06). It carries its own timeout (SSM `/skywalker/runtime/retrieval/ukb_timeout_ms`, launch 300 ms); exceeding it counts as arm failure for the fallback rules.

**Candidate budgets.** FAQ-only: 20 FAQ candidates into the pool. Dual-arm: ~10 per arm (SSM `/skywalker/runtime/retrieval/per_arm_candidate_budget`, launch 10), converging into one 20-candidate pool. Single-arm fallback: the survivor supplies 20.

**Normalization and handoff.** Arm-native candidates transform into the common envelope — FAQ fragments map nearly one-to-one from their index fields; UKB resources map per the [API_09] table (URI to `source_id`, resource text to `text`, `_meta.sourceUrl` to `source_url`, positional index to `arm_local_rank`, `policy_links` empty because UKB does not surface them — left empty, not fabricated). The reranker receives one unified pool plus the route record. There is no post-rerank sibling expansion and no parent reconstruction — candidates are whole-node fragments already (Section 03 Decision 5); an earlier draft's reconstruction handoff is gone with the chunk model. Section 07 returns either an answerable evidence package or a structured abstain; this layer finalizes the route record — gate outcome, path, survivals, counts, degradations — and the request leaves orchestration.

### 7. Failure behavior and abstain behavior

**Gate failure tolerance.** Stage-2 failure or timeout widens to dual-arm and records the widening. The gate is a routing aid, not a prerequisite. Stage-1 failure is in-process vector math and effectively impossible, but the same widening would apply.

**FAQ-arm rescue on the FAQ-only route.** If the gate routed FAQ-only and the FAQ retrieval fails or returns nothing usable, the runtime widens to UKB rather than converting an arm outage into a guaranteed no-answer (Section 01 §7). Routing is a quality policy, not a reason to waste a viable fallback. Note the rescue trigger's deliberate narrowness: it fires on *empty or unusable*, never on *mediocre-but-passing* — which is precisely the blindness Decision 10's shadow-routing exists to measure.

**One-arm survival on the dual-arm route.** One arm fails, the other succeeds: continue with the survivor, record `SINGLE_ARM_FALLBACK` with `surviving_arms` naming it.

**Total retrieval failure.** Both arms fail, or the survivor returns nothing usable: no meaningful pool exists, and the request proceeds to the abstain/failure path rather than pretending an empty pool is a thin answer opportunity. Whether that surfaces as `NO_USABLE_EVIDENCE` or a tool-execution error depends on whether the arms *ran and found nothing* versus *died* — the arm-failure-versus-retrieval-miss distinction is preserved per arm: a technically successful call returning zero candidates is a miss on that arm, not an outage, and is recorded as such.

**Degraded retrieval modes are visible.** Vector-only fallback, gate widening, single-arm survival — each rides in route metadata, because Section 10's calibration loop cannot distinguish bad routing from thin evidence from one-arm outages without it.

Abstention itself belongs to Section 07 — but this layer preserves it as a first-class possible outcome by handing over honest route-quality metadata rather than flattening every request into a generic success state.

Non-goals: PAPI retries or identity repair (Section 02), UKB internals, corpus publication, reranker scoring and the abstain floor, agent-side prompting, scope-broadening to salvage thin retrieval (rejected posture), session memory, hidden query rewriting.

### 8. Calibration surfaces

All gate and retrieval knobs are SSM control-plane values, tunable without redeploy, read at boot with a periodic refresh ([API_07]). That is the single most important operational fact about this section.

**Surfaces one through three — the gate thresholds.** One evidence caveat applies to all three: without Decision 10's shadow data, threshold calibration sees FAQ-only abstentions but not FAQ-only answers that were worse than dual-arm would have produced — half the failure surface. Shadow-routing is the evidence supply, not an optional extra.  `cosine_high_threshold` (0.80): lower it and more traffic short-circuits FAQ-only on cosine alone (latency for accuracy); raise it and more high-cosine-but-ambiguous cases get the stage-2 second opinion. `cosine_low_threshold` (0.30): raise it and more traffic routes dual-arm on cosine alone; lower it and stage 2 catches more cosine false-negatives. `rerank_floor` (0.50): raise for stricter FAQ-only admission, lower for more permissive. Re-litigation evidence: routed-path versus downstream-evidence-quality disagreement on judged traffic, in either direction.

**Surface four — the gate-rerank timeout** (300 ms launch). Moves with measured stage-2 latency on whatever endpoint shape the bake-off lands.

**Surface five — the branch policy itself.** Re-litigate toward always-query-both if the FAQ-only short-circuit proves to buy little, or toward stronger short-circuiting if it cleanly preserves quality while cutting noise.

**Surface six — per-arm candidate budget** (10) and **UKB timeout** (300 ms). Re-litigate the budget if one arm floods the handoff and starves the other; the timeout if UKB latency drags dual-arm p95 past budget.

**Surface seven — the hybrid weights** (`hybrid_bm25_weight` 0.30) and **over-retrieval `k`** (40). Reweight on judged evidence of systematic misranking on identifier-heavy or paraphrase-heavy queries; both tune in place, no rebuild, because both legs score at query time.

**Surface eight — the common envelope richness.** Re-litigate if evaluation shows arm-specific distinctions being lost in normalization or the reranker benefiting from a richer scored surface (Section 07 §8 tracks the same surface from the consumer side).

**Surface nine — the shadow sample rate** (`/skywalker/runtime/gate/shadow_sample_pct`, launch 0.10). Trades shadow-evidence accumulation speed against UKB and rerank spend on traffic that deliberately avoided them. Sample down on consistently safe short-circuits; sample up when the ambiguity band needs evidence faster than September volume supplies it.

### 9. Open questions

Until then, launch postures stand, and pre-launch pressure to move them resolves as a recorded non-change.

**UKB `additionalFilters`** *(disclaimer: does not gate the build; launch posture is decided — empty).* Whether Skywalker ever populates UKB's filter surface rather than relying on UKB-native personalization, and the final `client_id` issued at onboarding. Section 06 owns the integration detail.

**Envelope field promotion** *(disclaimer: does not gate the build; schema is frozen for launch).* Whether arm-level fields should be promoted from provenance into the reranker's scored text surface, pending Section 07 evaluation.

### Closing position

The online query path is where the two-arm architecture either becomes a coherent runtime system or stays two adjacent ideas. One scoped request enters. The query embeds once. The hybrid gate — cosine first, SageMaker-hosted Rerank v4 cross-encoder only on the ambiguous slice — routes against SSM thresholds operators tune without redeploy. The controlled arm retrieves hybrid (BM25 + FAISS k-NN through the `skywalker-faq-hybrid` pipeline) against the live index the SSM pointer names, scope-filtered with the employee-value-or-everybody-value semantics that make the corpus's coverage model real. The UKB arm runs concurrently when the route widens. Both arms normalize into one envelope with provenance and thinness preserved, and one routed retrieval package moves to reranking.

This section owns orchestration, route choice, concurrency, convergence, and degradation visibility. It claims neither UKB's internals, nor reranking, nor the conversational layer — and it keeps the runtime path sharp enough that later failures are diagnosed honestly instead of disappearing into "the system searched and answered."
