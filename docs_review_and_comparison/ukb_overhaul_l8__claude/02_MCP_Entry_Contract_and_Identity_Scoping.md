## Section 02. MCP Entry Contract and Identity Scoping

Section 01 fixed the runtime boundary: Skywalker is the identity-aware retrieval backend behind an MCP surface, not the conversational layer. This section takes the next load-bearing step. It defines how requests are allowed to enter the system, what identity and scope information the system must hold before retrieval begins, what the MCP surface actually promises callers, and how the entry path produces one canonical internal request shape regardless of whether the caller starts with an alias, an employee identifier, or already-resolved scope.

That is not a packaging concern. It is one of the places where correctness either survives or quietly fails. Travel, events, and expense answers are scoped responses: the same question produces different correct answers depending on the employee's country, job level, and employee class. Context-free is not neutral here — it is wrong-shaped (tenet 3). The entry contract therefore must be explicit about what information is required, where it comes from, and what the system refuses to do when that information is absent.

This section also fixes a discipline later sections depend on: retrieval is not allowed to improvise identity semantics after the fact. PAPI resolution, when needed, happens at the front of the request path — before similarity scoring, before retrieval, before reranking, before any answerability judgment. The routing logic and the two-arm design only make sense if the request entering those stages already carries the scope that makes the search space correct for the employee asking.

### 1. The contract boundary

This section owns the MCP search surface, the supported request modes, request validation at the contract boundary, identity-to-scope resolution through PAPI when the caller does not supply scope directly, the mapping from PAPI's attribute representation to the corpus's scope vocabulary, normalization of those inputs into one canonical internal request form, and the conditions under which a request is permitted to proceed into routing and retrieval.

It also owns the vocabulary for the scoping dimensions, and this is where Section 01's Decision 8 lands as contract text. The three decisive dimensions are **country/geography**, **job level**, and **employee class**. The third dimension was previously specified as a manager-versus-individual-contributor binary; that specification is dead. The production corpus scopes content on CoreX's `system_employee-class` vocabulary — `"All Employee Classes"` or specific classes such as `"F - Regular Full Time"`, `"I - Intern"`, `"B - Fixed Term Contractor - EU"` (`CorpusSchema.java:114-128`) — and a contract that speaks manager/IC to an index that speaks employee class filters nothing correctly. This section therefore replaces [API_01]'s `role` enum (`MANAGER | INDIVIDUAL_CONTRIBUTOR`, now flagged stale) with the employee-class vocabulary, and owns the mapping from PAPI's employment attributes to that vocabulary.

This section does not own the routing gate that decides whether a query is Top-50-shaped — that belongs to Section 04. It does not own candidate retrieval, reranking, or retrieval-layer abstain logic (Section 07). It does not own Slack-side or QuickSuite-side conversational behavior (Sections 08, 09). It does not own PAPI itself. It does not own caller authentication beyond consuming what MCP Gateway and the TA validator establish — the gateway authenticates and authorizes callers; this section consumes verified identity, it does not produce it. And it does not own the wording of any user-facing response.

The practical output is narrow and decisive: a request either enters Skywalker in a valid supported mode and becomes one canonical scoped internal request, or it fails at the boundary and does not proceed. Later sections build on that request object; they are not permitted to re-invent it. A bug about a request entering retrieval with wrong or missing scope files here. A bug about a correctly scoped request retrieving the wrong evidence files against Section 04. A bug about PAPI returning a wrong attribute on the employee record files against PAPI.

### 2. Inputs, outputs, and contracts

The MCP surface exists so multiple callers use one retrieval backend without each rebuilding policy-aware retrieval. The contract reflects the real entry conditions of the integration surfaces, and there are three supported request modes because the callers genuinely possess different identity information.

The first mode accepts a user question plus an Amazon alias — the expected entry mode for both launch clients, since Slack resolves its user ID to the alias and QuickSuite's Federate JWT carries the Midway login as `sub`. Skywalker uses the alias to resolve scope through PAPI before retrieval begins. The second mode accepts a question plus an employee identifier, for callers holding the stable identifier rather than the alias; resolution runs through PAPI on that key. The third mode accepts a question plus already-resolved scope fields and bypasses PAPI entirely; it exists because some callers already hold the decisive fields, and the architecture does not force a needless PAPI round-trip (it also serves controlled testing, which is how the UAT slice in Section 05 exercises it).

At the MCP surface these map to three tools: `skywalker.search.by_alias`, `skywalker.search.by_employee_id`, and `skywalker.search.by_explicit_scope` — discoverable via `tools/list`, invokable via `tools/call` under JSON-RPC 2.0. Skywalker targets MCP protocol revision `2024-11-05` at launch because that is what QuickSuite's MCP-connector implementation supports and what Amazon MCP Gateway carries through transparently; the revision moves only as a coordinated update across gateway, server, and clients.

**Input schemas.** `by_alias` requires `query_text` (string, `minLength: 1`) and `alias`; `by_employee_id` requires `query_text` and `employee_id`; `by_explicit_scope` requires `query_text`, `employee_id`, `country`, `level`, and `role`. All three accept an optional `correlation_id` (UUID-format string) for cross-system tracing. The `role` field on the explicit-scope tool carries an **employee-class value**, not a manager/IC enum. Two encoding questions remain open at the wire level (§9): whether `role` carries the letter code (`"F"`) or the full corpus label (`"F - Regular Full Time"`), and the canonical formats for `country` and `level`. The architectural commitment is to the three decisive dimensions and the employee-class vocabulary, not to a specific string format.

**Transport and identity channels.** All three production paths consume these tools through Amazon MCP Gateway (tenet 5). The gateway endpoint `api.mcp.asbx.aws.dev` is a public DNS name with auth enforced at the gateway — the public-with-auth posture inherited from the gateway's natural shape ([API_13]). Slack's inline-agent orchestrator and the UAT orchestrator use the CloudAuth-inbound route (`/ca/mcp/{registry}/{server}`), each registered as a CloudAuth-modeled AAA application with `canInvoke` on the Bindle resource `MCPGateway::{skywalker-server-id}`. On those paths, CloudAuth OBO carries the orchestrator's AAA application identity, and **TransitiveAuth carries the originating human's identity as a separate TA token** — Skywalker validates the TA token server-side and reads the human alias from TA claims, never from `arguments.alias`. QuickSuite uses the Federate-OAuth-inbound route (`/federate/mcp/{registry}/{server}`) with a Federate Prod Service Profile; the gateway does not currently propagate end-user identity on that combination, so QuickSuite carries identity in tool arguments. The argument-borne `alias` field remains in the schema for the Federate path and future non-TA callers. One thing must be said plainly rather than filed as routing detail: **argument-supplied identity is a strictly weaker trust guarantee than TA-verified identity** — Skywalker cannot verify the binding between the gateway-authenticated user session and the alias the integration asserts, which is exactly the exposure the CloudAuth paths fail closed against. Section 09 carries the full statement of that asymmetry and what bounds it ([API_14]).

The SigV4-inbound route (`/mcp/{registry}/{server}`) is closed to new MCP servers as of April 24, 2026, so for Skywalker the CloudAuth route is the only available shape for Slack and UAT — a structural improvement, since identity arrives through verified auth claims rather than through arguments the orchestrator chose to inject. No client has a wrapper Lambda in front of the MCP server; every client reads the same `tools/list` and chooses among the three tools directly.

**Output envelope.** All three tools return the same envelope through `structuredContent` (with a consistent `content[]` text rendition for clients that prefer the unstructured block). Six top-level fields:

- `result_kind` — `ANSWERABLE | ABSTAIN`. The single distinction agents consume to choose between grounded-answer rendering and abstain rendering; they are not permitted to infer it from `evidence` length.
- `route` — `path` (`FAQ_ONLY | DUAL_ARM | SINGLE_ARM_FALLBACK`), `surviving_arms`, `reranker_state` (`NORMAL | RERANKER_FAILURE_FALLBACK`), and per-arm counts `faq_counts` / `ukb_counts`. Route metadata is answer provenance, inspected during post-hoc review, not operational trivia.
- `scope_snapshot` — the `country`, `level`, and `role` (employee class) that actually shaped retrieval. The three entry modes take different paths to the triple; auditability requires the final filtering values be visible in every response regardless of how they were obtained.
- `evidence` — an array of candidate records when `result_kind` is `ANSWERABLE`. Each candidate carries `candidate_id`, `source_arm` (`FAQ | UKB`), `source_id`, `title`, `text`, `source_url`, `policy_links`, `arm_local_rank`, and `rerank_score`. For FAQ candidates, `text` is the whole-node fragment text the ingestion pipeline embedded — one CoreX node, one fragment, no reconstruction (Section 01 §2). An earlier draft of this contract specified a linked-Q&A citation chain: FAQ candidate text assembled from an anchor plus linked Q&As with Unicode superscript markers and a per-candidate `citations` array. That machinery is removed. It described parent/child reconstruction the implemented pipeline does not perform, it does not exist in [API_01]'s frozen candidate schema, and the index's `followup_fragment_ids` field — the surface that would feed it — is written empty by ingestion (`FragmentProcessor.java:112-125`). If linked-evidence rendering returns, it re-enters through a schema revision recorded in Section 10, not through quiet contract drift. The `evidence` array may still be present on abstain results for auditability, but agents are not allowed to render abstain responses from it.
- `abstain_reason` — populated on abstain: `NO_USABLE_EVIDENCE` (empty post-rerank shortlist) or `EVIDENCE_TOO_WEAK_AFTER_RERANK` (top rerank score below the SSM abstain floor). Deliberately narrow; earlier drafts carried four reasons and dropped the two that required semantic judgment Skywalker does not have or conflated arm identity with answerability.
- `correlation_id` — the caller-supplied or boundary-assigned trace identifier, propagated into every downstream log line, metric, and event so a request reconstructs end-to-end.

**Error model.** The split is load-bearing. JSON-RPC protocol errors (`{code, message}`) mean the request never entered the retrieval path: unknown tool, schema-invalid arguments, transport-layer auth failure, missing or malformed TA token on a CloudAuth path. Tool-execution errors (`isError: true`) mean the request reached Skywalker but could not be safely fulfilled: PAPI unable to resolve identity into a usable triple, caller-supplied scope failing validation, total retrieval failure with no fallback pool. A backend abstain is **not** an error — it is a successful result with `result_kind: ABSTAIN`. Agents that conflate abstain with error render the wrong user experience on a substantial fraction of live traffic, which is why the contract makes the difference structural rather than inferred.

**Sync at the MCP boundary.** Skywalker's MCP server returns a single JSON-RPC result; the gateway returns a single sync HTTP response; clients consume the envelope as one complete object. There is no time-to-first-token to optimize on a fixed-shape JSON object, and the gateway's HTTP transport on our auth combinations does not expose SSE framing. Streaming is a higher-layer concern: on the Slack and UAT paths the inline agent's final user-facing text streams from Bedrock as `chunk` events and the surface-specific renderer writes progressively (rate-limited `chat.update` for Slack, SSE for UAT — [API_12], Sections 05 and 08); on the QuickSuite path the agent belongs to QuickSuite and streaming is its concern.

**The canonical internal request.** The immediate output of this section is not the answer envelope — it is the internal request object that routing and retrieval consume, which exists to erase entry-mode variation. Every canonical request contains: the original `query_text` exactly as it arrived (query rewriting, if it ever exists, is a downstream decision, not a boundary mutation); the stable employee identity for traceability, with the alias preserved when alias mode was used; the resolved `country`; the resolved `level`; the resolved `role` carrying the employee class; and boundary metadata — entry mode, correlation identifier, caller-route markers — which supports control and audit but is not permitted to mutate retrieval semantics. Malformed requests are rejected here, missing scope is rejected here, unresolved identity is rejected here. Later stages receive a valid canonical request or nothing.

**The PAPI-to-corpus mapping contract.** Resolution alone is not enough; the resolved values must land in the vocabulary the evidence index actually filters on. The corpus carries `country` as geography labels (with `"Global"` as a real applies-to-everybody value), `level` as `L0`–`L12` (with `"All Job Levels"`), and `role` as employee-class labels (with `"All Employee Classes"`). PAPI returns employment attributes in its own representation. This section owns the deterministic mapping: PAPI's country representation to the corpus's geography label, PAPI's job level to the `L<n>` form, and PAPI's employee-class attribute to the corpus's class label. The mapping is code, not configuration — a wrong mapping silently de-scopes every request, which is the failure mode tenet 3 exists to prevent. The everybody-values are not produced by this mapping (no employee *is* `"All Job Levels"`); they are matched at query time by Section 04's `(employee value OR everybody value)` filter contract. Pinning the exact PAPI field for employee class against the PAPI reference is an execution item tracked in §9; the contract that the mapping exists, is owned here, and fails closed on unmappable values is fixed.

### 3. Fixed decisions

**Decision 1 — Three explicit request modes, not one overloaded payload.** Alias entry, employee-ID entry, and direct-scope entry are distinct contract surfaces because the information each caller possesses is meaningfully different. Binds the tool surface, client integration guides, and the gateway registration. Reopens if tool proliferation becomes materially more painful than contract explicitness — with the burden of proof on the more abstract interface (§4, alternative one).

**Decision 2 — PAPI resolution is immediate and unconditional when scope is absent.** No intermediate stage lets a request proceed with unresolved scope; PAPI is first in execution order after request acceptance. Binds the request lifecycle and Section 04's assumption that every request arrives scoped. Reopens never in direction; only the bypass path's trust posture is live (§8).

**Decision 3 — The scoping triple is country/geography, job level, and employee class.** Inherited from Section 01 Decision 8 and made contractual here, including the [API_01] enum replacement and the PAPI-to-corpus mapping ownership. Binds the explicit-scope tool schema, the `scope_snapshot` vocabulary, PAPI field selection, and Section 04's filters. Reopens only if corpus reality proves a fourth dimension necessary — a Section 10 re-litigation event, not a quiet extension.

**Decision 4 — Direct caller-supplied scope is a first-class path, not a debugging loophole.** A request carrying the explicit fields bypasses PAPI and proceeds on those values. Binds the UAT slice (Section 05) and any future scope-holding caller. Reopens with the trust-model calibration (§8) if real usage shows caller-supplied scope is frequently wrong.

**Decision 5 — The direct-scope path includes the employee identifier.** Traceability survives the PAPI bypass; the request still says who it is about. Binds the tool schema. Reopens only with calibration surface five (§8) if a real caller holds scope but only alias-level identity.

**Decision 6 — A TTL-bounded PAPI scope cache.** *(Revised June 10, 2026 — reverses the original no-cache posture; Section 10 D-44.)* Resolved scope is cached in-process, keyed by the lookup identity (alias or person ID), holding the **post-mapping** corpus-vocabulary triple plus the person identifier. A cache hit skips PAPI entirely; a miss resolves live and populates the cache. The design parameters: TTL launch default 24 hours, SSM-held (`/skywalker/runtime/papi/cache_ttl_seconds`) so the staleness window is an operator decision, not a redeploy; **no negative caching** — resolution failures are never cached, and the fail-closed rule (Decision 7) is unchanged on every miss; bounded size with LRU eviction; per-JVM in-memory only, never persisted (data minimization — the cached material is the scope triple, which exists transiently anyway). The rationale is the shape of real traffic: a multi-turn Slack conversation re-resolves the same employee's identical triple on every turn, paying dependency latency and failure exposure repeatedly for attributes that change on a cadence of months. The named cost is the staleness window: an employee promoted or relocated inside the TTL is served yesterday's scope — bounded wrongness, in the *previous-correct-scope* failure mode rather than the generic-scope mode tenet 3 forbids, and judged acceptable at 24 hours against HR attribute change cadence. The `scope_snapshot` always echoes the values actually used, so a stale-scope answer remains diagnosable from the response alone. Binds the request lifecycle (cache check precedes the PAPI call in §6) and adds calibration surface six. Reopens on staleness incidents traced to the cache, or evidence the TTL should move materially in either direction.

**Decision 7 — Fail closed on missing or unusable scope.** The boundary never degrades into unscoped retrieval because an alias lookup failed or a PAPI response came back incomplete. Binds §7's failure contract and the agents' error-rendering paths. Reopens never; a generic-retrieval fallback is a rejected posture, not an omitted feature.

**Decision 8 — Authentication is consumed, not built, here.** The gateway authenticates and authorizes callers against the Bindle resource; the TA validator establishes the human identity on CloudAuth paths; this section reads verified claims. The series does not spend its budget inventing an identity platform inside a system already operating behind authenticated surfaces. Binds the per-route identity-channel discipline in §2. Reopens if a caller arrives on an auth shape the gateway does not support — which is an AppSec review and a Section 10 event, not a contract patch.

**Decision 9 — No conversation state at the boundary.** One request in, one canonical request object out. Conversational continuity belongs to the agent layer (Section 01 Decision 7). Binds the MCP contract's statelessness. Reopens never under the current product shape.

### 4. Alternatives considered

**One generic search tool with a tagged identity union.** Attractive because it reduces public entry points to one. Rejected for clarity: MCP is the public contract, and explicit modes make it obvious to each caller which inputs are valid for its path — especially valuable when the integration surfaces naturally possess different identity information. Not dead forever; if tool proliferation later costs more than explicitness buys, the surface could collapse — but the burden of proof sits on the abstract interface.

**Mandatory PAPI resolution on every request, including explicit-scope.** One uniform runtime path, superficially tidy. Rejected because it defeats the direct-scope contract's purpose: callers that already hold the decisive fields would pay a dependency call for nothing.

**Alias-only entry.** Closer to launch reality than it appears — both launch clients arrive with an alias. Rejected because collapsing the surface forecloses the explicit-scope short-circuit and the employee-ID path for future callers, making the boundary less honest about what each caller actually holds.

**Direct-scope without caller identity.** Rejected as weaker: even when PAPI is bypassed, the system benefits from knowing who the request is about. Losing the identifier loses traceability for no gain.

**No PAPI cache (the original Decision 6).** The first draft resolved live on every request, favoring clarity over machinery. Reversed June 10, 2026: per-turn re-resolution of an unchanged triple is wasted dependency load and repeated failure exposure across multi-turn conversations, and the explicit-scope bypass does not help the launch clients, which are both alias-first. The no-cache design is now the rejected alternative; its honest advantage — zero staleness window — is preserved as the cache's calibration trigger rather than as the default posture.

**Scope as a soft signal consumed opportunistically downstream.** Already rejected by the project's correctness model: scope is part of whether the answer belongs to the employee asking, not decorative personalization. The boundary produces scope up front; the rest of the system never guesses at its importance.

### 5. Assumptions inherited from upstream

From Section 01: the system boundary (retrieval backend behind MCP), PAPI-before-retrieval ordering (Decision 2), the scoping vocabulary (Decision 8), statelessness (Decision 7), and the MCP Gateway transport posture (tenet 5). This section additionally assumes the corpus-side vocabulary is stable as characterized from the 56-record production export — geography labels, `L0`–`L12`, the known employee-class label set — and that PAPI's employment record exposes an employee-class attribute that maps deterministically onto that set. If the second assumption proves false, the consequence is that the alias and employee-ID paths cannot produce a complete triple, which would force either a PAPI enrichment request or a corpus-side vocabulary change — both Section 10 re-litigation events.

### 6. End-to-end data flow

The flow through this section is short but it is one of the most important control loops in the system, because every later stage relies on its output being valid and uniform.

A request arrives when a caller invokes one of the three tools through MCP Gateway: the Slack orchestrator (alias entry on the CloudAuth route, human alias arriving via TA claims), the UAT orchestrator (explicit-scope entry on the CloudAuth route, alias additionally via TA for audit), or QuickSuite's chat-agent runtime (alias entry on the Federate route, identity in arguments). The gateway has already authenticated the caller and authorized it against the Bindle resource before Skywalker sees anything.

The boundary first validates the contract shape: non-empty `query_text` and the fields the chosen mode requires. Alias mode must carry an alias (from TA claims on CloudAuth paths, from arguments on the Federate path); employee-ID mode an identifier; explicit-scope mode the identifier plus the full triple. Requests that fail validation stop as protocol errors. The boundary then assigns or adopts the correlation identifier and records the entry mode.

On the alias path, Skywalker first checks the scope cache (Decision 6); a fresh entry supplies the triple with no dependency call, and the request proceeds directly to normalization context. On a miss or expired entry, Skywalker calls PAPI — concretely, `peopleSearchV3` with a term expression on `login` keyed to the alias, returning a single `EmployeeV3` record — and populates the cache on success. It reads the login, the person identifier, the work-location country, the job level, and the employment attribute carrying the employee class, then runs the owned mapping into the corpus vocabulary: country to geography label, job level to the `L<n>` form, employee class to the corpus's class label. CloudAuth with TransitiveAuth is the authentication path to PAPI; the endpoint comes from SuperStar service discovery (`papi-beta.amazon.com`, `papi-gamma.amazon.com`, `papi.amazon.com`). Client-level retries (3 attempts) and the client timeout are properties of the PAPI client library; the architectural commitment here is that this layer fails closed and never fabricates scope. The employee-ID path is identical with `personId` as the lookup key.

On the explicit-scope path, Skywalker validates the supplied fields against the contract vocabulary, normalizes them through the same mapping where the caller supplied PAPI-shaped rather than corpus-shaped values, preserves the employee identifier, and constructs the canonical request without a PAPI call.

The normalization step is the point. Later stages never interpret a mixture of raw PAPI values, caller-specific spellings, and integration conventions; routing receives one canonical object whose `country`, `level`, and `role` are already in the vocabulary the evidence index filters on. The routing layer must never contain logic that says "if the request came from Slack, find scope one way; from QuickSuite, another." That drift is exactly what this section exists to prevent.

One downstream echo matters: the response's `route` metadata and `scope_snapshot` preserve how the request entered and what scope it ran under — not because retrieval branches on entry mode, but because a human reviewing a wrong answer needs to see, from the response alone, which path produced the scope that shaped it.

### 7. Failure behavior and abstain behavior

One principle governs: never let the rest of the system pretend a request is safely scoped when it is not.

**Malformed contract input.** Missing alias in alias mode, missing identifier in employee-ID mode, missing any required field in explicit-scope mode — these fail at the boundary as JSON-RPC protocol errors. They are invalid contract uses, not retriable retrieval events.

**Missing or invalid TA token on CloudAuth paths.** A protocol-level error with a descriptive message. Falling back to argument-supplied identity is deliberately not the launch posture ([API_14]); the client retries once after refreshing TA initiator state, and persistent failure is a hard error surfaced to the orchestrator's operators, not to retrieval.

**Identity lookup failure.** When PAPI cannot resolve the supplied key into a usable record — unknown alias, ambiguous result, dependency outage past retries — the request fails closed as a tool-execution error. The system does not continue with partially known identity.

**Incomplete or unmappable scope.** PAPI can return a record whose attributes do not map into the contract vocabulary (an employee class outside the known label set, a missing level); the explicit-scope path can receive values that are malformed or outside vocabulary. Either way the request stops here as a tool-execution error. The corrective action is repairing the input or the upstream identity data — not sending the query into retrieval and hoping reranking compensates for broken scope. The known employee-class set is maintained as a validated vocabulary (`CorpusSchema.KNOWN_EMPLOYEE_CLASSES`); a PAPI value outside it is treated as unmappable and logged with the unrecognized value, because the alternative — passing a novel label through — produces a filter that silently matches only everybody-scoped documents.

**Normalization failure.** If the boundary cannot construct the canonical form, the request fails here rather than passing an ambiguous hybrid object downstream.

Abstain does not exist in this section. The boundary cannot abstain — it can only refuse to construct a safe retrieval request. A request that fails here never reached the stage where answerability was judged, and the caller-visible surface must distinguish the two even when both produce a no-answer experience: one means the request could not be safely scoped; the other means it was safely scoped and the evidence was still inadequate. The error-model split in §2 is what makes that distinction structural.

The non-goals matter because boundary sections attract responsibilities they should not own. This section does not own deeper authentication design (the gateway and TA validator own it), conversational continuity (agents own it), routing or the Top-50 gate (Section 04), reranking or the abstain floor (Section 07), or any fallback that performs generic retrieval when scope is unavailable — a rejected posture, not an omitted feature.

### 8. Calibration surfaces

**Surface one — the public MCP shape.** Three explicit tools are the baseline because they keep caller responsibilities legible. Re-litigate if the explicit surface proves materially harder to integrate than a tagged union, or if supported modes grow enough to fragment the surface.

**Surface two — the direct-scope trust model.** The bypass exists and is fixed; how absolute its trust should be is intentionally open. Re-litigate if real integration behavior shows caller-supplied scope is frequently wrong, inconsistent, or operationally risky — the options then are validation-on-bypass or audit-after-the-fact, not removal of the path.

**Surface three — the normalization vocabulary.** The mapping from PAPI attributes to corpus labels assumes both sides hold still. Re-litigate if upstream identity data or the corpus's authoring conventions expose variation the current mapping cannot represent — new employee classes appearing in CoreX content are the most likely trigger, and the validated-vocabulary check in §7 is the detection mechanism.

**Surface four — the identity key policy.** Both alias and employee-ID keys are supported because the surfaces genuinely differ. Re-litigate only if the environment standardizes on one stable key and the dual surface stops earning its keep.

**Surface five — explicit-scope entry keyed by alias.** The current contract is `employee_id + scope`. Re-litigate if a real caller holds direct scope but only alias-level identity, forcing unnecessary translation.

**Surface six — the scope-cache TTL** (`/skywalker/runtime/papi/cache_ttl_seconds`, launch 86400). The knob trades dependency load against the staleness window. Re-litigate downward on any stale-scope incident traced to the cache (promotion or relocation served under prior scope inside the TTL); re-litigate upward only with evidence that PAPI load or tail latency still matters at the longer window. The cache's existence is Decision 6 and is not itself a calibration surface; the window is.

### 9. Open questions

Most open questions across this series share one precondition, stated in full in Section 10 §9: they are only answerable against real user data at meaningful volume — a few hundred actual users, arriving with the September production launch. Until then, launch postures stand, and pre-launch pressure to move them resolves as a recorded non-change.

**Wire encoding of the scope fields** *(disclaimer: does not gate the build; the internal vocabulary is fixed and the encodings are finalized as implementation settles).* Country: ISO 3166-1 alpha-2 versus alpha-3 versus the corpus's label form. Level: the canonical level string format mapping to `L0`–`L12`. Employee class: the letter code (`"F"`) versus the full corpus label (`"F - Regular Full Time"`) on the explicit-scope tool — the corpus carries full labels, which argues for full labels on the wire with the letter code rejected as ambiguous across HR systems, but the final call lands with the [API_01] contract revision this section owns.

**The exact PAPI field for employee class** *(disclaimer: execution item, not a design question; the mapping contract is fixed regardless of which field feeds it).* The alias and employee-ID paths read the employment attribute carrying class from the `EmployeeV3` record; pinning the field name and its value vocabulary against the PAPI reference, and confirming coverage for the populations the corpus scopes to (interns, fixed-term EU, agency), closes this. If PAPI cannot supply the attribute, that is a Section 10 re-litigation event (§5).

**Direct-scope trust posture** *(disclaimer: does not gate launch; the path ships as execution-authoritative).* Whether explicit-scope values remain fully authoritative by contract or become subject to later validation or audit hooks — calibration surface two carries the re-litigation triggers.

**JSON-RPC error-code numbers** *(disclaimer: does not gate the build; the error-model split is frozen).* The taxonomy is pinned — protocol errors for malformed input, unknown tool, auth and TA failures; `isError: true` for PAPI-unresolvable identity, post-resolution scope failure, total retrieval failure. The specific code numbers on the protocol branch are chosen at implementation.

**PAPI client wiring** *(disclaimer: does not gate the architecture; the commitment is independent of SDK choice).* Whether Skywalker reuses the existing `PapiClient` singleton pattern documented in the PAPI reference or adopts a JVM-native equivalent.

### Closing position

The MCP entry contract and the identity-scoping path are where Skywalker decides whether a request is valid for retrieval at all. The surface accepts three real request modes because the integration surfaces possess different identity information. It resolves scope through PAPI immediately when scope was not supplied, maps PAPI's attributes into the corpus's vocabulary — country/geography, `L0`–`L12`, employee class — through a mapping this section owns, bypasses PAPI when the caller supplied the decisive fields, normalizes every accepted request into one canonical form, and fails closed when that form cannot be built safely.

Later runtime logic is allowed to assume that a request entering routing is already valid, already scoped, and already normalized. If later sections find themselves compensating for fuzzy identity entry or half-resolved scope, this section has failed its job.
