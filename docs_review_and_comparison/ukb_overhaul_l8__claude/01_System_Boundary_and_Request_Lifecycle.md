## Section 01. System Boundary and Request Lifecycle

This section freezes the runtime boundary for Skywalker before later sections descend into storage, retrieval mechanics, and integration detail. The boundary must be explicit early. Left soft, later sections drift into the wrong system, assign ownership to the wrong component, or quietly make the retrieval backend responsible for conversational behavior that belongs to QuickSuite or to the Slack-side agent.

Skywalker is the retrieval backend behind an MCP surface for an internal Amazon assistant serving travel, events, and expense documentation. It is not the conversational brain. It does not own prompt strategy, dialogue flow, conversation memory, or user-facing prose generation. It owns identity-aware scoping, routing across two retrieval arms, normalization of candidate evidence into a common form, reranking, abstain behavior at the retrieval layer, and the structured MCP response that downstream agents consume.

That distinction is operational, not cosmetic. The backend is deliberately deterministic compared with the agent layer above it. The quality target for Skywalker is not "sound helpful in chat." The quality target is: return the right, properly scoped evidence for the employee asking, or abstain when the system cannot do so safely. A context-free answer is not neutral — for a question whose correct answer depends on the employee's country, job level, and employee class, an unscoped answer is wrong-shaped, and the architecture treats it as wrong.

This rewrite is grounded against three sources in priority order: the implemented ingestion code (`IngestionCodeReference/src/ATESkywalkerIngest`, `IngestionCodeReference/src/ATESkywalkerIngestCDK`), the API research series ([API_01] through [API_14], `research_grounding_docs/`), and the prior narrative. Where the code and the research docs disagree, the code wins and the stale research doc is flagged. Four such conflicts are resolved in this section; each is called out where it lands.

### 0. Series tenets

These tenets govern the entire architecture series. Later sections (02–10) reference them by number rather than restating them. They are decision-making tools, not values: each names what the architecture sacrifices when two good things conflict.

1. **Controlled accuracy over broad coverage.** When investing effort, the owned Top-50 FAQ path wins over the general arm. The program is judged most tightly on the questions it claims to own; UKB covers breadth.
2. **Abstention over confident overclaim.** When evidence is weak, the backend says so structurally (`result_kind: ABSTAIN`) rather than returning the least-bad candidate as though it were safe.
3. **Identity-scoped correctness over generic convenience.** Retrieval does not begin until the request carries the employee's scoping triple. A generic answer to a scoped question is a wrong answer, not a degraded one.
4. **Owned-corpus simplicity over incremental machinery.** The owned corpus is ~56 records. Daily polling and full all-or-nothing rebuild beat per-document mutation tracking at this scale, and the architecture refuses the extra machinery until corpus growth forces it.
5. **Paved-path platform primitives over bespoke transport.** Amazon MCP Gateway, CloudAuth OBO, TransitiveAuth, and OpenSearch Serverless replace the prior wrapper-Lambda-and-AgentCore-Gateway design. One transport, one Bindle authorization surface, no custom glue between clients and the MCP server.

An L8 reading these tenets should be able to predict every contested call in this series. Tenet 1 explains why the FAQ arm exists at all; tenet 2 explains the abstain contract; tenet 3 explains why PAPI sits before retrieval rather than after; tenet 4 explains why there is no chunking pipeline and no alias machinery the platform doesn't support; tenet 5 explains why the v2 wrapper design is dead.

### 1. The system boundary

Skywalker owns the following runtime responsibilities. It exposes an MCP search surface. It accepts a user question plus one of the supported identity entry modes. It resolves identity-derived scope through PAPI when the caller did not already provide the required scope directly. It runs a hybrid routing gate — cosine similarity against a static in-memory variant set, with a SageMaker-hosted Cohere Rerank v4 cross-encoder second stage, invoked only on cosine-ambiguous cases — to decide whether a request is Top-50-shaped. It decides whether the request runs through the FAQ arm alone or through both arms concurrently. The FAQ arm retrieves whole-node fragment documents from the owned evidence index in OpenSearch Serverless (collection `skywalker-faq-{stage}`, index resolved through an SSM live pointer — see the publication mechanics below). It gathers candidates from the participating arm or arms, normalizes them into a common candidate shape, reranks them through Cohere Rerank v4.0 Pro on SageMaker ([API_10] Part A), and returns a structured result bundle to the caller. It is also responsible for deciding when the retrieval layer should abstain rather than pretending the system has sufficient evidence.

Skywalker also owns the offline path that keeps the FAQ arm honest: the ingestion pipeline that polls CoreX daily, rebuilds the evidence index all-or-nothing on change, and atomically flips the live pointer. That pipeline is implemented as two Lambdas (`Poller`, `Processor`) plus CDK infrastructure, and Section 03 describes it in full; this section fixes the fact that it is inside Skywalker's ownership boundary.

Skywalker does not own the user-facing conversation. It does not manage multi-turn state. It does not decide how a final answer is worded for Slack or QuickSuite. It does not own prompt orchestration for those surfaces. It does not own UKB internals — UKB is an external MCP server Skywalker calls ([API_09]). It does not own PAPI. It does not own the human review process used after launch to validate answer quality in production, and it does not own the evaluation harness, which is out of scope for this series.

The result is a deliberate split. Skywalker is the scoped retrieval and evidence-selection layer. QuickSuite and the Slack-side agent are the conversational layers. PAPI is the identity enrichment dependency. UKB is the broad black-box retrieval dependency for the general arm. CoreX is the authoring source for the owned corpus.

One architectural wrinkle belongs in this boundary statement rather than only in the QuickSuite section: all three production client paths reach Skywalker's MCP server through **Amazon MCP Gateway**, the internal paved path for hosting and consuming MCP servers. Slack's inline-agent orchestrator and the UAT inline-agent orchestrator use the gateway's CloudAuth-inbound route at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry}/{server}`, each registered as a CloudAuth-modeled AAA application; QuickSuite uses the Federate-OAuth-inbound route at `https://api.mcp.asbx.aws.dev/federate/mcp/{registry}/{server}`. The gateway's legacy SigV4-inbound route (`/mcp/{registry}/{server}`) is closed to new MCP servers as of April 24, 2026, so for Skywalker the CloudAuth route is not a preference — it is the only available shape for the Slack and UAT paths ([API_14]). The gateway endpoint is a public DNS name with auth enforced at the gateway; the architecture inherits a public-with-auth posture from the gateway's natural shape, with the closed set of supported auth shapes (CloudAuth, Federate OAuth) keeping the authorization boundary against any caller outside Amazon's auth platforms ([API_13]). There is no wrapper Lambda between any client and the Skywalker MCP server, and no task-oriented translation tool — every client reads the same `tools/list` and chooses among the three core tools directly.

On the Slack and UAT paths, **CloudAuth OBO carries the calling orchestrator's AAA application identity** to Skywalker, and **TransitiveAuth carries the originating human's identity** as a separate TA token alongside CloudAuth, so Skywalker reads the human alias server-side from TA claims rather than from MCP tool arguments. On the QuickSuite path, MCP Gateway does not currently propagate originating end-user identity downstream on the Federate-inbound combination, so QuickSuite carries identity in MCP tool arguments. See [API_01] for the core MCP contract, [API_08] for the QuickSuite integration, and [API_14] for CloudAuth OBO + TransitiveAuth. The system boundary remains Skywalker behind MCP; Skywalker's MCP server registers with MCP Gateway, and clients reach it through the gateway.

**Quality test for this boundary:** a bug about wrong evidence for a correctly scoped employee files against Skywalker. A bug about a badly worded Slack reply built from correct evidence files against the Slack-side agent. A bug about stale general-domain content files against UKB. A bug about a wrong job level on the employee record files against PAPI. A bug about wrong scope labels on an owned FAQ document files against the CoreX content team. If any of those routings is ambiguous to a reader, this section has failed.

### 2. Inputs, outputs, and contracts

The MCP boundary exists so multiple callers can use one retrieval backend without each re-implementing policy-aware retrieval. The contract must reflect the real identity inputs the system receives, not an idealized one-size-fits-all request.

Skywalker supports three search entry modes, expressed as three MCP tools ([API_01]; tool names are placeholders pending final naming): `skywalker.search.by_alias` takes `query_text` plus an alias and resolves the employee's scoped metadata through PAPI — the expected entry mode for both launch clients. `skywalker.search.by_employee_id` takes `query_text` plus an employee identifier and resolves the same metadata through PAPI. `skywalker.search.by_explicit_scope` takes `query_text` plus the already-resolved scoping fields and bypasses PAPI entirely; it exists because some callers already know the scope they want, and the architecture should not force a needless PAPI round-trip when the caller intentionally supplies the decisive fields. All three accept an optional `correlation_id` (UUID) that flows through to the response for cross-system tracing. QuickSuite enters through the alias mode because Federate's JWT carries the user's Midway login as the `sub` claim and nothing more — the same alias-only identity shape Slack provides. On the Slack and UAT paths the alias is read from TA claims server-side; the `alias` argument remains in the schema for the Federate path and future non-TA callers.

**The scoping triple — corrected against the implemented data model.** The active correctness dimensions for retrieval are **country/geography**, **job level**, and **employee class**. This is a deliberate correction of the earlier narrative, which named the third dimension "type, meaning manager versus individual contributor," and of [API_01], whose `role` enum reads `MANAGER | INDIVIDUAL_CONTRIBUTOR`. The implemented corpus carries no such vocabulary. The real CoreX prod export grounding the ingestion code (`CorpusSchema.java`, built from 56 production records) scopes content on three metadata dimensions with these value sets:

- `country` ← CoreX `geography`: `"Global"` or country labels.
- `level` ← CoreX `system_job-level`: `"All Job Levels"` or `L0`–`L12`.
- `role` ← CoreX `system_employee-class`: `"All Employee Classes"` or specific employee classes (`"F - Regular Full Time"`, `"I - Intern"`, `"B - Fixed Term Contractor - EU"`, and ~14 others observed in production data — `CorpusSchema.java:114-128`).

A query-side filter of `role: "INDIVIDUAL_CONTRIBUTOR"` would never match a document scoped `["F - Regular Full Time"]`. The data vocabulary wins, and this is a closed decision, not a debate: the third scope dimension is **employee class**, and the index field named `role` carries employee-class labels. Two execution obligations follow. First, [API_01]'s `by_explicit_scope` enum is stale and is replaced with the employee-class vocabulary (Section 02 owns the contract revision; Section 10 records the decision). Second, PAPI resolution returns the employee's class, not a manager/IC bit — Section 02 owns specifying the mapping from PAPI's attribute representation to the corpus's label set.

The "applies to everybody" values are real data values, not synthetic catch-alls: a document scoped `["Global"]`, `["All Job Levels"]`, `["All Employee Classes"]` applies to every employee on that dimension. This shapes the **scope filter contract**: for each dimension, the retrieval filter must match documents carrying either the employee's own value or that dimension's everybody-value. A filter that requires an exact match on the employee's value alone silently excludes every globally scoped document — the most common kind in the corpus. Section 04 owns the query mechanics; the contract that the filter is `(employee value OR everybody value)` per dimension is fixed here, because getting it wrong inverts the corpus's coverage model.

The output contract matters as much as the input contract because it keeps Skywalker from drifting into agent behavior. Skywalker does not return an authored chat answer. It returns a structured evidence bundle ([API_01]): `result_kind` (`ANSWERABLE | ABSTAIN`); a `route` object recording which path produced the candidates (`path: FAQ_ONLY | DUAL_ARM | SINGLE_ARM_FALLBACK`, `surviving_arms`, `reranker_state: NORMAL | RERANKER_FAILURE_FALLBACK`, per-arm candidate counts); a `scope_snapshot` echoing the resolved triple the request was answered under; an `evidence` array of normalized candidates in ranked order; an `abstain_reason` when abstaining (`NO_USABLE_EVIDENCE | EVIDENCE_TOO_WEAK_AFTER_RERANK`); and the `correlation_id`. The `scope_snapshot` is not decoration — it is the auditable record of what identity context the answer was scoped to, and it is what makes a wrong-scope bug diagnosable from the response alone.

The internal contract between routing and reranking is also fixed here at the shape level. Both arms may participate in one request, but neither hands opaque arm-specific objects to the reranker. Before reranking, candidates are normalized into the common candidate form ([API_01]): `candidate_id`, `source_arm` (`FAQ | UKB`), `source_id`, `title`, `text`, `source_url`, `policy_links`, `arm_local_rank`, `rerank_score`. Without a common schema the reranker is a fiction rather than a true common scoring surface. The field-level schema is frozen in [API_01]; what remains open is whether evaluation reveals arm-specific distinctions being lost in normalization (Section 07 §8 tracks this).

One structural simplification against the earlier narrative: the owned FAQ arm retrieves **whole-node fragment documents** — one document per CoreX node, no parent/child chunking, no post-rerank sibling expansion. The implemented pipeline (`FragmentProcessor.java:69-97`) embeds each node's full extracted text as a single fragment; the `followup_fragment_ids` field exists in the index mapping as a reserved future surface and is written empty (`FragmentProcessor.java:112-125`). [API_06]'s chunk/parent/sibling-expansion model describes machinery that does not exist and, at ~56 short FAQ records, should not (tenet 4). [API_06] is flagged stale on this point. The reopen condition is concrete: if corpus growth or document length pushes individual nodes past the embedding model's useful input size or past reranker payload budgets, chunking re-enters as a Section 03 design change.

### 3. Fixed decisions

These decisions are fixed strongly enough that later sections treat them as architectural constraints, not live debates. Each states what it binds and what would reopen it.

**Decision 1 — Skywalker is a two-arm retrieval system.** One arm is the controlled Top-50 FAQ path; the other is the general path through UKB. The FAQ arm exists because the highest-value, highest-accuracy questions are precisely where the project cannot surrender control to a black box (tenet 1). UKB exists because it already exposes broad documentation retrieval through MCP, so Skywalker does not rebuild general retrieval for the wider travel, events, and expense domains. This binds the routing gate, the normalization layer, and the reranker contract. It reopens only if UKB's quality on Top-50-shaped questions is measured to match the owned arm's — at which point the owned arm's cost must be rejustified — or if UKB becomes unavailable as a dependency.

**Decision 2 — Identity-derived scoping happens before retrieval, not after.** PAPI is not a downstream enrichment step; it resolves the metadata that shapes retrieval in the first place. If a caller supplied only an alias or employee ID, Skywalker calls PAPI before any similarity matching, retrieval, or reranking. Only the explicit-scope entry mode bypasses PAPI. This binds the request lifecycle ordering and the fail-closed behavior in §6. It reopens never in direction (post-hoc scoping violates tenet 3 structurally); the only live question is the trust posture on the bypass path (§8, calibration surface four).

**Decision 3 — The routing policy is a hybrid gate with a genuine short-circuit.** The gate runs cosine similarity against the in-memory variant set first, with three SSM control-plane values: `/skywalker/runtime/gate/cosine_high_threshold` (launch default 0.80), `/skywalker/runtime/gate/cosine_low_threshold` (0.30), and `/skywalker/runtime/gate/rerank_floor` (0.50). Top cosine above the high threshold routes FAQ-only; below the low threshold routes dual-arm; the ambiguity band between them — roughly 20% of expected traffic — invokes a SageMaker-hosted Cohere Rerank v4 cross-encoder against the 50 variant texts, with the rerank floor deciding FAQ-only versus dual-arm. Gate-reranker timeout (300 ms, SSM `/skywalker/runtime/gate/rerank_timeout_ms`) falls through to dual-arm: the gate degrades toward the safer, broader path. **All reranking in Skywalker — the gate's cross-encoder stage and the evidence stage — runs Cohere Rerank v4 on SageMaker endpoints.** This supersedes [API_10] Part B, which placed the gate on Bedrock's Rerank 3.5 (`cohere.rerank-v3-5:0`); that design is dead and [API_10] is flagged stale on it. One model family, one hosting pattern, one operational surface. Whether the gate shares the evidence endpoint or runs a dedicated one is a fleet-sizing call folded into the instance-selection decision (§9); at gate payload sizes (one query against 50 short variants) either shape fits inside the 300 ms timeout on any candidate instance. This binds Section 04's query pipeline and the variant-set artifact contract ([API_11]). Reopen signals: repeated evidence that FAQ-only-routed traffic would have been better served dual-arm, or the reverse.

**Decision 4 — Concurrency eliminates sequential arm latency on the dual-arm path.** When both arms participate, they run at the same time; the architecture pays for parallel work, never for serial fan-out latency. Cost is established as a non-issue for this project, so the governing reason to revisit is answer quality, not compute spend. Binds the latency budget (Decision 9). Reopens only with that budget.

**Decision 5 — The reranker is the common scoring surface.** Arm-local scores are not globally meaningful. UKB has its own notion of relevance; the FAQ arm has its own retrieval scores. Skywalker does not normalize source scores into a synthetic number before reranking — it uses Cohere Rerank v4.0 Pro on a dedicated SageMaker endpoint (two-instance HA fleet in prod) as the place where heterogeneous candidates compete against the same query under one scoring model, capped at 20 candidates per request ([API_10] Part A). The instance type is deliberately not fixed here: the Marketplace package supports `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge` at a flat $3.50/host-hour software fee, and the cost-versus-latency selection among them is pending executive approval for the bake-off (§9) — HA-pair all-in cost ranges roughly $6.6K/month (g5.xlarge) to ~$19K/month (p5.4xlarge). Binds the common candidate schema and Section 07 entirely. Reopens if a measured systematic bias against one arm's candidate style survives calibration.

**Decision 6 — The FAQ arm is a retrieval system fed by an all-or-nothing rebuild, not an answer cache.** The corpus is small and owned, and the implemented pipeline treats freshness as a daily batch concern: an EventBridge cron (08:00 UTC launch default, `app.ts:54`) invokes the Poller, which reads a single high-water mark — the most recent `lastModifiedDate` ingested, SSM `/skywalker/ingestion/faq_evidence/last_snapshot_marker` — enumerates the corpus from CoreX, and exits no-op when the marker is unchanged. On change, it rebuilds the **entire corpus from the ground up** into the idle one of two physical indices (`faq_evidence_a` / `faq_evidence_b`) while the live index keeps serving, and promotes only on 100% completion: every enumerated node successfully processed (after per-node retries) and every document verified queryable by read-back polling, tolerating OpenSearch Serverless's ~10 s refresh interval and a measured ~64 s cold first refresh on freshly created indices (`RebuildCoordinator.java:127-167`). Promotion is an atomic flip of the SSM live pointer `/skywalker/ingestion/faq_evidence/live_index`; the query service reads that pointer to resolve which index to query. A partial or unverifiable build never promotes — the previous complete corpus keeps serving and the marker does not advance, so a systemic failure (auth, throttle, network) cannot ship a fragmentary index over a complete one. This two-physical-indices-plus-pointer design supersedes [API_06]'s alias-swap description: AOSS Serverless does not support index aliases, so the pointer is the atomicity primitive ([API_06] flagged stale). Binds Section 03 in full and the query service's index-resolution behavior in Section 04. Reopens if the corpus grows past the scale where daily full rebuild is acceptable — the trigger is rebuild duration approaching the Poller's 15-minute Lambda ceiling, not document count per se.

**Decision 7 — Skywalker is stateless per request.** Multi-turn continuity belongs to the model-facing agent layer. For Slack, the Slack-side agent and its model own conversation history; for QuickSuite, QuickSuite's chat-agent runtime owns it. Skywalker benefits when a better agent asks a better-scoped follow-up query, but it stores no conversation state. Binds the MCP contract (no session affinity) and the horizontal scaling model. Reopens never under the current product shape; server-side conversation state was considered and rejected (§4).

**Decision 8 — The active scope dimensions are country/geography, job level, and employee class.** Grounded in the production corpus as described in §2. Three further properties are fixed with it: scope values are **arrays** (a document may be scoped to several countries or levels); the everybody-values are real data, so the per-dimension filter contract is `(employee value OR everybody value)`; and ingestion **refuses to publish unscoped evidence** — a node missing any real value on any dimension is skipped and logged, never backfilled with a synthetic catch-all (`FragmentProcessor.java:79-92`). Across the real production set, all three dimensions are present on ~100% of nodes, so the skip rule is a malformed-node safety net, not a content filter. Binds the MCP contract revision (Section 02), PAPI attribute resolution (Section 02), index mappings (Section 03), and query filters (Section 04). Reopens if corpus reality proves a fourth identity dimension is required — a re-litigation event recorded in Section 10, not a quiet extension.

**Decision 9 — The system-level latency budget.** *(Revised June 10, 2026.)* The Skywalker retrieval pipeline — MCP request acceptance through normalized evidence package return — targets **p95 of 800–1000 ms**, raised from the earlier 250–450 ms target. The revision buys deliberate headroom in two places. First, worst-case component stacking (gate-rerank 300 ms + concurrent arms 300 ms + evidence rerank 350 ms ≈ 950 ms) now sits *inside* the ceiling rather than past it — the budget is a real envelope, not a typical-path hope. Second, the evidence-rerank timeout (350 ms launch default, SSM-held) gains explicit room to grow toward 600–700 ms within budget if the instance bake-off (§9) needs it, which materially widens the viable instance set: the cheaper A10G-backed types become realistic candidates rather than long shots, and the bake-off shifts from a feasibility question to a cost-versus-quality question. The end-to-end Slack experience still targets **under 4 s p95**, so the client side's share (interpretation, generation, transport) tightens to roughly 3 seconds — generation still dominates, and the trade is judged worth it for an honest retrieval envelope. These budgets are architectural constraints, not calibration surfaces: they determine why the dual-arm path is concurrent, why explicit-scope skips PAPI, and why agent-side generation latency matters as much as retrieval latency on the Slack path. A sustained breach is an architecture-class event, not a tuning inconvenience.

### 4. Alternatives considered

**Pure UKB-only.** Skywalker as a thin identity-scoping wrapper around UKB. Attractive because it deletes the owned corpus, the ingestion pipeline, and the routing gate — most of the system. Rejected because the program's tightest accuracy commitment is exactly the Top-50 set, and UKB is a black box: Skywalker controls none of its retrieval behavior, chunking, ranking, or indexing. Relying on it for the questions the team is judged on most tightly would put the least control on the hardest part (violates tenet 1).

**Build the general arm ourselves.** Owned retrieval over the full travel, events, and expense documentation instead of UKB. Attractive because it extends the control the FAQ arm enjoys to the whole domain. Rejected once UKB became available as an MCP-accessible knowledge base already built on the substrate the team would otherwise assemble. Ownership is kept only where it buys the most value: the Top-50 path.

**Always-query-both.** Every request hits both arms; the reranker sorts it out. Attractive for its simplicity — no gate, no thresholds, no routing calibration. Not adopted because the FAQ-only short-circuit is real value: when the system is confident a query belongs to the owned space, it answers from the controlled corpus without paying UKB's latency or admitting black-box candidates into the race. The gate exists to permit that short-circuit, not merely to admit FAQ candidates.

**Strict single-arm gate on every miss.** A non-FAQ query skips the FAQ arm entirely. Not adopted: the near-miss region is exactly where semantically adjacent Top-50 material may still win at the reranker even when FAQ-only routing can't be justified. The dual-arm path deliberately keeps the FAQ arm alive there.

**Server-side conversation state inside Skywalker.** Rejected. A deterministic retrieval system is the wrong place for multi-turn meaning; the agents already own conversation history and do it well enough. Adding state would pull Skywalker toward agent behavior and break Decision 7's scaling model.

**No PAPI caching (the original posture — reversed June 10, 2026).** The first draft kept resolution live on every request, favoring clarity over cache machinery. That posture is reversed: a TTL-bounded in-process scope cache is now the active design (Section 02 Decision 6, Section 10 D-44), because re-resolving an identical, slowly changing triple on every turn of a multi-turn conversation is wasted dependency load and repeated failure exposure for no correctness gain. The rejected alternative is now the no-cache design; the cache's staleness window is its named cost and calibration surface.

### 5. Assumptions inherited from upstream

This is the first architectural section, so it inherits program premises rather than technical detail. Later sections treat these as established unless explicitly revisited.

**Domain scope.** Skywalker serves travel, events, and expense documentation — not general Amazon documentation. Every later section assumes this unless a subsection is explicitly about future expansion.

**Owned corpus size — corrected.** The earlier premise ("effectively one document, upper bound on the order of ten") is replaced by ground truth: the production owned set is **~56 CoreX records** carrying the `Skywalker FAQ` content type, enumerated owner-first (CoreX `domainOwner` = the Travel and Events team's Abacus ID) then filtered by content type, with the content-type custom field resolved highest-version-wins from CoreX's versioned metadata keys (`CorpusSchema.java:66-78,205-218`). The corpus remains small enough that tenet 4's simplifications hold — daily polling, full rebuild on change, one fragment per node — but later sections must not size anything to "one document."

**CoreX is the authoring source; UKB is the general source.** Different dependencies with different control surfaces; later sections must not collapse them into one abstract "documentation source." CoreX access is GraphQL over per-stage hosts (`corex-api.beta.corex.pxt.amazon.dev` beta, `corex-api.corex.pxt.amazon.dev` prod) with cross-account STS role assumption and an API-key/external-ID secret in Secrets Manager (`Poller.java:73-80`, `app.ts:68-71,124-127`).

**Both launch clients provide an alias, not a scope tuple.** Slack supplies the user's Slack ID, which the bot resolves to the Amazon alias; QuickSuite supplies a Federate JWT whose `sub` is the Midway login. Both enter through the alias mode and require PAPI resolution. The employee-ID and explicit-scope tools remain in the contract for future callers holding richer identifiers.

**Cost is not a design driver — with one precise boundary.** Not a license for waste: the premise means quality, controllability, and correctness outrank optimization pressure that would push toward a thinner retrieval path, and dual-arm concurrency is accepted under it. The boundary: the premise governs design tradeoffs, not spending authority. The reranker fleet is a standing opex line (roughly $6.6K–$19K/month for the HA pair depending on instance), and standing opex requires explicit approval — which is why the instance bake-off (§9) needs executive sign-off even in a cost-is-not-a-driver program. Those two statements are consistent, and a reviewer noticing that the one build-gating question is a cost call deserves the reconciliation stated rather than implied: cost does not pick the architecture; it does get a signature before the meter starts.

### 6. End-to-end data flow

The runtime flow is end-to-end at the orchestration level; later sections explode individual steps, but the boundary version must be visible now so later detail stays attached to one request story.

A request begins when a caller invokes Skywalker's MCP server through Amazon MCP Gateway: the Slack inline-agent orchestrator (CloudAuth inbound, human alias via TransitiveAuth), the UAT inline-agent orchestrator (CloudAuth inbound, alias via TA, scope triple supplied as tool arguments — Section 05), or QuickSuite's chat-agent runtime (Federate inbound, identity in tool arguments). The gateway authenticates the caller, authorizes it against the Bindle resource `MCPGateway::{skywalker-server-id}`, and converts to CloudAuth on the outbound leg. The request arrives caller-shaped: a query plus identity material on the per-route channel (TA claims on CloudAuth paths, tool arguments on the Federate path). Skywalker's first job is to make it retrieval-shaped.

If the request carries only an alias or employee ID, Skywalker calls PAPI, which returns the scoped identity metadata: country, job level, employee class. If the caller supplied explicit scope, Skywalker skips PAPI and treats the supplied scope as authoritative for this request. Either way, retrieval does not begin until the scoping triple is attached. There is no generic-retrieval fallback path.

With scope resolved, Skywalker runs the routing gate against the in-memory variant set — 50 canonical Top-50 question variants with pre-computed Cohere Embed v4 embeddings (1024-dim, `input_type: search_document`), loaded at service boot from `s3://skywalker-config-{stage}/variants/top_50.json` and held in memory; the service hard-fails at boot on a missing or schema-invalid variant file rather than serving with a broken gate ([API_11]). The live query is embedded with `input_type: search_query` and cosine-scored against all variants in memory. High-band: FAQ-only. Low-band: dual-arm. Ambiguity band: the SageMaker-hosted Rerank v4 cross-encoder check decides, and on timeout the request falls through to dual-arm. The gate's verdict is a routing signal driven by SSM thresholds — it is not the answer and not the final confidence signal.

On the FAQ-only route, the owned arm queries the live evidence index — resolved through the SSM live pointer — with **hybrid retrieval from launch**: BM25 and k-NN over the query embedding, fused through the `skywalker-faq-hybrid` search pipeline (the collection's data-access policy carries the collection-level permissions that pipeline requires, `openSearchStack.ts:79-109`), plus the per-dimension scope filters (employee value OR everybody value on each of country, level, role). Hybrid-at-launch is a decision, not an open item; Section 04 owns the fusion weighting and query-time mechanics. Candidates come back as whole-node fragments and are normalized into the common candidate form.

On the dual-arm route, the FAQ arm runs as above while, concurrently, the UKB arm calls UKB's single `retrieve` tool over its IAM/SigV4 MCP endpoint with the query, `maxResults` (~10 launch default), and the resolved employee's `targetUser` identity so UKB applies its native personalization; the UKB call carries a 300 ms timeout ([API_09]). Both candidate sets are normalized into the common form and merged into one reranker pool.

Cohere Rerank v4.0 Pro then scores the merged pool (≤20 candidates) against the user query as one competition set. From this point the system no longer cares which arm produced a candidate except as route metadata; what matters is which evidence best answers the query for this scoped employee.

After reranking, Skywalker makes the answerability decision against the abstain floor (SSM `/skywalker/runtime/abstain/floor`, launch default 0.30). An empty post-rerank set abstains with `NO_USABLE_EVIDENCE`; a top relevance score below the floor abstains with `EVIDENCE_TOO_WEAK_AFTER_RERANK`; otherwise Skywalker returns the ranked evidence bundle with route metadata, scope snapshot, and source links.

The final step happens outside Skywalker: the caller-side agent turns the structured retrieval output into the user-facing response — conversational context, follow-up handling, surface-specific style. Those behaviors are downstream and must not be confused with the retrieval backend.

The offline flow runs beside this daily: EventBridge fires the Poller at 08:00 UTC; the Poller reads the high-water mark, enumerates CoreX, no-ops on no change; on change it creates the idle index empty, fans the node list out to the Processor in work items of 50, and each node is fetched from CoreX, text-extracted, scope-mapped, embedded through the `us.cohere.embed-v4:0` cross-region inference profile (the bare model ID is not invokable on Bedrock on-demand — pinned empirically, `serviceStack.ts:54-58`), and written as one fragment document. Per-node transient failures retry up to 4 attempts with exponential backoff and full jitter; content-shaped problems (blank text, missing scope) skip without retry (`Processor.java:35-45,123-155`). The flip gate then verifies completeness and queryability and atomically swaps the live pointer. One deterministic progression on each path; the runtime path summarizes as: request enters MCP → scope resolves → gate routes → one arm or both retrieve → candidates normalize → reranker chooses → answer or abstain → agent renders.

### 7. Failure behavior and abstain behavior

Failure behavior at the boundary preserves one contract: answers are scoped correctly or not returned as though they were safe.

**Fail-closed scoping.** If a request arrives without explicit scope and PAPI cannot resolve identity, Skywalker does not continue into retrieval as though a generic answer were acceptable — it is not (tenet 3). The request fails as a tool-execution error, visibly distinct from abstention. Likewise on the CloudAuth paths: a missing or malformed TA token is a protocol-level error, and falling back to argument-supplied identity is deliberately not the launch posture ([API_14]).

**Single-arm fallback when routing permits.** On the dual-arm path, if one arm fails while the other returns candidates, Skywalker continues with the survivor and records it: `route.path: SINGLE_ARM_FALLBACK`, `surviving_arms` naming the live arm. The reranker still gets a coherent, narrower pool. If a FAQ-only request's FAQ arm fails, the safer behavior is attempting the general arm rather than converting an arm outage into a guaranteed no-answer. Routing is a quality policy, not a reason to waste a viable fallback. UKB-arm failure handling is graded by cause ([API_09]): timeout/5xx fails the arm; an empty result set is a retrieval miss, not a fault; malformed items are discarded individually while good ones survive; 403 is an IAM misconfiguration that alerts; 429 aborts the arm for the request.

**Reranker failure is a flagged fallback, not total failure.** If the evidence reranker is unavailable (after its single 50 ms-backoff retry), Skywalker returns the best available normalized set in retrieval order with `reranker_state: RERANKER_FAILURE_FALLBACK`. Section 07 owns how much of that fallback set passes through; the boundary contract rejects the false dichotomy of "reranked perfectly" versus "nothing."

**Ingestion fails toward staleness, never toward partial truth.** Every offline failure mode — CoreX outage, embedding throttle, AOSS write failure, partial completion, failed read-back verification — leaves the previous complete corpus serving and the high-water mark unadvanced. The system serves day-old complete evidence rather than fresh fragmentary evidence, by construction (`RebuildCoordinator.java:127-155`). The operational signal is a rebuild that keeps declining to promote — Section 03 owns that alarm.

**Abstention is not failure.** A request may execute perfectly and still produce no answerable evidence; that is the system doing the correct thing when it cannot defend a grounded answer (tenet 2). The contract therefore separates three states cleanly ([API_01]): protocol-level JSON-RPC errors (unknown tool, malformed arguments, auth/TA failures), tool-execution errors (`isError: true` — PAPI unresolvable, double-arm failure with no fallback), and backend abstention, which is a **successful** result with `result_kind: ABSTAIN` and a machine-readable reason. In one case the path broke; in the other the path ran and declined to overclaim.

The non-goals that matter here keep later sections from smearing responsibility onto the wrong component: Skywalker does not own conversational memory, prompt engineering for either surface, final answer wording, the evaluation harness, or human-in-the-loop runtime review. SME review after launch is in scope for the program and remains an important practice; it is not part of Skywalker's runtime boundary.

### 8. Calibration surfaces

The series distinguishes three classes honestly: fixed decisions (§3 — changing one invalidates other subsystems), calibration-active settings (this section — intentionally empirical, tuned with real data), and open questions (§9 — deliberately undecided, with the evidence that would decide them). Calibration surfaces and open questions are deliberately separate sections: a calibration surface has a decided mechanism and an undecided value; an open question has an undecided mechanism.

**Calibration surface one — routing-gate thresholds.** `/skywalker/runtime/gate/cosine_high_threshold` (0.80), `cosine_low_threshold` (0.30), `rerank_floor` (0.50), and `rerank_timeout_ms` (300). Launch defaults, to be derived from real traffic and human review. They protect the FAQ-only route from over-eagerness; they are not the answerability threshold. Re-litigation signal: repeated evidence that FAQ-only-routed traffic would have been better served dual-arm, or the opposite.

**Calibration surface two — the abstain floor.** `/skywalker/runtime/abstain/floor` (0.30). A different question from routing: not "does this query belong to the Top-50 set" but "is the surviving evidence strong enough to return." Re-litigation signals: obviously weak evidence returned as answerable, or good requests pushed into abstain.

**Calibration surface three — candidate volume.** Evidence-reranker input capped at 20; UKB `maxResults` ~10; FAQ-arm k and over-retrieval factor owned by Section 04. The fixed fact is that both arms compete in one pool on the dual-arm path; the budgets feeding that pool are tuning.

**Calibration surface four — direct-scope trust.** The explicit-scope tool exists and is fixed. Whether Skywalker always trusts caller-supplied scope or later adds validation/audit hooks remains open; it affects contract behavior under real usage rather than any threshold.

**Calibration surface five — ingestion schedule and alarm thresholds.** The 08:00 UTC daily cron is a launch default to be tuned once the CoreX corpus's actual update timing is known; the deliberately loose monitoring thresholds tighten once a run-history baseline exists. Both are recorded in code comments as provisional, which is the correct epistemic posture.

Re-litigation discipline applies to all calibration surfaces: values move on clear behavioral evidence — threshold drift, routed-path/evidence-quality disagreement, persistent abstain misfires — not on vague discomfort.

### 9. Open questions

Most open questions across this series share one precondition, stated in full in Section 10 §9: they are only answerable against real user data at meaningful volume — a few hundred actual users, arriving with the September production launch. Until then, launch postures stand, and pre-launch pressure to move them resolves as a recorded non-change.

One question in this section gates a build decision; the rest do not. Each non-gating question carries an explicit disclaimer so a reader can tell at a glance what blocks work and what merely awaits evidence.

**Reranker instance selection — pending executive approval. This is the gating question.** All Skywalker reranking runs Cohere Rerank v4 on SageMaker (Decision 3), and the Marketplace package supports three real-time instance types — `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge` (the listing's recommended type) — at a flat $3.50/host-hour software fee, so the instance hour is the entire cost lever: roughly $6.6K/month HA-pair all-in on g5.xlarge versus ~$19K/month on p5.4xlarge. Executive approval is required to run the cost-versus-latency bake-off across the three. The same benchmark answers the latency question that was previously tracked separately: what timeout the chosen type can hold on Skywalker's real ~20K-token payloads. Public p50 estimates (180–300 ms) are H100-class; the A10G-backed g5 types will be slower and no per-instance benchmark exists, so the fit of the cheaper types is strictly empirical — but the revised 800–1000 ms pipeline budget (Decision 9) gives the rerank timeout room to grow toward 600–700 ms, which makes the cheaper types realistic candidates and turns the bake-off into primarily a cost-versus-quality decision rather than a feasibility test. Outcome binds Decision 5's endpoint build-out, Decision 9's timeout allocation, and the gate's endpoint-sharing call. If even p5.4xlarge cannot hold a budget-compatible timeout at production volume, the candidate pool shrinks or Decision 9 re-litigates.

**Reranker region topology — answered (researched June 10, 2026).** The question was whether rerank hosts must be deployed across regions to serve a globally distributed user base with minimal latency. The answer is no, on three findings. First, the rerank hop is service-to-service — query service to SageMaker endpoint — inside the request after it reaches the backend; user geography affects the front legs (user → Slack/QuickSuite → orchestrator → gateway), which no amount of rerank replication improves. Second, SageMaker endpoints are strictly regional; there is no global or anycast endpoint product, so "minimal latency without more hosts" does not exist as an option — every latency-motivated region adds at least one host (realistically an HA pair) at full instance-plus-fee cost. Third, per-instance throughput (~12–20 QPS) exceeds internal-assistant traffic by a wide margin, so the fleet is sized by availability, not capacity. **Resolution: one HA pair, co-regional with the prod query service. No global fleet.**

**Prod region consistency** *(disclaimer: does not gate the reranker decision or any current build; must close before the prod stage is laid down).* The region research surfaced an internal inconsistency that matters more than the question it answered: `ATESkywalkerQueryCDK` deploys alpha and beta to us-west-2 but gamma to us-east-1, with the Allegiance prod placeholder also us-east-1 — while UKB's v1 prod endpoint is us-west-2-only ([API_09]) and the implemented AOSS/ingestion stack runs in us-west-2. A us-east-1 prod query service would pay ~60–75 ms cross-region on every UKB call (against a 300 ms timeout) and every evidence-index read — eroding exactly the headroom a cheaper rerank instance needs. One region must be chosen for the entire prod data plane (query service, rerank endpoints, AOSS prod collection), and UKB's v1 constraint makes us-west-2 the gravity well. [API_10]'s us-east-1 endpoint placement is consistent with the current gamma topology but inherits the same tension. Deciding evidence: confirmation of UKB's region roadmap and the prod account/region assignment.

**Abstain message differentiation** *(disclaimer: staging-quality question; does not gate any build or the contract).* Whether QuickSuite and Slack produce meaningfully different user-facing messages from the two abstain reasons is tracked as Section 10 C-02. Earlier drafts carried four abstain reasons; two were dropped because they required semantic judgment Skywalker cannot provide or conflated arm identity with answerability.

### Closing position

Skywalker is the identity-aware retrieval backend behind MCP. It resolves scope before retrieval, runs a controlled owned arm beside a UKB general arm, short-circuits confidently Top-50-shaped requests to the owned arm, runs everything else through both arms concurrently, normalizes candidates into one form, reranks them as one set, and returns structured evidence or a structured abstention for downstream agents to render. Its owned corpus is rebuilt all-or-nothing from CoreX daily and published by an atomic pointer flip, so the runtime only ever sees a complete corpus.

That is the system boundary for the rest of the series. Later sections elaborate it. They do not drift from it silently.
