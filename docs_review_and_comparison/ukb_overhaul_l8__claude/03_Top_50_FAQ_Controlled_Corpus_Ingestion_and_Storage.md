## Section 03. Top 50 FAQ Controlled Corpus, Ingestion, and Storage

Section 02 fixed the request boundary: by the time a request leaves it, Skywalker knows who is asking, what country, level, and employee class shape the answer, and whether the request is valid for retrieval at all. This section moves to the first owned data plane behind that path: the controlled Top 50 FAQ arm.

That arm exists for one reason. The Top 50 questions are the highest-value question set in the system and the only place where the project is explicitly trying to hit a named accuracy target. UKB covers breadth, but UKB is a black box — it controls its own ingestion, indexing, retrieval, and ranking. Acceptable for the general arm; not acceptable as the only path for the small question set where the business value concentrates (tenet 1). The result is deliberately asymmetric: the general arm buys leverage from UKB, the owned arm buys control by owning a small corpus end to end.

Small does not mean trivial. A ~56-record corpus can still poison trust if change handling is sloppy, if documents publish inconsistently, or if a rebuild exposes half-finished state. This section defines the owned corpus as a real subsystem — and unlike most of this series, it is a subsystem that already exists as deployed code. The grounding here is the implementation itself: the `ATESkywalkerIngest` Brazil package (two Java 17 Lambdas) and its `ATESkywalkerIngestCDK` infrastructure. Where this section's earlier draft or the API research series disagrees with that code, the code wins and the stale source is flagged.

One point made plainly because it keeps getting blurred: the controlled FAQ arm is not a cache of prewritten answers. It is a retrieval-backed arm whose evidence comes from an owned corpus built from CoreX source material, embedded, stored, and retrieved under Skywalker's control.

### 1. The subsystem boundary

This section owns the complete controlled data plane behind the FAQ arm's evidence corpus: the authoritative source relationship with CoreX, the daily polling loop, the change-detection rule, the per-node processing path that turns CoreX nodes into retrieval documents, the publication mechanics that decide when a freshly built corpus becomes live, and the AOSS evidence surface (with its companion hybrid search pipeline) that query-time retrieval consumes.

This section does not own the routing gate — query-to-variant similarity is a runtime concern in Section 04, and the variant set itself is a static S3 artifact maintained by hand ([API_11]), entirely outside this pipeline. No variant index, no variant pipeline; variant maintenance is a manual control-plane activity. This section also does not own query-time thresholds, the reranker, the UKB arm, agent behavior, PAPI resolution (fixed in Section 02), or user-facing abstain behavior. Its job is structural: ensure the FAQ arm has a valid, current, queryable evidence corpus. A bug about stale or missing FAQ evidence files here; a bug about the wrong evidence being *selected* from a correct corpus files against Sections 04 or 07; a bug about wrong scope labels on the source content files against the CoreX content team.

### 2. Inputs, outputs, and contracts

**The CoreX source contract.** The authoritative input is the CoreX corpus owned by the Travel and Events team (CoreX `domainOwner` = `amzn1.abacus.team.looo53floubmzytmswva`). Enumeration runs against CoreX's `searchContent` GraphQL endpoint with a domain-owner filter, paginating in pages of 50 (CoreX's cap), and accepts a row only when its `nodeId` is a well-formed UUID — a deliberate security guard: CoreX node IDs are always UUIDs, so a non-UUID identifier in a response is treated as untrusted and rejected rather than propagated into downstream fetch and index calls (`CoreXSearchEnumerator.java:48-91`). Each enumerated node must also carry a non-blank `lastModifiedDate`, because that date feeds the change marker.

Per-node content arrives through CoreX's `getContentNodes` GraphQL endpoint. The production body shape is **PlateJS** — a Slate-derived rich-text JSON, proven against live CoreX content: the node's `content` field is a JSON string shaped `{"system_rte_v2": {"type": "RTE_V2", "content": "<JSON-string PlateJS array>"}}`, with text in leaf nodes across observed block types (`h1`, `h2`, `h3`, `p`, `ul`, `li`, `a`, `toc`, `dam-file`). The extractor walks the tree depth-first, collects every text leaf in document order, and degrades gracefully on unknown block types. In-text links are preserved as `"link text (https://url)"` — load-bearing, because FAQ answers frequently *are* a pointer to a policy page, so the URL is embedded into the vector and matchable by BM25 rather than discarded as formatting (`CoreXTextExtractor.java`). Access crosses accounts: STS role assumption against a CoreX-issued role, with an API-key/external-ID pair in Secrets Manager (`ATESkywalkerIngest/{stage}/corex`), over per-stage hosts (`corex-api.beta.corex.pxt.amazon.dev` beta, `corex-api.corex.pxt.amazon.dev` prod).

The corpus identification model is owner-first, then sub-type: the domain owner bounds enumeration, and the FAQ sub-type is the CoreX custom field `content-type` carrying the value `"Skywalker FAQ"`. CoreX custom metadata keys are versioned by numeric suffix (`content-type-16`, `applicable-policy-0`); the pipeline resolves by base name, highest version wins, on the working assumption that the content team does not ship breaking changes under a higher number (`CorpusSchema.java:166-218`). The resolved content type is stamped on every document as a first-class indexed `content_type` keyword field, so the corpus membership test is filterable at query time. The implemented enumeration currently filters by owner only; whether the content-type narrowing is enforced at enumeration or at query time is an open execution detail (§9).

**The AOSS evidence surface.** The published corpus lives in an Amazon OpenSearch Serverless **vector search** collection named `skywalker-faq-{stage}` (`openSearchStack.ts:63-68`), encrypted with an AWS-owned key, network-policied as public-endpoint-with-IAM-auth (the standard posture for internal AOSS collections — an earlier draft of this section claimed a private VPC endpoint; the deployed policy is `AllowFromPublic: true` with IAM as the enforced boundary, `openSearchStack.ts:47-60`), with standby replicas enabled in prod.

Publication uses **two physical indices plus an SSM live pointer**, not index aliases — AOSS Serverless does not support aliases, so the alias-swap model in [API_06] and [API_07] is stale and superseded. The Poller owns `faq_evidence_a` and `faq_evidence_b`; the SSM parameter `/skywalker/ingestion/faq_evidence/live_index` names the live one; a rebuild (re)creates the idle index empty and builds into it while the live index keeps serving; promotion is an atomic flip of the pointer (`OpenSearchIndexManager.java:28-112`, `serviceStack.ts:167-177`). The query service reads the pointer to resolve which index to query and reaches the collection through a read-only cross-account role (`ATESkywalkerIngest-OpenSearchQueryRole-{stage}`, limited to `aoss:DescribeCollectionItems`, `DescribeIndex`, `ReadDocument` — the query service structurally cannot write the corpus, `openSearchStack.ts:193-243`).

Each document is **one whole-node fragment** — one CoreX node, one document, no chunking. The mapping (`OpenSearchIndexManager.java:177-197`): `embedding` (`knn_vector`, dimension 1024, FAISS HNSW, `cosinesimil`, `m: 24`, `ef_construction: 128`); `fragment_id` and `source_id` (keywords, both currently the CoreX nodeId); `text` (analyzed `text` field — the BM25 leg of hybrid retrieval scores against it); `source_url` and `policy_links` (unindexed keywords, carried for the response envelope); `country`, `level`, `role` (indexed keyword **arrays** — the scope pre-filter surface, holding the real corpus vocabulary including the everybody-values); `corpus_version` (keyword — the snapshot marker of the run that produced the document); `followup_fragment_ids` (unindexed keyword array, **reserved and written empty** — the surface a future linked-evidence feature would use); `content_type` (indexed keyword, the promoted corpus-membership filter); and `source_metadata` (`flat_object` holding everything else about the node — all top-level API fields except the body, plus the parsed custom-metadata blob version-resolved — so provenance is never lost and new filters can be added later without re-ingesting, and the numbered custom keys never explode the mapping, `MetadataAssembler.java`). The HNSW build parameters and the mapping are fixed at index creation; the rebuild path absorbs this naturally, since every rebuild recreates the idle index with the current mapping.

There is no parent/child structure, no `chunk_id`/`parent_id`/`child_order`, no sibling reconstruction, and no linked-Q&A chain materialization. The earlier draft of this section specified a two-step chunker (hierarchical then 1000-token semantic child-splitting), children-only storage with query-time parent reconstruction, and a depth-2 cross-Q&A link chain with cycle-breaking and unresolved-edge tolerances. None of that machinery exists, and at this corpus's scale none of it should (tenet 4): FAQ nodes are answer-sized, the embedding model's context window (128K tokens) dwarfs any node, and 20 whole-node candidates fit the reranker's payload budget. The chunking and linking designs are preserved in git history and re-enter, if ever, as Section 10-recorded design changes with concrete triggers: chunking returns if node length outgrows embedding or reranker budgets; linking returns as a contract revision against the reserved `followup_fragment_ids` field (Section 02 carries the same flag on the envelope side).

**Scope mapping at ingest.** Each node's scope is mapped from real CoreX fields (`ScopeMapper.java`): `country` ← top-level `geography` (values like `"Global"` or country labels, tolerating both array and comma-joined forms); `level` ← metadata `system_job-level` (`"All Job Levels"` or `L0`–`L12`); `role` ← metadata `system_employee-class` (`"All Employee Classes"` or specific class labels). Values are cleaned — trimmed, de-duplicated, the AEM authoring placeholder `"No Value (Blank in AEM)"` stripped — and there is no synthetic catch-all token: "applies to everybody" is a real value the data carries. **Unscoped evidence is never published**: a node missing every real value on any dimension is skipped and logged rather than backfilled (`FragmentProcessor.java:79-92`). Measured across the production set, all three dimensions are present on ~100% of nodes, so this rule is a malformed-node safety net, not a content filter.

**The embedding contract.** Every document is embedded through Bedrock using the **`us.cohere.embed-v4:0` cross-region inference profile** — not the bare `cohere.embed-v4:0` model ID, which Bedrock rejects with HTTP 400 ("on-demand throughput isn't supported … use an inference profile"); this was pinned empirically and the IAM policy therefore grants `bedrock:InvokeModel` on both the account-scoped inference-profile ARN and the underlying foundation-model ARNs in every US region the profile can reach (`serviceStack.ts:53-58,179-196`). Output dimension 1024, matching the index. Ingestion sets `input_type: search_document` and `truncate: RIGHT` (`BedrockEmbeddingClient.java:64-76`); the runtime query path uses the same model with `input_type: search_query`; the variant set is pre-embedded with `search_document` ([API_11]). The input-type asymmetry is load-bearing — Cohere Embed v4 produces distinct embeddings for things-to-be-searched versus the-search-itself, and mismatching silently hurts recall. One embedding model across evidence, query, and variants means one vector space and no drift; a model change is an architecture-class coordinated event (rebuild the index, regenerate the variant artifact, change the runtime), never a configuration tweak.

**The SSM state surface.** Exactly two parameters persist for this subsystem, both under `/skywalker/ingestion/faq_evidence/`: `last_snapshot_marker`, the high-water mark, holding the single most-recent `lastModifiedDate` observed across the corpus (ISO-8601 strings compare lexically as chronologically, so a string max suffices — `CoreXSearchEnumerator.java:104-114`); and `live_index`, the pointer naming the live physical index. The marker is the only state that cannot be reconstructed from the collection itself; the pointer exists because AOSS offers no alias to read. [API_07] documents the marker correctly but describes alias-based publication and omits the pointer — flagged stale; its `/skywalker/runtime/*` control-plane parameter inventory remains valid for Sections 04 and 07. First-run behavior is defined: a missing marker means "no prior snapshot, build from scratch"; a missing pointer means "no live index yet, build into `_a`."

**The hybrid search pipeline.** Ingestion ensures, best-effort on every rebuild, a collection-level search pipeline named `skywalker-faq-hybrid` carrying a single `normalization-processor` (`min_max` normalization, `arithmetic_mean` combination) for fusing the BM25 and k-NN legs of the runtime's hybrid query (`OpenSearchIndexManager.java:154-208`). Best-effort is deliberate: the pipeline is search-time machinery, not write-path machinery, so its creation failing must not block a corpus publish — the failure logs loudly (including the data-access-policy fix if it is a 403) and ingestion continues. AOSS has no per-index default-pipeline binding in this design; the runtime references the pipeline at query time (Section 04 owns the query shape). Hybrid retrieval ships at launch (Section 01 §6).

**The runtime read surface.** The contract handed to Section 04: read the live pointer, query that physical index with the hybrid query through `skywalker-faq-hybrid`, pre-filter on `country`, `level`, and `role` with the `(employee value OR everybody value)` semantics Section 02 fixed, and trust that whatever the pointer names is complete, coherent, scope-filterable, and searchable on both legs. This section owns that guarantee; Sections 04 and 07 own the query shapes.

### 3. Fixed decisions

**Decision 1 — The Top 50 arm is owned because it needs control.** A targeted response to the project's asymmetry, not a preference for building infrastructure: the highest-value question set is exactly where black-box behavior is least acceptable. Binds the existence of this entire subsystem. Reopens only with Section 01 Decision 1.

**Decision 2 — CoreX is the authoritative evidence source.** The corpus is built from the organization's documentation source of truth, not a hand-authored side table, keeping the controlled arm tied to actual documentation reality. Binds the source contract, the credentials chain, and the content team's authoring responsibilities (scope fields, content type). Reopens only if the content program moves off CoreX.

**Decision 3 — Schedule-driven daily ingestion.** An EventBridge cron (08:00 UTC launch default, absent on personal stacks where the Poller is invoked by hand) triggers the run (`app.ts:52-54`, `serviceStack.ts:198-207`). No event-driven fabric for ~56 documents (tenet 4). Binds the freshness contract: the corpus is at most one day stale plus rebuild time. Reopens if the source starts changing fast enough that one-day staleness becomes unacceptable.

**Decision 4 — All-or-nothing rebuild with a verified flip gate.** On change, the system rebuilds the entire corpus from the ground up into the idle index and promotes only on 100% completion: every enumerated node successfully processed after per-node retries (no skips), and the full expected document count verified queryable by read-back polling before the pointer flips (`RebuildCoordinator.java:127-167`). Read-back tolerates AOSS's refresh behavior — ~10 s steady-state, with a measured ~64 s cold first refresh on freshly created indices — by polling up to 50 × 2 s (~100 s) before declaring a build unverifiable. A partial or unverifiable build never promotes; the previous complete corpus keeps serving and the marker does not advance, so a systemic failure (auth, throttle, network) cannot ship a near-empty index over a complete one. Binds the publication semantics every downstream consumer relies on and §7's failure posture. Reopens if corpus growth pushes rebuild duration toward the Poller's 15-minute Lambda ceiling — the trigger is duration, not document count.

**Decision 5 — Whole-node fragments, no chunking.** One CoreX node becomes one document carrying the node's full extracted text. The two-step chunker design is dead at this scale (rationale and reopen trigger in §2). Binds the index schema, the embedding payload, the reranker's per-candidate size expectations (Section 07), and the candidate `text` contract (Section 02). Reopens on the concrete trigger named in §2.

**Decision 6 — The arm remains retrieval-backed.** Routing toward this arm does not collapse the path into a lookup table; evidence still comes from retrieval over the corpus and still competes in the common reranking surface. Binds Sections 04 and 07. Reopens never under the current design — the project chose an owned retrieval corpus, not an answer table, and the distinction does not dissolve because the corpus is small.

**Decision 7 — Storage topology: one collection, two physical indices, one pointer.** No aliases (platform constraint), no versioned index chain with garbage collection (the idle index is recreated in place, so there are no orphans to sweep), no secondary storage surface, no variant index. Binds the query service's index-resolution behavior and the rollback model. Reopens only if the platform constraint changes *and* a positive reason to prefer aliases emerges.

**Decision 8 — Embedding discipline.** One model (`us.cohere.embed-v4:0`, 1024-dim) across evidence, query, and variants; input-type discipline (`search_document` at ingest and variant authoring, `search_query` live); model change is architecture-class. Binds ingestion, the runtime, and the variant artifact jointly. Reopens only by deliberate migration with coordinated cutover.

**Decision 9 — Never publish unscoped evidence.** The per-dimension skip rule in §2, enforced in code at the only place documents are created. Binds the corpus's coverage guarantees and gives Section 02's filter contract a corpus it can trust. Reopens never; weakening it reintroduces exactly the silent-genericity failure tenet 3 exists to prevent.

**Decision 10 — Preserve everything as metadata.** The embedding input is solo body text; everything else about the node rides in `source_metadata` (flat_object), with `content_type` promoted to an indexed field. Nothing is lost at ingest, and future filters need no re-ingestion. Binds the schema and the audit story (`corpus_version` plus full provenance per document). Reopens never — the cost is bytes, the benefit is never re-deriving the corpus by hand.

### 4. Alternatives considered

**UKB alone for the Top 50 set.** Rejected for the reason the arm exists: UKB's behavior is not controllable enough for the one question set where the system is explicitly measured (tenet 1).

**Event-driven or webhook-first change triggers from CoreX.** Optimizes detection latency the corpus does not need. Daily polling is the correct launch posture for this size and ownership model; the high-water-mark comparison makes the no-change case a cheap no-op.

**Incremental in-place mutation.** Right when full republish becomes expensive; wrong here. With ~56 documents, incremental mutation adds state-tracking complexity precisely where the system least needs it, and the all-or-nothing flip gate would be impossible to state as cleanly.

**The two-step chunking pipeline.** Considered, designed in an earlier draft, and removed against implementation reality: nodes are answer-sized, and chunking machinery without a size problem is pure liability. It remains the named fallback if node sizes outgrow budgets.

**A variant-ingestion pipeline.** Rejected; 50 short phrasings on a human change cadence are a static S3 artifact, not a pipeline product ([API_11]).

**Collapsing to a deterministic answer table after routing.** Rejected — it changes what the arm is (Decision 6).

**Alias-based atomic publication.** The earlier design; structurally unavailable on AOSS Serverless. The two-index pointer flip provides the same atomicity property through SSM. This is the one alternative rejected by platform constraint rather than judgment, and it is worth recording as such: if the constraint lifts, the pointer model still works and there is no pressure to migrate.

### 5. Assumptions inherited from upstream

From Section 01: the system boundary, the two-arm shape, the ~56-record corpus reality (Section 01 §5 — the earlier one-to-ten-document premise is dead), and the publication mechanics summarized in Decision 6 there. From Section 02: scope vocabulary and the never-unscoped contract this corpus must uphold for the filter semantics to work. Domain scope remains travel, events, and expense. Two assumptions are this section's own: CoreX's PlateJS body shape and versioned-custom-key conventions hold (the extractor degrades gracefully on new block types, and the highest-version-wins rule assumes non-breaking renumbering — if the content team ships a breaking change under a higher version number, validation at the value level is the defense); and the content team continues to populate all three scope dimensions on every node (the skip rule catches lapses, and a sustained lapse surfaces as shrinking corpus coverage, which is a content-program conversation, not an engineering fix).

### 6. End-to-end data flow

The flow is implemented end to end; this is a description of running code, not a plan.

EventBridge fires the Poller daily at 08:00 UTC. The Poller (a thin adapter — all logic lives in `RebuildCoordinator`) reads the high-water mark from SSM, then enumerates the corpus from CoreX: `searchContent` filtered by domain owner, paginated by 50, each row UUID-validated and date-validated. An empty enumeration is a guard condition, not a signal: CoreX returning zero nodes leaves the live corpus intact and the run exits no-op, because "the corpus vanished" is far more likely an upstream fault than a true mass-depublication (`RebuildCoordinator.java:88-92`). The run computes the current marker — the lexical max `lastModifiedDate` — and exits `NO_OP` when it matches the stored marker. Any publish, retraction, or modification advances some node's date and therefore the max, which triggers a rebuild.

On change, the coordinator calls `beginRebuild()`: the idle index (the one the live pointer does not name; `_a` on first run) is dropped and recreated empty with the current FAISS mapping — so removed content disappears and mapping changes deploy naturally — and the hybrid pipeline is ensured best-effort. The node list then fans out to the Processor Lambda in work items of 50 nodes via synchronous parallel invocation; the Poller blocks on the fan-out, which is why it runs at the 15-minute Lambda ceiling with 512 MB, and the Processor at 1024 MB (`serviceStack.ts:60-104`).

For each node, the Processor runs fetch → extract → scope-map → embed → write, with two distinct non-success shapes. Content-shaped problems — blank extracted text, a missing scope dimension — return zero fragments without retry, because retrying cannot change a data-quality outcome; they surface as skips. Infrastructure faults — CoreX fetch errors, Bedrock throttling, AOSS write failures (throttling surfaces as exceptions because the embed and write clients use a hand-rolled SigV4 HTTP path, not the SDK retry layer) — retry up to 4 attempts with exponential backoff and full jitter (base 500 ms, cap 8 s; full jitter de-synchronizes the many work items that throttle together), and only after exhaustion does the node land in the skip list (`Processor.java:35-155`). The write itself POSTs to `/_doc` with no client-supplied document ID — AOSS rejects explicit IDs on create — so idempotency is provided by the recreate-empty-then-build-everything model rather than by per-document upserts (`OpenSearchFragmentWriter.java:79-84`).

Then the flip gate. The coordinator refuses to promote unless every node succeeded: any skip, or a fragment count below the expected node count, leaves the run unpromoted with the live index unchanged and the marker unadvanced. Completeness confirmed at the work layer, the coordinator then verifies queryability: it polls the target's `_count` until the full expected count is searchable, tolerating the AOSS refresh interval, up to the ~100-second budget. Only then does it promote — one SSM write flipping `live_index` to the freshly built target (the manager refuses to promote any index name other than `_a` or `_b`) — and only after a successful promote does it write the new marker. The ordering is deliberate: a failure between promote and marker-write produces at worst a redundant rebuild next run (benign waste), whereas the opposite order could advance the remembered marker while the live surface did not move — the actual hazard, and the same reasoning the earlier alias-based design used, preserved across the mechanics change.

Rollback is an operator action with the same shape: flip `live_index` back to the previous physical index — which still holds the prior complete build, untouched by the failed or unwanted publish — and reset the marker to the value recorded in the run's structured logs. Skipping the marker reset is recoverable (one redundant rebuild); skipping the pointer flip is not.

There is no garbage-collection step (recreate-in-place leaves no orphans) and no parallel flow for variants (not this pipeline's product). A development-only path exists and is worth naming because it is a clean seam: the Processor accepts a `source: exemplar` work-item marker that swaps the CoreX fetcher for bundled real-production exemplar records, letting the full extract → embed → write path be proven without CoreX write access; the Poller never sets it, so scheduled production runs are always CoreX (`Processor.java:176-205`).

### 7. Failure behavior and abstain behavior

The governing principle is **publication conservatism: every failure mode resolves toward serving yesterday's complete corpus, never toward serving today's partial one.**

CoreX unreachable, or returning errors past per-node retries → affected nodes skip, the flip gate refuses promotion, the live corpus and marker stand, the next daily run retries from prior truth. CoreX returning zero nodes → the empty-enumeration guard exits no-op without touching anything. Bedrock or AOSS throttling → per-node retries with full jitter absorb transient bursts; sustained throttling exhausts retries, lands nodes in the skip list, and the flip gate does its job. Read-back verification failing its budget → no promotion, even though every write reported success, because "written" and "queryable" are different claims and the gate verifies the one the runtime actually depends on. SSM marker write failing after a successful promote → benign divergence, one redundant rebuild, convergence next run. The Poller dying mid-run → the idle index holds a partial build that the next run's `beginRebuild()` drops and recreates; the live surface never saw it.

Two failure modes deserve explicit operator framing because the system's correct behavior can mask them. First, **persistent decline-to-promote is silent staleness**: every individual run behaves correctly, but the corpus ages while CoreX moves on. The Poller's structured run outcome (`kind`, `fragmentsIndexed`, `nodesSkipped`) is the detection signal, and a sustained pattern of enumerated-changes-but-no-promote is an incident even though nothing is "down." Second, **a content-side scope lapse shrinks the corpus invisibly**: nodes skipped for missing scope are correct per-node behavior (Decision 9), but a batch of them after a content-model change means the published corpus quietly lost coverage; the skip logs carry the per-node dimension detail needed to route that conversation to the content team.

This section has no user-facing abstain behavior — it answers no users; it publishes or declines to publish. The nearest equivalent is the flip gate's refusal to make incomplete state live, and the distinction Section 02 draws between boundary failure and retrieval abstain has its offline analogue here: a stale corpus is not a broken pipeline, and the signals that distinguish them (run outcomes versus Lambda errors) are different on purpose.

Non-goals: this section does not own the UKB arm, routing policy, reranking, answer generation, conversational context, an authored-answer cache, real-time source propagation, incremental update machinery, the variant set, or the routing gate. Boundary decisions, not omissions.

### 8. Calibration surfaces

**Surface one — poll cadence.** Daily at 08:00 UTC is a launch default, recorded as provisional in the code; tune per stage once the CoreX corpus's real update timing is known. Re-litigate the *cadence*; the schedule-driven *model* is Decision 3.

**Surface two — work-item size and retry budget.** 50 nodes per work item; 4 attempts, 500 ms base, 8 s cap. Defaults sized for the current corpus; revisit with measured throttle behavior at real volume. Changing them is configuration, not architecture.

**Surface three — read-back budget.** 50 attempts × 2 s against AOSS's measured refresh behavior. If AOSS's cold-refresh characteristics shift, this budget moves with evidence; too short fails good builds, too long stretches the daily run for nothing.

**Surface four — HNSW build parameters.** `m: 24`, `ef_construction: 128` are launch defaults where recall matters more than build time on a small corpus; the per-query `ef_search` belongs to Section 04. Rebuild-class to change — which the daily-rebuild model makes cheap, an underrated property of Decision 4.

**Surface five — the BM25 analyzer.** The `text` field uses the standard analyzer. Re-litigate if review shows domain terms (policy codes, vendor names, currency abbreviations) tokenizing in ways that hurt the lexical leg on real queries; also rebuild-class, also cheap here.

### 9. Open questions

Most open questions across this series share one precondition, stated in full in Section 10 §9: they are only answerable against real user data at meaningful volume — a few hundred actual users, arriving with the September production launch. Until then, launch postures stand, and pre-launch pressure to move them resolves as a recorded non-change.

**Content-type enforcement point** *(disclaimer: does not gate the build; both candidate mechanisms exist in the schema today).* Enumeration filters by domain owner only; the `"Skywalker FAQ"` content-type test exists as schema knowledge and as the indexed `content_type` field. If the domain owner ever publishes non-FAQ content, the corpus ingests it. The closing decision is whether to narrow enumeration with a content-type filter or to add a `content_type` term to the runtime query (Section 04) — or both, defense in depth. Deciding evidence: whether the Travel and Events CoreX space carries non-FAQ content in practice.

**`source_url` and `policy_links` population — gates the June 30 UAT deliverable.** No disclaimer on this one; an earlier framing called it "data-wiring, does not gate the build," and that framing blurred exactly the distinction Section 10 warns against: it does not gate *ingestion builds*, but it gates the deliverable's fixed citation requirement (Section 05 Decision 8, Section 10 D-17), and a fixed requirement with an unwired data source is a gated deliverable. The implemented writer stamps both fields empty (`FragmentProcessor.java:99-126`). Two distinct things must be established, and only the first is mechanical. First, the wiring: a canonical per-node URL from CoreX into `source_url`, and the `applicable-policy` custom field into `policy_links`. Second — **the unverified assumption** — that the source data actually contains renderable URLs: if CoreX's `applicable-policy` values are policy codes or node references rather than links, "citations with policy links" is not achievable from the current corpus without a code-to-URL resolution table that does not exist, which is net-new design work, not wiring. Deciding evidence: inspect the production export and a live `getContentNode` response for both fields **now**, not at integration time. This is the open item most likely to slip June 30, and it is named as such.

**The deploy-time bootstrap index** *(disclaimer: cleanup item).* The CDK stack provisions an index named `faq_evidence` (no suffix) via custom resource at deploy time with the same mapping (`openSearchStack.ts:146-190`); the runtime data plane operates exclusively on `_a`/`_b`. The unsuffixed index appears vestigial — confirm and remove, or document a purpose.

**Prod-stage build-out** *(disclaimer: does not gate current work; alpha and beta exist).* The pipeline defines alpha and beta only; prod topology (account, region — entangled with the prod-region consistency question in Section 01 §9 — reserved concurrency, tightened thresholds on `isProd`) is reserved in code as TODO.

**CoreX custom-field evolution** *(disclaimer: monitored assumption, not a decision).* Highest-version-wins assumes non-breaking renumbering; value-level validation (e.g., content-type only counts when it carries the expected label) is the present defense. If the content team's authoring conventions change materially, the `CorpusSchema` fixture set needs a deliberate update.

### Closing position

The owned Top 50 FAQ arm is small on purpose and important on purpose, and it is the one subsystem in this series that exists as deployed code rather than design intent. CoreX is the evidence source, enumerated owner-first with UUID-validated rows and dated by a single high-water mark. Daily polling triggers an all-or-nothing rebuild into the idle one of two physical indices, with one whole-node fragment per CoreX node, scope mapped from real corpus vocabulary with unscoped evidence refused, everything else preserved as metadata, embeddings produced under one model with strict input-type discipline, and promotion gated on verified completeness before an atomic SSM pointer flip. Failures of every class resolve toward yesterday's complete corpus.
