## Section 01. System Boundary and Request Lifecycle

## 1. Tenets

We prefer scoped correctness over broad answering when they conflict. Skywalker exists to return evidence that is valid for the employee context in which the question is asked. A broadly relevant answer that ignores job level, geography, role, access tier, or circumstances is not a partial success; it is a correctness risk. This tenet is why scope resolution happens before retrieval and why missing scope is handled at admission rather than patched after evidence has already been selected.

We prefer source-backed abstention over fluent completion when they conflict. Skywalker is not the agent that writes final prose to the user. Its job is to determine whether defensible evidence exists and to return the evidence envelope a downstream agent can use. If the system cannot establish that the retrieved evidence answers the scoped question, it must return `ABSTAIN` rather than manufacture confidence.

We prefer explicit ownership boundaries over convenience coupling when they conflict. The request lifecycle crosses MCP Gateway, PAPI, the controlled FAQ corpus, UKB, reranking, and downstream answer rendering. Skywalker owns the boundary, normalization, routing, candidate unification, and evidence envelope. It does not own UKB internals, PAPI data quality, or the conversational agent's final wording. That separation is load-bearing because it determines where bugs are routed and which team must approve a change.

We prefer implemented behavior over aspirational documentation when they conflict. The source-of-truth hierarchy is `IngestionCodeReference`, implemented code paths, architecture documents, design discussions, and future proposals. If a narrative says one thing and the running code says another, the architecture document must be corrected or the code must be changed through an explicit decision; readers should never have to infer which one is authoritative.

We prefer fixed launch architecture with calibrated thresholds over open-ended architecture churn when they conflict. Hybrid retrieval and the dual-arm FAQ-plus-UKB design are launch architecture, not experiments hidden behind wording. Thresholds, candidate depths, weighting, confidence floors, and instance sizing remain empirical surfaces, but they tune the selected architecture rather than reopening the basic boundary on every measurement cycle.

## 2. Problem and architectural intent

Skywalker solves a narrow problem: given a user question and enough employee context to interpret that question safely, return structured, ranked evidence from the approved knowledge sources that a downstream agent can use to answer. The system is intentionally not a general conversational assistant, a policy authoring workflow, a human review queue, a memory system, or a replacement for UKB. It is the retrieval and evidence boundary inside the MCP path.

The architectural intent is to make the request safe and determinate before the first retrieval call. The system cannot retrieve first and scope later because the same surface-level question may have different correct evidence depending on job level, geography, role, access tier, and circumstances. For example, an answer that is valid for one region or role may be wrong for another. The architecture therefore treats scope as part of the query contract, not as optional metadata.

The live path has four obligations. First, it must admit only requests that match one of the supported entry modes. Second, it must produce a canonical scoped request before retrieval. Third, it must route the request across the fixed launch retrieval architecture: a controlled FAQ arm for the Top 50 path and UKB for broad documentation coverage, using hybrid sparse and semantic retrieval as complementary candidate-generation mechanisms. Fourth, it must return an evidence envelope that is explicit about answerability, route, scope, ranked evidence, and abstain reason when evidence is insufficient.

This document defines the system boundary and request lifecycle at the level needed to build against the architecture. It deliberately does not define API field types exhaustively; the wire-level API specification remains the companion artifact for exact serialization, validation details, and versioning. The architecture here defines why those contracts exist, what they bind, and how failures must behave.

## 3. System boundary and ownership

Skywalker is an MCP-bounded retrieval backend. It begins at MCP request admission for the supported caller path and ends when Skywalker returns the structured retrieval envelope. Inside that boundary, Skywalker owns schema validation for the supported entry modes, identity-to-scope resolution when the request does not already contain scope, canonical request normalization, retrieval route selection, hybrid candidate generation through the owned retrieval surfaces, candidate normalization across arms, common reranking, answerability classification, abstain classification, and the final evidence envelope.

Skywalker is not the system of record for employee identity, employment attributes, role metadata, regional eligibility, or access tier. Those fields are consumed as scope, either supplied directly by the caller under the explicit-scope path or resolved through PAPI when identity-driven entry modes are used. If PAPI returns missing, stale, or contradictory employee data, the defect routes to the owner of that source data or integration. Skywalker owns whether it accepted the data according to its contract and whether it failed closed when the scope was insufficient.

Skywalker is not UKB. UKB owns broad documentation indexing and the behavior of its own candidate generation. Skywalker owns when UKB is called, how UKB candidates are normalized into the common candidate shape, how those candidates compete with controlled FAQ candidates during reranking, and how the returned evidence is represented in the final envelope. A bug in UKB's internal ranking or corpus freshness routes to UKB; a bug in Skywalker's routing, normalization, reranking integration, or evidence envelope routes to Skywalker.

Skywalker is not the downstream conversational agent. It does not own final answer prose, multi-turn memory, prompt style, or human-facing abstain wording. It returns evidence and answerability status. If the downstream agent ignores an `ABSTAIN`, misquotes evidence, collapses scope distinctions, or produces unsupported prose, that defect routes to the downstream agent owner. If Skywalker returns an `ANSWERABLE` envelope with evidence that does not support the scoped question, that defect routes to Skywalker.

Skywalker is not the human review, evaluation, or approval system. Review workflows may inspect Skywalker outputs, and evaluation harnesses may drive calibration, but they do not change the boundary contract. A failed eval can reopen thresholds or implementation details only through the calibration and decision rules in this document; it does not implicitly move ownership of retrieval behavior out of Skywalker.

Escalations follow the same boundary. Admission failures, normalization defects, incorrect route selection, candidate unification defects, reranker integration defects, incorrect answerability, and malformed retrieval envelopes escalate to Skywalker. Identity and scope source defects escalate to the owning identity or PAPI path after Skywalker confirms it consumed the source as provided. Corpus content gaps escalate to the owner of the relevant corpus: controlled FAQ for Top 50 content and UKB for broad documentation. Final wording, refusal text, and conversation behavior escalate to the downstream agent.

## 4. Source-of-truth hierarchy

For lifecycle, boundary, and retrieval behavior, the authoritative order is:

1. `IngestionCodeReference`.
2. Implemented code paths.
3. Architecture documents, including adopted API and integration contracts.
4. Design discussions.
5. Future proposals.

This hierarchy is intentionally blunt. Architecture documentation is useful only when it stays tied to the system that runs. If this document conflicts with IngestionCodeReference or the implemented code paths, the conflict is not resolved by preference or recency of prose. The owner must either update the document to match implementation or make a deliberate code change that brings implementation back to the recorded decision. Future proposals have no authority until they are accepted as decisions and implemented.

The hierarchy also prevents evaluation or rollout observations from silently becoming architecture. A measurement can justify reopening a decision, tuning a calibration surface, or adding an open question, but it does not override the launch architecture by itself. The change must say what evidence was observed, what it changes, what it binds, and which downstream contracts are affected.

## 5. Assumptions and consequences

This section assumes that the supported launch entry modes are alias-driven request, employee-id-driven request, and explicit-scope request. If a new caller requires a materially different entry mode, the consequence is not just another schema branch; the admission contract, normalization rules, correlation strategy, and possibly ownership routing must be revisited.

This section assumes that PAPI is the authoritative scope source when the caller does not provide scoped fields directly. If PAPI cannot provide the needed job level, geography, role, access tier, or circumstances for a request, the consequence is a boundary failure or abstain depending on where the insufficiency is detected. The system must not compensate by retrieving unscoped evidence because that would violate scoped correctness.

This section assumes that explicit-scope callers are allowed to provide scoped fields directly and that Skywalker treats those fields as the scope input for the request. If post-launch evidence shows that direct-scope callers send stale, incomplete, or over-broad scope, the consequence is a revisit of validation hooks for that path, not a relaxation of the requirement that retrieval be scoped before candidate generation.

This section assumes the launch traffic mix contains both high-value Top 50 questions and broader documentation questions. If real traffic across hundreds of active users shows a materially different distribution, the consequence is calibration of router thresholds, candidate depths, and arm invocation policy. The launch dual-arm architecture remains fixed unless the observed distribution shows that an arm no longer has a justified role or that a missing arm is required for correctness.

This section assumes hybrid retrieval is the fixed launch architecture. Sparse and semantic retrieval are complementary mechanisms that feed candidate generation before common reranking. If production evidence shows one mechanism consistently harms answerability, latency, or cost after threshold and depth tuning, that evidence can reopen weighting and depth immediately and can reopen the hybrid decision only if the failure remains after calibration.

This section assumes reranking remains the common scoring surface for candidates produced by the controlled FAQ and UKB paths. If the reranker cannot meet latency, throughput, or quality requirements under launch traffic, the consequence is first to revisit instance sizing, batching, candidate counts, and thresholds. Replacing the common scoring surface would reopen a fixed decision because downstream answerability and evidence ordering depend on a unified score.

This section assumes production inference sizing has not yet been selected. GPU sizing remains benchmark-gated across `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge`, with final selection based on observed latency, throughput, concurrency behavior, and total operating cost. Production deployment of these instance classes requires L6 engineering and L6 finance approval because infrastructure spend is material. If benchmark results fail to produce an approvable option, the consequence is a capacity and latency decision review before launch-scale rollout.

## 6. Inputs, outputs, and contracts

Skywalker accepts three request entry shapes at the boundary. An alias-driven request provides a user alias plus query text and request metadata. An employee-id-driven request provides an employee identifier plus query text and request metadata. An explicit-scope request provides query text, request metadata, and the scoped employee context required for retrieval. The precise wire fields belong in the API specification, but the architectural contract is that every accepted entry mode must normalize to the same internal shape before retrieval.

The canonical internal request contains `query_text`, `identity`, `scope`, and request metadata. `query_text` is the normalized question to retrieve against. `identity` records the caller or employee identity used for traceability and scope resolution. `scope` is the retrieval-relevant employee context, including job level, geography, role, access tier, and circumstances where those fields affect answer validity. Request metadata includes the entry mode and `correlation_id` or the tracing identifier used by the caller path. Downstream retrieval stages consume the canonical fields and must not branch on transport-specific input encodings.

The scope contract is binding. Retrieval must not begin until the canonical request contains sufficient scope to decide which evidence is eligible. Scope may come from PAPI or from explicit caller-supplied fields, but once normalized it is treated as the context used for filtering, route selection, evidence selection, and envelope reporting. The envelope must include the scope snapshot used for retrieval so downstream systems and review workflows can understand which context produced the evidence.

The output contract is a structured retrieval envelope, not conversational prose. The envelope contains a retrieval result kind of `ANSWERABLE` or `ABSTAIN`, route metadata describing whether FAQ-only or dual-arm execution was used, the scope snapshot used for retrieval, ranked evidence records, and an abstain reason when the result is not answerable. Evidence records must carry enough source identity and ranking information for downstream rendering and audit of the answer path. The downstream agent may transform this envelope into user-facing text, but it may not treat missing evidence as permission to invent an answer.

The failure contract is also part of the boundary. Malformed input fails at admission. Missing or invalid scope fails before retrieval when the system cannot safely form a canonical scoped request. Dependency failures are represented as boundary failures when they prevent safe retrieval and as abstain only when the request can be safely processed but the retrieved evidence is insufficient. The caller must be able to distinguish "request not eligible for retrieval" from "request eligible, but no defensible answer found."

## 7. Fixed decisions

Decision: Skywalker supports alias-driven, employee-id-driven, and explicit-scope entry modes for launch. The rationale is that these modes cover the caller shapes needed at launch while still converging on one canonical internal request. This binds admission validation, identity resolution, tracing, and normalization. The decision reopens if a new caller cannot safely express its request through one of these modes or if post-launch evidence shows one mode creates unresolved ambiguity at material volume.

Decision: Scope is required before retrieval. The rationale is that context-aware answers depend on job level, geography, role, access tier, and circumstances, and retrieving without those constraints can select evidence that is correct for the wrong employee context. This binds request admission, PAPI resolution, explicit-scope validation, route selection, filtering, and abstain behavior. The decision reopens only if the product requirement changes so that Skywalker no longer returns employee-context-aware evidence, which would be a different system boundary.

Decision: PAPI is called before retrieval when scoped fields are not already present. The rationale is that identity-driven requests need a single source of employee scope before candidate generation begins. This binds alias and employee-id paths to the identity resolution stage and prevents route selection from treating identity as a proxy for scope. The decision reopens if PAPI is replaced as the scope authority or if a caller is approved to supply equivalent scoped context directly under the explicit-scope contract.

Decision: The launch retrieval architecture is dual-arm: controlled FAQ for the Top 50 path and UKB for broad documentation coverage. The rationale is that Top 50 questions need precision, ownership, and schema control, while broad documentation questions need UKB coverage. This binds route metadata, candidate normalization, reranking, answerability, and bug routing across corpus owners. The decision reopens if production evidence shows that one arm cannot meet its correctness role after calibration or if a new source of truth becomes required for a launch-critical class of questions.

Decision: Hybrid retrieval is fixed launch architecture. Sparse and semantic retrieval both feed candidate generation before reranking. The rationale is that lexical precision and semantic recall cover different failure modes, and choosing either one alone would make the system brittle before production evidence exists. This binds indexing assumptions, candidate depth calibration, retrieval latency budgeting, and normalization. The decision reopens if production data shows that hybrid retrieval cannot be made reliable or affordable after tuning weights, depths, and thresholds.

Decision: Reranking is the common scoring surface across FAQ and UKB candidates. The rationale is that arm-local scores are not comparable enough to drive final ordering once candidates come from different retrieval systems and scoring methods. This binds candidate normalization, final ranking, answerability thresholds, and the evidence order consumed downstream. The decision reopens if benchmark or quality data shows the common reranker cannot meet correctness, latency, or cost requirements and no sizing or candidate-depth calibration can resolve it.

Decision: Skywalker is stateless per request. The rationale is that retrieval correctness should be determined by the current query, identity, scope, request metadata, and source corpora, not by hidden conversational memory. This binds reproducibility, debugging, correlation, and downstream ownership. The decision reopens if the product requires session-aware retrieval, in which case memory ownership and contract boundaries must be designed explicitly rather than smuggled into retrieval.

Decision: Production GPU sizing remains benchmark-gated rather than preselected. The rationale is that `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge` have different latency, throughput, concurrency, and cost profiles, and the selected class has material financial impact. This binds deployment planning and approval flow, requiring L6 engineering and L6 finance approval before production deployment. The decision reopens when benchmark evidence is complete or when launch traffic assumptions materially change.

## 8. Alternatives considered

One alternative was to make Skywalker a conversational agent that retrieves, reasons, and writes the final answer. That design is attractive because it appears simpler to callers: they send a question and receive prose. It is rejected for this boundary because it couples retrieval correctness to prompt behavior, multi-turn memory, and final wording. It also makes abstain harder to enforce because evidence gaps can be hidden by fluent language. The chosen boundary lets Skywalker be held accountable for evidence while the downstream agent remains accountable for final expression.

Another alternative was to retrieve first and apply employee scope as a post-processing filter. That design is attractive because it minimizes admission-time dependency on PAPI and can produce candidates even when scope is incomplete. It is rejected because it violates scoped correctness. If candidate generation is unscoped, the ranking surface can be dominated by evidence that should never have been eligible for that employee context. Filtering after the fact does not repair ranking bias, and it makes failure behavior ambiguous when all high-ranked candidates are later removed.

Another alternative was to use UKB only for launch. That design is attractive because it reduces corpus ownership surfaces and avoids a separate controlled FAQ path. It is rejected because the Top 50 path has higher precision and ownership requirements than broad documentation retrieval alone can guarantee. The controlled FAQ arm exists so that high-value recurring questions can be represented with tighter schema control, known ownership, and more predictable evidence behavior.

Another alternative was to use the controlled FAQ corpus only and defer UKB. That design is attractive because it is easier to reason about, easier to test exhaustively, and likely cheaper at small scale. It is rejected because it would turn Skywalker into a narrow Top 50 responder rather than a retrieval backend for the broader documentation path. UKB is required for coverage outside the controlled FAQ set.

Another alternative was to choose sparse retrieval or semantic retrieval as the sole retrieval method. Sparse-only retrieval is attractive because it is explainable, fast, and strong for exact policy terms. Semantic-only retrieval is attractive because it is better at paraphrase and user language mismatch. Both single-method designs are deferred because each creates avoidable blind spots. Hybrid retrieval gives launch a broader candidate pool while leaving weighting, depth, and thresholds available for calibration.

Another alternative was to preserve arm-local scores and merge candidates by source priority. That design is attractive because it is mechanically simple and can encode an explicit preference for controlled FAQ content. It is rejected because source priority is not the same as evidence quality, and local scores from different retrieval systems are not directly comparable. A common reranker gives the system one scoring surface for answerability and evidence order while still allowing route metadata and source identity to remain visible.

Another alternative was to preselect the smallest GPU class and scale horizontally if needed. That design is attractive because it constrains initial cost and simplifies finance approval. It is deferred because the reranker may become the dominant latency and concurrency constraint, and premature instance selection could force architectural workarounds into retrieval policy. The launch decision is to benchmark `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge` with L6 engineering and L6 finance approval before production deployment.

## 9. End-to-end request lifecycle

The lifecycle begins when an MCP request arrives through the expected caller route. Skywalker first validates that the request matches one supported entry mode. If the request is malformed, carries an unsupported entry mode, or lacks the minimum fields required for that mode, the request fails at admission and does not enter retrieval. This branch is a boundary failure, not abstain, because the system did not have a valid retrieval request.

For an alias-driven request, Skywalker uses the alias as identity input and resolves the employee scope through PAPI before retrieval. For an employee-id-driven request, Skywalker uses the employee identifier as identity input and also resolves scope through PAPI. For an explicit-scope request, Skywalker validates and normalizes the provided scoped fields rather than calling PAPI for missing identity-derived context. In all cases, the goal of this stage is the same canonical internal request: query text, identity, scope, and metadata.

If identity resolution fails, if PAPI cannot return sufficient scope for an identity-driven request, or if explicit scope is incomplete or invalid, Skywalker stops before retrieval. It must not run a best-effort unscoped search. The caller receives the boundary failure shape defined by the API contract. The operational meaning is that the request could not be safely admitted to retrieval, not that the knowledge base lacked an answer.

Once canonical scope exists, Skywalker executes the routing gate before any retrieval call. Strong FAQ-gated signals may route to FAQ-only execution when the route can be decided with enough confidence. Otherwise, the system executes the dual-arm path so controlled FAQ and UKB candidates can both compete in the common ranking surface. The routing stage records route metadata because downstream debugging and review need to know which arm or arms were invoked.

Candidate generation then runs under the fixed hybrid architecture. Sparse and semantic retrieval produce candidate sets according to the calibrated depths and weighting rules for the chosen route. In the FAQ path, candidates come from the controlled FAQ corpus and preserve source identity needed for ownership and evidence reporting. In the UKB path, candidates come from UKB's broad documentation surface and are normalized into the Skywalker candidate shape. Retrieval failures in an invoked arm are handled according to whether enough safe evidence remains to continue; the system must not silently pretend a failed arm returned no relevant evidence unless that behavior is explicitly represented in the envelope or failure contract.

After candidate generation, Skywalker normalizes candidates into one common representation. Normalization removes transport-specific shape differences and preserves the fields needed for reranking, source reporting, route metadata, and evidence construction. Candidates that cannot be normalized are not allowed to leak into reranking as partial objects. If normalization failure prevents a safe candidate pool, the request fails or abstains according to whether the system has enough valid evidence to make an answerability decision.

The reranker scores the normalized candidate pool as the common ranking surface. Arm-local scores may be retained for diagnostics, but they do not determine final evidence order. The reranker output feeds answerability classification. If the top evidence meets the calibrated answerability threshold and supports the scoped query, the envelope returns `ANSWERABLE` with ranked evidence. If evidence is missing, contradictory, below threshold, out of scope, or otherwise insufficient for a defensible answer, the envelope returns `ABSTAIN` with an abstain reason.

The lifecycle ends when Skywalker returns the structured retrieval envelope to the downstream caller. The envelope is the handoff boundary. It contains the result kind, route metadata, scope snapshot, ranked evidence, and abstain reason when applicable. It does not contain final conversational prose, prompt instructions, or hidden memory state.

## 10. Failure and abstain behavior

Failure and abstain are different outcomes. A boundary failure means Skywalker could not safely process the request under the contract. `ABSTAIN` means Skywalker processed a valid scoped request but did not find evidence strong enough to support an answer. This distinction matters because callers, dashboards, and human review must not treat contract failures as knowledge gaps or treat abstains as system outages.

Malformed input fails at admission. Unsupported entry modes, missing query text, missing identity for identity-driven paths, malformed explicit scope, and invalid request metadata do not enter retrieval. These failures are fail-closed because letting malformed requests retrieve would make downstream evidence impossible to interpret.

Identity and scope failures also fail before retrieval when scope cannot be safely established. If PAPI is unavailable for an identity-driven request, if PAPI returns no matching employee, or if required context fields are missing, Skywalker cannot retrieve. If an explicit-scope request omits architecture-relevant scope fields, Skywalker cannot treat the caller's partial context as enough. These failures route to the identity, PAPI, caller, or Skywalker owner depending on whether the bad state came from source data, caller contract, or Skywalker validation.

Routing failures fail closed when the route cannot be determined safely. The system may fall from FAQ-only eligibility to dual-arm execution when the route confidence is not strong enough for FAQ-only. That is not a failure; it is the designed conservative branch. A routing implementation error, missing router model, or invalid route metadata is a Skywalker failure because it prevents the system from executing the lifecycle deterministically.

Retrieval-arm failures are handled according to the route and evidence state. If an arm required by the selected route is unavailable and the system cannot make a defensible answerability decision from the remaining valid evidence, the request must not return `ANSWERABLE`. For dual-arm execution, a partial dependency failure may still end in `ABSTAIN` if the system can safely say it lacks enough evidence, but it must not hide the fact that candidate generation was incomplete when that fact affects review or caller interpretation.

Candidate normalization and reranking failures are Skywalker failures when they prevent construction of a valid evidence pool or common score. These stages are inside the Skywalker boundary and cannot be delegated to corpus owners. The system must not return unmarked retrieval-order evidence as though it were reranked. Reranker failure behavior follows Section 07 and Section 10: either return explicitly marked `RERANKER_FAILURE_FALLBACK` evidence under a conservative rule or hard abstain.

Abstain occurs after a valid scoped request reaches evidence evaluation. Valid abstain reasons include no eligible evidence, evidence below confidence floor, conflicting evidence, evidence that is relevant to the question but not to the scoped employee context, and evidence that is too coarse to support the requested answer. Abstain is the correct behavior when the system has insufficient support; it is not a degraded form of success and should not be rewritten downstream as a confident answer.

## 11. Calibration surfaces

Calibration surfaces are intentionally empirical settings inside the fixed architecture. They may change with observed data without reopening the system boundary, provided the change preserves the decisions recorded above.

The router thresholds are calibration surfaces. FAQ-only eligibility, dual-arm fallback thresholds, cosine thresholds, and any second-stage gate thresholds should be tuned against observed traffic, false FAQ routing, missed FAQ opportunities, and abstain quality. Calibration can make FAQ-only stricter or looser, but it cannot remove the requirement that routing execute before retrieval.

Hybrid retrieval weights and depths are calibration surfaces. Sparse and semantic `k`, over-retrieve factors, deduplication behavior, and arm-specific candidate limits can change as production data shows which candidate pool gives the reranker enough useful evidence without exceeding latency and cost targets. These settings tune hybrid retrieval; they do not convert the launch architecture into sparse-only or semantic-only retrieval.

Reranker input counts, batching behavior, confidence thresholds, and abstain floors are calibration surfaces. They should be tuned against answerability precision, evidence support, latency, and throughput. Calibration can reduce candidate volume or raise the abstain floor when evidence quality is weak, but it cannot allow unsupported `ANSWERABLE` envelopes.

Evidence granularity is not an ordinary calibration surface. The launch architecture uses one indexed evidence document per eligible COREx node and does not perform parent/child reconstruction, sibling expansion, linked-parent traversal, or post-retrieval context assembly. If production review shows that one-document-per-node evidence is too coarse, the change reopens the ingestion and evidence contract rather than merely tuning a value.

Inference instance sizing is a calibration and approval surface. Benchmarking across `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge` must measure latency, throughput, concurrency behavior, and total operating cost. Final production sizing requires L6 engineering and L6 finance approval. Instance selection can change deployment characteristics and cost envelope, but it should not change the retrieval contract or answerability semantics.

## 12. Open questions

Open questions are unresolved architecture or product questions, not hidden decisions. They require stated evidence before closure, and many should not be closed before post-launch observations across hundreds of active users and real traffic distributions. Synthetic tests and pre-launch review are useful, but they are not enough to settle questions whose answer depends on how employees actually ask scoped policy and knowledge questions.

The first open question is the exact `correlation_id` and tracing strategy across future integrations. The launch path needs traceability, but additional callers may bring different propagation requirements. This should be closed when the next caller integration is concrete enough to show whether the current metadata contract is sufficient.

The second open question is whether explicit-scope mode remains a pure pass-through mode under all future caller sets. The current architecture accepts explicit scope as caller-provided context after validation. If hundreds of users expose recurring stale, incomplete, or over-broad explicit scope, the evidence standard for changing this mode is observed production variance tied to answerability or correctness defects, not isolated hypothetical concern.

The third open question is where secondary validation hooks for direct-scope input become necessary. The architecture does not add governance-only validation here, but production data may show that some callers need additional checks before retrieval. The evidence standard is repeated post-launch cases where the explicit-scope contract admits requests that produce wrong or unsafe evidence despite conforming to the current schema.

The fourth open question is how abstain vocabulary and downstream messaging should evolve. Skywalker owns abstain reason structure in the envelope, while the downstream agent owns user-facing wording. Changes should be driven by live human review cycles and user interaction data showing that abstain categories are too coarse, misleading, or insufficient for remediation.

The fifth open question is whether the Top 50 and broad-documentation traffic mix matches launch assumptions. This cannot be answered from pre-launch examples alone. It should be decided after the system has post-launch evidence from hundreds of active users, with enough volume to compare FAQ-only wins, dual-arm wins, UKB-only evidence, abstain reasons, latency, and escalation patterns.

The sixth open question is whether GPU sizing selected from the benchmark remains valid after real concurrency arrives. The benchmark gate across `ml.g5.xlarge`, `ml.g5.2xlarge`, and `ml.p5.4xlarge` is required before production deployment, but post-launch traffic may still differ from benchmark assumptions. Reopening sizing requires observed latency, throughput, concurrency, and cost data, plus renewed L6 engineering and L6 finance review if the change materially affects spend.
