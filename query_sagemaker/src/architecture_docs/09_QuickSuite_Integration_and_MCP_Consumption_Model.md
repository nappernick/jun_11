## Section 09. QuickSuite Integration and MCP Consumption Model

Section 08 fixed the Slack surface as the client path where the team itself is building the agent behavior, the prompt layer, and the user-facing turn handling. This section moves to the other client surface and fixes the asymmetry directly rather than pretending the two integrations are the same shape. They are not.

QuickSuite is different for one simple reason. **QuickSuite already handles MCP and already has its own agent runtime**, with its own chat-agent surface, its own knowledge spaces, and its own intent-based routing across registered MCP connectors. That means the work for QuickSuite is not to build a second conversation layer, not to recreate prompting in a new place, and not to turn the retrieval backend into a QuickSuite-specific answer generator. The work is to define how QuickSuite consumes Skywalker through MCP, what request shapes it should use, what response shapes to expect back, and where the boundary stays hard so the system does not drift into two different backends hiding behind one name.

The QuickSuite path is the cleanest expression of the architecture. A user interacts with their QuickSuite chat agent. The chat agent's intent routing decides Skywalker is the right tool for the turn. QuickSuite calls Skywalker through its registered MCP connector, which routes through **Amazon MCP Gateway** with Federate-OAuth inbound auth. Skywalker performs scoped retrieval. Skywalker returns a structured backend result. QuickSuite remains the conversational layer that decides how that result is presented or used in the larger interaction. The fact that Slack needs more work on our side does not change this — it only makes the QuickSuite shape easier to forget unless it is written down explicitly.

This section therefore defines the QuickSuite-facing subsystem as a real integration layer with its own ownership boundary, request contracts, route expectations, failure semantics, and re-litigation triggers. It is intentionally thinner than the Slack section, but thin does not mean vague. The whole point of this section is to make sure thin stays precise.

### 1. What this section owns

This section owns the QuickSuite-facing integration boundary that begins when QuickSuite decides to invoke Skywalker through its registered MCP connector and ends when Skywalker returns a structured backend result that QuickSuite can consume. In practical terms, that means it owns the QuickSuite-side consumption model for the MCP contract, the MCP Gateway transport configuration on the Federate-inbound route, the Federate Service Profile that authenticates QuickSuite into the gateway, and the Bindle-based authorization that controls which Federate principals may invoke Skywalker's MCP server.

It also owns the explicit decision that **there is no wrapper Lambda between QuickSuite and Skywalker**. Earlier drafts of this architecture imagined a Skywalker-owned plug-in wrapper hosted under Bedrock AgentCore Gateway that would translate one task-oriented MCP tool (`search_travel_events_expense_policy`) into one of three core Skywalker tools. That layer does not exist on the supported QuickSuite-to-MCP-Gateway path. QuickSuite reads Skywalker's actual `tools/list` directly through the gateway and chooses among the three core tools (`skywalker.search.by_alias`, `skywalker.search.by_employee_id`, `skywalker.search.by_explicit_scope`) based on what identity material the QuickSuite chat-agent integration is configured to supply. This is documented at the BuilderHub primary source: [Integration with QuickSuite](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/howto-quicksuite-client/), [MCP Gateway concepts](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/concepts/), and [MCP Gateway User Guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/). The grounding contract for this section is [API_08](done/API_08_QuickSuite_MCP_Gateway_Integration.md).

This section owns the QuickSuite-specific statement of asymmetry across client surfaces. The larger series already fixed that Skywalker sits behind MCP and that the MCP output feeds agents rather than being written as the final user-visible response directly. This section makes that concrete for QuickSuite. Unlike Slack, QuickSuite does not require the team to build the conversational layer, prompt layer, or turn-memory layer inside the integration section itself. Those behaviors already live on QuickSuite's side. The architecture work here is therefore not to make QuickSuite conversational. It is to keep the MCP seam clean enough that QuickSuite can be conversational without forcing retrieval logic to leak upward or agent logic to leak downward.

This section owns one important consequence of using MCP Gateway's Federate-inbound route as the transport. **MCP Gateway does not propagate the originating Federate-authenticated end-user identity downstream to the Skywalker MCP server on the Federate-inbound + CloudAuth-outbound combination.** The gateway's published delegated-identity patterns (TransitiveAuth on CloudAuth, Forward Access Sessions on AWSAuth, UnifiedAuth, Midway A5) are all flagged "in progress" on the [MCP Gateway concepts](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/concepts/) page as of this section's writing, and none currently cover the Federate-OAuth-inbound to CloudAuth-outbound combination this path requires. The CloudAuth-inbound + CloudAuth-outbound combination Sections 05 and 08 use does support OBO + TransitiveAuth natively per [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md), so the "identity does not propagate" statement is specific to QuickSuite's auth combination rather than a general property of MCP Gateway. Identity for retrieval scope on the QuickSuite path therefore has to be carried by the integration itself — supplied as MCP tool arguments by the QuickSuite chat-agent integration from its own session data — rather than read out of a propagated JWT claim on the Skywalker side. This is a real architectural decision, not a footnote, and §3 fixes it explicitly.

This section does not own the internal behavior of QuickSuite's chat-agent runtime, the QuickSuite chat-agent's behavioral prompt, QuickSuite's conversation memory, QuickSuite's rendering decisions, or QuickSuite's larger product behavior outside the Skywalker tool call. It does not own PAPI, the MCP entry contract itself, the Top 50 FAQ ingestion path, UKB integration, reranking, or the abstain logic already fixed in earlier sections. It also does not own the final user-visible answer phrasing for QuickSuite. The Skywalker side of the contract stops at the structured backend result.

A second non-ownership boundary matters just as much. This section does not turn QuickSuite into a special privileged caller that gets a different backend than other clients. QuickSuite is a first-class integration surface, but it is still a client of the same Skywalker MCP boundary. If the architecture starts to depend on one hidden QuickSuite-only path that bypasses MCP or bypasses normal request validation, then the backend has already started to fork.

### 2. Inputs, outputs, and contracts

The primary input to this section is a QuickSuite chat-agent decision to call Skywalker through its registered MCP connector for one user request. That means the true upstream input is not arbitrary conversation state and not a raw chat transcript. It is a tool invocation request that QuickSuite has already decided belongs on the Skywalker path.

The transport from QuickSuite to Skywalker's MCP server is **Amazon MCP Gateway on its Federate-inbound route**. The MCP server URL QuickSuite registers is `https://api.mcp.asbx.aws.dev/federate/mcp/{registry-id}/{skywalker-server-id}` (per BuilderHub [Integration with QuickSuite](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/howto-quicksuite-client/)). MCP Gateway validates the inbound Federate JWT directly — there is no AgentCore Gateway in the path, and there is no REQUEST interceptor Lambda. Bindle-based authorization (`canInvoke` on `MCPGateway::{skywalker-server-id}`) gates whether the QuickSuite Federate Service Profile's Amazon Teams group is allowed to invoke Skywalker's MCP server. On the gateway-to-Skywalker outbound leg, MCP Gateway converts Federate inbound to CloudAuth outbound; Skywalker's MCP server is registered as a CloudAuth-protected service.

The QuickSuite request contract is therefore narrow at the MCP-server boundary. QuickSuite's chat-agent runtime invokes one of the three core Skywalker tools fixed in Section 02 and pinned in API_01: `skywalker.search.by_alias` (`{ query_text, alias, correlation_id? }`), `skywalker.search.by_employee_id` (`{ query_text, employee_id, correlation_id? }`), or `skywalker.search.by_explicit_scope` (`{ query_text, employee_id, country, level, role, correlation_id? }`). The chat agent chooses among them based on what identity material the QuickSuite-side integration is configured to supply (§3 decision two). There is no fourth task-oriented tool, and no wrapper translates a single external tool name into one of these three.

Identity is not in headers and not in JWT claims. Per §1, MCP Gateway does not propagate Federate JWT claims downstream on this path. The Skywalker MCP server reads identity material from the request body's `arguments` object — `alias`, or `employee_id` plus the scope triple — exactly as it does for every other client. This is materially different from v1's AZA wrapper design (which used signed identity headers like `x-employee-id` and `x-amzn-transitive-authentication-token`) and from an earlier draft of this section that imagined an AgentCore Gateway REQUEST interceptor injecting `__authContext` into the request body. Neither is part of the current path. The integration is responsible for getting identity into the tool arguments through its own integration-specific channel.

The **launch posture for the QuickSuite integration's identity-carriage mechanism is open** (§9), because QuickSuite's chat-agent author has not yet committed to a specific approach. Realistic options the architecture supports without modification: the chat agent reads the user's alias from QuickSuite session metadata and passes it as a `skywalker.search.by_alias` argument; the chat agent prompts the user once per session for scope and passes those values to `skywalker.search.by_explicit_scope`; or a small QuickSuite-side data source maps the authenticated QuickSuite user to their alias before the tool call. Whichever the team picks, Skywalker's contract does not change — it validates the arguments against the input schema fixed in Section 02 and proceeds.

The QuickSuite-specific `_meta.additionalProperties` payload structure that v1's AZA wrapper had to deal with (`conversationHistory`, `attachments`, `userMessage`, `toolConversationId`) is not used. QuickSuite passes the current turn text as `query_text` and relies on its own conversation history on its side; Skywalker stays stateless per Section 02.

The output contract back into QuickSuite is the same structured envelope every other client receives: `result_kind` (`ANSWERABLE` or `ABSTAIN`), `route` (which arms ran and under what conditions, including `path` set to whatever Skywalker's runtime decided), `scope_snapshot` (the country/level/role used for retrieval, returned back for auditability), `evidence[]` (ranked candidates with title, text, source URL, policy links, arm-local rank, rerank score, and on linked FAQ candidates a `citations[]` resolution table — populated when answerable), `abstain_reason` (populated when abstaining), and `correlation_id`. The envelope flows unchanged through MCP Gateway to QuickSuite. There is **no wrapper layer reshaping the envelope into a different `content[]` convention** — QuickSuite's chat-agent runtime reads the structured response directly through MCP Gateway's standard MCP response shape.

A QuickSuite-specific UI quirk shapes the response contract enough to be worth naming explicitly. The QuickSuite chat-agent's "Sources" UI groups all citations under the MCP connector name rather than rendering per-document URLs from the response payload. This is a documented current limitation (the [Amazon Quick Suite — FAQs](https://w.amazon.com/bin/view/AmazonQuickSuite/FAQ/) explicitly addresses it). With no wrapper Lambda available to embed source titles and URLs literally in the response text body, the workaround has to live somewhere else. The launch posture is **chat-agent-side prompting**: the QuickSuite chat-agent author writes the agent's instruction prompt to require explicit source-listing in the chat agent's output text whenever it cites Skywalker evidence. If chat-agent prompting proves insufficient under production review, the alternative is for Skywalker's MCP server to populate a `_sources` text-body field in the structured envelope when the requesting client identifies as QuickSuite, but doing so would require client-identity awareness on the Skywalker side that we have otherwise tried to avoid. §8 calibration surface five tracks this.

The two-layer error model from the core contract is preserved across MCP Gateway. JSON-RPC protocol errors (unknown tool, malformed arguments, Federate JWT validation failure at the gateway, Bindle authorization failure) surface to QuickSuite as JSON-RPC errors. Tool-execution errors (PAPI unresolved identity on the alias path, post-resolution scope failure, total retrieval failure) surface to QuickSuite as a normal result with `isError: true` and a text block describing the failure. Backend abstention is **never** an error — always a successful `result_kind: "ABSTAIN"` response.

**The Skywalker MCP boundary to QuickSuite is sync.** Skywalker's MCP server returns a single sync JSON-RPC result; MCP Gateway returns a single sync HTTP response to QuickSuite; QuickSuite's chat-agent runtime consumes the full structured envelope as one complete object before composing its user-facing message. There are no SSE responses, no progressive token streaming through the MCP path, and no partial-result delivery on the QuickSuite consumption hop. This is true because Skywalker's evidence envelope is structured rather than narrative — there is no time-to-first-token to optimize on a fixed-shape JSON object — and because the MCP Gateway HTTP transport on the Federate-OAuth-inbound auth combination does not expose SSE framing today. Skywalker does not run the inline agent on the QuickSuite path the way it does on Slack and UAT (see [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md)); QuickSuite's chat-agent runtime is the agent layer here, and whether QuickSuite streams its own final generation to its own UI is a QuickSuite product behavior outside this architecture's control. The architecture's commitment is that Skywalker's MCP response to QuickSuite is sync and fully formed when it arrives.

One operational constraint: QuickSuite's MCP connector imposes a **299-second response timeout** on every tool invocation. Skywalker's retrieval p95 budget is 250–450 ms and the end-to-end target is well under 4 seconds, so the QuickSuite timeout sits two orders of magnitude above any normal request and is effectively never the binding constraint. It is documented here because it is a real upper bound — if a future Skywalker change ever pushed a single request near 299 seconds, the MCP server has to cut the call cleanly and return a JSON-RPC error rather than letting QuickSuite hit its own timeout, which renders ambiguously to the user.

### 3. Fixed decisions

The first fixed decision is that QuickSuite consumes Skywalker through MCP Gateway rather than through any wrapper layer or alternative gateway product. The transport is Amazon MCP Gateway on its Federate-inbound route at `https://api.mcp.asbx.aws.dev/federate/mcp/{registry-id}/{skywalker-server-id}`. There is no AgentCore Gateway, no Skywalker-owned wrapper Lambda, and no AppConfig CR equivalent. The grounding contract is API_08.

The second fixed decision is that QuickSuite reads Skywalker's actual `tools/list` and chooses among the three core MCP tools directly. There is no task-oriented translation tool at the gateway boundary. QuickSuite-side intent routing decides between `skywalker.search.by_alias` and `skywalker.search.by_explicit_scope` based on what identity material the QuickSuite chat-agent integration is configured to supply. The employee-ID path is part of the contract for completeness but is not an expected QuickSuite usage pattern at launch.

The third fixed decision is that **identity for retrieval scope on the QuickSuite path is carried by the integration itself, not propagated by MCP Gateway**. MCP Gateway's published delegated-identity patterns are all flagged "in progress" on the BuilderHub concepts page and do not currently cover the Federate-inbound to CloudAuth-outbound combination this path requires. The QuickSuite integration is responsible for getting the user's alias (or `country`/`level`/`role`) into the MCP tool arguments through its own integration-specific channel before the call leaves QuickSuite. This is **not** symmetric across all three production paths — Slack and UAT both moved to the CloudAuth-inbound route in their own sections, where CloudAuth OBO carries the orchestrator's AAA application identity and TransitiveAuth carries the human end-user's identity to Skywalker (see [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md)). UAT additionally still carries `(country, level, role)` as MCP tool arguments because it holds them server-side from custom Federate claims, but that's a Section 02 explicit-scope contract decision, not an identity-channel necessity. QuickSuite remains on argument-supplied identity at launch because the Federate-inbound combination has no published TA pattern; if the [MCP Gateway concepts](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/concepts/) page eventually publishes one, QuickSuite can move to the same OBO + TA shape Slack and UAT use, which would be a Section 09 calibration-active event rather than an architecture-class change.

The fourth fixed decision is that QuickSuite remains the conversational layer on its side of the boundary. Skywalker is not responsible for QuickSuite-ready phrasing, conversation-state handling, follow-up interpretation, or session continuity. Those belong to QuickSuite because QuickSuite is already the agent surface in that environment.

The fifth fixed decision is that Skywalker returns structured backend artifacts rather than final polished conversational output. The backend result may be rich, and it may contain enough material for QuickSuite to form a strong answer, but it remains a backend artifact. The exact shape is pinned in API_01 and is unchanged from the shape every other client receives.

The sixth fixed decision is that **the Skywalker MCP boundary to QuickSuite is sync**. Skywalker's MCP server returns a single JSON-RPC result; MCP Gateway returns a single sync HTTP response; QuickSuite's chat-agent runtime consumes the full envelope as one complete object. This is true symmetrically across all three production paths at the MCP boundary itself (Section 02, Section 05, Section 08). Streaming on the user-facing surface is a separate, higher-layer concern that lives between the inline agent and the rendering surface — it applies on Slack and UAT (where Skywalker runs the inline agent, see [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md)) but does not apply on QuickSuite because Skywalker does not run the agent there. Whether QuickSuite streams its own final generation to its own UI is a QuickSuite product behavior outside Skywalker's control.

The seventh fixed decision is that QuickSuite receives the same backend outcome classes as every other client surface. There is no special QuickSuite-only answerability semantics, no hidden source preference, and no separate notion of what backend abstention means. If Skywalker abstains, QuickSuite gets a structured abstain package just as any other proper client should.

The eighth fixed decision is the wrapper's authentication surface. Inbound auth is **Federate OAuth (Authorization Code flow with PKCE)** against a Federate Prod Service Profile created using the **"AWS QuickSuite Action Connectors"** pre-approved use case (per BuilderHub [Integration with QuickSuite](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/howto-quicksuite-client/)). The pre-approved use case does not require a security ticket. Federate Integ profiles are explicitly **not** used for production, because Integ profiles expire after one month and QuickSuite additionally auto-deletes connector entries backed by Integ profiles every 24 hours.

The ninth fixed decision is that Bindle-based authorization gates QuickSuite access to Skywalker's MCP server. MCP Gateway creates one Bindle resource per registered MCP server (`MCPGateway::{server-id}`); the QuickSuite Federate Service Profile's Amazon Teams group must be granted `canInvoke` on that bindle before QuickSuite chat agents can invoke Skywalker. Adding and removing chat-agent users is handled through Amazon Teams group membership, so access management stays centralized.

The tenth fixed decision is the QuickSuite connector registration mechanics. The connector is registered through the QuickSuite console (Integrations → Actions → MCP) by the QuickSuite-account administrator, not via a code-package CR. Registration carries: the MCP Gateway Federate-route URL, the Federate Client ID and Client Secret, the Federate Authorize URL (`https://idp.federate.amazon.com/api/oauth2/v1/authorize` for Prod), and the Federate Token URL (`https://idp.federate.amazon.com/api/oauth2/v2/token` for Prod). PKCE is enabled on the Federate profile. The QuickSuite OAuth callback URI for the QuickSuite-account region is registered in the Federate profile's redirect URIs list. Once registered, the connector lists the available MCP tools — Skywalker's three core tools surface directly through `tools/list`. Sharing controls on the chat agent are restricted by default; the chat-agent author opens access via the same Amazon Teams group that holds Bindle invocation permission so the two access lists stay synchronized.

The eleventh fixed decision is that there is no QuickSuite-specific PAPI trust or scope-injection logic in Skywalker. Every QuickSuite call follows the same Section 02 contract every other client follows. If QuickSuite invokes `skywalker.search.by_alias`, Skywalker resolves PAPI on its own side. If QuickSuite invokes `skywalker.search.by_explicit_scope`, Skywalker bypasses PAPI per the contract Section 02 already established. There is no QuickSuite-only short-circuit, no QuickSuite-only header trust, and no QuickSuite-only validation rule.

The twelfth fixed decision is the QuickSuite Sources-UI workaround posture. Launch is **chat-agent-side prompting**: the chat-agent author writes the prompt so the agent lists source titles and URLs explicitly in its output text whenever it cites Skywalker evidence. The alternative — Skywalker's MCP server populating a `_sources` text-body field on the structured envelope when the client identifies as QuickSuite — is the calibration-active fallback if launch behavior shows the prompt-side workaround is insufficient. §8 calibration surface five tracks this.

### 4. Alternatives considered or still live

The first alternative is a QuickSuite-specific direct integration that bypasses the shared MCP contract and lets Skywalker grow a QuickSuite-only service face. That alternative is rejected. The reason is not only reuse. It is discipline. Once one client has a hidden privileged entry path, the backend contract stops being the real boundary and starts becoming a suggestion.

The second alternative is registering Skywalker as a QuickSuite Knowledge Base or Space rather than as an MCP connector. QuickSuite supports Spaces (S3, SharePoint, file uploads) as one knowledge surface and MCP connectors as another. The Space surface would skip MCP Gateway entirely, but it would also force Skywalker's controlled retrieval logic into a static-document model that the FAQ corpus is explicitly not — it is retrieval-backed, not a static document collection. Spaces also do not support per-request scope resolution because they pre-index documents under the QuickSuite account's identity rather than under each user's identity. Rejected because Spaces collapse the architecture's identity-aware-scoping invariant into "every QuickSuite user gets the same retrieval surface."

The third alternative was a Skywalker-owned plug-in wrapper hosted under **Bedrock AgentCore Gateway** with Federate OAuth inbound and a REQUEST interceptor Lambda extracting identity claims from the JWT. This was the v2 design before the team confirmed MCP Gateway as the transport for all production paths. It is rejected because (a) MCP Gateway is the documented internal paved path for QuickSuite-to-MCP integration on BuilderHub, and the QuickSuite-client setup is materially simpler with no wrapper to maintain, (b) a wrapper Lambda introduces a deployment surface and a translation seam the architecture does not need now that QuickSuite can read Skywalker's actual `tools/list` directly, and (c) operating all three production paths on MCP Gateway gives the Skywalker MCP server one auth-onboarding story and one Bindle-permission surface rather than three. The v2 wrapper-and-AgentCore-Gateway design is preserved in git history but is not the active architecture.

The fourth alternative was QuickSuite's POPSTAR / phonetool-resolution path that some QuickSuite-native agents use, where the QuickSuite chat agent itself calls Phonetool to resolve user identity before invoking downstream MCP tools. That path could in principle let QuickSuite hand Skywalker a pre-resolved scope tuple via `skywalker.search.by_explicit_scope`, avoiding Skywalker's PAPI call. This remains live as one possible implementation of decision three's "identity carried by the integration" requirement. The launch posture stays open because the chat-agent author has not committed to a specific identity-carriage mechanism (§9).

The fifth alternative is making QuickSuite use a service-IAM identity rather than the end-user's identity. That alternative is rejected for the same reason it was rejected on the v1 AZA path: scope is part of correctness, the answer changes by user, and a service identity erases the per-user scope that the architecture exists to enforce. The Federate-OIDC-JWT path that QuickSuite supports is exactly the right shape because it carries a per-user identity claim through the OAuth handshake, even if MCP Gateway does not currently propagate that claim downstream.

The sixth alternative is letting Skywalker produce final QuickSuite-ready prose rather than one structured backend package. That alternative is rejected because it confuses the backend boundary and erases the fact that QuickSuite already has its own chat-agent runtime. The point of the integration is not to make Skywalker impersonate QuickSuite's agent.

The seventh alternative is **streaming the MCP envelope itself** through MCP Gateway to QuickSuite (or to any other client). Rejected for two reasons. First, Skywalker's evidence envelope is structured rather than narrative — there is no time-to-first-token to optimize on a fixed-shape JSON object that the consumer has to parse end-to-end before using. Second, MCP Gateway's HTTP transport on our auth combinations does not expose SSE framing today, and the gateway's published custom transport documentation describes JSON-RPC 2.0 in the body with auth in headers but no streaming envelope. Neither reason rules out streaming above the MCP boundary — Slack and UAT do exactly that, between the inline agent and the rendering surface (see [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md)) — but on QuickSuite specifically there is no inline agent in Skywalker's deployment to stream from.

A live alternative remains around the QuickSuite Sources-UI workaround. Decision twelve fixes chat-agent-side prompting as the launch posture. If production review shows the prompt approach is insufficient, the architecture can promote the Skywalker-side `_sources` field as a per-client envelope extension. That move would require Skywalker's MCP server to learn which client is calling, which the architecture has otherwise tried to avoid; the trade-off is real but is deliberately deferred until launch evidence justifies it.

### 5. Assumptions inherited from upstream sections

This section inherits the system boundary from Section 01 (Skywalker is a retrieval backend behind MCP, not the user-facing conversational brain), the MCP entry contract from Section 02 (the alias path, employee-ID path, and explicit-scope path already exist; this section decides how QuickSuite consumes them, not what the backend supports), and the fixed runtime behavior from Sections 04 through 07 (QuickSuite does not decide FAQ-only versus dual-arm routing, UKB behavior, hybrid retrieval, reranking, or backend abstention — those arrive here as stable backend truth).

It inherits Section 05's MCP Gateway transport posture: Amazon MCP Gateway is the transport for the UAT path on its SigV4-inbound route, and this section uses the same gateway product on its Federate-inbound route. The two routes share Bindle-based authorization on the same Skywalker MCP server resource, which means access additions and revocations on either route are managed against the same bindle.

It inherits Section 08's MCP Gateway transport posture for Slack: Slack also routes through Amazon MCP Gateway, on its SigV4-inbound route using the Slack application's IAM execution role. All three production paths share one transport.

Three further premises apply. The scoping triple — country, level, and manager-versus-IC role — is sufficient for the current system and not reopened here. QuickSuite-authenticated users are already strongly authenticated through Midway by the time their JWT reaches MCP Gateway, which keeps this section from turning into a broader authentication design document. And the client asymmetry made explicit in Section 08 holds: Slack required the team to build the agent boundary itself, while QuickSuite did not, so the architectural work here concentrates on the MCP seam rather than on a client-side agent the team owns.

### 6. End-to-end data flow for this section

The QuickSuite data flow is short, and the absence of a wrapper Lambda makes it shorter than earlier drafts of this section described.

**Step one.** A user interacts with their QuickSuite chat agent in QuickSuite's own conversational environment. The chat agent's intent routing determines that the request belongs on the Skywalker path and chooses a tool from Skywalker's `tools/list` (typically `skywalker.search.by_alias` at launch, depending on the integration's identity-carriage decision per §3 decision three).

**Step two.** QuickSuite checks whether the user already holds a valid Federate access token for the registered Skywalker MCP connector. If not, QuickSuite initiates the OAuth Authorization Code flow with PKCE against the wrapper's Federate Prod profile. The user authenticates through Midway. Federate issues a JWT scoped to the Skywalker connector's Federate Client ID with the user's Midway login in the `sub` (or `UID`) claim.

**Step three.** QuickSuite invokes Skywalker's MCP server through MCP `tools/call`. The HTTP request goes to `https://api.mcp.asbx.aws.dev/federate/mcp/{registry-id}/{skywalker-server-id}` with `Authorization: Bearer <federate-access-token>` as the inbound auth header and a JSON-RPC body carrying the tool name and arguments. The arguments include whatever identity material the chat-agent integration is configured to supply (alias, or `employee_id` plus the scope triple).

**Step four.** MCP Gateway validates the JWT against Federate's OIDC discovery document (signature, expiration, issuer, audience). If validation fails, the request is rejected at the gateway with a JSON-RPC protocol error before it ever reaches Skywalker.

**Step five.** MCP Gateway performs the Bindle authorization check. It extracts the principal from the validated Federate JWT and confirms the principal has `canInvoke` on `MCPGateway::{skywalker-server-id}`. If denied, the gateway returns an authorization error.

**Step six.** MCP Gateway forwards the authorized request to Skywalker's MCP server, converting Federate-inbound to CloudAuth-outbound on the gateway-to-Skywalker leg. The request body is unchanged from what QuickSuite sent — there is no REQUEST interceptor injecting `__authContext`, no wrapper Lambda reshaping the call, and no synthetic identity header insertion. Skywalker reads identity from the request body's `arguments` object.

**Step seven.** Skywalker validates the request against the MCP entry contract fixed in Section 02 (matching tool name, argument schema validation, mode-specific required fields). If the request is alias-shaped, Skywalker calls PAPI to resolve `(country, level, role)`. If the request is explicit-scope-shaped, Skywalker uses the supplied values directly. Either way, the canonical scoped request is constructed.

**Step eight.** The backend runtime executes per Sections 04 through 07 — routing gate, FAQ-only or dual-arm retrieval, normalization, reranking, sibling-and-linked-parent expansion, abstain judgment.

**Step nine.** Skywalker's MCP server returns one structured outcome: an answerable evidence package or a structured abstain package. The package shape is unchanged from Section 02 / API_01. It is a single sync JSON-RPC result. The MCP boundary itself does not stream on this path; QuickSuite's chat-agent runtime is the agent layer here, not Skywalker, so the streaming-vs-sync choice on the user-facing surface belongs to QuickSuite (see [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for the asymmetry across client surfaces).

**Step ten.** MCP Gateway returns the response unchanged to QuickSuite. The chat-agent runtime reads the structured envelope, composes the user-facing answer with citations drawn from `evidence[]`, and renders it. If the chat-agent prompt requires explicit source-listing in the output text (the launch posture for the Sources-UI workaround per §3 decision twelve), the chat agent surfaces source titles and URLs in its text body rather than relying on QuickSuite's "Sources" UI.

**Step eleven.** Any multi-turn continuation remains on the QuickSuite side until the chat agent decides a new Skywalker tool call is appropriate. Skywalker does not become stateful. QuickSuite's conversation history stays on QuickSuite's side of the boundary.

The important property of this flow is that there is only one place where QuickSuite-specific behavior should meaningfully exist: the decision to invoke the Skywalker tool and the decision of how to use the structured result afterward. The retrieval path in the middle is the same disciplined backend used for every client, and the transport in front of that backend is the same MCP Gateway that fronts every client.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

The first failure rule is that QuickSuite-side tool-selection failure is not Skywalker failure. If the chat agent decides not to call Skywalker when it should have, or calls Skywalker on the wrong turn, that is a client-side orchestration problem rather than a backend retrieval problem.

The second failure rule is that Federate JWT validation failure at MCP Gateway is a hard reject, not a retrieval event. An expired token, a missing audience claim, a signature mismatch — all surface to QuickSuite as a JSON-RPC protocol error from MCP Gateway, not as a Skywalker tool-execution result. The user's QuickSuite session may need to re-authenticate against Federate before the next call succeeds.

The third failure rule is that Bindle authorization failure is also a JSON-RPC protocol error. If the QuickSuite Federate Service Profile's Amazon Teams group does not have `canInvoke` on Skywalker's MCP server bindle, MCP Gateway returns the authorization error before the request reaches Skywalker.

The fourth failure rule is that PAPI failure is the same shape it has on the Slack path. Skywalker's PAPI client retries 3 times with a 30-second timeout and fails closed. A persistent PAPI failure surfaces back to QuickSuite as a tool-execution result with `isError: true` and a message indicating identity could not be resolved.

The fifth failure rule is that backend abstention (`result_kind: ABSTAIN`) is a successful tool-execution result, not an error. QuickSuite's chat-agent runtime must distinguish abstention from outage and render abstention as an evidence-grounded "I don't have enough to answer this for your situation" rather than as a system failure. This distinction is the same one Slack must observe and is preserved here unchanged.

The sixth failure rule is that the 299-second QuickSuite client timeout is enforced from the gateway side. Skywalker's p95 budget is 250–450 ms and the end-to-end target is well under 4 seconds, so the QuickSuite limit is effectively never the binding constraint at launch. If a future Skywalker change ever pushed a single request near 299 seconds, the architecture's response would be to cut the call cleanly with a JSON-RPC error rather than letting QuickSuite hit its own timeout, which renders ambiguously to the user.

Abstain behavior at the section level is therefore the same as on the Slack path. The retrieval backend is allowed to abstain. MCP Gateway forwards the abstain envelope unchanged. The chat agent renders it. Done.

The non-goals here matter because they keep the section from drifting into responsibilities that belong elsewhere. This section does not own QuickSuite's chat-agent prompt design, conversation memory, sharing controls, knowledge spaces, or any other QuickSuite-internal behavior. It does not own the deployment configuration of MCP Gateway itself (that belongs to the gateway team). It does not own the QuickSuite account administration that registers the connector. It does not own a separate QuickSuite-only retrieval path; QuickSuite is a client of the same Skywalker MCP boundary as Slack and the UAT inline-agent orchestrator.

A second non-goal: this section does not own streaming behavior on the QuickSuite user surface. Skywalker's MCP boundary to QuickSuite is sync, and Skywalker does not run the inline agent on this path — QuickSuite's chat-agent runtime does. Whether QuickSuite streams its own final generation to its own UI is QuickSuite's product behavior, not ours. The asymmetry across client surfaces is captured in [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md): Slack and UAT stream the inline agent's final response because Skywalker runs the agent on those paths, while QuickSuite does not appear in that contract.

### 8. Calibration surfaces and what would cause re-litigation

The first calibration surface is the QuickSuite chat-agent integration's identity-carriage mechanism. Launch posture is open per §9. Re-litigation is justified once the chat-agent author commits to a specific approach (alias from session metadata, scope from custom claims, prompted scope, or some other path) and either it works under production load or it does not.

The second calibration surface is whether QuickSuite's chat-agent intent routing chooses `skywalker.search.by_alias` or `skywalker.search.by_explicit_scope` correctly. The architecture is fixed that both are exposed and that the chat agent reads the actual `tools/list`. Whether the chat-agent prompt and tool descriptions reliably steer the agent to the right tool is calibration-active. Re-litigation is justified if production routing data shows the agent systematically picks the wrong tool.

The third calibration surface is the tool description text that QuickSuite's intent routing reads. Skywalker's `tools/list` publishes the descriptions for the three core tools directly (no wrapper layer in between), and QuickSuite reads them as-is. Tool description quality is therefore a backend concern that this section influences. Launch descriptions name the in-scope query classes (per-diem queries, Uber-home expensing, manager flight approval, home-internet eligibility, receipt submission, interview-meal expensing) and out-of-scope classes (time off → MyHR, payroll, AWS service docs). Re-litigation is justified if production review shows the chat agent is mis-routing — either calling Skywalker for queries Skywalker cannot answer, or failing to call Skywalker for queries it could answer.

The fourth calibration surface is the sharing-control granularity on the QuickSuite chat agent. Launch posture is to share access with a defined Amazon Teams group whose membership is also the Bindle `canInvoke` group, so the two access lists stay synchronized. Re-litigation is justified only if access control needs become finer-grained than group membership supports.

The fifth calibration surface is the QuickSuite "Sources" UI workaround. Launch posture is chat-agent-side prompting (§3 decision twelve). The fallback is for Skywalker's MCP server to populate a `_sources` text-body field on the structured envelope when the client identifies as QuickSuite, but doing so introduces client-identity awareness on the Skywalker side that the architecture has otherwise tried to avoid. Re-litigation is justified if production review shows the prompt-side workaround is insufficient.

The sixth calibration surface is whether the QuickSuite path eventually moves to MCP Gateway's published delegated-identity patterns (TransitiveAuth, FAS, UnifiedAuth, Midway A5) once any of them ship for the Federate-inbound + CloudAuth-outbound combination. Launch posture is "no, identity is carried by the integration through MCP tool arguments." If a delegated-identity pattern ships, the architecture can move identity propagation from arguments to header-level propagation, which would simplify the chat-agent author's job and make Skywalker's PAPI lookup consistent across all three production paths.

The seventh calibration surface is production review feedback once the system is live. Repeated review findings about QuickSuite mishandling abstention, over-smoothing single-arm-fallback responses, or failing to preserve sources and policy links are legitimate triggers for reopening the QuickSuite-facing contract or the guidance the chat agent receives about how to consume it.

### 9. Open questions, if any

The launch posture for the QuickSuite chat-agent integration's identity-carriage mechanism is open. The chat-agent author has not yet committed to a specific approach (alias from QuickSuite session metadata, alias from a small QuickSuite-side data source, scope from a custom Federate claims pattern, or scope prompted from the user). This is an integration-time decision and is the largest single open item on the QuickSuite path.

The Skywalker MCP server's identifier under MCP Gateway is operational and surfaces during MCP Gateway onboarding.

The Federate Prod Service Profile is required for production. The launch checklist includes ASR (AppSec) and privacy review against the integration's identity-handling behavior, registration of QuickSuite's region-specific OAuth callback URI in the redirect URIs list, and adding the Amazon Teams group that controls access to both the Federate profile and the Skywalker MCP server's bindle.

The MCP protocol version targeted by the gateway and Skywalker's MCP server is `2024-11-05` at launch, matching what QuickSuite's MCP-connector implementation supports today. If QuickSuite advances its supported MCP revision and we want to follow, that becomes a coordinated update across MCP Gateway, Skywalker, and any client expectations.

How QuickSuite's chat-agent runtime renders our structured envelope under its LLM-processing path — specifically whether the citation-listing requirement in the chat-agent prompt is sufficient to compensate for the Sources-UI grouping limitation — is an empirical question that lands during the chat-agent author's testing rather than at the gateway's design boundary.

Whether the QuickSuite path should eventually migrate to one of MCP Gateway's published delegated-identity patterns once they ship for the Federate-inbound + CloudAuth-outbound combination remains a calibration-active open question per §8 calibration surface six.

### Closing position

The QuickSuite integration is thinner than the Slack integration, but it is not conceptually lighter. It is the cleanest expression of the architecture because it keeps the boundary honest.

QuickSuite's chat-agent runtime decides when Skywalker is the right tool. QuickSuite calls Skywalker through Amazon MCP Gateway on its Federate-inbound route at `https://api.mcp.asbx.aws.dev/federate/mcp/{registry-id}/{skywalker-server-id}`, authenticated by a Federate Prod Service Profile created using the "AWS QuickSuite Action Connectors" pre-approved use case. MCP Gateway validates the JWT, performs Bindle-based authorization, converts Federate inbound to CloudAuth outbound, and forwards the call to Skywalker's MCP server unchanged. There is no wrapper Lambda, no AgentCore Gateway, and no REQUEST interceptor on this path. QuickSuite reads Skywalker's actual `tools/list` and chooses among the three core MCP tools directly based on what identity material the QuickSuite chat-agent integration is configured to supply. Skywalker performs scoped retrieval through the same hybrid AOSS path Slack and UAT use, and returns one structured backend result. The MCP boundary to QuickSuite is sync; Skywalker does not run an inline agent on this path, so the streaming-vs-sync choice on the user-facing QuickSuite surface belongs to QuickSuite's chat-agent runtime, not to Skywalker (see [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for how this differs from Slack and UAT). QuickSuite's chat agent reads the structured envelope, composes the answer with citations, and renders it.

If this seam stays clean, Skywalker remains one retrieval backend serving more than one client surface, on one transport, with one Bindle-based access surface. If it blurs, the system starts to fork. This section exists to keep it from blurring.
