## Section 03. Top 50 FAQ Controlled Corpus, Ingestion, and Storage

Section 02 fixed the request boundary. By the time a request leaves that boundary, Skywalker knows who the caller is, what geography, level, and role shape the answer, and whether the request is valid for retrieval at all. This section now moves to the first owned data plane behind that request path: the controlled Top 50 FAQ arm.

That arm exists for one reason and it is not subtle. The Top 50 questions are the highest-value question set in the system and the only place where the project is explicitly trying to hit a named accuracy target. UKB remains useful for broad coverage, but UKB is a black box. It controls its own ingestion, indexing, retrieval behavior, and ranking behavior. That is acceptable for the general arm. It is not acceptable as the only answer path for the small question set where the business value is concentrated and where lack of control would directly weaken the team's ability to improve accuracy intentionally.

The result is a deliberately asymmetric architecture. The general arm buys leverage from UKB. The Top 50 arm buys control by owning a small retrieval corpus end to end. Small here does not mean trivial. A one-to-ten-document corpus can still fail in ways that quietly poison trust if change handling is sloppy, if fragments are published inconsistently, or if a rebuild exposes half-finished state. This section defines the owned FAQ corpus as a real subsystem rather than as an afterthought tacked onto the side of the query pipeline.

One point needs to be made plainly because it keeps getting blurred in systems like this. The controlled FAQ arm is not a cache of prewritten answers and it is not a deterministic answer table wearing retrieval's clothes. It is still a retrieval-backed arm. The evidence it returns comes from an owned corpus built from CoreX source material, chunked, embedded, stored, and retrieved under Skywalker's control. That is the actual design.

### 1. What this section owns

This section owns the complete controlled data plane behind the Top 50 FAQ arm's evidence corpus. Concretely, that means the authoritative source relationship with COREx for this corpus, the polling loop that checks the source corpus on a daily cadence, the change-detection rules that decide whether publication work is required, the normalization and chunk-production path that turns source fragments into retrieval records, the resolution of author-asserted cross-Q&A links into a chain of `parent_id` references stamped onto each chunk, the publication rules that decide when a new FAQ corpus version becomes live, and the single AOSS evidence index (with its companion search pipeline) that later query-time retrieval consumes.

This section does **not** own the routing gate. Query-to-variant similarity comparison is a runtime concern defined in Section 04. The variant set itself is a static S3 artifact maintained by hand — a list of canonical Top 50 FAQ question phrasings with pre-computed embeddings — and lives entirely outside this section's ingestion pipeline. The artifact contract is pinned in [API_11](done/API_11_S3_Variant_Set.md). This section does not build, version, or publish variants. It does not own a variant index. It does not run any pipeline over the variant set. Variant maintenance is a manual control-plane activity, not an ingestion job.

This section does not own query-time routing thresholds, the reranker, the general UKB arm, the Slack or QuickSuite agent behavior, or any final answer formatting. It does not own PAPI resolution, which is already fixed upstream in Section 02. It does not own user-facing abstain behavior. Its job is narrower and more structural. It ensures that the controlled FAQ arm has a valid, current, queryable evidence corpus. Later sections decide how that corpus is used at runtime.

### 2. Inputs, outputs, and contracts

The authoritative input is the CoreX source corpus for the Top 50 FAQ arm. The current design assumes that CoreX will return source material as JSON objects carrying metadata and content, and may return content at fragment granularity rather than only as whole-document bodies. That is important because the ingestion path should not assume that every source document arrives as one monolithic text object. A fragment can already be an answer-sized unit. It can also be too large, too uneven, or too loosely structured to use raw. The contract here is therefore not merely "we receive documents." The contract is that the section receives a small, authoritative CoreX snapshot containing stable source identities, current metadata, and current textual content at document or fragment granularity.

At minimum, the owned source contract needs stable source-document identity, some current-state marker that lets the pipeline tell whether the source set changed, source-level metadata that the later retrieval path depends on, and the actual content bodies or fragments that will become the evidence corpus. The exact CoreX field names are still open, but the architecture is not waiting on those names to make its main decision. The system needs one authoritative current snapshot, not a stream of ambiguous partial hints.

The primary outputs of this section are threefold. First, it produces a published FAQ evidence corpus in a single AOSS index inside the `skywalker-faq-{stage}` vector collection, built from CoreX source material and shaped for hybrid (BM25 + vector) retrieval. Second, it produces a single persisted piece of state — the CoreX snapshot high-water mark — held in AWS Systems Manager Parameter Store so the next scheduled run can detect whether the source has changed. Third, it produces a runtime contract for later sections: one read surface that answers "what controlled FAQ evidence should compete in reranking for this scoped request."

The three named outputs — the AOSS index, the SSM parameter, and the runtime read surface — are concrete enough to deserve being described at the contract level rather than hidden behind a pointer.

**The AOSS evidence index.** The published corpus lives as a single versioned index named `faq_evidence_v<N>` inside an Amazon OpenSearch Serverless (AOSS) **vector search collection** named `skywalker-faq-{stage}`. The vector field is `knn_vector` of dimension 1024 under the **FAISS** engine with `cosinesimil` space and HNSW indexing. FAISS is the engine choice because AOSS vector collections support FAISS as their k-NN substrate (and not Lucene's nmslib substrate); it provides HNSW with `efficient_filter` clauses that pre-filter on the scope fields inside the k-NN query, which is what makes scope-aware retrieval correct rather than retrofit. The corpus is also small enough that raw vector throughput is not the bottleneck; what matters more is the pre-filter against `country`, `level`, and `role`. A single OpenSearch index alias named `faq_evidence_current` always points at the version that is live, and advancing that alias via the OpenSearch `_aliases` API (which AOSS supports on vector collections) is the one atomic publication primitive this section relies on.

Each document in the index is one child chunk. The schema carries fields for identity, structural reconstruction, retrieval content, scope filtering, provenance, the cross-Q&A link chain, and the vector itself. `chunk_id` and `parent_id` are keyword fields and indexed; `chunk_id` becomes the `candidate_id` echoed back in the MCP response envelope, and `parent_id` is the shared identifier across all children of one hierarchical unit that makes post-rerank sibling expansion possible. `child_order`, `child_count`, and `split_type` are stored in `_source` with `"index": false`, which means they are retrievable on a read but do not enter the inverted index and cannot be filtered or sorted against; that is deliberate, because these fields are reconstruction metadata, not retrieval axes, and the inverted-index space savings are worth having. `split_type` is a three-value enum (`paragraph_boundary`, `size_forced`, `other`) that drives the separator the sibling-reconstruction code uses when it concatenates children back into parent text. `source_id` is a keyword. `title` and `text` are explicitly mapped as `text` fields with the standard analyzer so they participate in BM25 scoring on the lexical half of hybrid retrieval; that is load-bearing — without explicit `text`-field mapping the BM25 leg of the hybrid query has nothing to score against. `source_url`, `policy_links`, and the source metadata the MCP response envelope ultimately returns to agents ride alongside. `country`, `level`, and `role` are keyword fields — arrays, because a chunk may apply to multiple countries or to both roles — and are the scope pre-filter surface that the live query path applies inside the FAISS `efficient_filter` clause. `corpus_version` records which publication the child belongs to, always matching the marker written to SSM when the alias was swapped to make this version live. `linked_parent_ids` is a keyword array stored in `_source` with `"index": false`, ordered, holding the parent_ids of the linked Q&As that ride along with this chunk's own parent at evidence-assembly time, in linked-list (chain) order, empty when the chunk's parent has no outgoing link; the chain is materialized at ingest from a single-valued custom metadata field on the FAQ content model in COREx (the "next" pointer per Q&A) and bounded at depth 2. `embedding` is the `knn_vector` field at dimension 1024 with FAISS HNSW parameters `m: 24`, `ef_construction: 128`, and a per-query `ef_search: 100`.

The HNSW parameters `m`, `ef_construction`, `dimension`, `space_type`, and `engine` are fixed at index-creation time and cannot be changed without rebuilding the index. They are architecture-class values, not control-plane values. The per-query knobs `ef_search`, `k`, and `size` are different — they are set on each query and may become control-plane values later if operator-tunable retrieval breadth becomes useful. Nothing in this section stores parents as separate documents. Parents are reconstructed at query time from the set of children that share a `parent_id`, via the post-rerank expansion query described in Section 07. That choice is what the chunker architecture was built to support.

The index is created with `"knn": true` in its settings, which activates AOSS's k-NN capability for the index, and the mapping is issued once at index-creation time as a single `PUT /faq_evidence_v<N>` call carrying the full schema above. There is no alter-after-create flow for the fields that matter; any change to mappings is a rebuild.

In addition to the index itself, ingestion creates one **search pipeline** named `skywalker-faq-hybrid` on the collection and binds it to the alias as the default search pipeline. The pipeline carries a single `normalization-processor` that applies `min_max` score normalization and `arithmetic_mean` combination; the runtime hybrid query (Section 04) supplies the per-leg weights. Section 04 owns the query construction; this section owns the fact that the pipeline exists, that it is created alongside the index, and that the alias swap promotes the pipeline binding too. Without the pipeline, a hybrid query against the collection runs but does not fuse scores correctly across BM25 and vector legs.

**The SSM ingestion parameter.** Exactly one piece of state persists between ingestion runs, and it is a single `String` parameter at the path `/skywalker/ingestion/faq_evidence/last_snapshot_marker`. It is a Standard-tier parameter, not `SecureString`, because the marker is not sensitive. Its value is opaque to everything except the ingestion job itself — the shape is determined by whichever change-marker strategy the pipeline settles on (per-node version map versus content hash), and downstream consumers do not interpret it. First-run behavior is well-defined: `GetParameter` returns `ParameterNotFound`, and the job treats absence as "no prior snapshot" and rebuilds from scratch.

Every other piece of publication state is derivable from the AOSS collection itself. What is live is whatever `faq_evidence_current` points at, discoverable via `GET /_cat/aliases`. What was previous is whichever versioned indexes still exist under the retention window, discoverable via `GET /_cat/indices/faq_evidence_v*`. Whether the last publish succeeded is implicit — if the alias advanced, it succeeded; if not, the candidate index is an orphan awaiting garbage collection. The high-water mark is the only thing that cannot be reconstructed from the AOSS collection, so it is the only thing that needs to persist elsewhere. That discipline keeps the publication state surface minimal and eliminates whole classes of state-divergence failure modes. if not, the candidate index is an orphan awaiting garbage collection. The high-water mark is the only thing that cannot be reconstructed from OpenSearch, so it is the only thing that needs to persist elsewhere. The variant set does not have an ingestion parameter. Variants are a static S3 artifact (API_11), updated by hand when the team decides to change them, loaded into memory at service boot. This section's pipeline has no awareness of them.

**Cross-Q&A links.** Authors curate contextual relationships between FAQ Q&As through a single-valued custom metadata field on the FAQ content model in COREx. The field's value is the COREx nodeId of the next linked Q&A in the chain, or empty when the Q&A is a chain tail. There is no separate authoring tool, no S3 artifact, and no external curation workflow — link information rides along with each Q&A through the same authoring path the rest of the FAQ content uses. The architecture's commitment is to the contract, not to the field name: the ingestion pipeline reads one custom metadata field on each FAQ Q&A whose value is interpretable as a single nodeId or empty/absent. Field name and the exact COREx custom-field type (Single Line Text holding the nodeId UUID is the obvious shape) are deferred to the team setting up the custom metadata model in COREx.

The ingestion pipeline materializes the chain at ingest, before embedding. For each Q&A node X in the freshly-built corpus, the pipeline reads X's "next" field, walks at most two hops, breaks cycles deterministically (stops on first revisit and emits `FAQLinkCycleBroken`), and stamps the resulting ordered list of linked parent_ids onto every child chunk of X under `linked_parent_ids`. A "next" pointer to a nodeId not in the published snapshot is an unresolved edge — log structured, emit `FAQLinkUnresolved`, treat that link as absent. If unresolved edges exceed `linked_unresolved_tolerance_pct` (default 5%, calibration-active under C-22) of total edges across the corpus, fail the build rather than publishing a corpus that is silently missing many links. A missing or empty custom metadata field is the same as "no chain": the chunks of that Q&A get `linked_parent_ids: []` and the build proceeds.

The runtime consumes `linked_parent_ids` purely as expansion metadata — it is not a retrieval axis and is not filtered or sorted against. Section 07 describes the post-rerank expansion query that consumes the field.

**The runtime read surface.** This is the contract handed downstream to Section 04. Runtime queries hit the `faq_evidence_current` alias (never a versioned index directly), apply a **hybrid query** that combines a `match` clause (BM25) on `text` and `title` with a `knn` clause on `embedding`, fuse the two legs through the `skywalker-faq-hybrid` search pipeline (`min_max` normalization, `arithmetic_mean` combination, weights set on the request), and apply a FAISS `efficient_filter` clause on `country`, `level`, and `role` inside the `knn` leg so only in-scope chunks compete in the vector search. The runtime over-retrieves (`k: 40` against a `size: 20` result, tunable) to absorb scope-filter selectivity. Post-rerank sibling expansion hits the same alias with one `terms` query on `parent_id` against the union of the winners' own parent_ids and the parent_ids in their `linked_parent_ids` chains. The query carries no scope filter; sibling expansion within an anchor's own parent inherits scope from the original retrieval, and linked-parent expansion is curatorial — author-asserted contextual relationships override scope filtering on the expansion query specifically (see §3 fixed decision ten and Section 07). This section does not own the query shapes themselves — Section 04 and Section 07 do — but it does own the guarantee that whatever version the alias currently points at is complete, coherent, scope-filterable, hybrid-searchable on both legs, link-chain-stamped on every chunk, and ready for those query shapes to succeed.

**IAM and SigV4.** AOSS is fronted by the standard `aoss` AWS service. The runtime's IAM execution role carries an AOSS data-access policy granting `aoss:CreateIndex`, `aoss:WriteDocument`, `aoss:UpdateIndex`, `aoss:DeleteDocument`, and `aoss:DescribeIndex` on `index/skywalker-faq-{stage}/faq_evidence*` for the ingestion job, and `aoss:DescribeIndex` plus the read action `aoss:ReadDocument` on the same resource pattern for the query path. SigV4 signing uses `service: "aoss"` and the collection's region. Network access is gated by an AOSS network policy: at launch the collection is private with a VPC endpoint placing access inside the query service's VPC; public access is not used.

**Embedding contract.** Every chunk written to the index is embedded through Amazon Bedrock's Cohere Embed v4 model (model ID `cohere.embed-v4:0`), invoked against the standard Bedrock runtime endpoint at `bedrock-runtime.<region>.amazonaws.com`. The output dimension is 1024, matching the `knn_vector` field dimension declared on the index. The ingestion path sets `input_type: search_document` on every embedding call because the text is being embedded for later retrieval; the live query path uses the same model with `input_type: search_query`. That asymmetry is important: Cohere Embed v4 produces distinct embeddings for "things to be searched" versus "the search itself," and mismatching the input type degrades retrieval quality even though the embeddings sit in the same vector space. The static variant set (API_11) is embedded at authoring time with `search_document` as well, since the variants are documents being matched against at query time.

The Bedrock invocation is a standard `InvokeModel` call carrying a JSON body with `texts` (array of strings being embedded), `input_type` (one of the two values above), and `embedding_types: ["float"]`. The response carries an `embeddings.float` array containing one 1024-float vector per input. Ingestion batches multiple children per call subject to Bedrock's per-request token budget; a single-chunk call is acceptable and simplifies failure handling. The service execution role carries `bedrock:InvokeModel` scoped to the Embed v4 model ARN in the region where Bedrock is invoked. Throttling during ingestion is treated as a hard abort rather than a partial-publish: if a `ThrottlingException` or a mid-batch failure occurs, the candidate index is not promoted and the next daily run retries from the prior published truth. Mixed-embedding corpora — where some children were embedded against one model version and others against another — are explicitly not produced. A model change would force coordinated regeneration of the variant S3 artifact, a rebuild of the evidence index, and a code-time change in the runtime, and is an architecture-class event rather than a configuration change.

### 3. Fixed decisions

The first fixed decision is that the Top 50 arm is owned because it needs control. This is not a philosophical preference for building more infrastructure. It is a targeted response to the project's asymmetry. The highest-value question set is exactly where black-box behavior is least acceptable, so the controlled arm remains in the architecture even though UKB exists.

The second fixed decision is that CoreX is the authoritative source of evidence content for this arm. The owned corpus is not manually authored answer text stored in some side table. It is built from the same source-of-truth documentation system that the organization already uses. That keeps the controlled arm tied to actual documentation rather than letting it drift into a separate content reality.

The third fixed decision is the trigger model. This pipeline is schedule-driven and runs daily. That decision is already anchored in the stated project goal of choosing simplicity somewhere in the system because the corpus is tiny and the project needs a fast build path. The design is not trying to build a complicated event-driven ingestion fabric for one to ten source documents.

The fourth fixed decision is rebuild posture. When the source set changes, the system does not perform fine-grained patch updates to the live FAQ corpus. It rebuilds the controlled FAQ corpus and republishes it as a whole. The size of the corpus makes that acceptable, and the simplicity dividend is real. When the entire evidence surface is tiny, correctness and explainability matter more than clever incremental mutation logic.

The fifth fixed decision is chunking posture. The ingestion pipeline runs chunking as two discrete, independently-testable steps rather than one monolithic pass.

The first step is **hierarchical chunking**. A dedicated `HierarchicalChunker` component reads the normalized CoreX snapshot and splits on structural hierarchy — headings, sections, logically coherent blocks — without regard for token count. Each hierarchical unit receives a freshly generated `parent_id` (UUID). This is purely a structural decision; no size enforcement happens here.

The second step is **semantic token-bound child chunking**. A dedicated `SemanticChildSplitter` component consumes each `HierarchicalChunk` and splits it into children whose length respects a **1000-token hard ceiling**. The splitter is semantic: it prefers paragraph boundaries, sentence boundaries, and structural breaks when producing children, and only resorts to mid-content splits when a single semantic unit exceeds the ceiling. Every child carries its parent's `parent_id`, a unique `chunk_id`, an integer `child_order` within the parent, the total `child_count`, and a `split_type` tag indicating how the child boundary was chosen (`paragraph_boundary` when a natural break was used, `size_forced` when the ceiling forced a mid-content cut, `other` for anything else). The 1000-token ceiling is driven by the reranker's practical budget: 20 candidates × ~1000 tokens fit comfortably in the Cohere Rerank 4 Pro 32K-token context window with room for the query and structural overhead.

Parents are **not** stored as separate documents. The single AOSS evidence index holds only children. At query time, after reranking, Skywalker recovers the parent context by issuing one sibling query against the evidence index filtering on the winners' `parent_id` values, then concatenating the returned children in `child_order` with separators chosen from `split_type`. This is the "children-only, reconstruct on read" pattern, detailed in Section 04 and API_06.

The two-step chunker design is deliberate: each step has one job, can be tested in isolation, and fails for understandable reasons. Combining them would create a chunker whose behavior depended on both structural semantics and token budgets simultaneously, which is much harder to reason about or debug.

The sixth fixed decision is that the FAQ arm remains retrieval-backed. Routing the query toward this arm does not collapse the rest of the path into a static lookup table. The evidence still comes from the FAQ corpus, is still subject to retrieval, and later still enters the common reranking surface alongside general-arm candidates when the runtime path calls for both arms.

The seventh fixed decision is storage topology. There is one owned AOSS surface: the evidence index inside the `skywalker-faq-{stage}` vector collection, fronted by the `skywalker-faq-hybrid` search pipeline and the `faq_evidence_current` alias. No variant index. No secondary storage surface inside this section. The routing variant set is a static S3 artifact (API_11) outside this section's ownership.

The eighth fixed decision is embedding discipline. Cohere Embed v4 is the single embedding model used across the system — for the evidence children embedded during ingestion, for the query vector computed at runtime, and for the pre-computed variant embeddings held in the static S3 artifact. A single live query embedding can interrogate both the AOSS evidence index and the in-memory variant vectors without vector-space drift. A model change is therefore an architecture-class coordinated event that affects ingestion, the S3 artifact, and the runtime — not a configuration change.

The ninth fixed decision is publication safety. A rebuild that has not completed does not become visible. The architectural requirement is clear: the system publishes a whole coherent FAQ evidence surface, not a partial one.

Concretely, atomicity is provided by the AOSS index alias (`faq_evidence_current` pointing at `faq_evidence_v42`) swapped via the OpenSearch `_aliases` API — which AOSS supports on vector collections — as the single atomic operation that makes a new version live. The daily ingestion job remembers what CoreX looked like last time through one SSM Parameter Store parameter at `/skywalker/ingestion/faq_evidence/last_snapshot_marker`. That parameter is the only persisted state the system keeps between runs; everything else is derivable from the AOSS collection. The job reads the parameter, fetches the current CoreX snapshot, and rebuilds only when the markers disagree. On successful publish, the alias swap happens first and the SSM update happens second — in that order so a failure between them produces at worst a redundant rebuild on the next run rather than a divergent live state. There is no revision counter and no optimistic-concurrency machinery because there is one scheduled writer. Rollback is an operator action that flips the alias back to the previous version and resets the parameter to the marker that produced that version; the runbook lives in API_07.

The tenth fixed decision is cross-Q&A linking semantics. Authors curate contextual relationships between FAQ Q&As as a directed, single-valued, depth-2-bounded chain. Each Q&A has at most one outgoing "next" pointer (a custom metadata field on the FAQ content model in COREx). Skywalker walks at most two hops from any anchor when materializing the chain, breaking cycles deterministically. The relationship is curatorial — it expresses contextual relevance asserted by the author, not a scope-applicability claim. The post-rerank expansion query carries no scope filter (Section 07 §3); a linked Q&A's `country`/`level`/`role` tags are not consulted at expansion time. This is a deliberate carve-out from the spirit of D-02 for author-curated links only and is recorded in Section 10 D-35. Linked items never enter the rerank pool (Section 07 §3 and Section 10 D-36); they ride along anchors that have already won rerank.

### 4. Alternatives considered or still live

The first rejected alternative is relying on UKB alone for the Top 50 question set. That path was considered and rejected for the same reason the owned arm exists at all. UKB is valuable, but its behavior is not controllable enough for the one question set where the system is explicitly measured and where the business value is concentrated.

The second rejected alternative is a webhook-first or event-first change trigger from CoreX. That path would optimize source-detection latency, but the corpus does not need sub-minute freshness and the project explicitly wants a fast, comprehensible implementation. Daily polling is therefore the correct launch posture for this corpus size and ownership model.

The third rejected alternative is incremental in-place mutation of the evidence corpus when a document changes. That approach is appropriate when the corpus is large enough that full republish becomes expensive or operationally risky. It is the wrong trade-off here. With one to ten source documents, incremental mutation adds complexity precisely where the system least needs it.

The fourth alternative was to treat CoreX fragments as atomic retrieval units and bypass chunking entirely. That remains a live simplification path for the future, but it is not the current baseline. Right now the system does not know enough about fragment quality, fragment length variance, or fragment coherence to assume that raw fragments are always safe retrieval units. The chunker stays in place until the corpus proves that assumption unnecessary.

The fifth rejected alternative is building a full variant-ingestion pipeline: daily polling, embedding, index publication, alias swap for a variant-routing surface. That was considered and explicitly rejected. The variant set changes on a human cadence, not a content-management-system cadence; it is 50 short question phrasings; pre-computing the embeddings once at authoring time and holding them in memory at service boot is simpler and faster than running any pipeline for them. A variant "pipeline" would be elaborate machinery for a tiny static artifact.

The sixth rejected alternative is collapsing the controlled FAQ arm into a direct lookup system after routing. That would make the arm faster on paper, but it would also change what the arm is. The project did not choose to maintain a small manual answer table. It chose to maintain a small owned retrieval corpus. That distinction is real, and the architecture should not dissolve it merely because the corpus is small.

A few alternatives remain live rather than rejected. If the corpus remains permanently tiny and CoreX fragments prove reliably answer-sized, the team can revisit whether the chunking pass should simplify toward one-fragment-one-chunk publication. If the Top 50 source set grows materially beyond the current one-to-ten-document regime, the full-rebuild posture can be revisited.

### 5. Assumptions inherited from upstream sections

This section inherits the system boundary fixed in Section 01 (Skywalker is the retrieval backend behind MCP, not the conversational layer) and the entry contract fixed in Section 02 (by the time a request reaches this arm, scope is already resolved and the request carries geography, level, and manager-versus-IC role). This section therefore builds a data plane rather than a dialogue manager and stores those applicability dimensions without owning their resolution.

Three further premises apply. The owned corpus is bounded to travel, events, and expense domains rather than all Amazon documentation. The controlled FAQ arm and the UKB general arm are separate subsystems, and this section supports only the FAQ evidence side. And the three current scoping dimensions are sufficient for launch; if later content models activate additional scope, the FAQ evidence schema can evolve, but geography, level, and manager-versus-IC role are what this corpus preserves now.

### 6. End-to-end data flow for this section

The CoreX evidence publication flow begins with a scheduled job on a daily cadence. On each run, the job reads the last published snapshot marker from SSM Parameter Store (`/skywalker/ingestion/faq_evidence/last_snapshot_marker`) and then requests the current authoritative FAQ source snapshot from CoreX. Because the corpus is small, the most important thing at this stage is not ultra-low detection latency. It is having a clean statement of current source truth.

Once the current snapshot is in hand, the pipeline compares it to the last published source state. If nothing material changed, the job records a no-op and exits without touching the live evidence surface. If the source state changed, the job does not try to surgically edit the current corpus in place. It builds a fresh candidate publication from the new source snapshot.

During that build, the pipeline normalizes the incoming source material into one canonical evidence snapshot. If CoreX already returned fragments, those fragments are preserved as the first structural units rather than discarded. If CoreX returned whole-document bodies, the pipeline derives its own internal fragment boundaries before chunking. In both cases, the goal is the same: establish a stable current evidence snapshot before embedding and publication.

The chunking stage runs as the two-step sequence fixed in §3. First, the `HierarchicalChunker` splits the normalized source snapshot on structural hierarchy — headings, sections, coherent blocks — and assigns a fresh `parent_id` UUID to each unit. Second, the `SemanticChildSplitter` consumes each hierarchical unit and produces children no larger than 1000 tokens, with semantic preference for paragraph and sentence boundaries. Every child carries its parent's `parent_id`, a unique `chunk_id`, `child_order`, `child_count`, and a `split_type` tag describing how the child boundary was chosen. Parents are not written anywhere; they are reconstructable at query time from the set of children that share a `parent_id`.

After chunking and before embedding, the pipeline runs the cross-Q&A chain materialization step. For each Q&A node X in the snapshot, it reads the value of X's "next" custom metadata field on the COREx FAQ content model. If empty or absent, X has no chain. Otherwise the pipeline walks the chain: it adds the next Q&A's `parent_id` to X's chain, then follows that Q&A's "next" field for one more hop, stopping when it has accumulated two hops, hits an empty field, hits a node already in the chain (cycle break — emit `FAQLinkCycleBroken` and stop), or hits a node not present in the current snapshot (unresolved edge — emit `FAQLinkUnresolved` and treat the link as absent). The resulting ordered list of zero, one, or two `parent_id` values is stamped onto every child chunk of X under `linked_parent_ids`. Whole-corpus validation runs at the end of this step: if more than 5% of edges across the corpus are unresolved (calibration-active under C-22, default `linked_unresolved_tolerance_pct = 0.05`), the build fails rather than publishing a corpus that is silently missing many links. At less than 5%, individual unresolved edges are logged and skipped.

After chunk production and chain materialization, the pipeline embeds the final children under Cohere Embed v4 and writes them into a new evidence publication target. The build is validated as a coherent whole before the live publication alias is advanced. However the implementation chooses to express the final cutover, the live evidence surface only moves once the new surface is complete.

Concretely, the pipeline builds `faq_evidence_v<N+1>` in full, writes every child with its embedding and metadata, validates the result, and then atomically advances `faq_evidence_current` via the AOSS `_aliases` API:

```
POST /_aliases
{
  "actions": [
    { "remove": { "index": "faq_evidence_v<N>",   "alias": "faq_evidence_current" }},
    { "add":    { "index": "faq_evidence_v<N+1>", "alias": "faq_evidence_current" }}
  ]
}
```

That single `_aliases` call is the one atomic publication primitive Skywalker uses. Immediately afterward — in that specific order, alias first then SSM second — the pipeline writes the new CoreX snapshot marker to `/skywalker/ingestion/faq_evidence/last_snapshot_marker` via `ssm:PutParameter` with `Overwrite: true`. If the SSM update fails after a successful alias swap, the live surface is already correct; the next daily run will observe `CoreX snapshot != SSM marker` and rebuild redundantly, which is benign waste rather than a correctness hazard. The opposite order (SSM first, alias second) would produce the actual hazard — a remembered marker advancing while the live surface did not — and is explicitly avoided.

The final step on a successful publish is garbage collection: the pipeline enumerates existing `faq_evidence_v*` indexes and deletes any beyond the retention window (launch default: keep the last three versions plus the current live one). Candidate indexes that were built but never promoted are caught by this same sweep as orphans.

Rollback is an operator action, not an automated path, and follows the same two-service pattern in the opposite direction. The operator first flips the alias back to the previous version via an `_aliases` call that removes `faq_evidence_v<N>` and re-adds `faq_evidence_v<N-1>`, then resets the SSM parameter to the snapshot marker that produced `v<N-1>` — a value recorded in the ingestion job's structured CloudWatch Logs at the time of the original publish — using `aws ssm put-parameter ... --type String --overwrite`. Skipping the SSM reset is recoverable because the next daily run would rebuild from the current CoreX snapshot; skipping the alias flip is not, because live traffic would continue to serve the bad version.

There is no parallel flow for variants. The variant set is not produced by this pipeline. It is a manually maintained S3 artifact (API_11) that changes only when the team decides it should. The ingestion pipeline has no awareness of it.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

The first failure rule is publication conservatism. If CoreX cannot be read, if the source snapshot is malformed, or if the pipeline cannot build a coherent new evidence surface, the current live FAQ evidence corpus remains authoritative. The system does not partially overwrite the live surface and it does not advance publication state to make the failure look smaller than it is. The next scheduled run retries from the prior published truth.

The second failure rule is that a candidate index whose alias was never swapped is an orphan. It must not be promoted on a future run; it should be cleaned up by the garbage-collection step. The architecture is unambiguous: the alias is the only authority for "what is live."

The third failure rule is that an SSM parameter update failure after a successful alias swap is a benign divergence, not a correctness hazard. The next run sees a stale marker, rebuilds redundantly, and converges. That is the trade-off the alias-first ordering buys us, and it is the correct trade-off.

The fourth failure rule is integrity behavior on the cross-Q&A link chain. A "next" pointer to a nodeId not present in the published snapshot is logged and treated as absent (`FAQLinkUnresolved`); a cycle in the chain is broken deterministically (`FAQLinkCycleBroken`); a missing custom metadata field on a Q&A is the same as "no chain." None of these conditions block a build at the per-edge level. Aggregate validation does: if more than `linked_unresolved_tolerance_pct` (default 5%) of edges across the whole corpus are unresolved, the build fails rather than publishing a corpus that is silently missing many links.

This section has no user-facing abstain behavior in the ordinary sense. It does not answer users. It publishes or declines to publish corpus state. User-facing abstain remains downstream and belongs to the retrieval and response path. The nearest equivalent here is a deliberate refusal to make incomplete state live.

Several non-goals matter because they keep this section from swallowing adjacent responsibilities. This section does not own the UKB general arm. It does not own final routing policy, reranking, or answer generation. It does not own conversational context. It does not own an authored-answer cache. It does not own real-time or near-real-time source propagation. It does not own a complex incremental update system for a corpus that is deliberately tiny. And it does not own the variant set, the routing gate, or any artifact that lives outside the AOSS collection. Those are not omissions. They are boundary decisions.

### 8. Calibration surfaces and what would cause re-litigation

The first calibration surface is poll cadence. Daily polling is the fixed baseline because the corpus is small and the build path values simplicity. Re-litigation would be justified only if the source corpus starts changing often enough that one-day staleness becomes operationally unacceptable or if the program shifts toward tighter freshness guarantees.

The second calibration surface is chunking behavior over CoreX fragments. The current baseline keeps the two-step chunker in place even when fragments are expected to be small. Re-litigation would be justified in either direction. If fragments almost always become exactly one child, the team can simplify. If fragments frequently explode into many children, then the upstream assumption that CoreX is handing back answer-sized fragments is weaker than expected and the content path needs a harder look.

The third calibration surface is full-rebuild posture. For the current one-to-ten-document regime, full republish is the cleanest design. Re-litigation would be justified if corpus size, rebuild time, or publication complexity grows enough that whole-corpus republish becomes a practical liability rather than a simplification.

The fourth calibration surface is the embedding contract itself. The current design uses Cohere Embed v4 across the evidence corpus, the query side, and the variant set. Re-litigation would be justified only by a deliberate model migration substantial enough to justify rebuilding the evidence index, regenerating the variant S3 artifact, and coordinating the cutover across both.

The fifth calibration surface is the BM25 analyzer choice for the `text` and `title` fields. The current design uses the standard analyzer because it gives reasonable English tokenization and is the safe default for a launch corpus that is mostly English travel/events/expense documentation. Re-litigation would be justified if reviewer feedback shows that domain-specific terms (policy codes, vendor names, currency abbreviations) are being tokenized in ways that hurt BM25 recall on real queries, in which case a custom analyzer with a synonyms list or a domain-aware tokenizer becomes worth its complexity. The change is index-rebuild-class because analyzer choice is fixed at mapping time.

The sixth calibration surface is FAISS HNSW build parameters. `m: 24` and `ef_construction: 128` are launch defaults chosen for a small corpus where recall matters more than build time; per-query `ef_search: 100` is the runtime knob and lives in SSM (`/skywalker/runtime/retrieval/ef_search`). Re-litigation of the build parameters is justified only if recall measurements on judged traffic show the chosen values leaving meaningful headroom on the table, and the change is rebuild-class.

The seventh calibration surface is the cross-Q&A link unresolved-edge tolerance, currently `linked_unresolved_tolerance_pct = 0.05` (calibration-active under Section 10 C-22). The right value is the one that fails noisy bad-link builds while tolerating the small handful of unresolved edges that depublish-races and authoring lag will routinely produce. Re-litigation is justified if production review shows that the tolerance is letting through builds with materially incomplete link chains, or in the opposite direction if it is failing builds for cosmetic edge-resolution noise.

### 9. Open questions, if any

The CoreX source contract is partially pinned against the CoreX API PDF in this directory: the ingestion job uses the `getContentNodes` GraphQL endpoint over HTTPS with SigV4-signed assume-role plus an API key, and reads `content`, `topics`, `geography`, `metadata`, `version`, and `nodeId` per node (with `returnTaxonomyValues: "LABEL"` so taxonomy IDs come back as labels rather than UUIDs). What remains open is the specific CoreX `modelSetId` values we filter against, the change marker we use for snapshot comparison (content hash computed on our side versus per-node version strings), and the exact `returnFieldVersions` choice. These are implementation-tuning decisions, not architecture-class.

The variant set authoring workflow is owned outside this section. What remains open is which internal tool holds the source list of variant phrasings, who edits it, and how the embedding step is invoked when the list changes. API_11 describes the S3 artifact contract but does not prescribe the authoring flow.

The final evidence-record schema exposed to the runtime is now pinned in API_06: `chunk_id`, `parent_id`, `child_order` (unindexed), `child_count` (unindexed), `split_type` (unindexed), `source_id`, `title`, `text`, `source_url`, `policy_links`, `country`, `level`, `role`, `corpus_version`, `linked_parent_ids` (unindexed), `embedding`. Parents are not stored as separate documents; they are reconstructed at query time from the children sharing a `parent_id`. What remains open is whether additional metadata fields from CoreX (for instance, `lastModifiedDate`, `createdBy`) should be carried on each child for auditability beyond their presence in the SSM high-water marker.

The exact name and COREx custom-field type for the cross-Q&A "next" pointer field on the FAQ content model are open at this layer. The architecture commits to "single-valued custom metadata field on each FAQ Q&A whose value is interpretable as a single nodeId or empty/absent." The team setting up the COREx custom metadata model picks the name and the field type (Single Line Text holding the nodeId UUID is the obvious shape).

Whether the Top 50 source set will remain in the one-to-ten-document range for long enough that the launch rebuild posture should be treated as durable rather than transitional is unchanged: the architecture is built for the current stated scale and will need another pass if the corpus grows materially.

### Closing position

The owned Top 50 FAQ arm is small on purpose and important on purpose. It exists because the project needs one place in the architecture where retrieval behavior is controlled rather than borrowed, and where the highest-value question set is not delegated to a black box. The result is a focused subsystem with one surface under this section's ownership: a controlled evidence corpus built from CoreX, embedded under Cohere Embed v4, chunked hierarchically and then token-bound, and stored in a single AOSS index inside the `skywalker-faq-{stage}` vector collection behind the `faq_evidence_current` alias and the `skywalker-faq-hybrid` search pipeline.

The section's decisions follow directly from that shape. CoreX is the evidence source. Daily polling is the launch trigger model. Rebuild-and-republish is the launch mutation model. Two-step chunking (hierarchical then semantic child-splitter with 1000-token ceiling) is the content shaping. The evidence index is a single AOSS surface behind `faq_evidence_current`, with FAISS as the k-NN engine and a search pipeline binding for hybrid (BM25 + vector) retrieval. Partial publication never becomes live. The arm remains retrieval-backed rather than collapsing into a static answer table. And the variant set that supports routing is deliberately out of scope for this pipeline — it is a static S3 artifact (API_11), not an ingestion product.

Later sections will decide how the runtime path uses this surface, how the FAQ arm competes with the UKB arm, and how the reranker sees the final candidate set. This section's job was earlier in the lifecycle and just as load-bearing. It defines how the owned FAQ evidence corpus becomes real, how it stays coherent over time, and how the runtime gets a controlled surface worth routing into at all.
