## Section 06. UKB General Retrieval Integration

Section 04 fixed the online orchestration layer. It established when the runtime stays inside the controlled Top 50 FAQ arm, when it fans out to both arms, and how those arms must converge before a common reranking surface can do honest work. This section now drills into the general arm itself: the UKB integration.

That integration matters because it is where Skywalker deliberately stops owning the whole retrieval stack. The FAQ arm is controlled end to end. The UKB arm is not. Skywalker does not control UKB's underlying indexing strategy, retrieval internals, or any internal ranking logic UKB may apply before returning results. At the same time, the general arm is not optional noise around the edges of the architecture. It is the broad-coverage retrieval surface for the travel, events, and expense domains outside the tightly controlled Top 50 FAQ path. If the integration layer is vague, the entire two-arm design becomes hard to reason about because every weakness in the general path gets blurred together with every weakness in routing, reranking, and downstream answer construction.

The purpose of this section is therefore very narrow and very practical. It defines what Skywalker is responsible for when it invokes UKB, what request shape must cross that boundary, what result shape must come back to remain useful inside Skywalker, what parts of the behavior remain black-box from our side, and which calibration surfaces would justify reopening the integration design later. The goal is not to pretend UKB is part of our owned retrieval pipeline. The goal is to make the seam with UKB explicit enough that the seam itself can be evaluated, debugged, and trusted.


### 1. What this section owns

This section owns the adapter boundary between Skywalker's runtime orchestration and the UKB general retrieval service. In practical terms, that means it owns four things.

The first owned concern is outbound request construction. By the time a request reaches this section, Skywalker already has a canonical query, a resolved identity scope, and a route decision saying the UKB arm should participate. This section owns the transformation of that internal runtime object into the request UKB will actually receive.

The second owned concern is invocation behavior. UKB is not a local library call. It is an external service boundary reached through MCP. This section therefore owns the mechanics that matter at that boundary: when the UKB call is made, how it participates in concurrent dual-arm execution, what latency budget applies to it, and how a timeout or service failure is represented back to the orchestration layer.

The third owned concern is inbound result normalization. UKB's native result shape is not assumed to be the same as Skywalker's owned FAQ retrieval shape. This section owns the translation from whatever UKB returns into the common candidate envelope that Section 04 already established as the handoff contract into the shared reranking surface.

The fourth owned concern is provenance preservation. Because UKB is a black-box service from Skywalker's perspective, the integration layer has to be especially disciplined about preserving what it did receive. It needs to carry through enough route metadata, source metadata, arm identity, and raw response structure that the rest of the pipeline can still understand what happened on the request and that later evaluation can distinguish a weakness in UKB retrieval from a weakness somewhere else.

This section does not own UKB's internal retrieval behavior. It does not own how UKB chunks content, scores content, or indexes content. It does not own the system-wide decision about whether the request is FAQ-only or dual-arm; Section 04 already fixed that. It does not own the final reranking logic, final abstain logic, or agent-side response behavior in Slack or QuickSuite. It also does not own PAPI resolution, alias or employee-ID handling, or any other upstream scoping work. If the request reaches this section, those things are already done.

That division of responsibility is important because UKB is valuable here precisely by not being another owned subsystem we have to build from first principles. Skywalker is supposed to benefit from UKB, not quietly recreate UKB inside its own service boundary.

### 2. Inputs, outputs, and contracts

The primary input to this section is the canonical scoped request produced by Section 02 and passed into the online decision layer described in Section 04. At minimum, that request carries the original user query, the resolved location, the resolved level, and the resolved role dimension that currently distinguishes manager from individual contributor. It also carries request-level trace context and the route state that explains why the UKB arm is being invoked at all.

The second input is the route decision itself. The UKB integration is not allowed to decide on its own whether it should be called. It is called because the runtime orchestration layer already decided the request belongs on the dual-arm path or because an FAQ-only attempt failed and the route was explicitly widened to the general arm as a rescue. That distinction matters because later telemetry and later evaluation need to know whether UKB participated as part of the normal route or as part of a rescue path.

The third input is the current publication and configuration state that governs UKB use. That includes the current timeout budget, the current candidate budget, and any domain restrictions or request-shaping rules that the active integration contract requires. These are part of the integration contract because a general retrieval arm that can roam without discipline will quietly expand the effective scope of the overall system.

The outbound contract from Skywalker to UKB is conceptually simple even though the final wire shape is still open. Skywalker needs to send a request that expresses the user query plus the three scope dimensions that matter for correctness: location, level, and manager versus individual contributor. Those dimensions are not decorative. They are the same dimensions that make answers correct on the controlled FAQ arm, and they are just as important here because the UKB arm is still answering within the same business domain. A broad retrieval surface that ignores scope would return plausible-looking but wrong-shaped evidence.

The outbound contract also needs to carry the fact that Skywalker is operating within a defined subject area. The presence of UKB does not expand Skywalker's product boundary to arbitrary internal knowledge. For Skywalker's purposes, the general arm is the broad retrieval path for travel, events, and expense questions that are not captured cleanly by the Top 50 FAQ route. The integration therefore needs some way, whether through explicit filtering, tool selection, or other request shaping, to keep the general arm aligned with the business domain the system is actually supposed to serve.

The inbound contract from UKB is similarly simple in principle even if the final field list is not yet locked. Skywalker needs rerankable evidence objects, not an opaque terminal answer. For each result, the integration needs text or passage content that can later be compared in a common reranking surface, plus whatever source metadata UKB can provide, such as title, URL, document identifier, or service-native rank information. If a field is unavailable, the integration should preserve that absence honestly rather than synthesizing false symmetry.

The output of this section is a UKB result package already shaped for the rest of Skywalker. That package contains a normalized list of UKB-origin candidates in the common runtime schema, a per-request UKB status record that says whether the call succeeded, failed, timed out, or returned no usable evidence, and a provenance object that preserves the raw UKB response or a lossy-minimal representation of it for later audit. This section's output is therefore not "the answer from UKB." It is the general-arm evidence package that the rest of Skywalker can reason about.

The concrete shape of the call Skywalker makes into UKB is explicit enough to be worth stating here rather than hidden behind a pointer. The tool is `retrieve`, invoked via the standard MCP `tools/call` method. The request body contains `arguments` with `query` (the user's text), `maxResults` (the per-arm candidate budget, launch default 10, SSM-backed as a calibration surface), `targetUser` (a two-field object with `LOGIN` set to the resolved employee alias and `PERSON_ID` set to the PAPI-resolved person UUID — populated so UKB applies its native personalization across `countryCode`, `stateCode`, `buildingCode`, `badgeColor`, `jobLevel`, and `payRateType`), and `additionalFilters` (left empty at launch). Five headers accompany every request: `x-acting-user` carrying a JSON-encoded `{"LOGIN": "<alias>"}` payload, `x-target-user` optionally carrying the same shape (omitted here because `targetUser` is already in the body and duplicating the alias must match exactly or UKB errors), `x-atoz-person-id` carrying the Person ID, `x-atoz-token-audience-type` carrying the AtoZ persona, and `x-amzn-transitive-authentication-token` carrying the caller's TA token. The request is transported as Streamable HTTP per the MCP spec and signed with SigV4 using `service: "execute-api"` and `region: "us-west-2"` against temporary credentials obtained by assuming the cross-account `kbs-mcp-role_{stage}_{client_id}` role that UKB issues during onboarding; our service's IAM policy carries `sts:AssumeRole` on that role ARN and nothing broader.

The response arrives as a standard MCP tool result whose `content[]` array contains entries of `type: "resource"`. Each resource carries `uri`, `mimeType: "text/plain"`, `text`, `name`, `title`, `annotations` (with `audience`, `priority`, and `lastModified`), and `_meta.sourceUrl`. The V1 response deliberately does not include `filters` or `nextToken` in `_meta` (unlike V0), so Skywalker does not pretend those fields exist. Item-level normalization into the common runtime envelope follows a fixed mapping: `candidate_id` is a generated runtime UUID, `source_arm` is `"UKB"`, `source_id` is `resource.uri`, `title` is `resource.title` with `resource.name` as the fallback, `text` is `resource.text`, `source_url` is `resource._meta.sourceUrl`, `policy_links` is empty (UKB does not surface a separate policy-link metadata field at the content level; any policy URLs are embedded in the text), `arm_local_rank` is the positional index in the `content[]` array, and `rerank_score` is populated later by the common reranker. Skywalker does not fabricate an arm-local score for UKB candidates — positional order is sufficient as arm-local rank because arm-local scores are not cross-arm comparable anyway.

Failure handling follows the rules already stated in this section's boundary principles and has concrete HTTP-level behavior on the wire. A 403 Forbidden on SigV4 is treated as IAM misconfiguration — alert and fail the request without retry because retrying will not fix a policy mismatch. A 504 Gateway Timeout is retried once with exponential backoff per UKB's documented recommendation; if the retry also fails, the arm is treated as failed and the Section 04 one-arm survival rule takes over. A 429 Too Many Requests is respected as a rate-limit signal — the arm aborts for this request rather than retrying inline and overwhelming the shared throughput budget. Any other 5xx or transport failure classifies as arm failure. An empty but structurally valid `content: []` response is a retrieval miss rather than a fault, and the empty UKB candidate set is handed to the reranker for the unified pool to stand or fall on its own merits.

The stage-specific endpoints follow one URL pattern: `https://api.us-west-2.{stage}.knowledge.pxt.amazon.dev/iam/v1/mcp`, with `{stage}` taking values `alpha`, `beta`, `gamma`, `preprod`, or `prod`. `us-west-2` is the only region at V1. Gamma does not carry production content. PreProd is the pre-production staging environment with a production content mirror. The UKB support path — Sev-2 for critical issues, resolver group `AET ML Engineering Seattle`, CTI `PXT/AETContentOptimization/Support` — is an operational dependency rather than a contract detail, but it is worth naming here so the integration's operational surface is not a surprise when the first production incident lands.

### 3. Fixed decisions

The first fixed decision is that UKB is the general retrieval arm, not the controlled high-accuracy arm. That sounds obvious, but it has architectural consequences. It means UKB exists to provide coverage, reach, and breadth across the travel, events, and expense domain where Skywalker does not want to own an end-to-end retrieval pipeline. It does not exist to displace the controlled Top 50 FAQ subsystem on the questions where the system is most tightly judged.

The second fixed decision is that the UKB arm is invoked only when the runtime route calls for it. On the normal path, that means the UKB arm is active on non-match queries that enter the dual-arm route. It may also be activated on an FAQ-rescue path when a request was initially allowed to stay FAQ-only but the controlled arm could not return usable evidence. What is fixed here is not the telemetry naming of those routes. What is fixed is that UKB participation is a route consequence, not a free-floating integration decision made inside this adapter.

The third fixed decision is that Skywalker treats UKB as a black-box retrieval surface. That does not mean the system is blind. It means the system is disciplined about where its visibility stops. Skywalker may know what request it sent and what results it got back. It does not claim to know why UKB internally ranked those results the way it did, how UKB internally indexed the source corpus, or how UKB internally reasons about its own confidence. That boundary keeps later diagnosis honest.

The fourth fixed decision is that UKB results are consumed as evidence candidates rather than as terminal answer objects. The system-wide design depends on a common reranking surface and on downstream answer construction that still has access to comparable evidence from whichever arm participated. For Skywalker's purposes, UKB is not the final speaker. It is a retrieval contributor. Even if UKB exposes richer response objects, the integration's job is to extract the evidence-bearing part that belongs in Skywalker's common runtime pipeline.

The fifth fixed decision is that the three scope dimensions remain load-bearing on the UKB arm. Location, level, and manager versus individual contributor are not only FAQ concerns. They shape correctness across the whole product. This section therefore treats scope-aware UKB invocation as fixed architecture even though the exact mechanics of expressing that scope to UKB are still open.

The sixth fixed decision is that raw UKB ranking signals are not compared directly with the owned FAQ arm's retrieval signals. The common reranker exists precisely so Skywalker does not have to pretend these surfaces are naturally score-compatible. UKB-native rank or confidence data may be preserved as provenance, but it is not the common decision language of the runtime.

The seventh fixed decision is that the integration remains stateless across requests. UKB is called based on the current scoped request. It is not fed hidden conversational history from Skywalker and it does not change retrieval behavior because the same user asked a different question five turns ago. That boundary stays consistent with the rest of the system.

The eighth fixed decision is that Skywalker does not use UKB to quietly broaden its product scope. UKB may expose a much wider documentation universe, but Skywalker remains a system for travel, events, and expenses. The integration layer has to preserve that boundary rather than letting the general arm turn into an all-purpose internal search escape hatch.

### 4. Alternatives considered or still live

The first major alternative was owning the general retrieval pipeline ourselves for the broader non-FAQ domain. That alternative was viable before UKB became available, but the availability of UKB changes the build-vs-integrate calculation materially. Skywalker already needs to own a tightly controlled retrieval path for the Top 50 highest-value questions. Owning a second broad retrieval stack for the rest of the same domain would create much more ingestion, indexing, and maintenance work without creating the same level of incremental control benefit. That alternative is therefore rejected under the current architecture.

The second major alternative was an all-UKB design where Skywalker would simply become a thin wrapper around UKB plus some identity shaping. That alternative is also rejected. The reason is not that UKB is bad. The reason is that the place where Skywalker is most strongly judged is the place where complete black-box dependence is least acceptable. The top-value, top-frequency question set is the exact place where the system needs direct ownership and direct control over the retrieval surface.

A third alternative, still technically live but not the current design, is to call UKB on every request even when the routing gate signal is strong enough to permit FAQ-only routing. The argument for that alternative is that UKB might still surface useful competing evidence and the reranker could decide whether it mattered. The argument against it is that it weakens the semantic meaning of the FAQ-only route and blurs a very intentional distinction in the runtime architecture. The current system keeps the FAQ-only path real rather than symbolic. Re-litigation would require evidence that the stronger path consistently benefits from UKB competition even when the query is plainly inside the Top 50 set.

A fourth alternative concerns how much structure Skywalker should send to UKB. One option is a minimal request that sends only query text and relies on UKB's own retrieval intelligence. The competing option is a strongly scoped request that attempts to inject or enforce location, level, role, and domain restrictions at the UKB boundary. This alternative remains live because the exact UKB contract is still open. The architecture, however, is clear about the direction: if UKB can accept structured scope, Skywalker should use it.

A fifth alternative concerns how Skywalker should treat UKB output. One possibility is to consume UKB as if it were already an answering system and simply forward its phrased results downstream. The other is the current architecture, which treats UKB output as a general-arm evidence source that still participates in Skywalker's own common reranking and later answer construction. The current architecture rejects the "just forward UKB answers" posture because it would create a structurally different pipeline for one arm and make later convergence much harder to justify.

A sixth alternative, also still live, concerns domain narrowing. If UKB can support strong domain restrictions for travel, events, and expenses, Skywalker should use them. If it cannot, Skywalker may need to express domain scope indirectly through request shaping and rely more heavily on later reranking and downstream answerability thresholds. The architecture prefers the former, but the final choice depends on the actual UKB surface rather than on wishful design.

### 5. Assumptions inherited from upstream sections

This section inherits the system boundary from Section 01 (Skywalker is the retrieval backend behind MCP, not a conversational planner that is free to improvise with hidden tools) and the entry contract from Section 02 (by the time UKB is called, the request already carries a stable and resolved location, level, and role shape — the UKB arm does not discover, infer, or repair missing scope). It also inherits the controlled FAQ arm from Section 03 as a real retrieval subsystem rather than a cache-like answer layer, which matters because the two-arm design depends on both arms contributing evidence rather than one arm contributing a prewritten answer.

It also inherits the runtime route policy from Section 04: a strong routing-gate signal means FAQ-only unless the FAQ arm later fails and the route is explicitly widened, and a weak signal means dual-arm. The UKB adapter is not allowed to rewrite that rule locally. Two deeper premises from the earlier sections apply as well. A context-free answer is not neutral but wrong-shaped, which is why the UKB integration cannot be a loose free-text passthrough even if the underlying service would allow it. And there is a common reranking surface after arm convergence, so this section's job is to produce rerankable UKB evidence rather than to act as a final answer channel that bypasses the rest of the runtime.

### 6. End-to-end data flow for this section

The end-to-end flow begins only after the route state says the UKB arm should run. That may happen because the request failed to cross the FAQ-routing threshold and entered the normal dual-arm path, or because a request that initially stayed FAQ-only later widened as part of a rescue path. In either case, the integration begins from an already scoped request and an already decided route reason.

The first operational step is request packaging. The integration adapter takes the canonical query text and the resolved scope dimensions and builds the UKB-facing request object. If the final UKB contract supports structured filters or fields for location, level, and role, they belong here. If the final contract instead requires those constraints to be expressed through a narrower request surface, the adapter still owns that shaping. What matters architecturally is that scope and domain are applied before the call, not as an afterthought after results return.

The second operational step is outbound execution. On the normal dual-arm path, the UKB call is launched concurrently with the controlled FAQ retrieval call. Neither arm should block the other from starting. If the UKB call is occurring as part of an FAQ rescue path instead, it runs because the runtime explicitly widened into it after the FAQ-only attempt proved insufficient. The adapter attaches request trace context and begins the timed service call.

The third operational step is inbound response validation. The integration receives whatever UKB returns and classifies the service outcome before it begins normalization. A transport failure, timeout, or contract-shape failure is first classified as an arm failure. A technically successful call that returns no usable evidence is classified as a retrieval miss rather than an outage. That distinction matters because the later runtime and later evaluation need to know whether the general arm was unavailable or merely unhelpful on this query.

The fourth operational step is item-level normalization. Each UKB result is examined for the fields Skywalker later requires. The adapter extracts the evidence-bearing text payload, source title where present, source URL where present, UKB-side identifiers where present, and any metadata that helps the later pipeline understand what the result actually is. Missing fields are preserved as missing. The adapter is not supposed to fabricate completeness.

The fifth operational step is runtime candidate construction. Every usable UKB result becomes one common candidate object carrying a runtime candidate identifier, source arm set to UKB, the original UKB result identity if one exists, the evidence text payload, source metadata, arm-local rank information, and the current request's scope snapshot. The runtime candidate identifier only needs to be stable for the life of the request and the later review of that request. It does not need to become a permanent cross-request document identifier because the permanent identity still lives in whatever UKB natively exposes.

The sixth operational step is result packaging. The normalized candidate list, the UKB status record, and the raw provenance payload are returned to the orchestration layer that owns arm convergence. At that point, this section's work is done. It has turned a scoped runtime decision into a UKB call and turned a UKB response into a general-arm evidence package the rest of Skywalker can actually use.

One important property of this flow is that it preserves the difference between service state and evidence state. A call can be healthy and useless. A call can be unhealthy and therefore absent. A general-arm integration that does not preserve that distinction will make later route analysis much less trustworthy because every weak request will look like the same kind of weak request.

Another important property is that the flow makes the general arm legible under audit. When a result from UKB later reaches the reranker or the answer envelope, the system should still be able to say that it came from UKB, what UKB result it came from, what route state led to the call, and what scope context was active when Skywalker asked for it. That is the minimum level of honesty required when one arm is black-box from our side.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

The first failure rule is that UKB call failure is not automatically system failure. If the request is on the dual-arm path and the FAQ arm succeeds, the overall runtime can still continue with a narrower candidate pool under the single-arm fallback route. The UKB integration must therefore report its own failure cleanly without forcing the rest of the request into a false hard-stop.

The second failure rule is that a timeout is a real failure mode, not a soft inconvenience. The UKB arm is valuable because it brings breadth, but it is still part of a user-facing online path. If the service exceeds its allowed budget, the integration should return a timed-out arm state and let the orchestration layer proceed with whatever other evidence is available rather than silently waiting until the route has already become unusable.

The third failure rule is that malformed UKB output should be handled at the narrowest responsible layer. If the outer response is unusable, the adapter should fail the arm. If the outer response is valid but a subset of the returned items are malformed, the adapter should discard those items, preserve the good ones, and record that the arm returned partial results at item level. The goal is to protect the common candidate pool without pretending that bad data is acceptable.

The fourth failure rule is that an empty but successful UKB result set is a retrieval miss, not a service fault. This distinction is essential because later system behavior differs. A retrieval miss still leaves open the possibility that the FAQ arm can carry the request or that the later pipeline should abstain because evidence is thin. A service fault says something very different about the runtime state of the system.

Abstain behavior enters this section only indirectly. This section does not decide whether the final system should answer. It decides whether the UKB arm contributed usable evidence. The adapter must therefore be honest about three states: no call because the route did not require UKB, call attempted but no usable evidence returned, and call attempted with usable evidence returned. Later sections need those distinctions to decide whether abstention was caused by route choice, by evidence weakness, or by later ranking behavior.

The non-goals matter just as much. This section does not retry retrieval logic by inventing a second custom search strategy over UKB results. It does not rewrite the query into a different meaning-bearing question in hopes that UKB will do better. It does not perform final answer generation. It does not merge UKB results with FAQ results itself; Section 04 owns route convergence and Section 07 will own common reranking. It does not try to infer new scope from the result set. It does not make policy judgments about whether the returned evidence is substantively correct for the user. Its job is narrower and stricter than that.

A second non-goal is worth stating explicitly because black-box integrations often tempt people into it. This section does not normalize away the uncertainty of the black-box boundary. If UKB cannot or does not return a field, the adapter should not manufacture a pseudo-equivalent. The rest of the system needs to know what is truly known and what is simply absent.

### 8. Calibration surfaces and what would cause re-litigation

The first calibration surface is the UKB timeout budget. This surface matters because the general arm adds real value only if its breadth survives contact with the online latency budget. Re-litigation would be justified if UKB routinely times out under normal traffic or if the current timeout forces the runtime to abandon useful general-arm evidence too early.

The second calibration surface is the arm-level candidate budget. Skywalker has to decide how many UKB candidates are worth carrying forward into the common reranking surface. Too few and the general arm may fail to contribute meaningful breadth. Too many and the general arm may flood the shared candidate pool with weak or repetitive evidence. Re-litigation would be justified if the general arm is consistently either underrepresented or overrepresented after reranking.

The third calibration surface is scope expression at the UKB boundary. The architecture is fixed that location, level, and role matter. What is not fixed yet is the best way to express them through the actual UKB contract. Re-litigation would be justified if the initial scope-shaping strategy produces too many wrong-shaped general-arm candidates or, conversely, if it overconstrains the general arm so aggressively that the broad path stops being broad in any meaningful way.

The fourth calibration surface is domain restriction. Skywalker wants UKB for breadth inside the travel, events, and expense space, not for arbitrary sprawl. Re-litigation would be justified if the current domain-shaping strategy still allows a large amount of irrelevant general content into the candidate pool or if a too-strict strategy starts cutting off obviously useful evidence that belongs inside the product boundary.

The fifth calibration surface is item-level normalization discipline. The adapter currently plans to preserve missing fields honestly and normalize only what UKB actually returns. Re-litigation would be justified if later reranking or later answer assembly proves that the current common candidate envelope is too thin to carry the value of UKB results forward effectively. In that case the system may need a richer normalized envelope, not a looser one.

The sixth calibration surface is rescue behavior frequency. The architecture allows the UKB arm to appear not only on the normal dual-arm path but also on widened FAQ rescue paths. Re-litigation would be justified if rescue-path UKB calls become common enough to suggest that the FAQ-only route is too brittle in practice or that the FAQ-routing threshold is allowing FAQ-only commitment too often.

The seventh calibration surface is provenance granularity. The current design intentionally carries detailed arm-origin and route-state metadata because the black-box nature of UKB makes later diagnosis dependent on that honesty. Re-litigation would be justified only if the retained provenance is operationally expensive without improving diagnosis, which is possible but should be treated as something evidence would need to prove rather than something to assume by default.

### 9. Open questions, if any

The UKB MCP contract is now pinned against the UKB Onboarding PDF and captured in API_09 (tool name `retrieve`, stage-specific endpoints under `api.us-west-2.{stage}.knowledge.pxt.amazon.dev/iam/v1/mcp`, IAM SigV4 assume-role auth, request and response shapes spelled out). What remains open is the final `client_id` issued at onboarding, the exact cross-account role ARN per stage, and the launch `maxResults` value.

How strongly UKB can be scoped at request time is now answerable: UKB applies its own personalization filters (`countryCode`, `stateCode`, `buildingCode`, `badgeColor`, `jobLevel`, `payRateType`) when `targetUser` is populated, and exposes three filter categories (`exactFilters`, `partialFilters`, `aclFilters`) for explicit constraints. Skywalker's launch posture is to pass `targetUser` (letting UKB apply its native personalization) and leave `additionalFilters` empty. Re-litigation is possible if UKB's personalization proves inconsistent with the Section 02 scoping triple.

Domain restriction is not expressed through a dedicated UKB filter at this time. UKB covers Amazon's unified knowledge bases; narrowing to travel, events, and expense happens implicitly through query semantics and through the FAQ gate's routing preference. If later evaluation shows UKB returning too-broad results on the dual-arm path, `partialFilters` on a domain-like attribute become a live calibration lever.

The UKB result shape is pinned per API_09: `content[]` of `type: "resource"` with `uri`, `text`, `title`, `name`, `_meta.sourceUrl`, and `annotations.lastModified`. No stable score is returned at the item level beyond positional order, which is why Section 04 §3 decision four treats native scores as non-comparable across arms.

The candidate budget per UKB call is calibration-active (launch default 10). Same calibration cluster as Section 04 §8 surface three.

Provenance fidelity: API_09 keeps the full `content[]` item's metadata on the normalized candidate, which is enough for later diagnosis without retaining the entire raw MCP response. That resolves the earlier open question in this section's favor of "keep provenance light but structured."

Whether any class of query should deliberately bypass UKB even on the nominal dual-arm route remains a Section 04 re-litigation trigger, not a Section 06 decision.

### Closing position

The UKB integration is the controlled seam between Skywalker's owned runtime and a black-box general retrieval surface. It exists to provide broad travel, events, and expense coverage without turning Skywalker into the owner of a second end-to-end retrieval pipeline. That benefit is real only if the seam itself is explicit.

This section therefore fixes the seam. A scoped request reaches the general arm only when the route policy says it should. The adapter constructs a scope-aware UKB request, executes it under a real latency budget, classifies the service outcome honestly, normalizes whatever usable evidence returns into Skywalker's common candidate envelope, and preserves enough provenance that the rest of the pipeline and later human review can still understand what happened.

That is the right architectural posture for this arm. Skywalker does not need to know how UKB works internally in order to integrate with it well. It does need to know exactly what it is responsible for at the boundary, exactly what it expects to cross that boundary, and exactly how the black-box general arm re-enters the rest of the system as comparable runtime evidence rather than as a special case the pipeline can no longer reason about.
