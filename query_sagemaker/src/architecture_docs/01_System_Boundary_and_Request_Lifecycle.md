## Section 01. System Boundary and Request Lifecycle

This section freezes the runtime boundary for Skywalker before later sections dive into storage, retrieval mechanics, and integration detail. That boundary has to be explicit early. If it is left soft, later sections drift into the wrong system, assign ownership to the wrong component, or quietly make the retrieval backend responsible for conversational behavior that actually belongs to QuickSuite or to the Slack-side agent.

Skywalker is the retrieval backend behind an MCP surface for an internal Amazon assistant serving travel, events, and expense documentation. It is not the conversational brain. It does not own prompt strategy, dialogue flow, conversation memory, or user-facing prose generation. It owns identity-aware scoping, routing across the two retrieval arms, normalization of candidate evidence into a common form, reranking, abstain behavior at the retrieval layer, and the structured MCP response that downstream agents consume.

That distinction is operational, not cosmetic. The backend is intentionally deterministic compared with the agent layer above it. The quality target for Skywalker is not “sound helpful in chat.” The quality target is “return the right, properly scoped evidence for the employee asking, or abstain when the system cannot do so safely.”

### 1. What this section owns

This section owns the system boundary of the live runtime path. That means it defines what counts as part of Skywalker, what remains outside of Skywalker, how a request enters the system, how identity context is resolved before retrieval begins, how the two retrieval arms participate in one request, how results converge into a single reranking surface, and what kind of output Skywalker returns to its callers.

Skywalker owns the following runtime responsibilities. It exposes an MCP search surface. It accepts a user question plus one of the supported identity entry modes. It resolves identity-derived scope through PAPI when the caller did not already provide the required scope directly. It runs a hybrid routing gate — cosine similarity against a static in-memory variant set, with a Cohere Rerank v4 cross-encoder second stage on a dedicated SageMaker endpoint, invoked only on cosine-ambiguous cases — to decide whether a request is Top-50-shaped. It decides whether the request runs through the FAQ arm alone or through both arms concurrently. The FAQ arm itself runs hybrid retrieval (BM25 + FAISS cosine) against the AOSS evidence index, with the two legs fused by an OpenSearch search pipeline. It gathers candidates from the participating arm or arms, normalizes them into a common candidate shape, reranks them through Cohere Rerank 4 Pro on SageMaker, and returns a structured result bundle to the caller. It is also responsible for deciding when the retrieval layer should abstain rather than pretending the system has sufficient evidence.

Skywalker does not own the user-facing conversation. It does not manage multi-turn state. It does not decide how a final answer should be worded for Slack or for QuickSuite. It does not own prompt orchestration for those surfaces. It does not own the UKB internals, since UKB is an external MCP server that Skywalker calls. It does not own PAPI. It does not own the human review process used after launch to validate answer quality in production. It also does not own the evaluation harness, which is out of scope for this architecture series.

The result is a deliberate split of responsibilities. Skywalker is the scoped retrieval and evidence-selection layer. QuickSuite and the Slack-side agent are the conversational layers. PAPI is the identity enrichment dependency. UKB is the broad black-box retrieval dependency for the general arm. Later sections will define the owned FAQ ingest and storage path in detail, but this section fixes the fact that the owned FAQ path is still part of Skywalker while the agent behavior is not.

One architectural wrinkle belongs in this boundary statement rather than only in the QuickSuite section: all three production client paths reach Skywalker's MCP server through **Amazon MCP Gateway**, the internal paved path for hosting and consuming MCP servers across Amazon. Slack's IAM-fleet-hosted inline-agent orchestrator and the UAT inline-agent orchestrator use the gateway's CloudAuth-inbound route at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry}/{server}` with each orchestrator registered as a CloudAuth-modeled AAA application; QuickSuite uses the gateway's Federate-OAuth-inbound route at `https://api.mcp.asbx.aws.dev/federate/mcp/{registry}/{server}`. The gateway's endpoint is a public DNS name with auth enforced at the gateway — the architecture inherits a public-with-auth posture from the gateway's natural shape, with the closed set of supported auth shapes (CloudAuth, Federate OAuth, SigV4 for any future caller it suits) keeping the authorization boundary against any caller external to Amazon's auth platforms (see [API_13](done/API_13_Network_Fabric_and_Internet_Exposure_Posture.md)). There is no wrapper Lambda between any client and the Skywalker MCP server, and there is no task-oriented translation tool — every client reads the same `tools/list` and chooses among the three core tools directly. On the Slack and UAT paths, **CloudAuth OBO carries the calling orchestrator's AAA application identity to Skywalker**, and **TransitiveAuth carries the originating human end-user's identity** as a separate TA token alongside CloudAuth, so Skywalker reads the human alias server-side from the TA claims rather than from MCP tool arguments. On the QuickSuite path, MCP Gateway does not currently propagate originating end-user identity downstream on the Federate-inbound combination, so QuickSuite carries identity in MCP tool arguments through its own integration-specific channel. This consolidation onto one transport with one Bindle authorization surface is the simplification that removes the v2 design's wrapper-and-AgentCore-Gateway complexity. See [API_01](done/API_01_MCP_Server_Protocol.md) for the core MCP contract, [API_08](done/API_08_QuickSuite_MCP_Gateway_Integration.md) for the QuickSuite-specific MCP Gateway integration, and [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md) for the CloudAuth OBO + TransitiveAuth integration that handles identity propagation on the Slack and UAT paths. The system boundary remains Skywalker behind MCP; Skywalker's MCP server registers with MCP Gateway, and clients reach it through the gateway.

### 2. Inputs, outputs, and contracts

The MCP boundary exists so multiple callers can use one retrieval backend without each caller re-implementing policy-aware retrieval. The contract at this boundary therefore needs to reflect the real identity inputs the system can receive, not an idealized one-size-fits-all request.

Skywalker supports three conceptual search entry modes. The first takes a query plus an alias and uses that alias to resolve the employee’s scoped metadata through PAPI. This is the expected Slack-side entry mode. The second takes a query plus an employee identifier and resolves the same scoped metadata through PAPI. The third takes a query plus the already-resolved scoping fields and bypasses PAPI entirely. That third path exists because some callers will already know the scope they want to use, and the architecture should not force a needless PAPI round-trip when the caller intentionally supplies the decisive fields up front. QuickSuite enters through the alias mode because Federate's JWT carries the user's Midway login as the `sub` claim and nothing more — the same alias-only identity shape Slack provides.

The required scope for the current system is the scoping triple already established in the project definition: location, level, and type, where type means manager versus individual contributor. This section treats those three as the active correctness dimensions for retrieval. A response that ignores them is not “generic.” It is wrong-shaped for the employee asking.

The contract between the MCP surface and the rest of Skywalker is therefore simple in concept even though the concrete schema will be finalized later. A valid request entering the orchestration layer contains a query and either an identity token that can be resolved into scope or the resolved scope itself. A valid request may also carry request metadata needed by the calling surface, but those fields do not change retrieval semantics in this section.

The output contract matters just as much as the input contract because it keeps Skywalker from drifting into agent behavior. Skywalker does not return a fully authored final chat answer. It returns a structured evidence bundle. At minimum that bundle contains the selected result set in ranked order, routing metadata that explains which path produced the candidates, identifiers or links that let the caller refer back to source material, and a retrieval-layer signal indicating whether the system found answerable evidence or is abstaining. Later sections will pin down the final response schema, but the boundary decision is fixed here: Skywalker returns retrieval output for agents to consume, not polished conversational prose for end users to read directly.

The internal contract between routing and reranking is also fixed here at a conceptual level. Both the owned FAQ arm and the UKB arm may participate in one request, but neither is allowed to hand opaque arm-specific result objects straight into the reranker. Before reranking, candidates must be normalized into a common representation so the reranker scores the same kind of object regardless of source arm. The exact candidate schema is an open design item, but the existence of a common schema is not optional. Without it, the reranker becomes a fiction rather than a true common scoring surface.

### 3. Fixed decisions

Several decisions are already fixed strongly enough that later sections should treat them as architectural constraints rather than live debates.

The first fixed decision is that Skywalker is a two-arm retrieval system. One arm is the controlled Top 50 FAQ path. That arm exists because the highest-value, highest-accuracy questions are precisely the questions where the project cannot afford to surrender control to a black box. The other arm is the general path through UKB. UKB is valuable because it already exposes broad documentation retrieval through an MCP surface, which means Skywalker does not need to rebuild the general retrieval path from scratch for the broader travel, events, and expense domains.

The second fixed decision is that identity-derived scoping happens before retrieval, not after. PAPI is not a downstream enrichment step. It is the step that resolves the metadata needed to shape retrieval in the first place. If a caller supplied only an alias or employee identifier, Skywalker calls PAPI before it attempts similarity matching, retrieval, or reranking. Only the direct-scope entry mode bypasses PAPI.

The third fixed decision is the routing policy. The system runs a hybrid gate against a static variant set: cosine similarity first, with a Cohere Rerank v4 cross-encoder on a dedicated SageMaker endpoint as the second stage, invoked only when cosine is ambiguous. If the gate's verdict is confident enough that the query belongs to the Top 50 question set, the request goes to the FAQ arm only. Otherwise the system does not simply abandon the FAQ arm. It queries both the FAQ arm and the general UKB arm concurrently and lets both compete in the same reranking stage. That policy matters because the near-miss region is exactly where the FAQ content may still be useful even when the system cannot justify short-circuiting straight to the FAQ path.

The fourth fixed decision is that concurrency eliminates sequential arm latency on the non-FAQ-only path. When both arms participate, they run at the same time. The architecture therefore pays for parallel work but not for serial fan-out latency. Cost has already been established as a non-issue for this project, so the governing reason to keep or change this behavior later will be answer quality, not compute spend.

The fifth fixed decision is that the reranker is the common scoring surface. Arm-local scores are not treated as globally meaningful across the system. UKB may expose its own notion of relevance. The FAQ arm will have its own retrieval behavior. Skywalker does not try to normalize those source scores into one synthetic number before reranking. It uses the reranker as the place where heterogeneous candidates compete against the same query under one scoring model.

The sixth fixed decision is that the FAQ arm remains a retrieval system, not a cache of deterministic answer blobs. The corpus is small, and the system will poll CoreX daily and rebuild its vector representation on change, but the architecture still treats Top 50 handling as retrieval over a tightly controlled corpus rather than as a simple keyed lookup over authored answers. The hybrid routing gate decides whether a user's wording belongs to the Top 50 question set. Retrieval over the owned FAQ corpus still does the evidence work inside that arm.

The seventh fixed decision is that Skywalker remains stateless per request. Multi-turn continuity belongs to the model-facing agent layer, not to the retrieval backend. For Slack, that means the Slack-side agent and its model handle conversation history. For QuickSuite, QuickSuite's chat-agent runtime handles that. Skywalker benefits from that conversation work because a better agent can ask it a better-scoped follow-up query, but Skywalker does not store or reason over conversation state itself.

The eighth fixed decision is that the active correctness dimensions for scoped retrieval are location, level, and manager-versus-individual-contributor type. Those three are sufficient for the current architecture. If later corpus reality proves that another identity dimension is required, that will be a re-litigation event. It is not the active design now.

The ninth fixed decision is the system-level latency budget. The Skywalker retrieval pipeline itself — from MCP request acceptance through normalized evidence package return — targets a p95 of 250 to 450 milliseconds. The reranker call on the H100-hosted SageMaker endpoint is the dominant cost in this budget (250–350 ms p95 at 20K-token payloads per API_10); the remaining ~100 ms covers scope resolution, the concurrent arm calls, candidate normalization, and the post-rerank sibling expansion. The end-to-end user experience on the Slack surface, measured from user message receipt to final Slack reply delivery, targets under 4 seconds p95. These budgets are architectural constraints, not calibration surfaces. They determine why the dual-arm path uses concurrency rather than sequential fan-out, why the explicit-scope request mode avoids a redundant PAPI round-trip, and why agent-side generation latency matters as much as retrieval latency on the Slack path. The budgets do not remove the need for measurement. They set the target the measurement is judged against, and a sustained breach of either budget is an architecture-class event rather than a tuning inconvenience.

### 4. Alternatives considered or still live

The most important rejected alternative is the pure UKB-only design. That design would have made Skywalker a thin wrapper around UKB with identity scoping layered on top. It was rejected for a specific reason. The place where the program is most exposed is the Top 50 FAQ set, because that is the highest-value question set and the place where the team is actually trying to hit a concrete accuracy target. UKB is a black box. Skywalker does not control its retrieval behavior, chunking, ranking, or indexing internals. Relying on it for the exact set of questions where the team is judged most tightly would make the hardest part of the program the part with the least control.

A second alternative existed earlier in the program and is no longer the chosen path: building the general retrieval arm ourselves over the relevant travel, events, and expense documentation instead of using UKB. That alternative became less attractive once UKB became available as an MCP-accessible knowledge base already built on the retrieval substrate the team otherwise would have been assembling for the broad general arm. The current design therefore keeps ownership only where ownership buys the most value: the Top 50 path.

A third alternative is the always-query-both design. Under that design, every request would hit the FAQ arm and the UKB arm, and the reranker would sort out everything every time. That design is still logically possible, but it is not the active one. The current design keeps a stronger FAQ-only route for cases where the system is confident that the query belongs on that path. The point of the similarity stage is not only to admit FAQ candidates into the race. It is also to allow a genuine short-circuit when the request very clearly belongs to the owned Top 50 space.

A fourth alternative is the strict single-arm gate on every miss. Under that design, a non-FAQ query would skip the FAQ arm entirely and go to the general arm alone. That alternative was also not adopted. The current routing policy intentionally keeps the FAQ arm alive in the near-miss region because semantically adjacent Top 50 material may still be valuable to the reranker even when the system cannot justify FAQ-only routing.

A fifth alternative is server-side conversation state inside Skywalker. That design is not the current one. The reason is straightforward. A deterministic retrieval system is not where multi-turn meaning is best handled, and trying to make Skywalker do that work would pull it toward agent behavior instead of scoped retrieval. The agents already do conversation history well enough to own that responsibility. The backend should remain focused on scoped evidence retrieval.

A sixth alternative was aggressive PAPI caching inside Skywalker. That is not part of the active design. PAPI is treated as the first dependency after the MCP boundary when scope must be resolved, and the current architecture is favoring clarity over extra machinery at that point in the request path.

### 5. Assumptions inherited from upstream sections

This is the first architectural section in the series, so it does not inherit technical detail from prior subsystem chapters. It does, however, inherit several program premises from the project definition, and later sections should treat those premises as established unless and until they are explicitly revisited.

The first premise is domain scope. Skywalker is not a general retrieval system over all Amazon documentation. The domain is narrower and operationally clearer than that. The system serves travel, events, and expense documentation. Every later section should assume that scope unless a subsection is explicitly about a future expansion.

The second premise is that the owned FAQ corpus is small. At present it is effectively one document and is expected to remain very small in the near term, with a rough upper bound on the order of ten documents. That premise justifies several later simplifications, including daily polling and full rebuild on change rather than elaborate incremental mutation machinery inside the owned FAQ path.

The third premise is that CoreX is the source of owned FAQ content and UKB is the source of broad general retrieval. Those are different dependencies with different control surfaces, and later sections must not collapse them into one abstract “documentation source.”

The fourth premise is that Slack and QuickSuite both naturally provide an alias rather than a pre-resolved scope tuple. Slack supplies the user's Slack ID, which the bot resolves to the Amazon alias. QuickSuite supplies a Federate JWT whose `sub` claim is the user's Midway login (alias). Both surfaces therefore enter Skywalker through the alias-based MCP tool, and both require PAPI resolution before retrieval unless the caller chooses to use the explicit-scope short-circuit deliberately. The employee-ID and explicit-scope paths remain part of the core MCP contract because the boundary should support them for any future caller that holds those richer identifiers, even if neither launch client uses them as the default.

The fifth premise is that cost is not a design driver here. That does not mean waste is a virtue. It means quality, controllability, and correctness take precedence over optimization pressure that would otherwise push the architecture toward a thinner retrieval path.

### 6. End-to-end data flow for this section

The runtime data flow described here is intentionally end-to-end at the orchestration level. Later sections will explode individual steps into much more detail, but the boundary-setting version needs to be visible now so the later detail stays attached to one request story.

The flow begins when an external caller invokes Skywalker's MCP server through Amazon MCP Gateway. The caller is the Slack inline-agent orchestrator (CloudAuth inbound on `/ca/mcp/...` with the human alias propagated via TransitiveAuth), the UAT inline-agent orchestrator (CloudAuth inbound on `/ca/mcp/...` with the human alias propagated via TransitiveAuth and the scope triple supplied as MCP tool arguments), QuickSuite's chat-agent runtime (Federate OAuth inbound on `/federate/mcp/...` with identity in MCP tool arguments), or another approved future caller. The incoming request contains a user query and one of the supported identity entry modes, with identity material arriving via the per-route channel (TA claims on the CloudAuth paths, MCP tool arguments on the Federate path). At this point the request is still caller-shaped. Skywalker's first job is to turn it into a retrieval-shaped request.

If the caller supplied only an alias or employee identifier, Skywalker immediately calls PAPI. PAPI returns the scoped identity metadata needed by retrieval: location, level, and type. If the caller supplied the scoped fields directly, Skywalker skips this step and treats the caller-supplied scope as authoritative for the current request. Either way, retrieval does not begin until the request has the scoping triple attached to it.

Once the scope exists, Skywalker runs the hybrid routing gate against the in-memory variant set (loaded from S3 at service boot, per API_11): cosine similarity first, with a Cohere Rerank v4 cross-encoder check on the gate's dedicated SageMaker endpoint only when cosine is ambiguous. This is the first routing decision in the live path. The gate's verdict is not the answer and it is not the final confidence signal. It is a routing signal driven by control-plane thresholds in SSM.

If the gate's verdict is strong enough for FAQ-only routing, Skywalker sends the request only to the owned FAQ retrieval arm. That arm retrieves candidates from the Top 50 corpus using the already-resolved scope. The output of that arm is not allowed to pass straight through to the caller. It is converted into the common candidate form the reranker expects.

If the similarity result does not clear the FAQ-only threshold, Skywalker fans out to both arms concurrently. The owned FAQ arm receives the query and scope. The UKB arm receives the query and the same resolved scope through its own request shape. Both arms return candidate evidence. Skywalker normalizes both sets into the same candidate representation and merges them into one reranker input pool.

The reranker then scores the normalized candidates against the user query as one competition set. At that point the system no longer cares which arm a candidate came from except as route metadata. What matters is which candidate set best answers the query for the scoped user context.

After reranking, Skywalker makes the retrieval-layer answerability decision. If no candidate clears the system’s bottom answerability bar, Skywalker abstains. In practical terms that means it returns a structured “no answer” result rather than pretending the evidence is stronger than it is. If one or more candidates do clear that bar, Skywalker returns the ranked evidence bundle plus route metadata and source metadata to the caller.

The final step happens outside Skywalker. The caller-side agent takes the structured retrieval output and turns it into the user-facing response. That may include conversational context, follow-up handling, or a surface-specific answer style. Those behaviors are downstream of Skywalker and should not be confused with the retrieval backend itself.

The runtime path can therefore be summarized as one deterministic progression. Caller request enters MCP. Scope is resolved. FAQ similarity is computed. One arm or both arms retrieve. Candidates are normalized. The reranker chooses across the combined set. Skywalker either returns a structured evidence bundle or abstains. The agent consumes that output and handles the conversational layer.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

Failure behavior at the system-boundary level should preserve the contract that answers are scoped correctly or not returned as though they were safe.

The first principle is fail-closed scoping. If a request arrives without direct scope and PAPI cannot resolve the identity needed for retrieval, Skywalker should not continue into retrieval as though a generic answer were acceptable. In this architecture, it is not. The system either receives the scoping triple directly or obtains it through PAPI. If neither happens, the request cannot be safely answered by the retrieval backend.

The second principle is single-arm fallback when the routing path permits it. If the system is on the dual-arm path and one arm fails while the other still returns candidates, Skywalker should continue with the surviving arm and mark the route accordingly in the structured output. The reranker still has a coherent candidate pool, just a narrower one. If the request was on the FAQ-only route and the FAQ arm fails, the safer fallback behavior is to attempt the general arm rather than convert an arm outage into a guaranteed no-answer outcome. Routing is a quality policy. It is not a reason to waste a viable fallback path.

The third principle is that reranker failure is a reranker-failure fallback event, not automatic total failure. The boundary-level contract here should allow Skywalker to return the best available normalized retrieval set in retrieval order with a reranker-failure fallback flag if reranking is unavailable. Later sections will decide how much of that fallback set is allowed through, but the architecture should not assume that the only two states are “reranked perfectly” and “nothing returned.”

Abstain behavior is distinct from component failure. A request may execute successfully and still produce no answerable evidence. That is not a system outage. It is the system doing the correct thing when it cannot defend a grounded answer. Skywalker should therefore distinguish between operational failure and evidence insufficiency. In one case the path broke. In the other case the path ran and decided not to overclaim.

The non-goals that matter here are equally important because they keep later sections from smearing extra responsibilities onto the wrong component. Skywalker does not own conversational memory. It does not own prompt engineering for Slack or QuickSuite. It does not own the final wording of the answer shown to users. It does not own the evaluation harness. It also does not own human-in-the-loop runtime review, even though subject-matter-expert review after launch is in scope for the overall program and remains an important operational practice. That review process matters, but it is not part of Skywalker’s runtime ownership boundary.

### 8. Calibration surfaces and what would cause re-litigation

This section sets several architectural shapes, but not every runtime knob is numerically fixed yet. The right way to handle those variables is to name them now, describe what job each one does, and define what evidence would justify changing them later.

The first calibration surface is the routing-gate thresholds. The hybrid gate (cosine + contingent Rerank 3.5) uses three control-plane values held in SSM: the cosine-high threshold, the cosine-low threshold, and the Rerank 3.5 floor. These decide when a query is similar enough to the Top 50 question set that Skywalker should short-circuit to the FAQ arm alone. They are not the same thing as the bottom answerability threshold later in the path. They exist to protect the FAQ-only route from becoming over-eager. They should be derived from real traffic and human review rather than guessed. Strong reasons to re-litigate: repeated evidence that traffic taking the FAQ-only route would have been better served by the dual-arm path, or the opposite.

The second calibration surface is the bottom answerability or abstain threshold after reranking. That threshold answers a different question. It does not ask whether a query belongs to the Top 50 set. It asks whether the evidence that survived retrieval and reranking is strong enough to justify returning a non-abstaining result. It therefore deserves its own calibration method and its own review triggers. A strong reason to re-litigate it would be a pattern of obviously weak evidence being returned as though answerable or, in the opposite direction, too many good requests being pushed into abstain when the evidence was actually adequate.

The third calibration surface is candidate volume into reranking. The architecture fixes the fact that the reranker is the common scoring surface, but it does not yet freeze the exact candidate budget that should be passed into that stage from the participating arms. That is a later tuning decision, not a boundary debate. The thing fixed here is that both arms must be allowed to compete in one reranker pool on the non-FAQ-only path.

The fourth calibration surface is the direct-scope trust model. The direct-scope request mode exists and is fixed. What remains open is whether Skywalker will always trust caller-supplied scope as authoritative or whether some later version will validate or audit it differently. That is not a routing threshold, but it is a calibration surface in the broader sense because it affects how the MCP contract behaves under real usage.

Re-litigation should not happen because the team has a vague feeling that something might be off. It should happen because the system’s actual behavior produces clear evidence that a current decision is mis-shaping the request path. Threshold drift, repeated disagreement between routed path and downstream evidence quality, or persistent abstain misfires are the kind of signals that justify reopening a decision. Mere discomfort is not.

### 9. Open questions, if any

The common candidate schema used between retrieval arms and the reranker is now frozen at the field level in API_01; what remains open is whether later evaluation reveals that important arm-specific distinctions are being lost in normalization and whether the schema needs additional optional fields for QuickSuite or Slack rendering (Section 07 §8 calibration surface five tracks this).

The trust posture for direct caller-supplied scope remains open. The architecture already allows a bypass-PAPI path through the `skywalker.search.by_explicit_scope` tool. What has not been finalized is whether the system will always trust that path absolutely or whether there will be later validation or audit hooks around it.

The abstain reason vocabulary is pinned to two values at launch (`NO_USABLE_EVIDENCE`, `EVIDENCE_TOO_WEAK_AFTER_RERANK`). Whether QuickSuite's chat-agent runtime and the Slack application produce meaningfully different user-facing abstain messages from these two classes is a staging-quality question (Section 10 C-02) rather than a contract question. Earlier drafts carried four abstain reasons; the top-two-margin-with-disagreement and single-arm-fallback-thinness branches were dropped because they either required LLM-style semantic judgment Skywalker cannot provide or conflated arm identity with answerability.

The launch transport posture for all three production client paths is **Amazon MCP Gateway**, on its SigV4-inbound route for Slack and UAT (`/mcp/{registry}/{server}`) and on its Federate-OAuth-inbound route for QuickSuite (`/federate/mcp/{registry}/{server}`). MCP Gateway validates inbound auth, performs Bindle-based authorization on `MCPGateway::{skywalker-server-id}`, and converts inbound auth to CloudAuth on the outbound leg to Skywalker's MCP server. There is no wrapper Lambda and no AgentCore Gateway in any production path. The v2 architecture's wrapper-and-AgentCore-Gateway design for QuickSuite is preserved in git history but is not the active architecture. See API_08 for the QuickSuite-side detail and Sections 05 and 08 for the UAT and Slack details.

Whether Cohere Rerank 4 Pro on SageMaker `ml.p5.4xlarge` consistently hits the 350 ms p95 timeout on Skywalker's 20K-token payloads is an empirical question to verify during Phase 4 shakedown. Launch estimates from public research put p50 at 180–300 ms and p95 in the 250–350 ms band, but no Cohere-published per-instance benchmark exists. If measured p95 exceeds 350 ms at production volume, either the timeout grows (relitigating the Skywalker p95 budget above) or the reranker pool shrinks. See API_10.

### Closing position

The architecture described in this section is intentionally narrow and load-bearing. Skywalker is the identity-aware retrieval backend behind MCP. It resolves scope before retrieval, uses a controlled Top 50 arm plus a UKB general arm, routes strongly matched Top 50 requests to FAQ-only retrieval, routes all other requests to both arms concurrently, normalizes candidates into a common representation, reranks them as one set, and returns structured retrieval output for downstream agents to turn into user-facing responses.

That is the system boundary for the rest of the series. Later sections are allowed to elaborate it. They are not allowed to drift away from it silently.
