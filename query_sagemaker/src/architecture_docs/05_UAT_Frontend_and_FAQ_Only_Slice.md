## Section 05. UAT Frontend and FAQ-Only Slice

Sections 01 through 04 fixed the production architecture: the system boundary, the MCP entry contract and identity scoping, the controlled Top 50 FAQ corpus, and the online query pipeline that decides between FAQ-only and dual-arm retrieval. This section steps deliberately to the side of that production picture and defines the **first major project deliverable**: a User Acceptance Testing (UAT) slice scoped to the FAQ arm only, exposed through a Skywalker-owned web frontend rather than through Slack or QuickSuite, due to clients on **June 30, 2026**.

That deliverable matters as architecture, not just as program management, because what gets built for UAT is not "the production system, smaller." It is a deliberately narrowed slice of the production system that exercises specific contracts under real user traffic before the rest of the architecture exists. Writing UAT as if it were a production-shape system in miniature would force premature commitments to the routing gate, UKB integration, and rerank fleet that the deliverable does not need. Writing it as if it were throwaway would lose the architectural value of treating the FAQ arm as a real, contracted retrieval surface from day one. This section keeps both errors out by fixing exactly what UAT is and what it is not.

The shape of the slice is straightforward. A small React single-page application is served from a Skywalker-owned domain inside Amazon's private network. A user authenticates through Midway and lands on a single chat input. The application calls a small **inline-agent orchestrator** which invokes a Bedrock Inline Agent running Claude Sonnet 4.6. The inline agent uses the **return-control loop** from Section 08 to call Skywalker's MCP surface through **Amazon MCP Gateway**, which the orchestrator reaches over CloudAuth on the gateway's `/ca/mcp/{registry}/{server}` route, with the orchestrator registered as a CloudAuth-modeled AAA application in ServiceLens. The MCP request hits exclusively the FAQ arm — no UKB, no routing gate, no two-arm convergence — and the response renders in the chat UI as a grounded answer with citations or as a structurally distinct abstain message. The reranker stage is **optional for UAT**: the slice is buildable without it and ships with whichever reranker posture the team can land before June 30.

One identity decision in this slice deserves to be flagged up front because it is the largest deviation from the production sections. UAT does **not** call PAPI. The Federate Service Profile that backs the UAT domain's Midway authentication is configured to mint custom claims for `country`, `level`, and manager-versus-IC `role` directly into the Midway token, sourced from the same authoritative HR data PAPI exposes. The orchestrator reads those claims server-side and calls `skywalker.search.by_explicit_scope` with them. The decision in Section 02 that scope is part of correctness still holds; this section just satisfies that requirement by carrying scope in the token rather than by resolving it through PAPI on each request.

### 1. What this section owns

This section owns the UAT deliverable as a coherent, time-boxed slice of the architecture: the React frontend hosted from a Skywalker-owned domain, the Midway authentication path that surfaces `country`, `level`, and `role` to the inline-agent orchestrator, the orchestrator that bridges between the authenticated browser session and Bedrock Inline Agents, the inline-agent return-control loop that calls Skywalker's MCP server through Amazon MCP Gateway, the FAQ-only retrieval surface that satisfies the request, the reranker-optional posture for UAT, and the structurally distinct rendering of answerable evidence packages versus abstain packages in the chat UI.

It also owns the boundary that separates the UAT slice from the surrounding production architecture rather than letting the slice silently absorb production assumptions. The routing gate from Section 04 is not invoked on this path. The UKB arm from Section 06 is not invoked on this path. The QuickSuite client surface from Section 09 is not on this path. Slack from Section 08 is not on this path. None of those omissions are technical regressions; they are scope decisions that reflect what the first deliverable is actually for.

This section owns the explicit difference between UAT and production. UAT validates the FAQ arm end-to-end against real users and against client-side review of grounded answers. It is not a test of the two-arm retrieval design, the routing-gate calibration surfaces, or the QuickSuite consumption model. Those remain valid sections of the architecture; UAT just does not exercise them. Calling that out here keeps later sections from having to defend their existence against "but UAT didn't use it," which is not the same statement as "UAT showed it was wrong."

This section does not own the FAQ corpus itself. Section 03 already fixed the controlled corpus, the daily polling loop, the chunking pipeline, the link materialization, and the AOSS evidence index. UAT consumes that surface as-is. It does not own the canonical scoped-request shape; Section 02 already fixed that. It does not own the inline-agent return-control loop in detail; Section 08 already fixed it for Slack and UAT reuses the same pattern with one MCP transport and one frontend swapped underneath. It does not own the production reranker fleet; Section 07 already fixed that for the production architecture, and UAT's reranker-optional posture is a slice property, not an alternative reranker design.

It also does not own the calibration, decision-log, and re-litigation governance from Section 10. UAT is allowed to surface evidence that later moves Section 10 entries — that is part of the point of running UAT — but it does not introduce a parallel governance track of its own.

### 2. Inputs, outputs, and contracts

The primary input to this section is a user typing a question into a web chat input. Behind that input the slice has four hops: browser to inline-agent orchestrator, orchestrator to Bedrock Inline Agents, inline agent (via return control) to MCP Gateway, MCP Gateway to Skywalker's MCP surface. Each hop has a real contract, and getting the contracts right is the whole architectural point of treating UAT as a section rather than as a side project.

**Browser to orchestrator.** The browser holds a Midway-authenticated session. The team's Federate Service Profile is configured so the Midway OIDC token includes custom claims for `country`, `level`, and `role` (`MANAGER` or `INDIVIDUAL_CONTRIBUTOR`) in addition to the standard `sub` (alias), email, and group claims a Midway token normally carries. The frontend itself does not read those claims directly. The orchestrator receives the request inside the same private domain, validates the Midway identity, extracts the four fields it needs (`alias`, `country`, `level`, `role`), and treats them as authoritative for the duration of the request. The frontend's contract to the orchestrator carries one field — the user's `query_text` — plus a session identifier that scopes the conversation. Every other field that matters to retrieval comes from the validated Midway identity rather than from the request body. That is deliberate: it is the architectural reason the frontend can stay simple.

**Orchestrator to Bedrock Inline Agents.** The orchestrator calls `InvokeInlineAgent` against `bedrock-agent-runtime` with the Claude Sonnet 4.6 model ID, an `instruction` carrying the stable UAT behavioral prompt, a `sessionId` derived from the chat session, and `inputText` carrying the user's current turn. The action group declared on the call exposes one function: `skywalker_search_by_explicit_scope`. The function shape mirrors the MCP tool shape from Section 02 — `query_text`, `employee_id` (filled in by the orchestrator, derived from the Midway alias), `country`, `level`, `role` — and `actionGroupExecutor` is set to `customControl: "RETURN_CONTROL"` so the orchestrator performs the actual MCP call rather than Bedrock invoking a Lambda. `idleSessionTTLInSeconds` starts at 900 seconds (15 minutes), revisited against UAT usage. Inference parameters start at `temperature: 0` and `maxTokens: 1024`. The single-tool action group is a deliberate UAT-only narrowing: production Slack (Section 08) declares `skywalker_search_by_alias` as the default plus `skywalker_search_by_explicit_scope` as the explicit-scope shortcut, but UAT only uses the explicit-scope tool because the orchestrator always already holds the four scope fields from the Midway token.

**Return-control hop: orchestrator to MCP Gateway to Skywalker.** When the inline agent decides to call retrieval, Bedrock returns control to the orchestrator with a `returnControl` event naming `skywalker_search_by_explicit_scope` and the scoped arguments. The orchestrator maps the function name to the MCP tool name `skywalker.search.by_explicit_scope` and issues an MCP `tools/call` against Skywalker — but not directly. The call goes through **Amazon MCP Gateway**, the internal paved path for hosting and consuming MCP servers across Amazon, documented in BuilderHub's [MCP Gateway User Guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/). MCP Gateway is the same gateway product that Section 09's QuickSuite path and Section 08's Slack path use; the three production paths share one transport with one Bindle-permission surface, differing in the inbound auth shape (UAT and Slack on **CloudAuth** over `/ca/mcp/{registry}/{server}`; QuickSuite on Federate OAuth over `/federate/mcp/{registry}/{server}`).

The orchestrator reaches MCP Gateway over its remote HTTP endpoint at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry-id}/{skywalker-server-id}` with **CloudAuth** credentials minted by the orchestrator's AAA application. The orchestrator additionally **initiates a TransitiveAuth token** for the human invoker — the alias resolved from the Midway `sub` claim plus an `AgenticContext` that names the agent involvement — and propagates the TA token through the CloudAuth MCP SDK alongside the CloudAuth call. MCP Gateway validates the inbound CloudAuth credentials, applies its **Bindle-based discovery and invocation permissions** to confirm the orchestrator's AAA application principal is allowed to invoke this Skywalker MCP server, and forwards the request to Skywalker on the gateway's CloudAuth-outbound leg with **CloudAuth OBO** (so Skywalker sees the orchestrator's AAA application identity rather than the gateway's) and the TA token preserved for downstream validation. Skywalker validates the TA token server-side, extracts the human alias from the TA claims, and combines it with the explicit-scope arguments the orchestrator supplied. Per the MCP Gateway user guide, the gateway adds approximately 50 to 150 milliseconds of latency for auth handling and protocol translation, with a per-client throttle of 50 transactions per second; both are well within UAT's expected request budget. CloudAuth OBO and TA validation add a small additional cost (low tens of milliseconds combined under measured production traffic on similar paths); both are well inside the 250–450 ms p95 retrieval budget. See [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md) for the full integration contract.

One nuance from the gateway documentation that this section has to encode honestly rather than around: as of the current published guide, MCP Gateway's **remote** connectivity is limited to "CloudAuth-protected, remotely hosted servers" — with AWSAuth and Midway-protected backends still rolling out. The Skywalker MCP server is registered with MCP Gateway as a CloudAuth-protected service, so it is in scope for remote MCP Gateway invocation today. MCP Gateway's CloudAuth-inbound + CloudAuth-protected-backend combination supports OBO + TransitiveAuth natively for the Slack and UAT paths; the Federate-inbound + CloudAuth-outbound combination QuickSuite uses does not yet have a published delegated-identity pattern, so QuickSuite continues to carry identity in MCP tool arguments.

**Skywalker MCP surface contract.** Inside Skywalker, the `skywalker.search.by_explicit_scope` call from Section 02 takes its normal canonical form. UAT exercises one path through the production architecture: scope is supplied directly, PAPI is not called, and the canonical scoped request `(query_text, employee_id, country, level, role, request_mode, correlation_id)` is constructed. From here downward, what happens differs from production:

- **No routing gate.** Section 04's hybrid gate (cosine first, contingent Cohere Rerank v4 stage 2) is not invoked. UAT is FAQ-only by configuration. The `route.path` value Skywalker stamps on the response is `FAQ_ONLY`.
- **No UKB arm.** Section 06's UKB MCP client is not invoked. The dual-arm path does not exist on the UAT request graph at all; it is not "skipped at runtime," it is not in the deployment.
- **FAQ retrieval runs in full.** The FAQ hybrid query against the `faq_evidence_current` AOSS alias, fused by the `skywalker-faq-hybrid` search pipeline, with the FAISS `efficient_filter` clause applied on `country`, `level`, and `role` — all of that runs unchanged from Section 03. Sibling-and-linked-parent expansion runs unchanged. Citations and evidence-text assembly run unchanged.
- **Reranker is optional.** This is the explicit UAT-versus-production divergence and §3 fixes the decision. The slice is shippable with or without Cohere Rerank 4 Pro on SageMaker; if the reranker is present, the final shortlist is reranker-judged exactly as Section 07 describes; if absent, the candidates emerge in their hybrid-fused FAQ order from the AOSS pipeline and that order is the shortlist.

**Output back into the chat UI.** The MCP response — the structured envelope from API_01 with `result_kind`, `route`, `scope_snapshot`, `evidence`, `abstain_reason`, and `correlation_id` — flows back through MCP Gateway to the orchestrator as **a single sync JSON-RPC result**. The MCP boundary itself does not stream: the gateway's HTTP transport on our auth combinations does not expose SSE framing, and Skywalker's envelope is a structured object rather than narrative text, so there is no time-to-first-token to optimize at that layer. The orchestrator feeds the serialized envelope back into the inline agent via a second `InvokeInlineAgent` call carrying the same `sessionId`, `inlineSessionState.returnControlInvocationResults`, and `streamingConfigurations: { streamFinalResponse: true }`. **The inline agent's final response streams**: as Claude Sonnet 4.6 generates the user-facing message from the evidence package, `chunk` events arrive on the Bedrock event stream and the orchestrator re-emits them to the React app over Server-Sent Events. The React app uses the browser-native `EventSource` API to receive chunks and renders the assistant text progressively into the chat bubble. On `result_kind: ABSTAIN`, the chunks still stream, but the React app applies its structurally-distinct abstain styling on the terminating event — so users can tell "the system did not have grounded evidence for your specific situation" apart from "the system failed," on the same architectural rule Section 08 sets for Slack. See [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for the streaming contract.

The contract back into UAT remains a structured backend result rather than final prose — the inline agent is responsible for composing prose from that result, exactly as on the Slack path. The frontend is not allowed to render evidence text directly without going through the agent. That keeps citation discipline (§3) honest. Streaming changes the time at which characters appear on screen; it does not change the contract that every claim in the rendered message has to trace back to the evidence package returned by Skywalker.

### 3. Fixed decisions

The first fixed decision is that UAT is a **deliberately narrowed slice**, not a production-shape system in miniature. Routing gate, UKB arm, dual-arm convergence, QuickSuite wrapper, and Slack are out of scope by intent. The slice exercises the FAQ arm end-to-end through a Skywalker-owned client surface so that the FAQ retrieval contract and the citation/abstain UX are validated against real users before the rest of the architecture lands. UAT is the first deliverable and is due to clients on June 30, 2026.

The second fixed decision is the **frontend hosting posture**: a static compiled React single-page application served behind Amazon's private network from a Skywalker-owned domain. The application is a single chat window with one input. There is no admin UI, no settings panel, no multi-tab navigation. That minimalism is architectural: every additional UI surface is one more place where the slice would have to take a position the production architecture has not yet finalized. The frontend's job is to demonstrate the FAQ arm and the citation/abstain UX, and nothing else.

The third fixed decision is the **identity model**: Midway authentication on the domain, with a Federate Service Profile configured to mint custom claims for `country`, `level`, and `role` into the Midway token. The inline-agent orchestrator reads those claims server-side and treats them as authoritative for retrieval scope. UAT does not call PAPI on the request path. This decision satisfies the Section 02 rule that scope is part of correctness — the fields are present and authoritative on every request — without forcing a PAPI call into a slice whose purpose is to validate the FAQ arm rather than the identity-resolution path. The production paths (Slack and QuickSuite) continue to call PAPI per Sections 02, 08, and 09; this section is the only path that bypasses it.

The fourth fixed decision is the **MCP transport layer**: Amazon MCP Gateway, accessed at `https://api.mcp.asbx.aws.dev/ca/mcp/{registry-id}/{skywalker-server-id}` over **CloudAuth** with the UAT inline-agent orchestrator registered as a CloudAuth-modeled AAA application in ServiceLens (suggested name `SkywalkerUATOrchestrator`, finalized at onboarding). Skywalker is registered as an MCP server in Amazon MCP Gateway as a CloudAuth-protected service. AAA relationships are established from the orchestrator's AAA application to MCP Gateway's `InvokeMcp` endpoint and to Skywalker's MCP server resource, per the OBO Decision Guide's "OBO Required (Shared / Cross-Boundary)" path. Bindle-based authorization grants the orchestrator's AAA application principal `canInvoke` on the Skywalker MCP resource. **CloudAuth OBO is on by default on this combination** — MCP Gateway invokes Skywalker on behalf of the orchestrator's AAA application identity. **TransitiveAuth additionally carries the human end-user's identity** as a separate TA token; the orchestrator becomes a TA initiator, and Skywalker becomes a TA validator. This is the same gateway product Section 09's QuickSuite path uses (on its Federate-inbound route) and Section 08's Slack path uses (also on the CloudAuth-inbound route, with its own AAA application `SkywalkerSlackOrchestrator`). The three production paths share one transport with one Bindle-permission surface; Slack and UAT also share the OBO + TA pair, while QuickSuite remains on the argument-supplied identity model because Federate-inbound + CloudAuth-outbound TA delegation is not yet published. Source: BuilderHub [MCP Gateway User Guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/), [UnifiedAuth MCP Server Builder Guidance](https://w.amazon.com/bin/view/Dev.CDO/UnifiedAuth/Agentic/MCP/Overview/), and grounding contract [API_14](done/API_14_CloudAuth_OBO_and_TransitiveAuth_Integration.md).

Earlier drafts of this section described the orchestrator as using SigV4 inbound at `/mcp/{registry-id}/{skywalker-server-id}`. That path is no longer available for new MCP servers per the [MCP server vendor guide](https://docs.hub.amazon.dev/docs/mcp-gateway/user-guide/howto-vendor/) — AWSAuth-inbound for CloudAuth-protected services was withdrawn for new servers after April 24, 2026, and the grace window for existing servers closed May 15, 2026. CloudAuth-inbound is the supported successor and is also the path with native OBO + TA support. The migration is captured in API_14.

The fifth fixed decision is the **agent layer**: a Bedrock Inline Agent running Claude Sonnet 4.6, invoked through `InvokeInlineAgent` with `RETURN_CONTROL` exactly as Section 08 fixes for the Slack path. The action group declares one function — `skywalker_search_by_explicit_scope` — because the orchestrator always already holds the four scope fields from the Midway token. The Converse API is not used. Reusing the inline-agent pattern from Section 08 means the UAT slice and the production Slack slice exercise the same agent-side mechanics, which keeps the two paths from drifting into incompatible contracts before the production Slack section ships.

The sixth fixed decision is **FAQ-only retrieval**. The routing gate is not invoked. The UKB arm is not invoked. The dual-arm path does not exist on this deployment. Skywalker's `route.path` value is `FAQ_ONLY` on every UAT response. This is the correct shape because UAT's purpose is to validate the FAQ arm; routing decisions and dual-arm convergence are exercises that belong to a deployment that actually has both arms.

The seventh fixed decision is the **reranker-optional posture for UAT**. The Cohere Rerank 4 Pro fleet on `ml.p5.4xlarge` H100 instances described in Section 07 is the production reranker. For UAT specifically, the reranker is allowed to be present or absent depending on what the team can land before June 30. If present, the final shortlist is reranker-judged on the contract Section 07 describes. If absent, the shortlist is the hybrid-fused FAQ order produced by the `skywalker-faq-hybrid` search pipeline. The abstain branch `EVIDENCE_TOO_WEAK_AFTER_RERANK` only fires when the reranker is present; in the no-reranker UAT posture, the only abstain branch is `NO_USABLE_EVIDENCE` (empty FAQ result). This optionality is a UAT-only property, not a production design alternative; Section 07 remains the production reranker contract.

The eighth fixed decision is that the **citation requirement from Section 08 §3 decision nine applies in full to UAT**. The final user-facing chat message must be constructible from the evidence package returned by Skywalker, and the message must surface citations to the specific FAQ evidence anchors and any linked Q&A segments that support its claims. The React app renders citations as inline markers in the assistant's text plus a sources list; the precise visual treatment is not load-bearing. The grounding discipline is. UAT is not allowed to ship with answers that look grounded but are not.

The ninth fixed decision is that **abstain UX is structurally distinct from service failure** in the UAT chat UI. On `result_kind: ABSTAIN`, the chat renders a clearly-shaped "Skywalker did not find grounded evidence for your specific situation" message that is visually and copy-distinct from a "the service failed, please try again" message. This is the same architectural rule Section 08 §3 decision seven applies to Slack, applied unchanged here.

The tenth fixed decision is that **UAT runs against production-shape FAQ data**, not against a synthetic test corpus. The CoreX → ingestion pipeline → AOSS index path described in Section 03 is the same one UAT consumes. UAT exercises real authored content under real scope filtering; that is the entire point of running it. UAT environment selection (which `{stage}` of the FAQ index UAT reads) is an operational choice and is not load-bearing here, but the data has to be authored content rather than a fixture set.

### 4. Alternatives considered or still live

The first alternative was building UAT as a production-shape system in miniature: routing gate present but loosely calibrated, UKB present but lightly used, dual-arm path enabled. That alternative was rejected because UAT does not have the calibration evidence to set those values honestly, and shipping under-calibrated production surfaces would either prematurely freeze them on weak evidence or train the team to ignore them. The narrower slice is the correct shape. The production sections remain unmodified.

The second alternative was building UAT against Slack rather than against a Skywalker-owned web frontend. That alternative was rejected because Slack onboarding (signing-secret HMAC, three event surfaces, Bolt for Java handlers, the latency budget interaction with Slack's 3-second ack rule from Section 08) is a meaningful body of architectural work that UAT does not need to validate in order to validate the FAQ arm. A simpler hosted React frontend gets to user feedback faster and exercises a cleaner contract.

The third alternative was building UAT against QuickSuite. That alternative was rejected because QuickSuite's inbound auth is Federate OAuth (Section 09) rather than Midway, the consumption model goes through MCP Gateway's `/federate/mcp/...` route with QuickSuite-side connector registration, and the visible-citation behavior depends on a QuickSuite-specific UI quirk. None of that is wrong, but none of it is what UAT is for. UAT is for FAQ-arm validation against an internal user base on an Amazon-internal domain.

The fourth alternative was bypassing MCP Gateway and having the orchestrator call Skywalker's MCP surface directly. That alternative is rejected for the same reason Sections 08 and 09 do not: MCP Gateway is the internal paved path for hosting and consuming MCP servers across Amazon, and routing UAT around it would create a non-paved-path UAT shape adjacent to MCP-fronted production paths. If MCP Gateway's currently published auth-support matrix turns out to be a blocker for UAT — for example, if Skywalker's CloudAuth setup needs adjustment to fit the gateway's supported pattern — the right answer is to align Skywalker's auth posture with MCP Gateway, not to introduce a per-client bypass.

The fifth alternative was bypassing PAPI by **hardcoding** UAT scope (a single fixed `country`, `level`, `role` for every UAT request) rather than by sourcing the four fields from custom Midway claims. That alternative is rejected because hardcoded scope would not exercise the production retrieval surface honestly: scope filtering would behave the same way for every UAT user, and UAT's value as evidence about the FAQ arm under real per-user scope would collapse. Custom Midway claims preserve per-user retrieval semantics without requiring a PAPI call.

The sixth alternative was making the reranker mandatory for UAT. That alternative is live but not fixed. If the team lands the Cohere Rerank 4 Pro fleet on SageMaker before June 30, UAT runs with it. If the team does not, UAT runs without it and the abstain rule narrows to `NO_USABLE_EVIDENCE` only. The optionality matters because the deliverable date is fixed and the reranker fleet is the heaviest infrastructure piece in the production architecture; coupling UAT to it would put the deliverable at programmatic risk for an architecture-validation gain that UAT does not need.

The seventh alternative was treating UAT as a throwaway prototype that does not need to maintain contract discipline (citations optional, abstain UX merged with failure UX, no API-shape commitment). That alternative is rejected. UAT is the first time real users see the system, and the citation and abstain disciplines are the strongest accuracy-preserving rules in the architecture. Letting UAT ship without them would teach the team and the early users that grounded retrieval is aspirational rather than required.

### 6. End-to-end data flow for this section

The UAT request graph is short, but each hop is a real contract boundary that this section is responsible for naming.

**Step one: the user types a question.** A logged-in employee on the UAT domain (a Midway-authenticated, private-network-only host such as `https://uat.skywalker.<internal-suffix>` — the precise hostname is an operational detail rather than load-bearing here) types a question into the single chat input on the React SPA. The frontend posts the question to the inline-agent orchestrator on the same domain over a normal authenticated session.

**Step two: the orchestrator reads scope from the Midway token.** The orchestrator receives the request, validates the Midway identity, and reads four claims from the token: `sub` (alias), `country`, `level`, `role`. It does not call PAPI. It does not synthesize scope. It does not fall back to defaults. If any required claim is missing or malformed, the orchestrator stops the request here and returns a structurally distinct "we could not establish your identity scope" error to the frontend rather than sending an under-scoped request into Skywalker. (This mirrors Section 02's "fail closed on missing or unusable scope" rule, applied at the UAT orchestrator rather than at Skywalker's own boundary.)

**Step three: the orchestrator invokes the inline agent.** The orchestrator calls `InvokeInlineAgent` against `bedrock-agent-runtime` with the Claude Sonnet 4.6 model, the stable UAT instruction, the session ID derived from the chat session, and `inputText` carrying the user's current turn. The action group declares `skywalker_search_by_explicit_scope` with `customControl: "RETURN_CONTROL"`. The agent returns either a clarification turn (the agent decided to ask the user something, no tool call) or a `returnControl` event naming the function and arguments.

**Step four: the orchestrator maps return-control to an MCP call through MCP Gateway.** The orchestrator maps the function name `skywalker_search_by_explicit_scope` to the MCP tool name `skywalker.search.by_explicit_scope`. It builds the MCP `tools/call` request with the JSON-RPC envelope from API_01 and the `params.arguments` shape from Section 02. Before issuing the call, it **initiates a TransitiveAuth token** carrying the human invoker's alias (the `sub` claim from the validated Midway token) and an `AgenticContext` describing the agent involvement. It POSTs to `https://api.mcp.asbx.aws.dev/ca/mcp/global/<skywalker-mcp-server-name>` with **CloudAuth** credentials minted by the orchestrator's AAA application, with the TA token propagated alongside CloudAuth via the CloudAuth MCP SDK. MCP Gateway validates the inbound CloudAuth credentials, checks the orchestrator AAA application's `canInvoke` permission on Skywalker's MCP resource bindle, and forwards the request to Skywalker's MCP surface on the gateway's CloudAuth-outbound leg with **CloudAuth OBO** (so Skywalker sees the orchestrator's AAA application identity) and the TA token preserved for downstream validation.

**Step five: Skywalker handles the request through the FAQ-only path.** Skywalker accepts the explicit-scope request (Section 02), constructs the canonical scoped request, and proceeds directly to FAQ retrieval (Section 03). Embedding the query once with Cohere Embed v4 (`input_type: "search_query"`) per Section 04 still applies — the FAQ hybrid query consumes that vector. The routing gate from Section 04 is not invoked. The UKB MCP client from Section 06 is not invoked. The FAQ hybrid query against `faq_evidence_current` runs, fused by the `skywalker-faq-hybrid` pipeline, with FAISS `efficient_filter` applied on `country`/`level`/`role`.

**Step six: candidates flow through the (optional) reranker and post-rerank assembly.** When the reranker is present in UAT, candidates are normalized to the common envelope and reranked through Cohere Rerank 4 Pro per Section 07; the top-N shortlist proceeds. When the reranker is absent, the FAQ hybrid output order is the shortlist directly. Either way, post-rerank sibling-and-linked-parent expansion (Section 07 §6) runs unchanged: the runtime issues the AOSS `terms` query against the union of the winners' own parent_ids and their `linked_parent_ids` chains, reconstructs anchor texts with citation markers, and builds the parallel `citations[]` envelope per anchor.

**Step seven: Skywalker returns the structured envelope.** The MCP response carries `result_kind` (`ANSWERABLE` or `ABSTAIN`), `route.path: "FAQ_ONLY"`, `scope_snapshot` echoing the four scope fields the orchestrator sent, `evidence[]` with anchors and their citation tables, `abstain_reason` where applicable, and `correlation_id`. The response flows back through MCP Gateway to the orchestrator.

**Step eight: the orchestrator feeds the envelope back into the inline agent and streams the response.** The orchestrator serializes the structured envelope and calls `InvokeInlineAgent` a second time on the same `sessionId` with `inlineSessionState.returnControlInvocationResults` populated and `streamingConfigurations: { streamFinalResponse: true }`. As Claude Sonnet 4.6 generates the user-facing message — grounded answer with inline citation markers and a sources list, or an abstain message with the abstain reason exposed in user-friendly form — `chunk` events arrive on the Bedrock event stream. The orchestrator re-emits each chunk to the React app over Server-Sent Events on a long-lived HTTP connection, then sends a terminating SSE event when the inline-agent stream ends. See [API_12](done/API_12_Bedrock_Inline_Agent_Streaming.md) for the streaming contract.

**Step nine: the React app renders progressively.** The browser-native `EventSource` API receives chunks as they arrive and appends them to the chat message bubble with a typewriter-style cursor while generation is in progress. On the terminating SSE event, the React app applies the appropriate final styling. On `ANSWERABLE`, the final styling renders the grounded answer with citations as clickable source links resolved from `citations[]`. On `ABSTAIN`, the final styling renders a structurally distinct abstain message — different visual shape and different copy from a service-failure message. On a true service failure (any hop returned an error rather than an `ABSTAIN` envelope, or the Bedrock stream errored mid-generation), a final SSE event signals the failure and the React app overwrites whatever partial text was visible with a service-failure message that is also structurally distinct from both answerable and abstain shapes. Three UI states, three distinguishable user experiences.

### 7. Failure behavior, abstain behavior, and non-goals that matter here

UAT failure handling has to be more conservative than production failure handling, not less, because UAT is the first surface real users see and a confusing failure shape early teaches users to distrust the system.

The first failure class is **identity failure at the orchestrator**. If the Midway token is missing, expired, or missing any of the required custom claims (`country`, `level`, `role`), the orchestrator returns an identity-shaped error to the frontend without calling Bedrock or Skywalker. The frontend renders a "we could not establish your identity scope" message. This is structurally distinct from both `ABSTAIN` and from a generic service failure — the user is being told that the system could not even attempt retrieval for them.

The second failure class is **MCP Gateway authorization failure**. If MCP Gateway returns a 403 or otherwise rejects the call, the orchestrator treats this as a service failure rather than as an abstain or an identity failure. The most likely causes are missing Bindle permissions on the orchestrator's IAM role or a misconfigured MCP server registration; both are operational issues, not user-facing ones. The frontend renders a service-failure message.

The third failure class is **Skywalker tool-execution error**. Per Section 02 and API_01, this surfaces as a normal JSON-RPC result with `isError: true`. The orchestrator feeds this back into the inline agent in the return-control result with the `isError` flag preserved; the behavioral prompt instructs the agent that this is a system-failure state rather than permission to answer from memory. The frontend renders a service-failure message.

The fourth failure class is **inline-agent prompt non-compliance**. If the model ignores the prompt and tries to answer from training rather than from the structured backend result, that is a prompt-compliance issue handled at calibration (this is an instance of the same surface Section 08 §8 calibration surface one tracks for Slack). It is not a per-request architectural failure; it is a tuning surface visible against UAT traffic.

**Abstain behavior is unchanged from Section 07's contract**, scoped to the branches that apply on the UAT path. With the reranker present, both `NO_USABLE_EVIDENCE` and `EVIDENCE_TOO_WEAK_AFTER_RERANK` are reachable. Without the reranker, only `NO_USABLE_EVIDENCE` is reachable — the FAQ hybrid query returned no candidates, so the shortlist is empty before any rerank stage could exist. In both cases, the agent and the React app render an abstain message that is structurally distinct from a service failure. Backend abstain remains a successful result, never an error.

**Non-goals for this section.** UAT does not introduce a new MCP tool. UAT does not introduce a new abstain branch. UAT does not introduce a new common candidate envelope. UAT does not modify the FAQ corpus, the chunking pipeline, the link materialization, or the AOSS index schema. UAT does not run a parallel reranker fleet; if the production fleet exists by June 30, UAT uses it; if not, UAT runs without a reranker. UAT does not produce the calibration evidence Section 10 needs for the routing gate, because the routing gate is not exercised on this path; production-system calibration evidence comes from the post-UAT deployment that turns on the routing gate and the UKB arm.

### 8. Calibration surfaces and what would cause re-litigation

UAT is bounded in time (June 30) and in scope (FAQ-arm slice through a Skywalker-owned frontend). Most of its surfaces therefore close rather than open after the deliverable lands. A small number remain calibration-active even within the slice.

The first calibration surface is the **inline-agent inference parameters** (`temperature: 0`, `maxTokens: 1024`, `idleSessionTTLInSeconds: 900`). These are launch defaults shared with the Slack path (Section 08 §3 decision six) and the same calibration argument applies — they exist so the system can run, and they move against real evaluation. UAT traffic is the first place those values are exercised against real users; movement under UAT evidence is the legitimate use of this surface, not an architecture-class re-litigation event.

The second calibration surface is the **reranker presence-or-absence posture**. The decision is fixed for the UAT deliverable (optional, ship with whichever is buildable), but it is also the surface most likely to flip during the run-up to June 30. Re-litigation is justified only if reaching the deliverable date forces an explicit choice between "ship with no reranker" and "delay to ship with reranker"; that is a program decision rather than an architecture decision, and Section 10's governance does not cover the timeline tradeoff.

The third calibration surface is the **frontend rendering of citations**. The architectural rule (citations must be present and traceable, abstain must be structurally distinct) is fixed. The visual treatment is not. Inline superscript markers, footnoted lists, expandable source cards — any of those satisfy the rule. UAT user feedback is the right evidence class for moving among them.

The fourth calibration surface is the **MCP Gateway latency budget**. The published gateway adds approximately 50 to 150 ms for auth handling and protocol translation. UAT measurement is the first concrete evidence about where in that band Skywalker actually lands and whether the resulting end-to-end latency keeps the chat experience snappy. Re-litigation is justified only if measured gateway-attributable latency consistently exceeds the published upper bound under UAT traffic; the resolution then is a conversation with the MCP Gateway team rather than a Skywalker-side bypass.

The fifth calibration surface is the **scope of the custom Midway claims**. UAT works because the Federate Service Profile mints `country`, `level`, and `role` directly. If the production Slack and QuickSuite paths later turn out to want the same claims for the same reason, that is a re-litigation event for Section 02 (whether the alias-and-employee-ID paths should retire in favor of explicit-scope on every client) and for Section 10's decision log. UAT does not pre-empt that conversation; it just establishes that the custom-claim path is technically viable.

One thing is **not** a calibration surface here: whether UAT should grow into a long-lived production frontend. The architecture's production client surfaces are Slack and QuickSuite (Sections 08 and 09). UAT is a deliberately narrowed slice with a specific deliverable date. If the team later decides Amazon needs a dedicated web frontend in addition to Slack and QuickSuite, that would be a new client-surface section, not a UAT scope expansion.

### Closing position

UAT is the first major project deliverable, due to clients on June 30, 2026, and is a deliberately narrowed slice of the production architecture rather than a production-shape system in miniature. It exercises the FAQ arm end-to-end through a Skywalker-owned React frontend hosted behind Amazon's private network, authenticated through Midway with custom claims that supply `country`, `level`, and `role` directly so PAPI is not on the request path. A small inline-agent orchestrator runs Bedrock Inline Agents on Claude Sonnet 4.6 with the same return-control loop Section 08 fixes for Slack, and it reaches Skywalker's MCP surface through Amazon MCP Gateway — the internal paved path documented on BuilderHub — using **CloudAuth** with the orchestrator registered as a CloudAuth-modeled AAA application and **CloudAuth OBO + TransitiveAuth** carrying both the orchestrator's service identity and the human end-user's identity through to the Skywalker backend. The routing gate, the UKB arm, the dual-arm convergence, the QuickSuite wrapper, and Slack are all out of scope for UAT by intent rather than by omission. The reranker is optional for UAT and ships with whichever posture the team can land before June 30. The citation requirement and the structurally-distinct abstain UX from Section 08 apply unchanged. By the time UAT lands, the team has validated the FAQ retrieval contract, the citation and abstain UX, and the inline-agent-to-MCP-Gateway-to-Skywalker invocation path under real users — and the rest of the architecture, sized in Sections 04, 06, 07, 08, 09, and 10, has not been forced to commit to calibration evidence it does not yet have.
